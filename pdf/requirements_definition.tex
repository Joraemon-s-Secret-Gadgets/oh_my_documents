\DocNeedspace{8\baselineskip}
\subsection*{1.1 목적}
\addcontentsline{toc}{subsection}{1.1 목적}


\DocNeedspace{3\baselineskip}
본 문서는 로브(Lovv) 서비스의 아이디어 기획 단계에서 제품의 핵심 기능,\newline API 연동 요구사항, 비기능 요구사항을 정의한다.

\DocNeedspace{3\baselineskip}
개발 착수 전 이해관계자 간 요구사항에 대한 공통 이해를 확보하는 것을 목적으로 한다.


\DocNeedspace{24\baselineskip}
\subsection*{1.2 범위}
\addcontentsline{toc}{subsection}{1.2 범위}


\DocNeedspace{3\baselineskip}
본 문서는 다음 범위를 포함한다.


\begin{itemize}
\item 대화형 챗봇 인터페이스 및 기본·확장 테마 기반 소도시 집중 일정 추천 엔진
\item 한국 소도시 추천 --- PoC/P1은 한국 강원·경북 데이터 기반 추천으로 고정하고, 한국 전국 확장은 P2로 둔다
\item 일본 소도시 추천(일본 데이터 기반 추천, 일본 현지 데이터/API 연동)은 일본 데이터 수집 실패 리스크를 반영해 P5 재검토 범위로 보류하며, 수집 경로가 복구된 뒤 재승격을 검토한다
\item 여행 시기(달/계절) 입력에 따른 계절 적합 소도시 및 축제·행사 연동
\item 메인 지도 기반 소도시 탐색, 월·테마 필터, 지도 마커 기반 추천 진입점
\item 온보딩 선호 프로필, 추천 피드백, 저장 일정, 북마크, 외부 링크 클릭 기반 개인화
\item 사용자 행동 데이터와 외부 검증 루트를 활용한 cold start 보완 및 세션 기반 개인화
\item KICK(유사 취향 공개 일정·1탭 복제)은 P2 설계 범위로 관리
\item 지역 문화 체험, 스키 액티비티 등 P3\textasciitilde{}P4 범위의 확장 테마 추천
\item 여행 시기(월)를 직접 입력해 Wikipedia 기반 월별 날씨 경향이 좋은 소도시를 추천받는 챗봇 진입점
\item 별 관측, 치유·웰니스, 예산 역산 추천은 이번 프로젝트 대상에서 제외하고\newline 향후 단계에서 검토
\item RAG(검색증강) 기반 grounding 및 멀티스텝 에이전트 추천 파이프라인
\item Google Maps Platform, Kakao Maps, OpenRouteService 연동 (Yahoo Japan 등 일본 현지 플랫폼 연동은 P5 재검토 범위)
\item 지도 표시, 장소 상세 정보, Wikipedia 기반 월별 날씨 경향, 숙소 검색 링크, 외부 탐색 링크 제공 기능
\end{itemize}

\begin{itemize}

\item 서비스 관리자는 API Key 값을 직접 관리하지 않고, 키 발급·교체·만료 대응이 필요할 때 개발·운영 담당자에게 요청한다.

\subsection*{2.6 관리자 RBAC 권한 경계}
\addcontentsline{toc}{subsection}{2.6 관리자 RBAC 권한 경계}


\DocNeedspace{3\baselineskip}
RBAC(Role-Based Access Control, 역할 기반 접근 제어)는 사용자 역할에 따라 접근 권한과 작업 범위를 제한하는 인가 방식이다.\newline 로브 관리자 콘솔의 인증·인가 경계는 다음 원칙으로 고정한다.\newline 대상 관리 역할은 서비스 관리자(\texttt{R-\allowbreak{}ADMIN}), 데이터 제공 기관(\texttt{R-\allowbreak{}DATA-\allowbreak{}PROVIDER}), 관광 운영자(\texttt{R-\allowbreak{}LOCAL-\allowbreak{}OPERATOR})다.


\begin{itemize}
\item \textbf{역할 Source of Truth:} 최종 권한은 로브 서비스 DB의 활성 역할 할당을 기준으로 판단한다. Cognito Group 등 외부 인증 그룹 정보는 최초 역할 연결·운영 편의용 보조 신호로만 사용하고 단독 권한 근거로 신뢰하지 않는다.
\item \textbf{서버 재검증:} API Gateway 인증 이후에도 각 관리자 요청마다 역할과 리소스 범위를 서버 데이터로 다시 검증한다. 요청 본문의 \texttt{roles}, \texttt{region\allowbreak{}Ids}, \texttt{organization\allowbreak{}Id}, \texttt{reviewer\allowbreak{}Id} 등 권한·소유권 필드는 신뢰하지 않는다.
\item \textbf{범위 정의:} \texttt{Own}은 본인 또는 본인 소속 기관 소유 리소스, \texttt{Assigned}는 활성 지역 할당이 있는 담당 지역, \texttt{All}은 전역 범위를 의미한다.
\item \textbf{역할 비상속:} 역할은 자동 상속하지 않으며, 권한은 활성 역할별 허용 작업의 합집합으로 계산한다. 범위 제한은 역할마다 따로 적용한다.
\item \textbf{기본 역할:} 일반 로그인 사용자는 \texttt{R-\allowbreak{}USER}만 가진다. 관리자 역할은 명시적으로 할당해야 하며, 소셜 로그인이나 이메일 도메인만으로 자동 부여하지 않는다.
\item \textbf{실패 원칙:} 역할 또는 범위를 확인할 수 없으면 fail-closed로 거부한다.
\item \textbf{권한 분리:} 데이터 제안 승인과 월간 여행지(\texttt{이번 달 여행지}) 게시는 분리한다. 승인된 데이터도 별도 게시 결정 전에는 노출하지 않는다.
\item \textbf{이해충돌 방지:} 제안 작성자 본인은 동일 제안을 승인·반려할 수 없다.
\end{itemize}

개인화 루프 & 온보딩 선호, 저장 일정, 북마크를 다음 추천 조건에 반영한다. & 좋아요/싫어요 피드백과 외부 링크 클릭은 세션 내 후보 조정에 활용한다. \\
혼잡 회피 & 단순 인기순이 아니라 혼잡도 낮은 대체 여행지를 우선 추천한다. & 오버투어리즘 회피 목적에 맞춰 후보 정렬 기준에 반영한다. \\
RAG 정확성 & 직접 구축한 소도시 데이터를 검색·근거로 활용한다. & 환각 없는 사실 기반 추천을 목표로 한다. \\
에이전트 추론 & 분류$\rightarrow$검색$\rightarrow$선정$\rightarrow$일정 구성을 단계별로 추론한다. & 추천 품질과 근거 일관성을 확보한다. \\
일본 트랙 & 한국 소도시 추천을 PoC/P1 핵심으로 제공한다. & 일본 소도시 트랙은 데이터 수집 경로 복구 후 P5에서 재검토한다. \\

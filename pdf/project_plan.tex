로브 (Lovv)


\DocNeedspace{3\baselineskip}
Lovv는 Local(소도시), Vibe(분위기·감성), Voyage(여행)의 의미를 결합한 소도시 여행 추천 서비스다.


\DocNeedspace{24\baselineskip}
\subsection*{1.2 한 줄 정의}
\addcontentsline{toc}{subsection}{1.2 한 줄 정의}


\DocNeedspace{3\baselineskip}
로브는 여행 시기와 국내 지역 조건부터 시작해 오버투어리즘을 피할 수 있는 소도시 대체 여행지를 발견하게 해주는 설명 가능한 여행 추천 에이전트다.


\DocNeedspace{3\baselineskip}
\textbf{서울·오사카 말고, 지금은 이곳}


\DocNeedspace{3\baselineskip}
\textbf{지도에서 흐릿한 곳, 여행에선 가장 빛나는 곳}


\DocNeedspace{24\baselineskip}
\subsection*{1.3 기획 배경}
\addcontentsline{toc}{subsection}{1.3 기획 배경}


\DocNeedspace{3\baselineskip}
국내외 여행 수요는 회복되고 있지만, 인기 관광지로의 집중은 여행 만족도 저하와 지역 수용력 문제를 동시에 만든다.\newline 여행자는 덜 붐비는 목적지를 원하지만, 방문객 수가 낮은 관광지와 그 주변 소도시 정보는 흩어져 있고 이동성, 계절성, 축제, 날씨, 주변 관광지 정보를 한 번에 비교하기 어렵다.


\DocNeedspace{3\baselineskip}
로브의 1차 검증 타깃은 한국 강원·경북 소도시 여행자다.\newline PoC는 일본 데이터 수집 실패 리스크를 반영해 한국 강원·경북의 실제 수집·검증 가능한 데이터를 기준으로 추천 경험을 먼저 검증하고, 한국 전국 확장은 P2로 둔다.\newline 일본 데이터 기반 추천과 일본 현지 데이터 연동은 데이터 수집 경로가 복구된 뒤 P5에서 재검토한다.


\DocNeedspace{3\baselineskip}
로브는 오버투어리즘 회피와 소도시 발견이라는 문제를 여행자 관점의 추천 경험으로 해결한다.\newline 사용자가 여행 시기와 국내 지역 조건을 먼저 입력하면, 서비스는 그 조건을 기준으로 사용자의 자연어 취향과 Wikipedia 기반 월별 날씨 경향을 해석하고 추천 근거가 있는 소도시 여정과 주변 정보를 함께 제공한다.


\DocNeedspace{3\baselineskip}
일반 여행 일정 서비스가 사용자가 이미 선택한 대도시 안에서 동선과 일정을 최적화하는 데 강점이 있다면, 로브는 어디를 갈지부터 정하지 못한 사용자에게 국내 지역과 시기 조건에 맞는 대체 소도시를 발견하게 하는 목적지 탐색 엔진으로 차별화한다.


\DocNeedspace{3\baselineskip}
PoC 단계에서는 많은 지역을 얕게 나열하기보다 한국 강원·경북의 제한된 소도시를 깊게 다룬다.\newline 축제, 기상, 교통, 체험, 외부 링크를 고밀도로 채워 Lovv다운 목적지 추천 경험을 먼저 검증한다.

\subsection*{1.4 시장 포지셔닝과 차별점}
\addcontentsline{toc}{subsection}{1.4 시장 포지셔닝과 차별점}


\DocNeedspace{3\baselineskip}
트리플이 인기 도시 안에서 여행 일정을 최적화하는 슈퍼앱이라면, Lovv는 여행 시기와 국내 지역 조건부터 시작해 오버투어리즘을 피할 수 있는 소도시 대체 여행지를 발견하게 해주는 설명 가능한 추천 에이전트다.


\DocNeedspace{3\baselineskip}
Lovv는 “어디를 가라”에서 끝나는 서비스가 아니라, 왜 이 소도시인지, 어떤 데이터에 근거했는지, 지금 실제로 갈 수 있는지를 함께 보여주는 목적지 탐색 엔진이다.

\begin{itemize}

\item 출처와 외부 링크를 UI에서 적극적으로 노출해 “LLM이 만든 이야기”가 아니라 데이터 기반 큐레이션임을 드러낸다.

\section*{2. 문제 정의}
\addcontentsline{toc}{section}{2. 문제 정의}


\DocNeedspace{8\baselineskip}
\subsection*{2.1 사용자 문제}
\addcontentsline{toc}{subsection}{2.1 사용자 문제}


\DocNeedspace{5\baselineskip}
\subsubsection*{유명 관광지 편중}
여행자는 검색 결과, SNS 후기, 패키지 상품에서 반복적으로 노출되는 대도시와 유명 관광지 위주로 정보를 찾게 된다. 이 과정에서 실제 취향이나 여행 시기에 더 잘 맞는 소도시 후보는 비교 대상에 오르기 어렵고, 성수기 혼잡이나 높은 비용을 감수한 채 익숙한 목적지를 선택하게 된다.

\DocNeedspace{5\baselineskip}
\subsubsection*{소도시 정보 부족}
방문객 수가 낮은 관광지와 주변 소도시는 공식 정보, 블로그 후기, 지도 정보, 축제 일정, 교통편이 여러 채널에 흩어져 있다. 여행자는 목적지의 계절성, 이동 난이도, 주변 관광 자원, 비수기 매력을 한 번에 파악하기 어려워 계획 수립에 많은 시간을 쓰게 된다.

\DocNeedspace{5\baselineskip}
\subsubsection*{여행 조건 비교 어려움}
날씨, 교통, 혼잡도, 동행 유형, 체험 요소, 여행 기간을 동시에 고려해야 하지만 일반 여행 검색은 조건별 정보를 분리해서 제공하는 경우가 많다. 특히 짧은 일정에서는 비가 올 때의 대체 코스, 이동 시간이 긴 목적지의 부담, 축제 포함 여부 같은 의사결정 기준을 사용자가 직접 조합해야 한다.

\DocNeedspace{5\baselineskip}
\subsubsection*{추천 신뢰도 부족}
일반 생성형 답변은 그럴듯한 일정은 제시할 수 있지만, 어떤 데이터와 출처를 근거로 추천했는지 확인하기 어려운 경우가 있다. 축제 개최 여부, 날씨 정보, 장소 운영 상태처럼 변동성이 큰 정보가 검증되지 않으면 사용자는 추천 결과를 실제 여행 계획에 바로 적용하기 어렵다.

\DocNeedspace{5\baselineskip}
\subsubsection*{선택 불안}
낯선 소도시는 정보가 적어 실제 저장·예약·이동으로 전환되기 어렵다.

\DocNeedspace{24\baselineskip}
\subsection*{2.2 시장·운영 문제}
\addcontentsline{toc}{subsection}{2.2 시장·운영 문제}


\DocNeedspace{5\baselineskip}
\subsubsection*{지역 관광 수요 불균형}
관광 수요가 일부 대도시와 대표 관광지에 집중되면 숙박·교통·상권 혼잡이 심해지고, 여행 만족도와 지역 수용력이 함께 낮아질 수 있다. 반면 주변 소도시나 비수기 목적지는 충분한 매력을 갖고 있어도 노출 기회가 부족해 방문 수요를 만들기 어렵다. 로브는 인기 지역의 대체 목적지와 비수기 방문 가치를 함께 제안해 관광 수요를 더 넓은 지역과 시기로 분산시키는 역할을 한다.

\DocNeedspace{5\baselineskip}
\subsubsection*{데이터 분산}
여행 추천에 필요한 관광지 정보, 축제 일정, 교통 접근성, 날씨, 지도, 숙박·맛집 탐색 링크는 서로 다른 기관과 플랫폼에 흩어져 있다. 이 데이터들은 형식과 갱신 주기가 달라 한 번에 비교하기 어렵고, 사용자가 직접 취합하면 정보 누락이나 오래된 정보를 사용할 가능성이 높다. 서비스는 목적지 단위로 필요한 데이터를 묶고 출처와 갱신 기준을 관리해 추천 결과의 일관성과 신뢰도를 높여야 한다.

\DocNeedspace{5\baselineskip}
\subsubsection*{운영 검증 필요}
지자체, 관광 운영자, 데이터 제공 기관이 제안하는 지역 정보는 실제 서비스에 반영되기 전에 정확성, 최신성, 공개 적합성을 검토해야 한다. 검증 절차가 없으면 폐업한 장소, 종료된 축제, 과장된 홍보 정보가 추천 결과에 포함될 수 있고 서비스 신뢰도가 낮아진다. 따라서 데이터 제안, 관리자 검토, 승인·반려 이력, 갱신 상태를 관리하는 운영 흐름이 필요하다.

\DocNeedspace{5\baselineskip}
\subsubsection*{성과 측정 어려움}
관문 대도시 방문과 실제 지역 소도시 방문 준비 행동을 분리해 보지 않으면 수요 분산 효과를 잘못 판단할 수 있다.

\DocNeedspace{10\baselineskip}
\section*{3. 프로젝트 목표}
\addcontentsline{toc}{section}{3. 프로젝트 목표}


\DocNeedspace{8\baselineskip}
\subsection*{3.1 핵심 목표}
\addcontentsline{toc}{subsection}{3.1 핵심 목표}


\begin{itemize}
\item 방문객 수가 많은 혼잡 관광지 대신 대체 가능한 소도시 여행지를 추천한다.
\item 추천 1건은 소도시 1곳을 최종 목적지로 삼고, 생성 일정은 그 도시 서비스 범위 안에서 완성한다.
\item PoC는 한국 강원·경북 소도시를 기준으로 검증하고, 한국 전국 확장은 P2 범위로 둔다.
\item 일본 데이터 기반 추천과 일본 현지 데이터 연동은 수집 경로가 복구된 뒤 P5에서 재검토한다.
\item 여행 시기, 축제, Wikipedia 기반 월별 날씨 경향, 이동 경로, 주변 관광 정보를 결합한 여정을 제안한다.
\item RAG 기반 검색과 멀티스텝 에이전트로 추천 근거와 안정성을 확보한다.
\item 추천 이유, 출처, 검증 상태를 함께 보여주는 설명 가능한 추천 경험을 제공한다.
\item KICK(유사 취향 공개 일정·1탭 복제)을 P2 설계 범위에 포함해 선택 불안을 줄이고 저장·외부 링크 전환 가능성을 검증한다.
\end{itemize}

\subsection*{6.3 Production 범위}

\DocNeedspace{3\baselineskip}

\begin{itemize}
\item P1 구체화: 한국 강원·경북 데이터 기반 추천 흐름의 추천 품질, 일정 구성, 설명 가능성, 데이터 검증, 운영 안정성을 보강한다.
\item P2 구현: 한국 전국 확장과 KICK(유사 취향 공개 일정·1탭 복제)을 구현 검토 대상에서 실제 구현 대상으로 올린다.
\item P2 고도화: 공개 일정 운영, 개인정보·공개 범위, 저장·복제 전환, 재방문 개인화 흐름을 Production 수준으로 정교화한다.
\end{itemize}

\begin{itemize}
\item 사용자 입력 수집
\item 국내 지역·여행 시기·일정 길이·테마·진입점 조건 분류
\item RAG 기반 후보 검색
\item 후보 필터링 및 선정
\item 메인 관광지·서브 관광지·식사 슬롯 기반 일정 구성
\item 추천 이유와 외부 정보 보강
\item 출처·갱신 시점·검증 상태 표시
\item P2 KICK 흐름 또는 후속 질의응답
\item 조건 변경 기반 재추천
\end{itemize}


\DocNeedspace{24\baselineskip}
\subsection*{9.2 시스템 원칙}
\addcontentsline{toc}{subsection}{9.2 시스템 원칙}


\begin{itemize}
\item 추천 결과는 근거 데이터와 함께 제공한다.
\item 추천은 단순 생성형 답변이 아니라 검색된 데이터와 검증 상태를 기반으로 구성한다.
\item API Key는 개발·운영 담당자가 서버 환경 변수 또는 백엔드 프록시에서 관리한다.
\item 클라이언트에서 직접 호출하기 어려운 API는 딥링크 또는 백엔드 프록시로 대체한다.
\item 데이터 출처, 최신성, 검증 상태를 표시한다.
\item 사용자가 후속 질문에서 조건을 바꾸면 동일한 추천 파이프라인을 재실행해 대체 소도시 또는 수정 일정을 제안한다.
\item 초기 일정은 소도시 선정 근거 관광지만으로 채우지 않고, 선정 근거 관광지 기반 메인 관광지와 DynamoDB 재인출 기반 서브 관광지를 조합한다.
\item 장소 타입이 과도하게 중복되지 않도록 균형을 두고, 식당은 별도 식사 슬롯에서 사용자가 선택하게 한다.
\item 일정 수정에서는 메인 관광지는 테마 유지 조건에서 교체하고, 서브 관광지는 테마 고정 없이 사용자가 자유롭게 바꿀 수 있게 한다.
\item 첫 화면은 가능한 한 사전계산·캐시를 활용하고, 이유 문구는 템플릿 또는 정형 근거를 우선 사용한다.
\item 공개 일정 복제와 부분 수정은 P2 상세 설계에서 권한, 개인정보, 버전 관리 기준을 정리한다.
\item PoC에서는 빠른 검증을 위해 핵심 흐름을 우선 구현한다.
\end{itemize}

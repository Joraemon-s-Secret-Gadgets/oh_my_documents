\section*{1. 목적}
\addcontentsline{toc}{section}{1. 목적}
\addtocontents{toc}{\protect\TocNoChildTight}


\DocNeedspace{3\baselineskip}
본 문서는 여행 추천 Multi-Agent 서비스에서 한국 소도시 추천에 필요한 도시·관광지·축제·방문객 통계 데이터를 취득하기 위한 범위, 데이터 소스, 검증 방식, 저장 구조를 정의한다.


\DocNeedspace{3\baselineskip}
이번 문서의 기준은 실제 수집과 TourAPI 명세 확인을 거친 강원·경북 40개 도시 데이터다.

\DocNeedspace{3\baselineskip}
로컬 \texttt{data/\allowbreak{}KR/\allowbreak{}*.\allowbreak{}json} 파일은 실제 수집 검증 산출물이자 S3 Raw Bucket 적재 전 입력으로 관리한다.

\DocNeedspace{3\baselineskip}
서비스 조회용 정규화 데이터는 S3 Raw 누적 후 Lambda 전처리를 거쳐 DynamoDB에 적재한다.


\DocNeedspace{3\baselineskip}
핵심 방향은 다음과 같다.


\begin{itemize}
\item City, Attraction, Festival, 방문객 통계의 정의된 항목을 모두 최초 취득 대상으로 둔다.
\item 한국은 강원·경북 40개 도시를 실제 수집 검증 범위로 둔다.
\item City는 Wikipedia/Wikidata 기반으로 취득하고, \texttt{KR-\allowbreak{}\{\allowbreak{}도\_\allowbreak{}코드\allowbreak{}\}-\allowbreak{}\{\allowbreak{}CITY\_\allowbreak{}EN\allowbreak{}\}} 형식의 안정 ID를 사용한다.
\item 관광지와 축제는 한국관광공사 TourAPI 4.0(KorService2)을 중심으로 취득한다.
\item 방문객 통계는 한국관광공사 관광 빅데이터 API(DataLabService)를 활용한다.
\item 운영시간, 입장료, 사진처럼 실제 제공률이 낮거나 변동성이 큰 값은 \texttt{missing} 또는 \texttt{needs\_\allowbreak{}review} 상태로 관리한다.
\item 자동 수집, 공식 사이트 확인, Web Search Worker, 수동 검수를 하나의 취득 파이프라인으로 묶어 처리한다.
\end{itemize}

\section*{2. 목표 데이터 모델}
\addcontentsline{toc}{section}{2. 목표 데이터 모델}


\DocNeedspace{8\baselineskip}
\subsection*{2.1 관계 구조}
\addcontentsline{toc}{subsection}{2.1 관계 구조}


\DocNeedspace{3\baselineskip}
대표 데이터 수집 계획서의 공통 모델은 \texttt{Administrative\allowbreak{}Area 1:\allowbreak{}N City}를 기준으로 한다.

\DocNeedspace{3\baselineskip}
한국 데이터에서는 이 상위 행정 단위를 \texttt{Prefecture}로 구체화하고, 강원(\texttt{KR-\allowbreak{}42})과 경북(\texttt{KR-\allowbreak{}47}) 아래의 40개 시·군·구를 추천 기준 지역인 \texttt{City}로 관리한다.


\DocNeedspace{3\baselineskip}
\texttt{City}는 추천 결과를 묶는 중심 엔티티다.

\DocNeedspace{3\baselineskip}
관광지(\texttt{Attraction})와 축제·행사(\texttt{Festival})는 사용자가 실제로 방문하거나 일정에 넣을 수 있는 추천 소재이며, 방문객 통계(\texttt{Visitor\allowbreak{}Statistics})는 월별 수요와 계절성을 판단하는 보조 지표로 City에 연결한다.

따라서 한국 데이터 취득 결과는 \texttt{data/\allowbreak{}KR/\allowbreak{}cities.\allowbreak{}json}을 중심으로 \texttt{attractions.\allowbreak{}json}, \texttt{festivals.\allowbreak{}json}, \texttt{visitor\_\allowbreak{}statistics.\allowbreak{}json}이 같은 \texttt{city\_\allowbreak{}id}를 공유하는 구조가 된다.

\DocNeedspace{3\baselineskip}
이 구조를 유지해야 S3 Raw 적재 후 Lambda 전처리와 DynamoDB 정규화 단계에서도 City 기준 추천 데이터셋을 일관되게 만들 수 있다.

\subsection*{5.4 데이터 정규화 규칙}
\addcontentsline{toc}{subsection}{5.4 데이터 정규화 규칙}


\begin{itemize}
\item \texttt{city\_\allowbreak{}id} 형식은 \texttt{KR-\allowbreak{}\{\allowbreak{}도\_\allowbreak{}코드\allowbreak{}\}-\allowbreak{}\{\allowbreak{}CITY\_\allowbreak{}EN\allowbreak{}\}}을 사용한다.
\item \texttt{attraction\_\allowbreak{}id} 형식은 \texttt{KR-\allowbreak{}\{\allowbreak{}도\_\allowbreak{}코드\allowbreak{}\}-\allowbreak{}\{\allowbreak{}CITY\_\allowbreak{}EN\allowbreak{}\}-\allowbreak{}ATT-\allowbreak{}\{\allowbreak{}content\allowbreak{}id\allowbreak{}\}}를 사용한다.
\item \texttt{festival\_\allowbreak{}id} 형식은 \texttt{KR-\allowbreak{}\{\allowbreak{}도\_\allowbreak{}코드\allowbreak{}\}-\allowbreak{}\{\allowbreak{}CITY\_\allowbreak{}EN\allowbreak{}\}-\allowbreak{}FES-\allowbreak{}\{\allowbreak{}content\allowbreak{}id\allowbreak{}\}}를 사용한다.
\item 강원은 \texttt{KR-\allowbreak{}42}, 경북은 \texttt{KR-\allowbreak{}47} 광역 코드를 사용한다.
\item 포항시 남구·북구 레코드는 \texttt{KR-\allowbreak{}47-\allowbreak{}POHANG}으로 통합한다.
\item 도시명은 \texttt{city\_\allowbreak{}name\_\allowbreak{}ko}, 필요 시 \texttt{city\_\allowbreak{}name\_\allowbreak{}en}을 분리 저장한다.
\item 관광지와 축제는 반드시 \texttt{city\_\allowbreak{}id}를 가진다.
\item TourAPI의 \texttt{content\allowbreak{}id}, \texttt{content\allowbreak{}type\allowbreak{}id}, \texttt{areacode}, \texttt{sigungu\allowbreak{}code}, \texttt{lcls\allowbreak{}Systm1}, \texttt{lcls\allowbreak{}Systm2}, \texttt{lcls\allowbreak{}Systm3}, \texttt{cat1}, \texttt{cat2}, \texttt{cat3}를 원본 참조 필드로 저장한다.
\item 기간 정보는 원문 문자열(\texttt{period\_\allowbreak{}text})과 정규화 값(\texttt{start\_\allowbreak{}date}, \texttt{end\_\allowbreak{}date}, \texttt{month})을 함께 저장한다.
\item 외부 설명문은 원문 복제가 아니라 내부 요약문으로 재작성하거나 TourAPI \texttt{overview} 값을 출처와 함께 저장한다.
\end{itemize}

포항 통합 & 포항시 남구·북구 레코드는 \texttt{KR-\allowbreak{}47-\allowbreak{}POHANG}으로 통합되어 있어야 한다. \\
출처 기록 & 모든 자동 수집 데이터는 \texttt{source\_\allowbreak{}name}, \texttt{source\_\allowbreak{}url}, \texttt{collected\_\allowbreak{}at}을 가진다. \\
전체 필드 상태 & 정의된 모든 필드는 \texttt{collected}, \texttt{needs\_\allowbreak{}review}, \texttt{missing}, \texttt{blocked} 중 하나의 상태를 가진다. \\
수량 검증 & 도시 40개, 관광지 3,709건, 축제 106건, 월별 통계 12개월 범위를 기준으로 누락을 확인한다. \\
최신성 & 운영시간, 입장료, 축제 기간은 확인일을 기록한다. \\
기후 정합성 & \texttt{climate\_\allowbreak{}table}은 Wikipedia와 기상청 자료를 비교하고 불일치 여부를 검수 메타데이터로 남긴다. \\
저작권 & 사진과 설명문은 사용 가능 조건을 확인한다. 설명문은 내부 요약문으로 저장한다. \\
API 키 보안 & API Key는 문서나 산출 JSON에 직접 저장하지 않는다. \\
\bottomrule
\end{longtable}

\section*{10. 법적·운영 제약}
\addcontentsline{toc}{section}{10. 법적·운영 제약}
\addtocontents{toc}{\protect\TocNoChildTight}


\begin{itemize}
\item Wikipedia API 및 크롤링 기반 설명은 라이선스 조건에 따라 출처와 원문 링크를 기록한다.
\item TourAPI와 공공데이터포털 데이터는 공공누리 유형, API 이용 조건, 출처 표기 조건을 확인한다.
\item TourAPI 일일 쿼터 제한을 준수한다.
\item API Key는 환경변수 또는 보안 저장소로 관리하고 문서와 산출물에 직접 기록하지 않는다.
\item 지자체 관광 페이지는 이용약관과 크롤링 허용 범위를 확인한다.
\item 사진은 저작권 위험이 크므로 자동 수집 값이 있더라도 사용 가능 조건을 확인하고, 불명확하면 \texttt{needs\_\allowbreak{}review} 또는 \texttt{blocked}로 관리한다.
\item 운영시간과 입장료는 변경 가능성이 높으므로 서비스 화면에 "확인일 기준" 문구를 표시한다.
\end{itemize}

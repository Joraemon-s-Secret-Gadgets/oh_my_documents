1. 프로젝트 개요

1.1 프로젝트명

로브 (Lovv)

Lovv는 Local(소도시), Vibe(분위기·감성), Voyage(여행)의 의미를 결합한 소도시 여행 추천 서비스다.

1.2 한 줄 정의

로브는 국가와 여행 시기부터 시작해 오버투어리즘을 피할 수 있는 소도시 대체 여행지를 발견하게 해주는 설명 가능한 여행 추천 에이전트다.

서울·오사카 말고, 지금은 이곳

지도에서 흐릿한 곳, 여행에선 가장 빛나는 곳

1.3 기획 배경

국내외 여행 수요는 회복되고 있지만, 인기 관광지로의 집중은 여행 만족도 저하와 지역 수용력 문제를 동시에 만든다. 여행자는 덜 붐비는 목적지를 원하지만, 방문객 수가 낮은 관광지와 그 주변 소도시 정보는 흩어져 있고 이동성, 계절성, 축제, 날씨, 주변 관광지 정보를 한 번에 비교하기 어렵다.

로브의 1차 검증 타깃은 한국인 재방일객의 일본 지역 여행이다. 기준 상세 문서는 2025년 한국에서 일본으로 향한 여행 흐름이 일본에서 한국으로 향한 흐름보다 큰 비대칭 시장임을 근거로 삼고, 소도시 여행 수요를 전체 대중시장이 아니라 재방문·근거리·목적형 조건에서 커지는 고의도 하위시장으로 본다.

로브는 오버투어리즘 회피와 소도시 발견이라는 문제를 여행자 관점의 추천 경험으로 해결한다. 사용자가 국가와 여행 시기를 먼저 입력하면, 서비스는 그 조건을 기준으로 사용자의 자연어 취향과 날씨 조건을 해석하고 추천 근거가 있는 소도시 여정과 주변 정보를 함께 제공한다.

일반 여행 일정 서비스가 사용자가 이미 선택한 대도시 안에서 동선과 일정을 최적화하는 데 강점이 있다면, 로브는 어디를 갈지부터 정하지 못한 사용자에게 국가와 시기 조건에 맞는 대체 소도시를 발견하게 하는 목적지 탐색 엔진으로 차별화한다.

PoC 단계에서는 많은 지역을 얕게 나열하기보다 한국·일본의 제한된 소도시를 깊게 다룬다. 축제, 기상, 교통, 체험, 외부 링크를 고밀도로 채워 Lovv다운 목적지 추천 경험을 먼저 검증한다.

1.4 시장 포지셔닝과 차별점

트리플이 인기 도시 안에서 여행 일정을 최적화하는 슈퍼앱이라면, Lovv는 국가와 시기부터 시작해 오버투어리즘을 피할 수 있는 소도시 대체 여행지를 발견하게 해주는 설명 가능한 추천 에이전트다.

Lovv는 “어디를 가라”에서 끝나는 서비스가 아니라, 왜 이 소도시인지, 어떤 데이터에 근거했는지, 지금 실제로 갈 수 있는지를 함께 보여주는 목적지 탐색 엔진이다.

PoC 보완 방향은 다음과 같다.

  • 소도시를 넓게 나열하기보다 제한된 후보 도시를 깊게 구성한다.
  • 출처와 외부 링크를 UI에서 적극적으로 노출해 “LLM이 만든 이야기”가 아니라 데이터 기반 큐레이션임을 드러낸다.
  • 첫 질문은 “어디로 갈까요?”보다 “언제, 어느 나라로 떠나세요?”로 설계한다.
  • 소도시 후보 리스트보다 소도시 1곳을 깊게 이해하는 상세 흐름을 우선한다.
  • 지자체·공공 파트너가 볼 수 있는 추천 노출, 클릭, 관심도 지표를 장기 확장 방향으로 둔다.

2. 문제 정의

2.1 사용자 문제

문제설명
유명 관광지 편중여행자는 익숙한 대도시와 유명 관광지 위주로 정보를 찾게 된다.
소도시 정보 부족방문객 수가 낮은 관광지와 그 주변 소도시의 계절성, 축제, 이동 경로, 주변 관광 정보를 한 번에 파악하기 어렵다.
여행 조건 비교 어려움날씨, 교통, 혼잡도, 동행 유형, 체험 요소를 동시에 고려하기 어렵다.
추천 신뢰도 부족일반 생성형 답변은 출처와 근거가 약해 실제 여행 계획에 바로 쓰기 어렵다.
선택 불안낯선 소도시는 정보가 적어 실제 저장·예약·이동으로 전환되기 어렵다.

2.2 시장·운영 문제

문제설명
지역 관광 수요 불균형일부 지역에 관광객이 집중되고 다른 지역은 노출 기회가 부족하다.
데이터 분산관광지, 축제, 교통, 날씨, 지도 정보가 여러 플랫폼에 흩어져 있다.
운영 검증 필요지자체·관광 운영자·데이터 제공 기관이 제안한 정보를 검토하고 반영할 흐름이 필요하다.
성과 측정 어려움관문 대도시 방문과 실제 지역 소도시 방문 준비 행동을 분리해 보지 않으면 수요 분산 효과를 잘못 판단할 수 있다.

3. 프로젝트 목표

3.1 핵심 목표

  • 방문객 수가 많은 혼잡 관광지 대신 대체 가능한 소도시 여행지를 추천한다.
  • 추천 1건은 소도시 1곳을 최종 목적지로 삼고, 생성 일정은 그 도시 서비스 범위 안에서 완성한다.
  • 한국과 일본 소도시를 나라별 독립 트랙으로 제공하되, PoC는 한국 강원·경북 및 일본 관동 우선 수집 범위를 기준으로 검증하고 최종 목표는 한일 전국 도시로 확장한다.
  • 여행 시기, 축제, 날씨, 이동 경로, 주변 관광 정보를 결합한 여정을 제안한다.
  • RAG 기반 검색과 멀티스텝 에이전트로 추천 근거와 안정성을 확보한다.
  • 추천 이유, 출처, 검증 상태를 함께 보여주는 설명 가능한 추천 경험을 제공한다.
  • KICK(유사 취향 공개 일정·1탭 복제)을 P2 설계 범위에 포함해 선택 불안을 줄이고 저장·외부 링크 전환 가능성을 검증한다.

3.2 성공 기준

구분기준
추천 경험사용자가 국가와 여행 시기, 여행 일정 길이, 자연어 취향 또는 지도 마커를 입력하면 조건에 맞는 소도시 1곳과 day-by-day 일정이 생성된다.
근거 제공추천 결과에는 추천 이유, 지도 링크, 날씨 정보, 외부 탐색 링크, 출처·검증 상태 중 핵심 정보가 함께 제공된다.
KICK 전환유사 취향 공개 일정이 노출되고 사용자가 1탭 복제 후 저장 또는 부분 수정으로 이어지는 흐름을 설계 산출물로 확보한다.
범위 통제PoC에서는 P1 기능을 구현하고, P2(KICK 포함)~P3는 설계 중심으로 정리하며, P5는 제외 범위로 관리한다.
확장성신규 지역, 축제, 촬영지, 체험 정보, 외부 데이터 소스를 추가할 수 있는 공통 데이터 모델과 운영 구조를 유지한다.

4. 서비스 개념

4.1 핵심 서비스

로브는 공개 추천 흐름에서는 선택 로그인 또는 온보딩을 통해 테마와 기본 국가 선호를 입력받고, 카드 슬롯·지도 마커·AI 챗봇으로 일정 생성에 진입하게 한다. 챗봇은 국가, 여행 시기(월), 여행 일정 길이(당일치기~2박 3일)를 먼저 확인한 뒤 사용자의 자연어 취향을 테마 또는 추천 진입점으로 해석한다.

소도시는 행정구역의 인구수만으로 판단하지 않는다. 관문 대도시(서울·도쿄·오사카·후쿠오카 등)는 교통 안내에만 사용하고, 지역 거점도시와 인접 소도시·농산어촌을 분리해 관광객 집중도가 낮고 실제 여행 경험을 만들 수 있는 목적지와 주변 지역을 중심으로 본다.

PoC 기준 핵심 설계 원칙은 다음과 같다.

  • 추천 1건의 최종 목적지는 소도시 1곳이다.
  • 생성 일정의 주요 관광지와 체험은 추천 소도시 서비스 범위 안에서 구성한다. 숙소는 직접 추천보다 검색 링크로 연결한다.
  • 공항·역·관문도시는 교통 안내에만 사용하고 일정의 목적지로 섞지 않는다.
  • 적합 소도시를 확정할 수 없으면 여러 도시를 섞지 않고 추가 조건을 요청한다.

로브의 첫 질문은 “어디로 갈까요?”보다 “언제, 어느 나라로 떠나세요?”에 가깝다. 사용자가 도시를 이미 정했다고 가정하지 않고, 국가와 여행 시기부터 고정한 뒤 조건에 맞는 소도시를 좁혀 간다.

추천 경험의 핵심 aha는 “이런 곳이 있는 줄 몰랐는데, 지금 내 조건에 맞고, 진짜 갈 수 있겠다”이다. 이 문장은 세 요소로 나뉜다.

요소의미
발견사용자가 몰랐던 소도시를 만난다.
납득왜 이 도시가 지금 내 조건에 맞는지 이해한다.
실행가능성일정, 지도, 날씨, 외부 링크로 실제 행동이 가능해진다.

추천 여정 funnel은 다음과 같다.

단계목표
0. 첫 진입온보딩 직후 행동 이력이 없어도 타이밍·축제·혼잡 정보로 첫 발견을 만든다.
1. 추천 기준왜 이 도시인지 설명해 납득을 만든다.
2. 후속 질의세션 내 신호로 미니 개인화를 제공한다.
3. 일정 생성·저장day-by-day 일정과 저장으로 실행가능성을 만든다.
4. 재방문선언·행동·세션 신호를 누적해 개인화 정확도를 높인다.

행동 이력이 없어도 타이밍, 날씨, 축제, 혼잡은 누구에게나 현재 존재한다. Lovv는 초기 추천에서 이 공통 신호를 활용하고, 사용자의 세션 내 선택과 후속 질의를 빠르게 반영한다. 갔던 곳 기반 개인화는 재방문 단계의 무기로 분리한다.

취향 프로필은 세 계층으로 다룬다.

  • 선언 신호: 온보딩에서 입력한 테마·국가·여행 스타일
  • 행동 신호: 저장, 클릭, 복제, 외부 링크 이동
  • 세션 신호: 현재 대화에서 나온 여행 시기, 일정 길이, 동행, 이동 강도

KICK은 발견·납득·실행 다음의 선택 전환 장치다. 사용자는 나와 취향이 비슷한 공개 일정을 참고하고, 필요한 경우 1탭 복제와 챗봇 부분 조정으로 내 일정에 가깝게 바꿀 수 있다. 대표 기획서에서는 KICK을 추천 축 자체가 아니라 선택 불안을 줄이는 P2 설계 검토 범위로 다룬다.

KICK의 핵심 후보 흐름은 다음과 같다.

  • 로그인·온보딩으로 기본 취향을 확보한다.
  • 공개 일정 카드에서 유사 취향 근거를 보여준다.
  • 사용자는 공개 일정을 1탭으로 복제하거나 참고한다.
  • 챗봇이 일정 길이, 이동 강도, 동행 등 일부 조건을 조정한다.
  • 조정된 일정은 저장 또는 재추천 흐름으로 이어진다.

KICK 운영 원칙은 상세 설계에서 다음 방향으로 검토한다.

  • 공개 일정 부족 시 가짜 사용자 일정을 만들지 않는다.
  • 초기에는 명시적인 Lovv 큐레이션으로 노출한다.
  • 유사성 표현은 실제 일치 근거만 표시하고 과장하지 않는다.
  • 공개 일정 복제와 부분 수정은 개인정보, 공개 범위, 버전 관리 기준을 먼저 확정한 뒤 설계한다.
  • 개인정보는 선택 공개, 익명화, 공개 범위 제한을 기본 방향으로 검토한다.

KICK 성공 기준은 다음과 같다.

  • 공개 일정 노출
  • 유사 취향 근거 확인
  • 1탭 복제
  • 챗봇 부분 수정
  • 일정 저장
  • 외부 링크 클릭
  • 재방문

6. 프로젝트 범위

6.1 PoC 범위

PoC는 P1 구현을 핵심 범위로 한다. P2와 P3는 설계 산출물을 우선 만들고, 일정 여유가 있을 경우 일부 구현한다. KICK은 추천 축 자체가 아니라 선택 전환 장치이므로 P2 설계 범위에 별도 포함한다.

우선순위기능 범위PoCProduction상태
P1혼잡 회피 대체 여행지 추천, 한일 축제 기반 여정 추천, 일주일 이내 여행의 날씨 기반 추천구현고도화핵심
P2미디어 촬영지·성지순례 추천, KICK(유사 취향 공개 일정·1탭 복제)설계구현 검토선택 구현
P3철도 경로 기반 주변 관광지 안내, 지역 문화 체험 추천설계구현 검토설계
P4스키 액티비티 여행 추천정의Production 구현 검토Production
P5별 관측 조건 기반 여행지 추천, 치유·웰니스 여정 추천, 예산 역산 추천제외검토제외

PoC 데이터는 한국 강원·경북 및 일본 관동을 우선 수집 범위로 삼는다. 전국을 선제 구축하지 않고 3~5개 검증 후보 도시부터 깊게 구성한 뒤, 같은 공통 데이터 모델을 한국·일본 전체 도시로 확장한다.

6.2 제외 범위

다음 기능은 PoC 상세 요구기능에서 제외하고, 우선순위 및 단계별 추진 범위에만 기록한다.

  • 별 관측 조건 기반 여행지 추천
  • 치유·웰니스 여정 추천
  • 예산 역산 추천
  • B2C 과금·가격 모델
  • B2G 영리화 시나리오

7.4 비즈니스 모델

B2C·B2B·B2G는 서로 다른 수익화 단계가 아니라 수요 검증, 성과형 제휴, 공익 협력을 분리해 관리하는 사업 트랙이다. 가격 실험, 수익성 시뮬레이션, 파트너별 제안 전문은 상세 문서에 둔다.

B2C 과금·가격 모델과 B2G 영리화 시나리오는 대표 기획서 범위에서 제외한다. 비즈니스 모델 세부 내용은 보조 문서 supplemental/business_model.md에서 관리한다.

B2C: 수요·가치 검증 트랙

1차 타깃은 한국인 재방일객의 일본 지역 여행이다. 서비스 가치는 정보 열람이 아니라 제한된 시간 안에서 실행 가능한 일정, 이동 실패 방지, 예약 가능한 다음 행동에 둔다.

핵심 KPI는 다음과 같다.

  • 일정 저장률
  • 공유·내보내기율
  • 예약 파트너 클릭률
  • 재방문율

B2B: 성과형 제휴 트랙

B2B는 취향·일정 맥락을 예약 가능한 상품 탐색으로 연결하고, 성과형 제휴 가능성을 검토하는 추천 중개 트랙이다.

사용자우선 검토 제휴
한국인 방일 여행자KKday·클룩 액티비티 병렬, 라쿠텐 숙박 조건부
일본인 방한 여행자라쿠텐 숙박, KKday 액티비티

수익 인식 기준은 총 예약액보다 승인 매출 기준으로 검토한다. 성과 구간, CPA, 추천 모듈·API, 화이트라벨, 공동브랜드 같은 장기 확장 후보는 보조 문서에서 관리한다.

B2G: 비과금 공익 협력 트랙

B2G는 지자체·공공기관과 검증된 관광 정보를 “이번 달 여행지” 월간 큐레이션으로 노출하는 비과금 협력이다.

성과 지표는 재정 수치보다 다음 비재정 지표를 우선한다.

  • 노출
  • 클릭
  • 일정 생성
  • 일정 저장
  • 공식 정보 이동
  • 방문 의향

공식 정보 링크와 제휴 링크는 분리한다. 관문도시 방문은 소도시 성과로 보지 않고, 지역 숙박·지역 상품 클릭·공식 정보 이동을 별도로 집계한다.

8. 데이터 및 외부 연동

8.1 필요 데이터

우선 데이터는 소도시 기본 정보, 축제·행사, 혼잡도·방문 통계, 월별 기후 경향, 지도·장소, 출처·검증 정보로 구성한다. 실시간 날씨는 상세 화면 표시용으로 두고, 미디어 촬영지, 철도·교통, 지역 문화 체험, 공개 일정 데이터는 P2 이후 확장 데이터로 검토한다.

우선혼잡도·방문 통계대체 여행지 추천, 수요 분산 근거, 관문도시 성과 오인 방지
우선월별 기후 경향월별 추천 적합도와 계절 정합성 검증

확장철도·교통 정보주요 역, 환승, 소요 시간, 패스 활용 가능 여부
확장지역 문화 체험 정보체험 유형, 예약 필요 여부, 난이도, 우천 적합성
확장공개 일정 데이터익명화된 일정, 복제·저장 전환 측정

PoC는 한국 강원·경북과 일본 관동을 우선 수집 범위로 삼는다. 현재 범위는 전국 선제 구축이 아니라 검증 후보 도시를 깊게 구성하기 위한 출발점이다. 최종 목표는 한국·일본 전체 도시 확장이다. 확장 시에도 AdministrativeArea, City, Attraction, Festival, VisitorStatistics를 중심으로 한 공통 데이터 모델을 유지한다.

8.2 외부 연동

외부 연동은 지도·장소, 날씨, 관광·방문 통계, 예약 딥링크, 공공 관광 데이터로 나뉜다. 클라이언트에서 직접 호출하기 어려운 API는 딥링크 또는 백엔드 프록시로 대체한다.

출처역할
TourAPI / DataLab한국 관광지·축제·방문 통계
JNTO / 일본 관광청(JTA) / e-Stat일본 방문 통계와 지역 검증

클룩·라쿠텐 트래블·KKday예약·숙박·액티비티 탐색 링크 연결
공공 관광 데이터관광지, 축제, 교통, 지역 공식 정보

WeatherAPI는 상세 화면에서 현재 날씨를 보여주는 역할이다. 월별 기후 경향, 추천 적기, 기후 정합성 검증은 기상청 API허브와 일본기상청(JMA)을 기준으로 한다. 두 데이터는 역할이 다르므로 함께 사용한다.

9. 기술 방향

9.1 추천 구조

Lovv의 기술 방향은 Explainable RAG 기반 소도시 추천 엔진이다. 추천 파이프라인은 다음 단계로 구성한다.

  • 사용자 입력 수집
  • 국가·여행 시기·일정 길이·테마·진입점 조건 분류
  • RAG 기반 후보 검색
  • 후보 필터링 및 선정
  • 일정 구성
  • 추천 이유와 외부 정보 보강
  • 출처·갱신 시점·검증 상태 표시
  • P2 KICK 흐름 또는 후속 질의응답
  • 조건 변경 기반 재추천

9.2 시스템 원칙

  • 추천 결과는 근거 데이터와 함께 제공한다.
  • 추천은 단순 생성형 답변이 아니라 검색된 데이터와 검증 상태를 기반으로 구성한다.
  • API Key는 개발·운영 담당자가 서버 환경 변수 또는 백엔드 프록시에서 관리한다.
  • 클라이언트에서 직접 호출하기 어려운 API는 딥링크 또는 백엔드 프록시로 대체한다.
  • 데이터 출처, 최신성, 검증 상태를 표시한다.
  • 사용자가 후속 질문에서 조건을 바꾸면 동일한 추천 파이프라인을 재실행해 대체 소도시 또는 수정 일정을 제안한다.
  • 첫 화면은 가능한 한 사전계산·캐시를 활용하고, 이유 문구는 템플릿 또는 정형 근거를 우선 사용한다.
  • 공개 일정 복제와 부분 수정은 P2 상세 설계에서 권한, 개인정보, 버전 관리 기준을 정리한다.
  • PoC에서는 빠른 검증을 위해 핵심 흐름을 우선 구현한다.

9.3 데이터 파이프라인

데이터 파이프라인은 수집, 로컬 검증, 원본 보존, 정규화, 추천·검색 활용 계층으로 나눈다. S3 Raw Bucket, Lambda, DynamoDB 같은 구체 구현 방식은 데이터 수집 계획서, 데이터베이스 설계 명세서, 기술 명세서에서 관리한다.

  • 자동 수집
  • 로컬 검증 산출물 생성
  • 원본 데이터 보존
  • 배치 전처리
  • 정규화 데이터 적재
  • 추천·검색 계층에서 RAG와 운영 조회에 활용

11. 기대 효과

로브를 통해 여행자는 유명 관광지 중심의 반복적인 여행 탐색에서 벗어나, 덜 붐비면서도 자신의 취향과 여행 시기에 맞는 소도시 대체 여행지를 더 쉽게 발견할 수 있다. 단순히 장소 목록을 보여주는 것이 아니라 추천 이유, 날씨, 지도, 외부 링크, 출처·검증 상태를 함께 제공하므로 실제 여행 계획으로 이어지는 의사결정 부담도 줄어든다.

유사 취향 공개 일정과 1탭 복제는 낯선 소도시 선택의 불안을 낮추고, 일정 저장, 외부 링크 클릭, 재방문으로 이어지는 전환 경로를 만든다.

지역 관광 운영자는 상대적으로 노출이 부족했던 소도시 관광 자원과 축제, 지역 문화 체험을 여행자에게 소개할 수 있다. 이를 통해 특정 대도시나 유명 관광지에 집중된 수요를 주변 지역으로 분산시키고, 지역 관광 자원의 활용도를 높이는 기회를 확보할 수 있다.

B2B 파트너는 고의도 맥락형 트래픽과 증분 거래를 확보할 수 있고, 지자체·공공기관은 B2G “이번 달 여행지” 노출을 통해 비재정 성과를 측정할 수 있다.

서비스 운영자는 RAG 기반 추천 근거, 사용자 피드백, 공개 일정 복제 퍼널, 데이터 검증 상태를 바탕으로 추천 정책을 점진적으로 개선할 수 있다.

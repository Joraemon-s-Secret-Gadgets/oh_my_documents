## 1. 대화기록 저장 방식 결정

### 결정 내용

대화 자체(채팅 로그)는 저장하지 않는다.
추천 결과로 도출된 여행 일정과 사용자가 명시적으로 표현한 선호(좋아요/싫어요, 온보딩 답변)만 저장한다.
PoC 단계는 로컬 스토리지, Product 단계는 회원가입 기반 서버 저장으로 전환한다.

### 현재 문서 상태

- `NFR-013`: "사용자 대화 내용은 로컬에서만 처리하며 별도 서버에 저장하지 않는다" — 저장 대상 구분이 없어 의도가 불명확
- `FR-REC-005`: 저장/비교 기능만 언급, 저장 데이터 범위와 구현 방식 미정의

### 수정·추가 내용

| 구분 | 대상 | 변경 방향 |
| --- | --- | --- |
| 수정 | NFR-013 | 저장하지 않는 대상(대화 로그)과 저장하는 대상(일정, 선호)을 명확히 분리 |
| 추가 | FR-STORE-001 | 저장 데이터 범위 정의 |
| 추가 | FR-STORE-002 | PoC 로컬 스토리지 구현 조건 |
| 추가 | FR-STORE-003 | Production 회원가입 기반 저장 조건 |

**NFR-013 개정안:**
```
NFR-013 | 사용자 대화 로그(채팅 내용 전문)는 저장하지 않으며, 서버나 외부 저장소로 전송하지 않는다.
         단, 추천으로 생성된 여행 일정, 사용자가 명시적으로 표현한 선호(온보딩 답변, 좋아요/싫어요)는
         PoC 단계에서는 브라우저 로컬 스토리지에, Production 단계에서는 로그인 사용자 계정에 저장할 수 있다.
```

**추가 요구사항 초안:**
```

                브라우저 로컬 스토리지에 저장하며 서버 전송 없이 클라이언트에서 관리한다.

FR-STORE-003 | Product 회원가입 기반 저장 | Production 단계에서 사용자는 회원가입 후

## 2. 소도시 데이터 지도 시각화 (메인 페이지)

### 결정 내용

메인 페이지에 소도시 마커가 표시된 지도를 제공한다.
필터: 한국/일본 국가, 월, 테마 세 가지.
마커 클릭 시 해당 소도시를 목적지로 설정해 여행 추천을 시작한다.

### 현재 문서 상태

- `FR-MAP-001`: 추천 결과와 혼잡도/목적지 위치를 지도 기반으로 표시 — 메인 페이지 맥락 없음
- `FR-MAP-004`: 상세 패널 제공 — 마커 클릭 흐름 미정의

### 수정·추가 내용

| 구분 | 대상 | 변경 방향 |
| --- | --- | --- |
| 추가 | FR-MAP-005 | 메인 페이지 소도시 마커 지도 |
| 추가 | FR-MAP-006 | 메인 지도 필터 (국가/월/테마) |
| 추가 | FR-MAP-007 | 마커 클릭 → 여행 시작 진입점 |

**추가 요구사항 초안:**
```
FR-MAP-005 | 메인 소도시 지도 | 메인 페이지는 추천 대상 소도시의 위치를 마커로 표시한
             지도를 제공해야 한다. 마커는 한국·일본 소도시를 시각적으로 구분 표시한다.

FR-MAP-006 | 지도 필터 | 메인 지도는 국가(한국/일본), 월(1~12월), 테마를 기준으로

             챗봇 추천 흐름이 시작된다.

```

## 3. 개인화 온보딩 질문 (필수)

### 결정 내용

서비스 진입 시 대도시-테마 매칭 기반 온보딩 질문을 필수로 제공한다.
사용자가 선호하는 대도시 스타일을 고르면 아래 **6개 기본 테마** 중 해당하는 테마를 추출해 소도시 매칭에 활용한다.

**6개 기본 테마 (온보딩 매핑 기준):**
온천·휴양 / 바다·해안 / 역사·전통 / 미식·노포 / 자연·트레킹 / 예술·감성

**한국 대도시 → 기본 테마 매핑:**
| 대도시 스타일 | 매핑 기본 테마 |
| --- | --- |
| 아산/온양 같은 온천/휴양 도시 | 온천·휴양 |
| 부산 같은 바다/해안 도시 | 바다·해안 |
| 경주 같은 역사/전통 도시 | 역사·전통 |
| 전주 같은 미식/노포 도시 | 미식·노포 |
| 제주 같은 자연/트레킹 여행지 | 자연·트레킹 |
| 강릉 같은 예술/감성 여행지 | 예술·감성 |

**일본 대도시 → 기본 테마 매핑:**
| 대도시 스타일 | 매핑 기본 테마 |
| --- | --- |
| 벳푸 같은 온천/휴양 도시 | 온천·휴양 |
| 오키나와 같은 바다/해안 여행지 | 바다·해안 |
| 교토 같은 역사/전통 도시 | 역사·전통 |
| 오사카 같은 미식/노포 도시 | 미식·노포 |
| 닛코 같은 자연/트레킹 여행지 | 자연·트레킹 |
| 가나자와 같은 예술/감성 도시 | 예술·감성 |

### 현재 문서 상태

- `FR-REC-004`: 개인화 언급, 온보딩 흐름과 매핑 체계 미정의
- 온보딩 전용 요구사항 없음

| 추가 | FR-ONBOARD-001 | 온보딩 질문 필수 제공 |
| 추가 | FR-ONBOARD-002 | 대도시-테마 매핑 체계 (6개 기본 테마 기준) |
| 추가 | FR-ONBOARD-003 | 온보딩 결과와 추천 연결 |
| 추가 | 7.1절 데이터 항목 | 대도시-테마 매핑 데이터 항목 추가 |

**추가 요구사항 초안:**
```

                  사용자가 대화 중 명시적으로 다른 조건을 요청하면 온보딩 기본값보다 우선 적용한다.

## 3.5 추천 시작 직전 — 축제·행사 포함 여부 확인

### 결정 내용

여행 일정 추천으로 넘어가기 직전, 두 진입점 모두에서 축제·행사 포함 여부를 묻는다.

| 진입점 | 질문 타이밍 |
| --- | --- |
| 지도 마커 클릭 | 일정 유형 선택 후, 추천 시작 직전 |
| 챗봇 대화 | 여행 조건(국가·시기·테마 등) 파악 후, 일정 생성 직전 |

질문 형식: "이 시기 현지 축제나 행사를 여행 일정에 포함할까요?" (예 / 아니오)
- **예**: 해당 지역·월의 축제·행사 정보를 일정 카드에 포함해 추천
- **아니오**: 축제·행사 없이 소도시 여정만 추천

### 현재 문서 상태

- `FR-FES-001·002`: 축제 추천 기능 정의 — 사용자가 직접 축제 추천을 요청하는 흐름
- 추천 시작 직전 포함 여부를 묻는 흐름 없음
- `FR-RAG-005`: 챗봇 직접 질의 진입점 — 축제 포함 여부 단계 미정의

### 수정·추가 내용

| 구분 | 대상 | 변경 방향 |
| --- | --- | --- |
| 추가 | FR-FEST-ENTRY-001 | 두 진입점 공통 축제·행사 포함 여부 확인 단계 |
| 수정 | FR-MAP-007 | 마커 클릭 흐름에 FR-FEST-ENTRY-001 단계 추가 (반영 완료) |
| 수정 | FR-RAG-005 비고 | 챗봇 흐름에서 일정 생성 직전 FR-FEST-ENTRY-001 단계 추가 명시 |

**추가 요구사항 초안:**
```
FR-FEST-ENTRY-001 | 축제·행사 포함 여부 확인 | 시스템은 지도 마커 클릭과 챗봇 대화
  두 진입점 모두에서 여행 일정 생성 직전 "현지 축제·행사를 일정에 포함할까요?"를
  사용자에게 확인해야 한다.
  - 예: 해당 지역과 여행 월에 맞는 축제·행사 정보를 일정 카드에 포함해 추천한다.
  - 아니오: 축제·행사 없이 소도시 여정만 구성한다.
  선택 결과는 해당 세션의 추천 조건으로만 사용하며 장기 저장하지 않는다.
```

**챗봇 진입점 흐름:**
```
대화 시작
  → 여행 조건 파악 (국가, 시기, 테마 또는 대도시 스타일)
  → "현지 축제·행사를 일정에 포함할까요?" (FR-FEST-ENTRY-001)
  → 일정 유형 확인 (FR-TRIP-001, 챗봇이 질문하는 것이 기본값)
  → 추천 생성
```

## 4. 이전 대화 기반 추천 (좋아요/싫어요 피드백)

### 결정 내용

추천받은 여행 일정(소도시 단위)에 좋아요/싫어요를 선택할 수 있다.
이 피드백은 다음 대화 또는 세션에서 추천 조건에 반영된다.

### 현재 문서 상태

- `FR-REC-005`: 저장/비교 기능 정의, 피드백 루프 명세 없음
- 멘토 피드백 플랜의 Action 1에서 FR-REC-006 추가 예정

### 수정·추가 내용

멘토 피드백 플랜 Action 1의 FR-REC-006과 통합하되,
좋아요/싫어요 UI와 적용 기준을 구체화한다.

| 구분 | 대상 | 변경 방향 |
| --- | --- | --- |
| 추가 | FR-PERS-001 | 추천 결과 피드백 UI (좋아요/싫어요) |
| 추가 | FR-PERS-002 | 피드백 데이터의 다음 추천 반영 기준 |

**추가 요구사항 초안:**
```
FR-PERS-001 | 추천 피드백 UI | 사용자는 추천받은 각 소도시에 대해 좋아요 또는 싫어요를
               선택할 수 있어야 한다. 선택 결과는 FR-STORE-001에 정의된 저장 범위에 따라 보관된다.

FR-PERS-002 | 피드백 반영 추천 | 시스템은 사용자가 이전 세션에서 싫어요를 선택한 소도시를
               이후 추천에서 우선도를 낮추고, 좋아요를 선택한 소도시의 테마 태그를
               다음 추천의 가중치 조건으로 반영해야 한다.
               (FR-REC-006 개인화 루프와 연동)

## 9. 날씨 — 월별 지역 기상 경향 기반 추천 + 대체 일정 토글

### 결정 내용

추천 로직에서 날씨를 반영할 때 실시간 예보 API가 아닌
**여행하는 달의 해당 지역 기상 경향(역사적·계절적 패턴)**을 사용한다.

- 기본: 추천 여행 일정 하나를 제시한다.
- 조건부 토글: 해당 달에 우천·폭설 경향이 있는 목적지의 경우,
  "기상 악화 시 대체 일정 보기" 토글을 제공해 실내 중심 대체 여정을 제시한다.

**기상 경향 기준 예시:**

| 국가 | 기상 악화 경향 구분 | 대상 지역·시기 |
| --- | --- | --- |
| 한국 | 장마 | 남해·제주·남부 내륙 — 6~7월 |
| 한국 | 태풍 | 남해안·제주 — 8~9월 |
| 한국 | 폭설 | 강원 산악 — 12~2월 |
| 일본 | 장마(梅雨) | 혼슈·규슈·시코쿠 — 6~7월 |
| 일본 | 태풍 | 규슈·오키나와 — 8~9월 |
| 일본 | 폭설(豪雪) | 호쿠리쿠·도호쿠 — 12~3월 |

## 유저 서비스 흐름 (전체)

> 이 플랜에서 결정된 사항이 반영된 전체 흐름이다.
> `[PoC]` = PoC 단계 구현, `[P2]` = Production 단계 구현

---

### 1. 서비스 진입

```
서비스 접속
  ├─ 신규 사용자 (온보딩 미완료)
  │    └─ 온보딩 화면 → [2. 온보딩]
  └─ 재방문 사용자 (온보딩 완료)
       └─ 메인 화면 (지도) → [3. 메인 화면]
            [PoC] 로컬 스토리지에서 선호 프로필 로드
            [P2]  로그인 계정에서 선호 프로필 로드
```

---

### 2. 온보딩 (신규 사용자 필수, 건너뛰기 불가)

```
"어떤 스타일의 여행지를 선호하나요?"

  ┌─ 한국 선택지 (각 1개 기본 테마 대응)
  │    아산/온양 같은 온천/휴양 도시   → 온천·휴양
  │    부산 같은 바다/해안 도시        → 바다·해안
  │    경주 같은 역사/전통 도시        → 역사·전통
  │    전주 같은 미식/노포 도시        → 미식·노포
  │    제주 같은 자연/트레킹 여행지    → 자연·트레킹
  │    강릉 같은 예술/감성 여행지      → 예술·감성
  │
  └─ 일본 선택지 (각 1개 기본 테마 대응)
       벳푸 같은 온천/휴양 도시        → 온천·휴양
       오키나와 같은 바다/해안 여행지  → 바다·해안
       교토 같은 역사/전통 도시        → 역사·전통

## 10. 추천 결과 — 소도시 1곳 집중 일정

### 결정 내용

추천 결과는 소도시 **1곳**에 대한 집중 여행 일정을 제공한다.
기존 "2~4개 소도시를 여정으로 묶어 추천"하는 방식에서 전환한다.
사용자가 진입점(지도 마커 또는 챗봇)에서 목적지를 설정하면,
해당 소도시 1곳의 선택 일정 유형(당일치기~4박5일) 기준 일별 세부 일정을 추천한다.

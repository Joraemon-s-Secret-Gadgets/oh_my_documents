# 4.6 서비스별 세부 기능 최신화 계획

## 목적

`docs/01_requirements/01_requirements.md`의 4.6 서비스별 세부 기능 아래에 있는 기존 기능 요구사항 표를 최신 우선순위 및 단계 정의와 일치하도록 정리한다.

동일한 내용은 `pages/01_requirements.html`에도 반영하여 Markdown 요구사항 정의서와 HTML 산출물이 서로 어긋나지 않도록 한다.

## 적용 원칙

- 요구사항 정의서에는 기능 요구사항 수준의 내용만 작성한다.
- 구현 방식, 코드 구조, API 상세 설계, DB 스키마 수준의 내용은 작성하지 않는다.
- 기능 ID가 있는 표는 우선순위 순서대로 정렬한다.
- 상세 기능 표의 비고는 원래 기능 설명 중심으로 유지하고, PoC/Production 단계 설명은 상단 요약 표에서 관리한다.
- P1은 PoC 구현 대상, P2는 PoC 설계 및 가능 시 구현 대상, P3는 PoC 설계 대상, P4는 Production 구현 대상, P5는 이번 프로젝트 제외 및 향후 검토 대상으로 유지한다.
- P5 기능은 상세 기능 요구사항 표에서 제외하고, 우선순위 및 단계별 추진 범위 표에만 기록한다.
- `FR-JP-*`처럼 우선순위가 명시되지 않은 공통 기반 기능은 별도 판단 없이 임의로 P1~P5에 편입하지 않는다.

## 작업 계획

1. 기준 정보 확인
   - `docs/01_requirements/14_priority_plan.md`의 최신 우선순위 정의를 확인한다.
   - `docs/01_requirements/01_requirements.md` 4.6의 요약 표와 상세 기능 요구사항 표가 같은 기준을 쓰는지 비교한다.
   - `pages/01_requirements.html`의 4.6 HTML 표가 Markdown과 같은 행, 같은 순서, 같은 문구를 가지는지 확인한다.

2. 4.6 상단 요약 표 기준 확정
   - P1: 혼잡 회피 대체 여행지 추천, 한일 축제 기반 여정 추천, 일주일 이내 여행의 날씨 기반 추천
   - P2: K-Drama 촬영지 탐색, 애니메이션 성지순례
   - P3: 철도 패스 기반 소도시 경로 추천, 공예·장인 체험형 여행 추천
   - P4: 스키 액티비티 여행 추천
   - P5: 별 관측 조건 기반 여행지 추천, 치유·웰니스 여정 추천, 예산 역산 추천

3. 상세 기능 요구사항 표 정렬
   - 기능 ID가 있는 표를 P1, P2, P3, P4 순서로 정렬한다.
   - 같은 우선순위 안에서는 서비스 흐름이 자연스럽게 읽히도록 기존 기능 묶음을 유지한다.
   - P5 기능 ID는 상세 기능 요구사항 표에서 제외한다.
   - `FR-JP-*` 공통 기능은 우선순위 미지정 항목으로 상세 표 하단에 유지하거나, 별도 공통 기반 기능 표로 분리할지 검토한다.

4. 상세 기능 요구사항 문구 최신화
   - 각 기능명과 설명이 최신 서비스 범위와 맞는지 확인한다.
   - PoC 단계에서 구현하지 않는 기능은 상세 설명이 실제 구현 대상처럼 보이지 않도록 표현을 정리한다.
   - P5 항목은 상단 요약 표에만 기록하고 상세 기능 요구사항 표에서는 제외한다.
   - 비고 칸은 기존 기능 설명 중심으로 유지하고, `PoC 구현`, `PoC 설계`, `Production 진행`, `이번 프로젝트 제외` 같은 단계 문구를 상세 표 비고에 반복하지 않는다.

5. Markdown과 HTML 동기화
   - Markdown 표 수정 후 같은 순서와 문구로 HTML 표를 갱신한다.
   - HTML의 `<td>` 텍스트가 Markdown과 의미상 동일한지 확인한다.
   - 긴 설명이 HTML 화면에서 잘리지 않도록 필요 시 문장 길이를 조정하되, 요구사항 의미는 바꾸지 않는다.

6. 검증
   - 기능 ID 누락 여부를 확인한다.
   - 우선순위 순서가 깨졌는지 확인한다.
   - 상세 표 비고에 단계 문구가 불필요하게 남아 있는지 확인한다.
   - Markdown과 HTML의 기능 행 수 및 주요 문구가 일치하는지 확인한다.

## 수정 계획

### 1. 4.6 설명 영역 수정

- 4.6 상단 설명은 PoC와 Production의 적용 범위를 먼저 설명하는 구조로 유지한다.
- 긴 단계 설명은 표 안에 반복하지 않고, 표 위의 문단에서 한 번만 정리한다.
- P1~P5의 의미가 4.6 상세 기능 요구사항 표를 읽기 전에 먼저 이해되도록 문장 순서를 정리한다.
- P5는 상세 기능 요구사항 표가 아니라 우선순위 및 단계별 추진 범위 표에서만 확인되도록 설명한다.

### 2. 우선순위 요약 표 수정

- 현재 요약 표의 우선순위 순서는 유지한다.
- `기능 범위`, `PoC`, `Production`, `상태` 컬럼은 유지한다.
- P5는 삭제하지 않고 `제외`, `검토`, `제외`의 의미가 향후 검토 대상임을 설명 문단에서 보완한다.
- P5 기능 범위 문구는 상세 기능 요구사항 표와 매칭하지 않고, 추진 범위 기록으로만 유지한다.

### 3. 상세 기능 요구사항 표 수정

- 상세 표의 행 순서를 P1, P2, P3, P4 순서로 정리한다.
- 같은 우선순위 안에서는 기존 기능 ID 묶음을 유지한다.
- 기능 ID는 변경하지 않는다.
- 기능명은 상단 요약 표의 기능 범위와 충돌하지 않도록 최신화한다.
- 설명은 요구사항 정의서 수준으로 유지하고 구현 방식은 넣지 않는다.
- 비고는 원래 기능 보충 설명 중심으로 원복 또는 유지한다.
- P5에 해당하는 `FR-ASTRO-*`, `FR-THEME-WELLNESS-*`, `FR-ENTRY-BUDGET-*`는 상세 표에서 제외한다.

### 4. 공통 기반 기능 처리

- `FR-JP-*` 항목은 현재 우선순위가 명확하지 않으므로 P1~P5에 임의 배치하지 않는다.
- 4.6 상세 표 하단에 유지하되, 필요하면 `공통 기반 기능`으로 구분하는 소제목 또는 별도 표 분리를 검토한다.
- 공통 기반 기능을 별도 표로 분리할 경우에도 기능 ID와 기존 설명은 유지한다.

### 5. HTML 산출물 수정

- Markdown 수정이 끝난 뒤 `pages/01_requirements.html`의 4.6 영역을 같은 내용으로 수정한다.
- HTML의 행 순서, 기능 ID, 기능명, 설명, 비고가 Markdown과 동일한 의미인지 확인한다.
- 긴 설명이 화면에서 잘리지 않도록 필요하면 문장만 다듬고 요구사항 의미는 바꾸지 않는다.

### 6. 수정 후 확인

- 4.6 상세 표 안에 단계 문구가 과도하게 반복되지 않는지 확인한다.
- P5 항목이 상세 기능 요구사항 표에 남아 있지 않은지 확인한다.
- 4.7의 Auth, Review, OPS 항목이 4.6 수정 과정에서 이동되거나 변경되지 않았는지 확인한다.
- 변경된 Markdown과 HTML을 비교하여 누락된 기능 ID가 없는지 확인한다.

## 완료 기준

- 4.6 상세 기능 요구사항 표가 최신 우선순위 기준과 충돌하지 않는다.
- 기능 ID가 있는 표가 P1~P4 우선순위 기준으로 정렬되어 있다.
- P5 기능은 우선순위 및 단계별 추진 범위 표에만 기록되어 있다.
- 상세 표의 비고가 기능 설명 중심으로 유지되어 있다.
- Markdown 요구사항 정의서와 HTML 산출물의 4.6 내용이 동기화되어 있다.
- `git diff --check`에서 공백 오류가 발생하지 않는다.

## 주의 사항

- Auth, Review, OPS 기능은 4.7 운영·관리·검증 기능 범위이므로 4.6 최신화 작업에 섞지 않는다.
- `FR-JP-*` 항목은 일본 여행 정보 기반의 공통 기능으로 보이며, 우선순위가 명확히 확정되기 전까지 임의 재분류하지 않는다.
- P5 항목을 우선순위 및 단계별 추진 범위 표에서 삭제하지 않는다. 다만 상세 기능 요구사항 표에서는 제외한다.

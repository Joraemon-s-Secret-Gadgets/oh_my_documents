# 프로젝트 기획서 기준 요구사항 정의서 반영 계획

## 목적

`docs/00_project_plan/00_project_plan.md`의 최신 기획 기준을 `docs/01_requirements/01_requirements.md`와 `pages/01_requirements.html`에 반영한다.

프로젝트 기획서에서 확정한 핵심 기준은 다음이다.

- 사용자는 국가와 여행 시기를 먼저 입력한다.
- 이후 자연어 취향을 반영한다.
- 날씨는 추천 조건이 아니라 목적지 판단을 돕는 정보로 제공한다.
- 소도시는 행정구역 인구수가 아니라 관광지 방문객 수와 혼잡도를 기준으로 판단한다.
- 멀티스텝 에이전트는 국가·여행 시기를 고정 조건으로 사용한다.
- P5는 상세 요구기능이 아니라 우선순위 및 단계별 추진 범위에만 기록한다.
- `FR-RAG-005` 직접 질의 진입점은 날씨 추천 조건 제거에 따라 상세 요구기능에서 제외한다.
- `FR-API-007`은 추천 조건이 아니라 일정에 포함된 지역별 날씨 정보를 알려주는 기능으로 정리한다.
- `FR-API-008` 비용 추정 데이터는 예산 역산 P5 제외 기준에 맞춰 제거한다.

## 수정 계획

### 1. 용어 정의 정리

현재 요구사항 정의서의 `소도시` 정의는 관광객 집중도가 낮고 로컬 경험이 풍부한 지방 도시 및 읍·면 단위 지역으로 되어 있다.

수정 방향:

```text
소도시는 행정구역 인구수만으로 판단하지 않고, 관광지 방문객 수가 상대적으로 낮아 혼잡 회피 대안이 될 수 있는 목적지와 주변 지역을 기준으로 본다.
```

목표:

- 소도시 판단 기준을 인구수 기준으로 오해하지 않게 한다.
- 방문객 수와 혼잡도 기준을 요구사항 정의서의 공통 기준으로 둔다.

### 2. 서비스 소개 수정

현재 요구사항 정의서에는 다음 흐름이 남아 있다.

```text
사용자가 자연어로 취향과 여행 시기를 입력하면...
```

수정 방향:

```text
사용자가 국가와 여행 시기를 먼저 입력하면, 멀티스텝 에이전트는 이를 고정 조건으로 사용하고 이후 입력된 자연어 취향을 테마 또는 추천 진입점으로 해석한다. 날씨는 추천 조건이 아니라 추천 결과에서 확인하는 정보로 제공한다.
```

목표:

- 입력 순서를 프로젝트 기획서와 동일하게 맞춘다.
- 국가·여행 시기를 사전 입력 조건으로 명확히 한다.
- 자연어 취향은 후속 조건으로 반영한다.

### 3. 이해관계자 설명 수정

현재 최종 사용자 설명:

```text
취향·시기·동행·이동 범위를 입력하고 소도시 테마 여정을 추천받는다.
```

수정 방향:

```text
국가와 여행 시기를 먼저 입력하고, 이후 취향·동행·이동 조건을 추가해 소도시 여정을 추천받는다. 날씨는 추천 결과에서 확인하는 정보로 제공받는다.
```

### 4. 서비스 흐름 요약 수정

현재 3.4 서비스 흐름은 `취향·시기·예산·날씨 입력`으로 시작하고, 예산도 현재 흐름처럼 포함되어 있다.

수정 방향:

| 단계 | 입력 | 처리 / 출력 |
| --- | --- | --- |
| 1 | 국가, 여행 시기 입력 | 추천 대상 국가와 시간 조건 고정 |
| 2 | 자연어 취향 입력 | 취향을 테마 또는 추천 진입점으로 해석 |
| 3 | RAG 검색·군집 | 방문객 수, 계절, 축제, 이동 기준으로 후보 선별 |
| 4 | 여정 구성 | 2~4곳 소도시와 방문 순서를 여정 카드로 표시 |
| 5 | 상세 정보 조회 | 지도·장소 정보·날씨 정보·외부 링크 확인 |
| 6 | 외부 연동 | Kakao / Yahoo Japan 딥링크로 추가 정보 확인 |

목표:

- 예산 역산은 P5이므로 현재 서비스 흐름의 첫 입력에서 제거한다.
- 방문객 수 기준을 후보 선별 단계에 반영한다.

### 5. 기능 요구사항 수정

수정 대상:

| ID | 현재 문제 | 수정 방향 |
| --- | --- | --- |
| FR-COM-001 | 자연어, 조건 입력이 뭉뚱그려져 있음 | 국가와 여행 시기 선입력 + 자연어 취향 입력 구조로 명확화 |
| FR-REC-001 | 사용자 입력 분류에 국가·여행 시기 고정 조건이 약함 | 국가·여행 시기를 먼저 고정하고 취향을 테마/진입점으로 분류 |
| FR-REC-002 | 대체 관광지 중심 표현 | 방문객 수 적은 여행지를 추천 |
| FR-REC-004 | 예산과 날씨가 현재 개인화 조건처럼 보임 | 예산은 P5, 날씨는 정보 제공이므로 현재 범위에서는 취향·이동·일정·동행 중심으로 정리 |
| FR-MAP-003 | 패스권·예상 비용 중심 | 철도 경로, 주요 역, 환승, 소요 시간, 주변 관광지, 패스 활용 가능 여부 중심으로 정리 |
| FR-RAG-002 | 추천 파이프라인에 국가·여행 시기 고정 조건이 없음 | 국가·여행 시기 고정 → 취향 해석 → 검색 → 후보 선별 → 결과 구성 흐름 반영 |
| FR-RAG-005 | 날씨 직접 질의 진입점이 현재 추천 조건처럼 남아 있음 | 상세 요구기능 표에서 제외하고, 날씨는 결과 상세 정보로만 제공 |
| FR-API-007 | 날씨 데이터가 후보 비교와 우천 대안 추천에 쓰이는 것처럼 표현됨 | 추천 일정에 포함된 지역별 날씨 정보를 알려주는 기능으로 정리 |
| FR-API-008 | 비용 추정 데이터가 현재 기능처럼 남아 있음 | 예산 역산 P5 제외 기준에 맞춰 제거 |

### 6. 데이터 요구사항 수정

수정 대상:

- `목적지 기본 정보`
- `데이터 품질 기준`
- 필요 시 혼잡도/방문객 수 관련 데이터 항목 추가

수정 방향:

- 소도시 판단 기준에 관광지 방문객 수 데이터를 포함한다.
- 후보 선별과 추천 근거에 방문객 수 기준을 사용할 수 있게 한다.
- 날씨 데이터는 후보 선별 기준이 아니라 추천 일정에 포함된 지역별 날씨 정보와 사용자 판단 보조 정보로 제공한다.
- 비용 추정 데이터는 예산 역산 기능과 연결되므로 현재 상세 요구사항에서 제거한다.
- 인구수 기준이라는 표현이 있다면 제거한다.

## 완료 기준

- 요구사항 정의서에서 국가와 여행 시기를 먼저 입력받는 흐름이 명확하다.
- 자연어 취향은 국가·여행 시기 이후 반영되는 조건으로 설명된다.
- 날씨는 추천 조건이 아니라 결과 상세 정보로 제공된다는 점이 명확하다.
- 소도시는 인구수 기준이 아니라 관광지 방문객 수 기준으로 설명된다.
- 혼잡 회피 추천은 `방문객 수 적은 여행지 추천` 방향으로 표현된다.
- 예산 역산은 현재 서비스 흐름이나 현재 기능 요구사항에 포함되지 않고 P5 검토 항목으로만 유지된다.
- `FR-RAG-005`는 상세 기능 요구사항 표에서 제외되어 있다.
- `FR-API-007`은 추천 일정에 포함된 지역별 날씨 정보를 알려주는 기능으로 정리되어 있다.
- `FR-API-008`은 상세 기능 요구사항 표에서 제거되어 있다.
- Markdown과 HTML의 내용이 같은 의미로 동기화되어 있다.

## 주의 사항

- 프로젝트 기획서 자체는 이번 계획에서 수정하지 않는다.
- 요구사항 정의서의 기능 ID는 변경하지 않는다.
- P5 상세 요구기능 ID를 다시 추가하지 않는다.
- `FR-RAG-005`는 삭제가 아니라 현재 범위 제외로 처리한다.
- `FR-API-008`은 예산 역산과 연결된 비용 추정 데이터이므로 현재 범위에서 제거한다.
- HTML은 Markdown 반영 산출물로 보고, Markdown 수정 후 동기화한다.

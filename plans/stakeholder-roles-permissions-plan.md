# Implementation Plan: 이해관계자 기준 역할 및 권한 정의

## Overview
로브(Lovv) 서비스의 이해관계자를 기준으로 사용자 역할(Role), 접근 권한(Permission), 데이터 관리 책임, 운영 책임을 명확히 정의한다. 이 계획은 요구사항 정의서의 `2. 이해관계자` 섹션을 역할·권한 모델로 확장하고, 이후 사용자 시나리오, API 명세, 관리자 기능, 보안 요구사항에 일관되게 반영하기 위한 작업 절차를 정리한다.

## Current Context
현재 요구사항 정의서에는 다음 이해관계자가 정의되어 있다.

| 이해관계자 | 현재 설명 |
|---|---|
| 자유여행자 (FIT) | 취향·시기·동행·이동 범위를 입력하고 소도시 테마 여정을 추천받는 최종 사용자 |
| 지자체·관광 운영자 | 소도시 관광 집중도를 모니터링하고 수요 분산 전략을 검토하는 운영 사용자 |
| 서비스 관리자 | 소도시·축제 데이터, 추천 정책, 공지사항을 관리하는 내부 사용자 |
| 공공 데이터 제공 기관 | 관광지·축제·날씨·교통 데이터를 API 형태로 제공하는 연계 기관 |

현재 문서는 이해관계자별 목적은 제시하지만, 인증 필요 여부, 읽기/쓰기 권한, 관리 기능, 데이터 접근 범위, 승인 책임은 아직 분리되어 있지 않다.

## Target Role Model

| Role ID | 역할 | 주요 사용자 | 인증 | 권한 수준 |
|---|---|---|---|---|
| `R-USER` | 일반 여행 사용자 | 자유여행자 (FIT) | 선택 또는 비로그인 | 추천 조회 중심 |
| `R-LOCAL-OPERATOR` | 관광 운영자 | 지자체·관광 운영자 | 필수 | 지역 데이터 조회·분석 중심 |
| `R-ADMIN` | 서비스 관리자 | 내부 운영/기획 담당자 | 필수 | 서비스 데이터·정책 관리 |
| `R-DATA-PROVIDER` | 데이터 제공 기관 | 공공 데이터 제공 기관/API 제공자 | API Key 또는 기관 인증 | 데이터 제공·연동 상태 확인 |
| `R-SYSTEM` | 시스템/외부 API | Google Maps, Kakao Maps, Yahoo Japan, WeatherAPI 등 | 시스템 키 | 자동 조회·동기화 |

## 권한 원칙

1. 최소 권한 원칙
   각 역할은 수행 목적에 필요한 최소 범위의 데이터와 기능만 사용할 수 있어야 한다.

2. 공개 데이터와 운영 데이터 분리
   여행자에게 노출되는 목적지·축제·날씨 정보와 내부 운영 지표, 관리자 정책 데이터는 분리한다.

3. 제안과 승인의 분리
   지자체·관광 운영자와 데이터 제공 기관은 데이터를 제안하거나 제공할 수 있지만, 공개 반영은 서비스 관리자의 승인 후 적용한다.

4. 개인정보 최소 수집
   일반 여행 사용자의 취향 입력과 대화 내용은 추천 목적에 한정해 처리하며, 저장이 필요한 경우 익명화·보관 기간·삭제 기준을 명확히 한다.

5. API Key 보호
   외부 API Key는 서비스 관리자 또는 서버 환경에서만 관리하고, 클라이언트 노출을 최소화한다.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| 이해관계자와 시스템 Role이 1:1로만 고정됨 | Medium | 하나의 기관/사용자가 여러 역할을 가질 수 있음을 명시한다. |
| MVP 범위 대비 권한 모델이 과도하게 복잡해짐 | Medium | MVP에서는 조회/제안/승인/관리 4단계 권한으로 단순화한다. |
| 데이터 제공 기관 권한이 관리자 권한과 혼동됨 | High | 제공 기관은 데이터 제공·출처 확인까지만 허용하고 공개 반영은 관리자 승인으로 제한한다. |
| 사용자 대화 로그 접근 범위가 불명확함 | High | 본인 세션 또는 익명 집계만 허용하고 원본 로그 접근은 관리자에게도 제한적으로 둔다. |
| 외부 API Key가 클라이언트에 노출됨 | High | 배포 환경에서는 서버 환경 변수 또는 프록시 관리 원칙을 요구사항에 명시한다. |

## Open Questions

- MVP에서 일반 여행 사용자를 완전 비로그인으로 둘지, 여행 기록 저장을 위해 선택 로그인을 둘지 결정이 필요하다.
- 관광 운영자의 권한 범위를 `담당 지자체 단위`로 제한할지, `권역 단위`까지 허용할지 결정이 필요하다.
- 데이터 제공 기관이 직접 데이터를 수정할 수 있는지, 항상 제안/제공만 가능한지 정책 결정이 필요하다.
- 사용자 대화 로그를 저장하지 않는 현재 방향을 유지할지, 품질 개선을 위한 익명 로그 저장을 허용할지 결정이 필요하다.

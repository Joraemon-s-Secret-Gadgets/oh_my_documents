# 추천 실행 및 재추천 버튼 요구사항 반영 계획

## 1. 목적

요구사항 정의서에는 여행 일정 추천 기능, 대체 추천, 후속 질문, 실패 시 재시도 흐름이 일부 정의되어 있다.
다만 사용자가 명시적으로 누르는 `여행 일정 추천 받기` 버튼과 추천 결과 화면의 `추천 다시 받기` 버튼은 UI/UX 요구사항으로 분리되어 있지 않다.

본 계획은 해당 버튼 요구사항을 메인 요구사항 정의서(`docs/01_requirements/01_requirements.md`)와 HTML 생성물(`pages/01_requirements.html`)에 반영하기 위한 작업 범위를 정의한다.

## 2. 현재 반영 상태

| 항목 | 현재 상태 | 판단 |
| --- | --- | --- |

| 추천 결과 표시 | 카드, 표, 지도, 패널 형태의 결과 표시가 정의되어 있음 | 기능은 있음 |
| 대체 추천 | 혼잡, 조건 불일치, 데이터 부족 시 대체 관광지·축제·동선 추천이 정의되어 있음 | 일부 있음 |
| 후속 질문 | 추천 이후 사용자가 추가 질문을 할 수 있음 | 일부 있음 |

## 4. 반영 방향

### 4.1 공통 UI/UX 기능 보강

`4.1 공통 UI/UX 기능`에 추천 실행 CTA 요구사항을 추가한다.

권장 요구사항:

| ID | 기능명 | 요구사항 | 비고 |
| --- | --- | --- | --- |
| `FR-COM-005` | 추천 실행 CTA | 필수 조건이 충족되면 사용자는 `여행 일정 추천 받기` 버튼을 눌러 추천 파이프라인을 실행할 수 있어야 한다. 필수 조건이 부족하면 버튼은 비활성화되거나 추가 질문을 유도해야 한다. | 추천 시작 버튼 |

반영 기준:

- 버튼은 추천 파이프라인을 시작하는 사용자 명령으로 정의한다.
- 필수 조건 미충족 시 무조건 실패시키지 않고 추가 질문 흐름으로 연결한다.
- 자연어 대화 진입과 지도 마커 진입 모두에서 사용할 수 있도록 특정 화면에만 묶지 않는다.

### 4.2 추천·탐색 기능 보강

`4.2 추천·탐색 기능`에 추천 다시 받기 요구사항을 추가한다.

권장 요구사항:

| ID | 기능명 | 요구사항 | 비고 |
| --- | --- | --- | --- |
| `FR-REC-006` | 추천 다시 받기 | 사용자는 추천 결과 화면에서 `추천 다시 받기` 버튼을 눌러 현재 조건을 유지한 채 다른 소도시 또는 변형 일정을 다시 추천받을 수 있어야 한다. 시스템은 직전 추천과 동일한 결과를 반복하지 않거나, 동일 결과가 필요한 경우 그 사유를 설명해야 한다. | 결과 재생성 |

반영 기준:

- 기존 `대체 추천`과 구분한다. 대체 추천은 시스템 판단에 따른 보완이고, 추천 다시 받기는 사용자가 명시적으로 요청하는 재추천이다.
- 현재 조건 유지, 일부 조건 변경, 추천 후보 제외 또는 점수 재계산 중 어떤 방식으로 재추천하는지 문장으로 남긴다.
- 추천 이력과 피드백 데이터가 향후 개인화에 활용될 수 있음을 데이터 요구사항과 연결한다.

### 4.3 추천 파이프라인 보강

`5.1 추천 파이프라인` 또는 유사 섹션에 다음 흐름을 보강한다.

- 필수 조건 충족 후 `여행 일정 추천 받기` 버튼을 통해 추천 파이프라인이 실행된다.
- `추천 다시 받기` 요청 시 기존 입력 조건과 직전 추천 결과를 함께 참조한다.
- 재추천 시 직전 추천 후보를 제외하거나 점수를 낮춰 결과 다양성을 확보한다.
- 조건이 부족하면 추천을 즉시 실행하지 않고 추가 질문으로 되돌린다.

### 4.4 데이터 요구사항 보강

`7.1 필요 데이터 항목` 또는 개인화 관련 데이터 항목에 다음 내용을 추가한다.

| 데이터 구분 | 필요 내용 |
| --- | --- |
| 추천 실행 이벤트 | 추천 요청 시각, 입력 조건, 진입 방식, 필수 조건 충족 여부 |
| 재추천 이벤트 | 직전 추천 결과, 재추천 요청 시각, 유지 조건, 변경 조건, 제외 또는 감점 후보 |

반영 기준:

- PoC에서는 이벤트 로그 수준으로 정의한다.
- Production에서는 추천 품질 개선, 개인화, A/B 테스트의 기반 데이터로 확장 가능하다고 정리한다.

# 프로젝트 기획서 포지셔닝 보강 반영 플랜

## 1. 목적

`docs/00_project_plan/update_project_plan.md`에 정리된 Lovv의 차별화 방향과 보완 포인트를 프로젝트 기획서 원본(`docs/00_project_plan/00_project_plan.md`)과 HTML 생성물(`pages/00_project_plan.html`)에 반영한다.

현재 프로젝트 기획서는 Lovv의 기본 문제 정의, 기능 범위, 데이터·기술 방향을 설명하고 있으나, 트리플 등 기존 여행 서비스와의 차별점, 대체 여행지 탐색 엔진으로서의 포지셔닝, 설명 가능한 RAG 추천의 브랜딩 메시지는 별도 보강 문서에 분리되어 있다.

본 계획의 목표는 `update_project_plan.md`의 핵심 내용을 메인 프로젝트 기획서에 자연스럽게 통합해, Lovv가 어떤 서비스와 어떻게 다른지 문서만 읽어도 명확히 이해되도록 만드는 것이다.

## 3. 반영 기준

- `update_project_plan.md`의 출처 표기(`[triple]`, `[fnnews]`, `[ppl-ai-file-upload.s3.amazonaws]` 등)는 최종 기획서 본문에 그대로 옮기지 않는다.
- 외부 서비스 비교는 과도한 경쟁사 분석 문서가 아니라 Lovv의 포지셔닝을 설명하는 수준으로 요약한다.
- 프로젝트 기획서에는 GitHub 이슈 번호, Epic 번호, 세부 구현 태스크를 직접 넣지 않는다.
- 기존 `00_project_plan.md`의 문서 구조와 톤을 유지한다.
- HTML은 생성물이므로 Markdown 원본과 내용이 어긋나지 않게 동기화한다.

## 4. 유지할 내용

- 프로젝트명, 한 줄 정의, 문제 정의, 핵심 목표의 기존 방향은 유지한다.
- PoC / Production 범위, P1~P5 우선순위 표는 이번 작업에서 직접 변경하지 않는다.
- 요구사항 정의서(`docs/01_requirements/01_requirements.md`)의 기능 요구사항 ID 체계는 건드리지 않는다.
- 예산, 인력, WBS 일정은 별도 문서 범위로 보고 이번 작업에서는 수정하지 않는다.

## 5. 변경할 내용

### 5.1 카피 문구 반영

`1.2 한 줄 정의` 또는 `3.1 핵심 목표` 앞뒤에 Lovv의 대표 카피를 추가한다.

반영 후보:

| 구분 | 문구 |
| --- | --- |
| 메인 카피 | 서울·오사카 말고, 지금은 이곳 |
| 서브 카피 | 지도에서 흐릿한 곳, 여행에선 가장 빛나는 곳 |

반영 방식:

- 본문 초반에 브랜드 메시지로 짧게 넣는다.
- HTML에서는 기존 문서 스타일 안에서 강조 문장 또는 인용형 블록으로 표현한다.

### 5.2 목적지 포커스 차별점 반영

`1.3 기획 배경` 또는 `4.1 핵심 서비스`에 다음 의미를 반영한다.

- Lovv는 이미 정한 대도시 안의 일정 최적화 도구가 아니라, 국가와 여행 시기부터 시작해 덜 붐비는 소도시 목적지를 발견하게 하는 서비스다.
- 트리플류 서비스가 “선택한 도시 안의 일정 최적화”에 강점이 있다면, Lovv는 “어디를 갈지부터 고르는 목적지 탐색 엔진”으로 차별화한다.
- PoC에서는 한국·일본 소도시를 넓게 늘리기보다, 제한된 소도시를 깊게 다루는 방향을 명시한다.

### 5.3 설명 가능한 추천 포지셔닝 반영

`3.1 핵심 목표`, `7.2 추천 기능`, `9.1 추천 구조`, `9.2 시스템 원칙`에 다음 내용을 보강한다.

- 추천 이유, 날씨, 지도, 외부 링크, 출처·검증 상태를 함께 제공한다.
- Lovv는 단순 생성형 답변이 아니라 RAG 기반 검색과 멀티스텝 에이전트로 근거를 제공하는 설명 가능한 추천 엔진이다.
- PoC에서도 출처 링크와 검증 상태를 사용자에게 드러내 “데이터 기반 큐레이션” 인상을 강화한다.

### 5.4 입력·흐름 차별점 반영

`4.2 사용자 흐름`에 다음 흐름을 보강한다.

- 첫 질문은 “어디로 갈까요?”보다 “언제, 어느 나라로 떠나세요?”에 가깝게 설계한다.
- 국가와 여행 시기를 먼저 고정하고, 자연어 취향 또는 지도 마커로 추천 기준을 좁힌다.
- 여러 도시를 얕게 훑기보다 소도시 1곳을 깊게 탐색하는 집중형 UX를 강조한다.

### 5.5 데이터·운영 구조 차별점 반영

`5. 주요 사용자 및 이해관계자`, `7.3 운영·관리 기능`, `8.1 필요 데이터`, `8.2 외부 연동`에 다음을 보강한다.

- Lovv는 지자체·관광 운영자·공공 데이터 제공 기관과 함께 만드는 소도시 추천 플랫폼으로 확장 가능하다.
- 데이터 출처, 갱신 시점, 검증 상태를 UI에 노출하는 방향을 명시한다.
- 향후 지자체/공공 파트너용 Lovv 대시보드 또는 리포트 가설을 “운영 확장 후보” 수준으로 추가한다.

### 5.6 기술·AI 차별점 반영

`9. 기술 방향`에 다음 내용을 보강한다.

- Lovv의 기술 태그라인을 “Explainable RAG 기반 소도시 추천 엔진”으로 정리한다.
- 후속 Q&A는 기존 추천 결과를 기반으로 조건을 바꿔 재추천하는 에이전트 흐름으로 설명한다.
- “이 도시 말고 더 조용한 곳?”, “축제 없는 곳으로 바꿔줘” 같은 후속 조건 변경 흐름을 예시로 넣을 수 있다.

### 5.7 기대 효과 보강

`11. 기대 효과`에 다음 관점을 추가한다.

- 여행자는 유명 관광지 중심 탐색에서 벗어나, 국가·시기 조건에 맞는 소도시 대체지를 발견할 수 있다.
- 지역 관광 운영자는 소도시별 노출, 클릭, 관심도 같은 운영 지표를 통해 향후 협업·지원사업으로 연결할 수 있다.
- 서비스 운영자는 RAG 근거와 검증 상태, 사용자 피드백을 이용해 추천 품질을 반복 개선할 수 있다.

## 8. 주의사항

- `update_project_plan.md`는 보강 메모이므로, 원문을 그대로 복사하지 않고 프로젝트 기획서 문체에 맞게 요약·통합한다.
- 경쟁 서비스 비교는 Lovv의 차별점을 설명하는 데 필요한 수준으로만 사용한다.
- 외부 출처가 명확히 정리되지 않은 bracket 표기는 최종 산출물에 남기지 않는다.
- HTML은 `00_project_plan.md`와 같은 내용을 보여주는 생성물로 관리한다.
- 이번 계획은 실행 전 계획이며, 이 파일을 작성하는 것만으로 실제 기획서가 수정된 것은 아니다.

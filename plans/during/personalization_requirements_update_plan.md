# 개인화 요구사항 정의서 반영 계획

## 1. 목적

본 계획서는 `docs/01_requirements/16_personalization.md`에 정리된 신규 개인화 항목을 메인 요구사항 정의서(`docs/01_requirements/01_requirements.md`)에 어떻게 반영할지 정의한다.

반영 전 메인 요구사항 정의서에는 온보딩 선호, 저장 일정, 좋아요/싫어요 피드백 기반 개인화가 일부 반영되어 있었다.
반면 북마크, 실제 방문 기록, 별점, cold start 보완, 소도시 랭킹, PoC 머신러닝, 현재 위치 반경 추천, 여행 중 즉흥 일정 조정은 명확히 반영되어 있지 않았다.

본 계획서는 메인 요구사항 정의서 반영 방향과 반영 시 정합성 점검 기준을 기록한다.

## 3. 메인 요구사항 문서 반영 위치

### 3.1 문서 개요 / 범위

`1.2 범위`에 다음 항목을 추가한다.

- 소도시·대도시 북마크, 실제 방문 기록, 별점 기반 개인화 데이터 수집
- 사용자 행동 데이터와 외부 검증 루트를 활용한 cold start 보완
- 사용자 계정 기반 개인화 데이터 누적 및 개인화 랭킹 1차 운영
- 현재 위치와 사용자 지정 반경 기반 추천
- 여행 중 예산·일정 변화에 따른 즉흥 일정 조정

반영 방식:

- 기존 `온보딩 선호 프로필, 추천 피드백, 저장 일정 기반 개인화` 항목을 확장한다.
- PoC와 Production 범위가 섞이지 않도록 문장 안에서 단계 구분을 명시한다.

### 3.2 서비스 개요 / 핵심 가치 제안

`3.2 핵심 가치 제안`의 `개인화 루프` 항목을 보강한다.

반영 내용:

- 온보딩, 저장 일정, 좋아요/싫어요에 더해 북마크, 실제 방문 기록, 별점을 개인화 신호로 포함한다.
- 신규 사용자의 초기 추천 품질을 위해 cold start 보완을 명시한다.
- 개인화가 단순 선호 저장이 아니라 추천 후보 랭킹에도 영향을 준다는 점을 추가한다.

### 3.4 추천 파이프라인

기존 후보 필터링·순위 단계에 다음을 보강한다.

- 개인화 피드백뿐 아니라 북마크, 방문 기록, 별점, 사용자별 소도시 적합도 점수를 랭킹 신호로 사용한다.
- PoC에서는 규칙 기반 가중치와 경량 ML 실험을 병행할 수 있다.
- Production 1차에서는 사용자 계정 기반 행동 데이터를 누적하고, PoC에서 검증한 규칙 기반 점수 또는 사용자별 소도시 적합도 점수를 실제 추천 순위에 반영한다.
- Production 고도화 단계에서는 정기적인 성능 평가, 모델 재학습, A/B 테스트, 협업 필터링 등 운영형 개인화 랭킹 모델 개선 기준을 포함한다.

## 4. 우선순위 제안

### 4.1 우선순위 기준

우선순위는 다음 기준으로 산정한다.

| 기준 | 설명 |
| --- | --- |
| 추천 품질 영향도 | 개인화 추천 품질을 직접 개선하는가 |
| PoC 검증 가능성 | 짧은 기간 안에 입력, 저장, 랭킹 흐름을 검증할 수 있는가 |
| 구현 복잡도 | 데이터 구조와 UI 구현 부담이 과도하지 않은가 |
| 개인정보 리스크 | 위치, 방문 기록 등 민감도가 높은 데이터를 다루는가 |
| Production 의존성 | 로그인, 서버 저장, 실시간 위치 등 운영 환경이 필요한가 |

### 4.2 우선순위 역할 분리

메인 요구사항 정의서에는 이미 “P1 단계에서 personalization, ranking, explainability가 충분히 검증되지 않으면 P2 구현을 시작하지 않는다”는 문장이 있다.
신규 개인화 항목을 반영할 때 이 문장과 충돌하지 않도록 개인화와 랭킹의 단계별 역할을 다음처럼 분리한다.

| 우선순위 | 역할 | 반영 방식 |
| --- | --- | --- |
| P1 | 개인화 입력 데이터 수집 및 단순 가중치 검증 | 소도시 북마크, 좋아요/싫어요, 방문 기록, 방문 소도시 별점, cold start 기준 데이터를 수집하고 단순 가중치로 추천 결과가 달라지는지 검증한다. |
| P2 | PoC 랭킹 실험 | 사용자별 소도시 적합도 점수, 유사도 기반 랭킹, 경량 머신러닝 실험을 수행한다. 운영 모델이 아니라 추천 가능성과 feature 중요도 검증이 목적이다. |
| P3 | Production 개인화 1차 운영 | 사용자 계정 기반 데이터 누적과 규칙/점수 기반 개인화 랭킹을 실제 추천 순위에 반영한다. |
| P4 | Production 랭킹 모델 고도화 | 모델 재학습, A/B 테스트, 협업 필터링, 모델 모니터링 등 운영형 ML 고도화를 수행한다. |

메인 문서의 우선순위 표를 수정할 때는 P3 기능 범위에 `사용자 계정 기반 개인화 데이터 누적`과 `개인화 랭킹 1차 운영`을 추가한다.
P4 기능 범위에는 `개인화 랭킹 모델 고도화`, `현재 위치 반경 추천`, `여행 중 즉흥 일정 조정`을 추가한다.

### 4.3 랭킹 운영 수정안

랭킹 운영은 한 번에 “머신러닝 모델 운영”으로 반영하지 않고, 다음 단계로 쪼개어 반영한다.

| 단계 | 랭킹 방식 | 설명 |
| --- | --- | --- |
| PoC P1 | 단순 가중치 랭킹 | 북마크, 좋아요/싫어요, 방문 기록, 방문 소도시 별점을 점수화해 추천 순위에 영향을 주는지 확인한다. |
| PoC P2 | 경량 ML/유사도 랭킹 실험 | 데이터가 부족할 수 있으므로 규칙 기반 점수와 비교하는 실험으로 제한한다. |
| Production P3 | 개인화 랭킹 1차 운영 | PoC에서 검증한 규칙 기반 점수 또는 사용자별 소도시 적합도 점수를 실제 추천 순위에 반영한다. |
| Production P4 | 모델 고도화 | 누적 행동 데이터를 기반으로 재학습, A/B 테스트, 협업 필터링, 세션 기반 추천, 모델 모니터링을 검토한다. |

Production P3의 “개인화 랭킹 1차 운영”은 복잡한 ML 운영이 아니라 서비스 추천 결과에 개인화 점수식을 안정적으로 반영하는 단계다.
따라서 P3로 올려도 무리가 적으며, 무거운 ML 운영 요소만 P4로 분리한다.

## 5. 단계별 반영 순서

### 5.1 1차 반영: PoC 필수 개인화

메인 요구사항 정의서에 먼저 반영할 항목:

1. 소도시 북마크
2. 추천 일정 좋아요/싫어요 보강
3. 실제 방문 기록
4. 소도시 별점
5. Cold Start 기준 데이터

목표:

- 개인화 입력 데이터를 확보한다.
- PoC에서 추천 결과가 사용자 행동에 따라 달라질 수 있음을 검증한다.

### 5.2 2차 반영: PoC 랭킹 실험

1차 반영 이후 추가할 항목:

1. 사용자별 소도시 적합도 점수
2. 내가 갔던 곳 / 남이 갔던 곳 랭킹
3. 경량 머신러닝 또는 유사도 기반 랭킹 실험

목표:

- 수집된 사용자 행동 데이터를 추천 후보 순위에 반영한다.
- 규칙 기반 점수와 ML 기반 점수를 비교할 수 있는 구조를 만든다.

### 5.3 3차 반영: Production 개인화 1차 운영

Production 범위로 반영할 항목:

1. 사용자 계정 기반 개인화 데이터 누적
2. 개인화 랭킹 1차 운영
3. 규칙 기반 점수 또는 사용자별 소도시 적합도 점수의 실제 추천 순위 반영

목표:

- PoC에서 검증한 개인화 입력과 랭킹 실험을 실제 서비스 추천 순위에 반영한다.
- 사용자 개인 행동 데이터와 전체 사용자 행동 데이터를 운영 계정 기준으로 누적할 수 있는 구조를 만든다.

### 5.4 4차 반영: Production 개인화 랭킹 고도화

Production 고도화 범위로 반영할 항목:

1. 추천 모델 성능 평가 기준
2. 모델 재학습 기준
3. A/B 테스트
4. 협업 필터링 또는 세션 기반 추천 확장
5. 모델 모니터링

목표:

- 1차 운영에서 쌓인 사용자 행동 데이터를 기반으로 개인화 랭킹 품질을 개선한다.
- 단순 점수 기반 추천을 운영형 랭킹 모델로 확장한다.

### 5.5 5차 반영: Production 상황 기반 추천 확장

Production 범위로 반영할 항목:

1. 현재 위치 또는 기준 위치 입력
2. 추천 반경(km) 설정
3. 현재 여행 중 예산·일정 변화 기반 재추천

목표:

- 사용자의 현재 상황을 반영하는 실시간 개인화로 확장한다.
- 위치·여행 상태 데이터의 동의, 저장, 삭제 정책을 함께 정의한다.

## 6. 반영 시 주의사항

- 대도시 북마크는 “대도시 스타일 추론”이 아니라 대도시 자체를 저장하는 기능으로 정의한다.
- 위치 기반 추천은 “한국 사용자는 한국 추천, 일본 사용자는 일본 추천”이 핵심이 아니라, 현재 위치 기준 추천 반경을 사용자가 입력하는 기능으로 정의한다.
- PoC 머신러닝은 완성 모델 운영이 아니라 경량 모델 실험 및 feature 중요도 확인 수준으로 제한한다.
- Production 개인화는 `사용자 계정 기반 데이터 누적`과 `규칙/점수 기반 개인화 랭킹 1차 운영`을 P3으로 먼저 반영한다.
- Production 머신러닝 고도화는 모델 재학습, A/B 테스트, 협업 필터링, 모델 모니터링을 포함하는 P4 범위로 정의한다.
- Production 개인화 1차 운영은 위치 반경 추천이나 즉흥 일정 조정보다 먼저 반영할 수 있는 핵심 개인화 확장 항목으로 관리한다.
- 예산 역산 추천은 현재 메인 문서에서 P5 제외로 정의되어 있으므로, Production의 즉흥 일정 조정은 “전체 여행 예산 역산”과 구분해 남은 예산 기반 재조정으로 표현한다.
- 방문 기록과 위치 정보는 민감도가 높으므로 저장 여부, 삭제 가능 여부, 동의 범위를 함께 정의한다.

### 8.2 P1 문장과의 충돌 방지

메인 문서에는 다음 취지의 문장이 있다.

> P1 단계에서 recommendation flow, personalization, ranking, explainability가 충분히 검증되지 않으면 P2 구현을 시작하지 않는다.

신규 반영 시 이 문장과 충돌하지 않도록 `ranking`의 의미를 단계별로 나눈다.

| 단계 | ranking 의미 |
| --- | --- |
| P1 | 좋아요/싫어요, 북마크, 방문 기록, 방문 소도시 별점이 추천 순위에 영향을 줄 수 있는지 단순 가중치로 검증 |
| P2 | 사용자별 소도시 적합도 점수와 경량 ML/유사도 기반 랭킹 실험 |
| P3 | Production에서 규칙/점수 기반 개인화 랭킹을 실제 추천 순위에 반영 |
| P4 | 모델 재학습, A/B 테스트, 협업 필터링, 모니터링 등 운영형 랭킹 고도화 |

### 8.3 예산 역산 추천과 즉흥 일정 조정 구분

메인 문서에서 `예산 역산 추천`은 P5 제외 항목이다.
따라서 Production의 즉흥 일정 조정은 다음처럼 구분해 작성해야 한다.

| 구분 | 포함 여부 | 설명 |
| --- | --- | --- |
| 예산 역산 추천 | 제외 | 전체 여행 예산을 먼저 입력하고 여행지·일정을 역산 추천하는 기능 |
| 즉흥 일정 조정 | Production 확장 | 현재 여행 중 남은 예산, 남은 일정, 현재 위치를 기준으로 기존 일정을 재조정하는 기능 |

요구사항 문구에는 “남은 예산·남은 일정 기반 재조정”이라는 표현을 사용하고, “예산 역산 추천”으로 오해될 수 있는 표현은 피한다.

### 8.4 위치 기반 추천 의미 분리

기존 메인 문서의 위치 관련 요구사항은 주로 지도 표시, 목적지 위치, 장소 상세 패널에 관한 내용이다.
신규 위치 기반 추천은 “현재 위치 또는 기준 위치와 추천 반경(km)을 사용자에게 입력받는 기능”이므로 별도 요구사항으로 분리한다.

반영 위치:

- `4.3 지도·위치·경로 기능`에 `FR-MAP-008`로 추가

권장 표현:

> 사용자는 현재 위치 또는 직접 입력한 기준 위치와 추천 반경(km)을 입력해 해당 범위 안의 소도시를 추천받을 수 있어야 한다.

주의:

- “한국 사용자는 한국 추천, 일본 사용자는 일본 추천”이 핵심이 아니다.
- 핵심은 사용자가 지정한 반경 안에서 이동 가능한 소도시 후보를 좁히는 것이다.

### 8.5 대도시 관련 용어 충돌 방지

메인 문서에는 이미 `대도시-테마 매핑` 온보딩 요구사항이 있다.
신규 `대도시 북마크`는 온보딩 스타일 선택이 아니라 대도시 자체를 저장하는 기능이다.

| 항목 | 의미 |
| --- | --- |
| 대도시-테마 매핑 | 온보딩 질문에서 대도시 스타일 보기를 기본 테마와 매핑하는 기능 |
| 대도시 북마크 | 사용자가 관심 있거나 다시 확인하고 싶은 대도시 자체를 저장하는 기능 |

반영 시 두 기능을 섞지 않는다.
대도시 북마크는 대도시 기준 주변 소도시 탐색, 비교, 연계 일정 구성에 활용한다.

### 8.6 별점 대상 통일

별점은 `실제로 방문한 소도시`에 한해 남기는 것으로 통일한다.

반영 시 다음 표현을 사용한다.

- 방문 소도시 별점
- 실제로 방문한 소도시에 한해 별점을 남길 수 있어야 한다.
- 별점 데이터는 방문 기록 ID와 연결한다.

관심 소도시나 북마크한 소도시에 대한 평가와 혼동하지 않는다.

### 8.7 기존 오탈자 수정

메인 요구사항 정의서의 `FR-STORE-003` 명칭은 다음처럼 수정한다.

| 현재 | 수정 |
| --- | --- |
| Product 회원가입 기반 저장 | Production 회원가입 기반 저장 |

내용 자체는 Production 단계 설명이므로 명칭만 고치면 된다.

# 요구사항 정의서 WBS 기준 일정 수정 플랜

## 1. 목적

`docs/01_requirements/01_requirements.md`의 `9. 예산안·인력 분배·개발 기한` 중 개발 일정과 마일스톤을 GitHub Projects `Lovv WBS` 기준으로 수정한다.

현재 요구사항 정의서는 `PoC / Production` 2단계 중심으로 작성되어 있으나, GitHub WBS는 `M1 PoC 완료 / M2 중간 발표 / M3 Production`과 `Sprint 1~9` 기준으로 관리되고 있다. 따라서 요구사항 정의서의 일정 섹션을 실제 프로젝트 관리 기준과 일치시키는 것이 이 플랜의 목표다.

## 4. 유지할 내용

- `9.1 예산 산정 기준`, `9.2 예산안`, `9.3 인력 역할 정의`의 예산·역할 내용은 이번 수정 범위에서 유지한다.
- AWS 인프라 `300,000원`, Runpod AI 실행 `$300`, OpenAPI LLM/API `$300` 분리 관리 원칙은 유지한다.
- 기능 요구사항 `FR-*`, 비기능 요구사항 `NFR-*` ID 체계는 건드리지 않는다.
- `pages/01_requirements.html`은 생성물이므로 직접 수정하지 않고, 필요 시 Markdown 원본 수정 후 생성 절차로 갱신한다.

### 5.4 `9.6 예산·인력·일정 관리 원칙`

다음 방향으로 문구를 수정한다.

- M1 PoC는 기획·데이터·DB·AI 모델·시스템 아키텍처와 화면/UX 기초 설계 산출물 완료를 목표로 한다.
- M2 중간 발표는 PoC의 마지막 항목이 아니라 별도 마일스톤으로 관리한다.
- 2026-06-16 중간 발표에서 Sprint 5~9 Production 범위와 리스크를 확정한다.
- M3 Production은 2026-07-16 최종 발표·피드백 반영·산출물 최종 제출까지 완료하는 일정으로 관리한다.
- 외부 API Key 승인, 데이터 확보, RAG 품질 검증, 배포 리스크가 지연되면 해당 Sprint 안에서 우선순위 기능을 조정한다.

## 8. 주의사항

- 이 플랜은 아직 실행 전 계획이므로, 이 파일만 수정해도 `docs/01_requirements/01_requirements.md`가 변경된 것으로 보지 않는다.
- 요구사항 정의서의 일정 섹션을 수정할 때는 Markdown 원본을 먼저 수정하고, HTML은 생성물로만 갱신한다.
- GitHub Project 실제 WBS 저장소는 `Joraemon-s-Secret-Gadgets/Lovv`이고, 현재 문서 저장소는 `Joraemon-s-Secret-Gadgets/oh_my_documents`이므로 출처를 혼동하지 않는다.
- `docs/lovv_github_projects_wbs.md`는 보조 참고로만 사용하고, 최종 일정 기준은 GitHub Project #3 조회 결과로 둔다.
- GitHub Project의 Epic과 하위 이슈 구성은 현재 상태를 유지하며, 요구사항 정의서에는 Epic 번호, 이슈 번호, 개별 화면 구현 목록을 직접 쓰지 않는다.
- M2 중간 발표는 PoC 하위 단계가 아니라 별도 마일스톤으로 표기한다.
- 2026-07-17 이후의 검증·정리 기간은 M3 Production 일정에 포함하지 않는다. 필요하면 사후 운영 또는 회고 범위로 분리한다.

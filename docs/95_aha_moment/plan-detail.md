## 0. 이 화면과 단계의 관계

PLAN DETAIL은 **일정 생성 결과 화면**이다. 여정 단계로는 **단계 3 — 실행가능 + peak-end**. 핵심은 *진입 경로가 여러 개여도 결국 같은 일정 생성 파이프라인·같은 화면으로 수렴*한다는 점이다.

**진입 경로 (모두 동일한 PLAN DETAIL로 수렴)**

- 테마 카드 → THEME DETAIL → "이 테마로 일정 계획하기" (단계 1 경유)
- 지도에서 **마커 클릭**
- **AI 챗봇으로 일정짜기**

세 진입점은 모두 **일정 생성 채팅으로 수렴**한다. 채팅에 들어오면 **챗봇이 국가·여행 시기(월)를 가장 먼저 묻고**(국가·시기 선입력), 그 답 + 취향(또는 마커 지역, 자연어)으로 소도시 day-by-day 일정을 생성해 이 화면을 띄운다. "이거 그대로 떠나면 되겠다"를 만든 뒤 저장·피드백으로 개인화 씨앗을 심는다. 여행 월이 확정된 이후 화면이므로 타이밍 표현은 **"선택하신 N월에 …"**를 쓴다(월 미선택 화면인 카드 슬롯·THEME DETAIL은 "추천 적기/현재 달"로 쓴 것과 구분). 따라서 이 문서의 화면·일정·저장 설계는 진입점과 무관하게 공통 적용된다. (단, "왜 이 테마"라는 납득은 테마 카드 경로에서만 THEME DETAIL이 담당 — `theme-detail.md`. 지도·챗봇 경로는 THEME DETAIL을 거치지 않을 수 있다.)

## 1. 현재 화면 구조 (관찰)

`강릉시 · 미식·노포 / 자연·트레킹 / 바다·해안 2박 3일 초안` 화면 기준.

- **헤더** — PLAN DETAIL, 도시명, 테마·기간 요약, 배지(테마칩 · 2박 3일 · AI 추천 여행 코스), 한 문단 요약.
- **일차별 시간대 흐름** — "현재 대화에서 정리한 조건으로 만든 전체 일정 초안입니다." + 3일 구성 배지.
- **일자 블록** — 예: "경포생태저류지, 순포습지 중심 일정" + 동선 텍스트(→ 화살표) + N개 코스.
- **타임라인 아이템** — 번호, 시간대(아침/점심/저녁), "다음 장소까지 0분", 장소명, 한 줄 설명, **추천 이유** 블록.
- **사이드바 일정 액션** — "이 일정은 어땠나요? 좋아요"(저장 토글), 마이페이지에 저장, 채팅으로 돌아가기, 마이페이지로 이동.

## 2. aha 매핑 (현재 vs 보강)

| aha 요소 | 현재 화면 | 판단 |
| --- | --- | --- |
| 막연 → 구체 (일정) | 일자·시간대·장소·순서 제시 | ✅ 골격은 있음 |
| 동선 | "다음 장소까지 0분" | ⚠️ 0분은 미연결로 보임 → 신뢰 저하 |
| 날씨 | 없음 | ❌ 실행가능성에 필요 |
| 지도·외부 링크 | 없음 | ❌ "진짜 갈 수 있다" 확신에 필요 |
| 장소별 통찰 이유 | "테마에 맞아 적합합니다" 류 | ⚠️ generic — 통찰 원칙 미적용 |
| peak-end 씨앗 | 좋아요·저장 있음 | ✅ 있으나 취향 시드 연결 필요 |

## 3. 일정·동선·날씨 (실행가능)

"이 화면만 보고 실제로 떠날 수 있는가?"가 판단 기준이다.

- **동선(이동시간) 실데이터.** "다음 장소까지 0분"은 미연결 placeholder로 보인다. 실제 이동 시간을 넣거나, 값이 없으면 그 줄을 **숨긴다**(0분이 그대로 노출되면 신뢰를 깎음).
- **날씨.** 일자별(또는 야외 장소) 월별 기상 경향을 표시하고, 우천 가능 달이면 실내 대안 한 줄을 같이 둔다.
- **지도·외부 딥링크.** 각 장소에 지도 링크, "식사 자유 선택" 항목엔 맛집·카페 검색 딥링크(Kakao/Google/일본 현지)를 붙인다. 더해 **투어·체험·입장권 등 예약이 필요한 항목은 클룩(Klook) 연동 딥링크**를 제공해, 일정에서 바로 예약 흐름으로 넘어갈 수 있게 한다(예: 지옥 순례 투어, 스파·온천 입장권).

## 4. 장소별 추천 이유 (통찰형)

타임라인 각 장소의 "추천 이유"도 generic 묘사가 아니라 **왜 이 장소를, 왜 이 순서·시간대에** 두었는지의 통찰로 쓴다(원칙은 `theme-detail.md` 4장과 동일).

- ❌ "자연·트레킹 테마에 맞아 편안한 산책과 풍경 감상을 원하는 여행자에게 적합합니다."
- ✅ "경포생태저류지를 오전 첫 코스로 둔 이유 — 호수에 물안개가 남아 있고 사람도 적어요. 오후엔 방문객이 몰립니다."

트레이드오프 또는 타이밍 1개 + 근거를 포함하고, 근거 데이터가 없으면 그 문장을 생성하지 않는다(환각 방지).

## 4.1 화면 인터랙션 — 긴 스크롤 해소 (Day 탭 · 지도 동선 · 접이식)

저장 일정을 마이페이지에서 클릭하면, 일자·시간대별 모든 카드(설명·이유·링크 포함)가 한 번에 펼쳐져 세로 스크롤이 과도하게 길어진다. 일정이 길수록(3박·4박) 더 심하다. 이를 **점진적 노출(progressive disclosure)**로 해소한다. (와이어프레임: `plan-detail-wireframe.html`, UI 규칙: `../09_ui_ux_guide/09_ui_ux_guide.md`)

- **Day 탭으로 일자 분할.** 한 번에 한 일자만 렌더한다. 상단 `1일차 / 2일차 / 3일차` 탭으로 전환하며, 전체 스크롤 길이를 일자 수만큼 줄인다. 일정이 짧으면(당일치기) 탭 없이 단일 일자만 노출.
- **카드 기본 접힘.** 타임라인 카드는 기본 상태에서 **요약 한 줄**(시간대 · 장소명 · 다음 장소까지 이동시간)만 보인다. 탭하면 한 줄 설명 · 통찰형 추천 이유 · 외부 링크가 펼쳐진다. "전체 펼치기/접기" 토글로 일괄 제어.
- **지도 + 점선 동선.** 일자별 지도에 그날 장소를 **마커**로 찍고 방문 순서대로 **점선**으로 잇는다. 마커 사이에는 구간 **이동시간 라벨**을 둔다(없으면 숨김 — 3장 동선 규칙과 동일). 식사 등 좌표 없는 항목은 마커를 만들지 않는다.
- **마커 ↔ 카드 양방향 연동.** 마커를 누르면 해당 카드가 펼쳐지고 강조되며, 카드를 펼치면 대응 마커가 강조된다.
- **원본 레이아웃 유지.** 상단 소도시 소개(헤더)와 우측 일정 액션(피드백·저장)은 그대로 두고, 중앙 "일차별 시간대 흐름"만 위 방식으로 대체한다.

## 5. 저장·peak-end (씨앗 심기)

사람은 **마지막 인상**으로 경험을 기억한다(peak-end rule). 이 화면이 여정의 마무리이자 다음 방문 개인화의 씨앗을 심는 곳이다.

- 좋아요·저장은 **취향 시드로 기록**한다(어떤 분위기·테마·장소를 좋아했는지). 이 시드가 단계 4 「개인화」 상태의 입력이 된다. (시드/프로필 스키마는 대표 문서 "공통 데이터 모델" 참고)
- "이 일정은 어땠나요?"의 좋아요/해제뿐 아니라, **분위기 단위 피드백을 4개 정도**(예: 한적함 / 자연 위주 / 동선 편함 / 일정 알참) 1탭으로 받아 시드 품질을 높인다.
- **부정 피드백도 시드로 저장한다.** "이 일정은 관심 없어요" 같은 버튼을 누르면 그 신호를 기록해 해당 테마·분위기의 노출을 낮춘다(긍정만 모으면 취향이 한쪽으로 쏠림). 긍정·부정 모두 단계 4 「개인화」 입력이 된다.
- 취향 시드는 **로그인 계정에 저장**한다(로브는 진입 시 회원가입·로그인이 필수이므로 익명·로컬 보관은 해당 없음).
- 저장 완료·"다음에 이어서 보기" 같은 긍정적 마무리 인상을 남긴다.

## 6. 레이턴시 대응

일정 생성 요청 직후(진입 경로 무관 — 테마 카드·지도 마커·챗봇) 일정을 만드는 구간이 가장 무겁다. 기다림을 기대로 바꾼다.

- **스트리밍 출력** — 일정·이유를 토큰 단위로 흘려보낸다. (기술 명세상 동기 호출 29초 한도를 넘을 수 있는 작업은 스트리밍 API로 분리하도록 이미 설계됨)
- **스켈레톤 카드** — 일정 자리를 먼저 그려 빈 화면을 없앤다(순수 프론트엔드).
- **단계적 노출** — 일정 골격(빠른 조회분)을 먼저 띄우고 장소별 이유를 뒤이어 채운다.
- **사전계산·캐시** — 후보·데이터 레이어는 미리 만들어 두고 요청 시엔 조회만 한다(6.1).

목표 지표: RAG 검색 1초 이내, 멀티스텝 Agent 전체 8초 이내, 동기 API 29초 한도(06_technical_spec).

### 6.1 사전계산·캐시 구조 (아키텍처 근거)

**사전계산 (요청 전, 오프라인 배치).** 수집 → 전처리(ELT) 후 여러 저장소에 미리 적재·인덱싱해, 런타임엔 *생성*이 아니라 *조회·검색*만 하게 한다.

- 통합 목적지 DB — 축제명·장소 개요·이미지·좌표 등 무거운 마스터. 1차 조회 레이턴시 확보용.
- S3 vector RAG Index — 장소·테마·자연어 임베딩 검색.
- DynamoDB 인접 리스트 + **사전계산 후보 테이블** — 도시·테마·축제·인접도시 관계를 미리 계산(그래프DB 대체). 런타임엔 Lambda가 읽어 후보 확장·재랭킹, 깊은 관계만 Lambda 인메모리 그래프로 보조.
- 정적 통계 — 월별 기상 경향(장마·태풍 플래그), 혼잡도 점수.

**캐시 (요청 중, 결과 재사용).**

- **query embedding cache** — 쿼리 벡터를 캐시 조회 → 없으면 생성.
- **축제 검증 캐시** — 키 `festival_id + travelYear`. 조회(hit) 시 재사용, miss 시 DynamoDB 축제 `start_date` 연도 일치 확인(초기)·공식 웹 검증(확장) → 검증 JSON만 저장, 상태별 TTL 보관. 상위 K개만 검증.

**caveat — LLM 산출물은 사전계산 대상이 아니다.** 후보 evidence 검색·scoring·이유 생성·Planner 일정 배치는 **런타임 추론**이다. 사용자별 일정·이유를 통째로 미리 굽는 건 현 설계 밖이며, 사전계산이 빠르게 만드는 것은 "후보 소도시 + 근거 데이터"까지다.

## 7. 일정 생성 파이프라인

기획서 9.1을 이 화면 기준으로 정리한다.

1. 진입점에서 조건 수신 — 테마 카드(테마·첫 동선 단서) / 지도 마커(지역) / AI 챗봇(자연어) 중 어디서 왔든 국가·시기·취향으로 정규화
2. 조건 분류 → RAG 기반 후보 검색
3. 후보 필터링 및 소도시 1곳 선정
4. day-by-day 일정 구성 + 장소별 이유 + 외부 정보 보강
5. 출처·갱신 시점·검증 상태 표시
6. 저장 시 취향 시드 기록 / "채팅으로 돌아가기"로 후속 질의(단계 2) 연결

## 8. 필요 데이터/전제

- 소도시 근거 데이터: 축제, 날씨(월별 경향), 혼잡도, 교통.
- 일정 템플릿, 동선·교통, 출처·갱신 시점·검증 상태.
- 지도·장소 정보, 외부 탐색 딥링크(Kakao/Google/일본 현지), 투어·예약 딥링크(클룩/Klook).
- 저장 스키마(취향 시드).

## 9. 측정 이벤트

- 일정 완독률·일자 펼침률.
- 지도·외부 링크 클릭률.
- 좋아요·저장 전환율, 분위기 피드백 응답률(긍정 4종·부정 각각), "관심 없어요" 클릭률.
- 후속 질의 전환율(단계 2 진입), 저장 → 재방문(단계 4) 전환율.
- 일정 생성 레이턴시(P50·P95).

## 10. 할 일 체크리스트

1. **동선 데이터 연결** — 실제 이동시간 표시 또는 미연결 시 줄 숨김(0분 노출 금지).
2. **날씨·지도·외부 딥링크** — 일자 날씨(우천 실내 대안), 장소별 지도, 식사 검색 링크, **투어·예약은 클룩(Klook) 연동 딥링크**.
3. **장소별 통찰 이유** — 장소·순서·시간대 근거 프롬프트(generic 금지, 근거 없으면 미생성).
4. **스트리밍·스켈레톤·사전계산 활용** — 일정 생성 구간 레이턴시 가리기.
5. **취향 시드 저장(긍정·부정)** — 좋아요·저장·분위기 긍정 4종 + "관심 없어요" 부정 신호를 모두 로그인 계정에 기록 → 단계 4 개인화 입력 스키마(부정은 노출 다운웨이트).
6. **peak-end 마무리** — 저장 완료·이어보기 등 긍정 마무리 UI.
7. **긴 스크롤 해소(progressive disclosure)** — Day 탭 분할 + 카드 기본 접힘 + 지도 점선 동선·마커↔카드 연동(§4.1, `../09_ui_ux_guide/09_ui_ux_guide.md`).

## 11. 엣지케이스·실패 상태

- **일정 생성 실패·timeout(동기 29초 초과).** 폴백 UI를 띄운다 — 부분 결과 노출 + 재시도, 또는 안전한 안내(요구사항 NFR-005의 타임아웃 폴백 규정과 연결).
- **추천할 소도시 없음(빈 결과).** 빈 화면을 띄우지 않는다. 조건 완화 제안("월/테마를 바꿔볼까요?")이나 인접 대안을 안내한다.
- **장소별 통찰 이유 근거 없음.** 그 문장을 생성하지 않는다(짧은 기본 설명만, generic 강제 금지).
- **동선·날씨·딥링크 데이터 없음.** 해당 줄/버튼을 숨긴다(0분·빈 링크 노출 금지, 3장 동선 규칙과 동일).
- **클룩 등 외부 예약 링크 없음.** 예약 버튼을 숨긴다(죽은 링크 금지).

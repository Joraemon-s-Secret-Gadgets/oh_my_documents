# 로브(Lovv) 후속 질의·재추천 설계 (단계 2)

> 문서 성격: 보조 Markdown
> 대표 문서: `95_aha_moment.md`

> 문서 버전: v0.6
> 문서 상태: 초안 (Draft)
> 적용 범위: 추천 이후 "더 조용한 곳" 같은 조건 변경에 따른 대화형 재추천(단계 2)
> 상위 문서: `95_aha_moment.md`(Aha Moment 여정 단계별, 대표 문서)
> 관련 문서: `theme-detail.md`("다른 추천 보기" 진입점), `plan-detail.md`("채팅으로 돌아가기" 진입점)
> 관련 산출물: `followup-requery-wireframe.html` (채팅 후속 질의 와이어프레임)

## 1. 목적

추천을 받은 뒤 사용자가 조건을 바꿔 다시 묻는 구간이다. 여기서 터뜨릴 aha는 **응답성** — "얘가 내 말을 알아듣네"이다. 동시에 이 구간은 **cold start의 다리**다. 행동 이력이 없어도 세션 안에서 쌓이는 신호(어떤 카드를 펼쳤는지, 무엇을 저장했는지, 어떤 follow-up을 했는지)를 **세션 임시 취향 벡터**로 모아 다음 추천을 좁히고, 그 신호가 단계 4 개인화의 씨앗이 된다.

판단 기준: "조건을 바꿨을 때 *정말* 바뀐 추천이 나오는가, 그리고 그걸 사용자가 체감하는가?"

## 2. 화면·인터랙션 구조

- **대화 입력창** — 자연어로 조건 변경 입력("더 조용한 곳").
- **추천 칩(quick reply)** — 자주 쓰는 follow-up을 버튼으로 제시(간판 5개, 3장 참고). 입력 부담을 줄이고 "무엇을 바꿀 수 있는지"를 알려준다.
- **인정 문구** — 재추천 위에 "아까 조용한 곳을 좋아하셨으니 이번엔 더 한적한 곳으로" 한 줄(6장).
- **재추천 미니 카드** — 채팅 안에 바뀐 테마/소도시 후보 + 이유 한 줄을 미니 카드로 보여주고, "이 일정으로 계획하기"로 **PLAN DETAIL(단계 3)** 진입한다. (별도의 추천 결과 화면은 없다 — 실제 일정·이유·저장은 PLAN DETAIL에서.)

## 3. 간판 follow-up (집중 튜닝 대상)

모든 자연어를 완벽히 처리하기보다, **자주 쓰이고 데이터로 답할 수 있는 5개를 간판으로 정해 집중 튜닝**한다. 여기서 막히면 응답성 aha가 신뢰 붕괴로 바뀌므로 우선 보장한다. (가격 데이터는 없으므로 예산 관련 follow-up은 제외)

| 간판 follow-up | 처리(데이터) |
| --- | --- |
| "더 조용한 곳" | 혼잡도 낮은 후보로 재정렬 |
| "축제 있는/없는 곳" | 여행 월(선택됐으면 그 달, 아니면 추천 적기) 축제 개최 여부로 필터 |
| "비 와도 되는 곳" | 월별 기상 경향(장마 플래그) + 실내 적합 체험 우선 |
| "이동 짧은 곳" | 교통·접근성 기준 후보 조정 |
| "다른 테마로" (예: 자연 말고 미식) | 테마 전환 후 동일 파이프라인 재실행 |

각 follow-up은 의미를 분류기로 해석하고, 데이터로 답할 수 없는 요청이면 솔직히 한계를 알리고 가능한 대안을 제시한다(환각 금지).

## 4. 세션 임시 취향 벡터 (cold start의 다리)

로브는 진입 시 이미 회원가입·로그인을 마친 상태다. 다만 **첫 방문이면 저장·클릭 같은 행동 이력이 아직 없다.** 그런 사용자도 세션 안에서는 신호가 쌓이므로, 이를 **로그인 계정에 묶어** 모은다.

- follow-up 내용(예: 조용함 선호)
- 카드 펼침·'왜 여기' 정독 등 관심 신호
- 세션 내 저장

이 벡터로 다음 추천을 좁혀, **첫 방문자도 세션이 길어질수록 개인화 aha를 미리 맛본다.** 세션이 끝나고 저장·피드백이 남으면(단계 3) 그것이 단계 4 「개인화」의 입력이 된다. 즉 단계 2의 세션 벡터 → 단계 3의 취향 시드 → 단계 4의 개인화로 이어지는 다리다. (세션/영속 계층과 스키마는 대표 문서 "공통 데이터 모델" 참고 — `sessionTaste` → `tasteProfile` 병합)

## 5. 재추천 파이프라인

새 검색을 처음부터 다시 하는 게 아니라, **바뀐 조건만 반영해 동일 파이프라인을 재실행**한다(기획서 9.1 step 8).

1. follow-up 파싱 → 변경된 조건 추출(기존 조건은 유지, 테마/혼잡/축제/날씨 등 변경분만)
2. 세션 임시 취향 벡터 갱신
3. 동일 추천 파이프라인 재실행(RAG 후보 검색 → 선정 → 이유 생성)
4. 재사용 가능한 캐시 활용(query embedding cache, 축제 검증 캐시) — 변경 안 된 부분은 재계산하지 않음
5. 인정 문구 + 갱신된 추천 결과 노출

## 6. 인정 문구 ("얘가 내 말 알아듣네")

재추천이 *왜 바뀌었는지*를 한 줄로 드러내 응답성을 체감시킨다.

- "아까 조용한 곳을 좋아하셨으니 이번엔 더 한적한 곳으로 골랐어요."
- "축제 없는 곳을 원하셔서, 축제가 없는 한적한 소도시로 바꿨어요."

템플릿화하되 실제 변경 조건을 채워 넣어 "그냥 새 추천"이 아니라 "내 말을 반영한 추천"으로 보이게 한다.

> **진입 시점에 따른 월 표현 주의.** 후속 질의는 PLAN DETAIL "채팅으로 돌아가기"(여행 월 이미 선택됨)와 THEME DETAIL "다른 추천 보기"(아직 월 미선택) 양쪽에서 들어온다. 월이 선택된 뒤면 "선택하신 N월"을, 아직이면 "추천 적기/현재 달"을 쓴다(대표 문서 ①타이밍 규칙과 동일).

## 7. 측정 이벤트

- follow-up 발생률(추천 → 재질의 전환).
- follow-up 해석 정확도(간판 5개 기준 성공률).
- 재추천 후 저장 전환율(단계 3 진입).
- 세션 내 follow-up 횟수와 만족(저장·완독) 상관.

## 8. 할 일 체크리스트

1. **간판 follow-up 5개 분류기·튜닝** — "더 조용한/축제/비/이동/다른 테마" 의미 해석 + 데이터 매핑.
2. **세션 임시 취향 벡터** — 세션 단위 신호 수집 구조(로그인 계정에 귀속, 단계 3·4로 전달).
3. **재추천 재실행 경로** — 변경 조건만 반영, 캐시 재사용으로 지연 최소화.
4. **인정 문구 템플릿** — 변경 조건을 채우는 동적 한 줄.
5. **한계 응답** — 데이터로 답할 수 없는 요청은 솔직히 알리고 대안 제시(환각 금지).
6. **추천 칩 UI** — 간판 follow-up을 버튼으로 노출.

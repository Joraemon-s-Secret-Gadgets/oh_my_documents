# 로브(Lovv) THEME DETAIL 화면 설계 (단계 1)

> 문서 버전: v0.7
> 문서 상태: 초안 (Draft)
> 적용 범위: 카드 클릭 후 도착하는 "이 테마를 추천하는 기준" 화면(단계 1)
> 상위 문서: `95_aha_moment.md`(Aha Moment 여정 단계별, 대표 문서)
> 관련 문서: `aha-card-slot.md`(앞 단계 카드 슬롯), `plan-detail.md`("이 테마로 일정 계획하기" 다음 화면)
> 관련 산출물: `theme-detail-wireframe.html` (THEME DETAIL 와이어프레임)

## 1. 목적

메인 카드(테마)를 클릭하면 처음 도착하는 화면이다. 여기서 터뜨릴 aha는 **납득** — "왜 이 테마·이 방향을 추천하는지 알겠다"이다. 발견(슬롯)에서 시작된 흥미를 **"이 테마로 일정 계획하기"라는 행동**으로 미는 다리 역할을 한다. (실제 day-by-day 일정·저장은 그다음 화면 `plan-detail.md`에서.) 참고로 PLAN DETAIL은 지도 마커·AI 챗봇으로도 진입하는 공통 화면이고, 이 CTA는 그 진입 경로 중 하나(테마 카드 경로)다.

## 2. 현재 화면 구조 (관찰)

`온천·휴양` 테마 화면 기준.

- **헤더·이미지** — THEME DETAIL, 테마명·대표 관광지 배지(온천·휴양 / 아산·온양·벳푸), 큰 제목("온천으로 회복하는 느린 여행"), 한 줄 소개.
- **RECOMMENDATION BASIS "이 테마를 추천하는 기준"** + 한 문단("Lovv는 도시 이름보다 먼저 여행 분위기를 정리하고, 해당 테마와 가까운 소도시 후보를 일정 대화로 연결합니다.").
- **기준 테마** — 온천·휴양 (#온천 #휴양).
- **추천 이유** — "온양온천과 벳푸처럼 뜨거운 물, 숙소 체류, 느린 회복감을 중심에 둬요."
- **첫 동선 단서** — 온양온천 · 스파 휴양 · 지옥 순례 ("AI 일정은 이 동선 단서를 기준으로 여행 기간과 세부 취향을 좁힙니다.").
- **CTA** — "이 테마로 일정 계획하기"(→ 단계 3 PLAN DETAIL), "다른 추천 보기"(→ 단계 2 후속 질의·재추천, 채팅).

## 3. aha 매핑 (현재 vs 보강)

| 요소 | 현재 화면 | 판단 |
| --- | --- | --- |
| 추천 이유 (왜 이 테마) | "뜨거운 물·숙소 체류·느린 회복감" | ⚠️ 분위기 묘사 위주 — 통찰로 한 끗 |
| 타이밍 (왜 지금) | 없음 | ❌ "온천 적기 10~11월" 한 줄 여지 |
| 첫 동선 단서 | 있음 | ✅ 일정 생성 시드로 좋음 |
| CTA(일정 계획하기) | 있음 | ✅ 단계 3 진입 명확 |

## 4. 통찰형 추천 이유 (납득의 핵심)

이 화면의 "추천 이유"가 단계 1 aha의 핵심이다. generic 분위기 묘사보다 **트레이드오프·타이밍 통찰 1개 + 근거**를 담는다.

| 구분 | 예시 |
| --- | --- |
| ❌ generic | "뜨거운 물, 숙소 체류, 느린 회복감을 중심에 둬요." (분위기 나열) |
| ✅ 통찰(트레이드오프) | "성수기 벳푸 온천가는 붐비지만, 같은 '뜨거운 물 + 느린 체류'를 덜 알려진 온천 소도시에서 한산하게 누릴 수 있어요." |
| ✅ 통찰(타이밍·추천 적기) | "온천은 10~11월이 적기예요 — 물·날씨가 가장 좋은 시기." |

원칙(generic 금지 · 트레이드오프/타이밍 강제 · 근거 의무 · 길이 상한)은 PLAN DETAIL의 장소별 이유에도 똑같이 적용한다(`plan-detail.md` 4장).

## 5. 타이밍 배지 (선택)

THEME DETAIL에도 "왜 지금" 한 줄을 얹을 수 있다. **단, 이 화면에서는 아직 여행 월을 안 고른 상태**이므로 "선택하신 N월"이라고 쓰면 안 된다. **추천 적기**("온천 적기: 10~11월") 또는 **현재 달**("지금(10월)이 가기 좋은 시기") 기준으로 쓴다. 여행 월은 "이 테마로 일정 계획하기"를 누르면 들어가는 일정 생성 채팅에서 챗봇이 가장 먼저 물어보고, 그 답을 받은 뒤 화면(PLAN DETAIL)에서야 "선택하신 N월" 표현을 쓴다.

## 6. 레이턴시

THEME DETAIL은 **테마 단위 정보**(기준 테마·추천 이유·동선 단서)라 대부분 사전계산/템플릿으로 즉시 노출할 수 있어 빠르다. 무거운 LLM 일정 생성은 "이 테마로 일정 계획하기" 이후(단계 3)에서 발생하므로, 레이턴시 대응(스트리밍·스켈레톤·사전계산 범위)은 `plan-detail.md` 6장에서 다룬다.

## 7. 필요 데이터/전제

- 테마 정의·태그, 테마 → 대표 관광지 매핑(`aha-card-slot.md` 2.1).
- 첫 동선 단서(대표 관광지·축제 시드).
- 테마별 추천 적기(월), 통찰형 이유 생성 근거(혼잡·계절·축제 데이터).

## 8. 측정 이벤트

- THEME DETAIL 도달률(슬롯 카드 클릭 대비).
- **"이 테마로 일정 계획하기" 클릭률**(단계 3 전환) — 이 화면의 핵심 지표.
- "다른 추천 보기" 비율, 추천 이유 정독률.

## 9. 엣지케이스·빈 상태

- **추천 이유 근거 없음.** 통찰을 뽑을 데이터(혼잡·계절·축제)가 없으면 generic 문장을 억지로 만들지 말고, 안전한 기본 한 줄(테마 정의 수준)로 폴백한다.
- **첫 동선 단서 데이터 없음.** 단서를 빈 채로 두지 말고 생략하고, "일정 계획하기에서 조건을 더 좁혀요" 안내로 대체한다.
- **추천 적기 데이터 없음.** 타이밍 한 줄을 생략한다(억지 적기 금지).

## 10. 할 일 체크리스트

1. **통찰형 테마 추천 이유** — 트레이드오프/타이밍 강제, generic 금지, 근거 의무.
2. **타이밍 한 줄(선택)** — 추천 적기 또는 현재 달 기준("온천 적기 10~11월"). "선택하신 N월" 금지(월 미입력).
3. **첫 동선 단서 → 일정 시드 연결** — 단계 3 일정 생성 입력으로 전달.
4. **CTA 전환 추적** — "이 테마로 일정 계획하기" 클릭 이벤트.

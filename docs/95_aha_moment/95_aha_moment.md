## 핵심 aha 정의

로브의 추천 경험에서 사용자가 "이런 곳이 있는 줄 몰랐는데 가보고 싶다"고 느끼는 순간(aha moment)을 더 자주·더 빨리·여정의 더 많은 지점에서 끌어내는 것이 이 문서군의 목표다. 새 기능을 더하는 게 아니라, 이미 있는 흐름(온보딩에서 테마·기본 국가 선택 → 카드/지도/챗봇으로 일정 생성 채팅 진입 → 챗봇이 국가·여행 시기를 먼저 질문 → 소도시 추천)에서 aha가 터지는 지점을 늘린다.

로브의 핵심 aha는 다음 한 문장으로 정의한다.

> "이런 곳이 있는 줄 몰랐는데, 지금 내 조건에 딱 맞고, 진짜 갈 수 있겠다."

이 aha는 세 성분으로 분해된다. 추가 aha를 만든다는 것은 이 세 축 중 하나를 더 세게 자극하거나, 여정의 다른 지점에 새 봉우리를 심는 일이다.

| 성분 | 의미 | 강화 수단 |
| --- | --- | --- |
| 발견(놀람) | 내가 못 찾았을 소도시를 만남 | 맛보기 카드, 발견 카드, 비선택 테마 노출 |
| 납득(왜 여기) | 추천 이유가 개인적 통찰로 다가옴 | 트레이드오프·타이밍 통찰형 설명 |
| 실행가능성 | 실제로 갈 수 있다는 확신 | 일정·지도·날씨·외부 링크 |

## aha를 늘리는 두 갈래 전략

aha 강화는 성격이 다른 두 갈래로 나뉜다. 둘은 별개이며 1:1로 대응하지 않는다.

### 첫 aha를 더 세게 (방법)

첫 추천 경험 자체의 봉우리를 높이는 방법론이다.

1. **Time-to-value 단축.** 입력 게이트 뒤에만 가치가 있으면 절반이 이탈한다. 입력 전 맛보기 카드로 첫 aha를 앞당긴다.
2. **회피한 혼잡의 가시화.** 사용자가 무엇을 피했는지 보여줘야 추천 가치가 체감된다. 단, 유명지↔유명지 대치가 아니라 **유명 대표 관광지 → 덜 붐비는 소도시** 대조여야 한다(카드 모델은 `aha-card-slot.md` 2장). 유명지를 무조건 배제하지 않고 비수기 유명지도 추천 대상이므로, 가시화 기준은 '유명도'가 아니라 '지금 이 조건에서 갈 만한가(혼잡도·계절 적합도)'로 둔다.
3. **이유를 통찰로.** 밋밋한 설명 대신 "비수기라 단풍은 끝났지만 대기 없이 온천을 즐길 수 있다" 같은 비직관적 트레이드오프·타이밍 통찰을 제시한다.

### 여정 곳곳에 새 봉우리 (적용 지점)

여정의 단계마다 별도의 aha를 심는다. 아래는 ROI(효과 대비 비용) 높은 순서다.

| 우선순위 | 후보 | aha 성격 | 핵심 장치 |
| --- | --- | --- | --- |
| ① | 타이밍/희소성 | "지금 안 가면 놓친다" | 날씨·축제를 필터가 아닌 넛지로 |
| ② | 후속 질의 응답성 | "얘가 내 말을 알아듣네" | "더 조용한 곳" 같은 follow-up 정밀 튜닝 |
| ③ | 개인화 | "나를 기억하네" | 저장·행동 기반 재추천 |
| ④ | 막연 → 구체 | "이거 그대로 떠나면 되겠다" | day-by-day 일정·동선·날씨 |

**① 타이밍/희소성 aha — "지금 안 가면 놓친다."** 날씨·축제를 '필터'에서 '넛지'로 재해석한다. 여행 날짜는 월 단위로만 받고, **여행 월(시기)은 일정 생성을 위해 채팅에 진입하면 챗봇이 가장 먼저 묻는다**(국가·시기 선입력). 그래서 타이밍 표현은 화면 시점에 따라 다르다 — **여행 월 입력 전(카드 슬롯·THEME DETAIL)에는 "추천 적기/현재 달"**(예: "온천 적기 10~11월"), **여행 월 입력 후(PLAN DETAIL)에는 "선택하신 N월"**. 날씨는 월별 기상 경향, 축제는 월 단위 개최로 다룬다. 데이터를 이미 다루므로 추가 비용이 작고 효과가 커서 가장 먼저 둔다.

**② 후속 질의 응답성 aha — "얘가 내 말을 알아듣네."** follow-up에 파이프라인이 잘 pivot하면 그 순간이 지능을 증명한다. 여기서 막히면 신뢰가 무너지므로 간판 follow-up을 정해 집중 튜닝한다(단계 2 / `followup-requery.md`).

**③ 개인화 aha — 두 번째 추천에서 "이거 나를 아네."** 저장·피드백을 다음 추천에 반영한다. cold start 때문에 재방문 사용자에게만 작동한다(단계 4 / 카드 슬롯 「개인화」 상태, `aha-card-slot.md` 3.2).

**④ 막연한 바람 → 구체적 계획 aha.** day-by-day 일정 + 동선 + 날씨로 "이거 그대로 떠나면 되겠다"가 되는 전환. 소도시 1곳 집중 구조의 강점을 살린다(단계 3 / `plan-detail.md`).

## 개요

cold start부터 재방문까지를 **funnel 단계로 깔고, 거기에 aha 후보들을 각자 맞는 자리에 배치**한다. 개인화는 "기능"이 아니라 *방문이 쌓이며 자라는 곡선*이라, 단계로 봐야 cold start 문제가 자연히 풀린다.

각 단계는 아래 6항목으로 구체화한다. "할 일"은 실제 구현 단위다.

- **트리거** — 언제 발생하는가
- **사용자가 보는 것** — 화면·카피 예시
- **시스템 동작** — 입력 → 처리 → 출력
- **필요 데이터/전제** — 무엇이 준비돼야 하는가
- **측정 이벤트** — 무엇을 찍어 검증하는가
- **할 일 체크리스트** — 만들어야 하는 것

> 카드 슬롯 상태(「기본」 / 「개인화」)와의 연결: **단계 0 = 「기본」 상태**(6장), **단계 4 = 「개인화」 상태**(맨 앞 보라 카드 추가, 7장). 단계 2의 세션 신호는 단계 4 개인화를 예열한다. (단계 1·3은 카드 슬롯이 아니라 THEME DETAIL·PLAN DETAIL 화면이다.) 카드 슬롯 상세는 `aha-card-slot.md`.

## 화면 흐름과 구현 단위

**단계는 5개, 실제 화면은 4개**다. 단계 0·4는 같은 화면(카드 슬롯, 상태만 다름)을 쓰므로 "단계 = 화면"이 아니다.

- **메인 카드 슬롯** = 단계 0(첫 진입)·단계 4(재방문)가 같은 화면(상태만 「기본」/「개인화」).
- **THEME DETAIL "이 테마를 추천하는 기준"** = 카드 클릭 후 도착. 단계 1(왜 이 테마=납득). "이 테마로 일정 계획하기"로 다음 화면.
- **PLAN DETAIL** = 일정 생성 결과 화면. 단계 3(day-by-day 일정·저장=실행가능·peak-end). **테마 카드(THEME DETAIL의 "이 테마로 일정 계획하기")·지도 마커 클릭·AI 챗봇으로 일정짜기 어느 경로로 진입해도 같은 화면·같은 일정 생성**으로 수렴한다(지도·챗봇 경로는 THEME DETAIL을 거치지 않을 수 있음).
- **채팅 후속 질의** = 단계 2. THEME DETAIL "다른 추천 보기" 또는 PLAN DETAIL "채팅으로 돌아가기"에서 진입.

### 단계 ↔ 화면 ↔ 문서 매핑

| 단계 | 터지는 aha | 실제 화면 | 보조 문서 |
| --- | --- | --- | --- |
| 0 첫 진입 | 발견 | 메인 카드 슬롯(「기본」, 6장) | `aha-card-slot.md` |
| 1 추천 이유 | 납득 | THEME DETAIL "이 테마를 추천하는 기준" | `theme-detail.md` |
| 2 후속 질의 | 응답성 | 채팅 후속 질의 | `followup-requery.md` |
| 3 저장·peak-end | 실행가능 | PLAN DETAIL (테마/지도/챗봇 진입 공통) | `plan-detail.md` |
| 4 재방문 | 개인화 | 메인 카드 슬롯(「개인화」, 7장) | `aha-card-slot.md` |

### 화면 흐름

1. **단계 0** — 메인 진입 → 카드 슬롯(「기본」, 6장)
2. (카드 클릭) → **단계 1** — THEME DETAIL: 왜 이 테마인지 납득 + 첫 동선 단서 → "이 테마로 일정 계획하기"
3. (일정 계획하기 클릭) → **단계 3** — PLAN DETAIL: day-by-day 일정·동선·날씨 → 저장·peak-end
4. **단계 2** — "더 조용한 곳" 등 후속 질의 → 재추천(채팅)
5. (재방문) → **단계 4** — 메인 진입 → 카드 슬롯(「개인화」, 7장)

> **단계 3 진입점은 셋이다.** 위 흐름의 테마 카드 경로 외에, **지도 마커 클릭**과 **AI 챗봇으로 일정짜기**로도 PLAN DETAIL(단계 3)에 바로 진입할 수 있다 — 진입점만 다르고 일정 생성·화면은 동일하다. 지도·챗봇 경로는 단계 1(THEME DETAIL)을 건너뛸 수 있다.

각 화면의 세부 설계와 "할 일"은 위 표의 보조 문서, 그리고 아래 단계별 절의 **할 일 체크리스트**에 있다.

## 단계 0 — 첫 진입 (온보딩 직후, 행동 0)

개인화 aha를 못 쓰는 구간(행동 이력 0). **조건 기반 발견 aha**로 첫 인상을 책임진다. 핵심은 "이력 없어도 *타이밍*은 항상 있다"는 것 — 날씨·축제·혼잡은 누구에게나 지금 존재한다.

- **트리거.** 가입 + 온보딩 설문(테마 + 기본 국가 선호) 완료 후 메인 첫 진입. 여행 시기 미입력(국가는 온보딩 기본값 사용).
- **사용자가 보는 것.** 상단 맛보기 카드 슬롯(「기본」 상태). 메인 멘트 "수아 님을 위한 추천(FOR YOU)" 고정. 소도시 payoff는 기본 국가 소도시. 카드에 혼잡 대조 한 줄(월·계절 기준) — "10월 오사카는 가장 붐빔 → 같은 분위기, 한적한 군산".
- **시스템 동작.** 온보딩 테마 + 기본 국가 선호 + 오늘 날짜·계절·현재 혼잡/날씨 조건 → **사전계산된 후보·근거 데이터**(통합 목적지 DB·S3 vector·사전계산 후보 테이블)를 조회해 즉시 노출. 후보·데이터는 실시간 RAG 생성 없이 조회만 하므로 빠르지만, '이유' 문구는 LLM 실시간 생성 대신 **템플릿/정형 문구**로 처리한다(사전계산·캐시 범위는 `plan-detail.md` 6.1 참고).
- **필요 데이터/전제.** 온보딩 테마·기본 국가 선호 저장값, 소도시별 계절 적합도 점수, 현재 혼잡·날씨, 테마 → 대표 관광지 매핑(`aha-card-slot.md` 2.1, 국가 반영은 2.3).
- **측정 이벤트.** 맛보기 카드 노출·클릭, 진입부터 첫 클릭까지 시간(time-to-first-click), 비선택 테마 카드 클릭률.
- **할 일 체크리스트.**
  1. 온보딩 테마·기본 국가 선호를 사용자 프로필에 저장하는 필드.
  2. 테마·국가별 소도시 추천을 미리 계산해 캐시하는 배치 작업(일/주 단위).
  3. 혼잡 대조 카피 템플릿("지금 OO 혼잡 → 한적한 XX").
  4. 「기본」 카드 슬롯 컴포넌트(오렌지=선호 / 초록=발견, 메인 멘트 고정).

## 단계 1 — 추천 기준 (THEME DETAIL)

발견 다음은 **납득**이다. 후보①(타이밍/희소성) + 틀③(이유→통찰)이 붙는다. 이 단계는 **THEME DETAIL "이 테마를 추천하는 기준"** 화면이며, 화면 단위 상세 설계는 보조 문서 `theme-detail.md` 참고. (아직 여행 월·소도시는 정하지 않은 테마-level 화면.)

- **트리거.** 메인 카드(테마)를 클릭. **여행 월 미입력 상태.**
- **사용자가 보는 것.** 기준 테마 + '왜 이 테마' 통찰형 이유 + 첫 동선 단서 + 타이밍 한 줄(아직 월 미선택이라 **추천 적기/현재 달** 기준, 예: "온천 적기: 10~11월") + "이 테마로 일정 계획하기" CTA.
- **시스템 동작.** 테마 정보·추천 이유·동선 단서를 사전계산/템플릿으로 즉시 노출(무거운 RAG 일정 생성은 단계 3에서). 소도시를 아직 확정하지 않음.
- **필요 데이터/전제.** 테마 정의·태그, 테마 → 대표 관광지 매핑, 첫 동선 단서, 통찰 이유 근거, 테마별 추천 적기.
- **측정 이벤트.** "이 테마로 일정 계획하기" 클릭률(단계 3 전환), 추천 이유 정독률, "다른 추천 보기" 비율.
- **할 일 체크리스트.**
  1. 통찰형 테마 이유 프롬프트(분위기 나열 금지, 트레이드오프·타이밍 1개 이상 강제).
  2. 타이밍 한 줄 — **추천 적기/현재 달** 기준(월 미선택 단계라 "선택하신 N월" 금지).
  3. CTA 전환 추적, 첫 동선 단서를 단계 3 일정 시드로 전달.

## 단계 2 — 후속 질의 (세션 내 미니 개인화 = cold start의 다리)

cold start를 푸는 진짜 열쇠. **이력이 없어도 세션 안에서는 신호가 쌓인다.** 후보②(응답성 aha). 화면 단위 상세 설계는 보조 문서 `followup-requery.md` 참고.

- **트리거.** 사용자가 "더 조용한 곳", "축제 없는 곳", "비 와도 되는 곳" 같은 follow-up을 입력.
- **사용자가 보는 것.** 조건이 반영된 재추천 + 인정 문구 — "아까 조용한 곳을 좋아하셨으니 이번엔 더 한적한 곳으로".
- **시스템 동작.** follow-up 파싱 → 세션 임시 취향 벡터 업데이트(어떤 카드를 펼쳤는지·저장했는지 포함) → 동일 추천 파이프라인 재실행.
- **필요 데이터/전제.** 세션 내 행동 로그(펼침·저장·follow-up), 조건 변경을 해석하는 분류기.
- **측정 이벤트.** follow-up 발생률, 재추천 후 저장률, follow-up 해석 정확도.
- **할 일 체크리스트.**
  1. **간판 follow-up 5개 정의 + 집중 튜닝**("더 조용한", "축제 있는/없는", "비 와도 되는", "이동 짧은", "다른 테마로"). 가격 데이터가 없으므로 예산 관련은 제외. 여기서 막히면 신뢰가 무너지므로 우선 보장.
  2. 세션 임시 취향 벡터 자료구조(세션 단위, 로그인 계정 귀속 — 진입 시 이미 로그인 상태).
  3. 인정 문구 템플릿("아까 OO 좋아하셨으니…").

## 단계 3 — 일정 생성·저장 (PLAN DETAIL)

소도시 1곳을 정해 day-by-day 일정으로 떨어뜨리는(막연→구체, 후보④) + 마지막 인상(peak-end)으로 저장·씨앗을 심는 단계. 이 단계는 **PLAN DETAIL** 화면이며, 상세 설계는 보조 문서 `plan-detail.md` 참고. **테마 카드(THEME DETAIL CTA)·지도 마커·AI 챗봇 어느 경로로 진입해도 동일.**

- **트리거.** "이 테마로 일정 계획하기"(또는 지도 마커·챗봇 일정짜기)로 **일정 생성 채팅 진입** → 챗봇이 **국가·여행 시기(월)를 가장 먼저 질문** → 일정 생성. 이 답을 받은 시점에 여행 월이 확정된다.
- **사용자가 보는 것.** 소도시 1곳 + day-by-day 일정 + 동선 + 날씨 + 장소별 '왜 여기' 통찰 이유 + 지도·외부 딥링크(투어·예약은 클룩) + 타이밍("**선택하신 N월**에 …" — 이제 월이 확정됐으므로 사용) + 분위기 피드백(긍정 4종 + "관심 없어요" 부정) 1탭 + 저장.
- **시스템 동작.** 진입 조건 + 여행 월 → RAG 후보 검색 → 소도시 1곳 선정 → 일정·장소별 이유 생성(스트리밍·스켈레톤으로 대기 가림) → 저장 시 **취향 시드(seed) 기록**.
- **필요 데이터/전제.** 소도시 근거 데이터(축제·날씨·혼잡·교통), 일정 템플릿, 동선, 저장 스키마.
- **측정 이벤트.** 일정 완독률, 저장률, 피드백 응답률, 저장→재방문 전환율, 일정 생성 레이턴시.
- **할 일 체크리스트.**
  1. 챗봇 첫 질문으로 국가·여행 시기(월) 수집 + 소도시 선정·일정 생성.
  2. 통찰형 장소 이유 + 동선·날씨·지도 링크 + 투어·예약 클룩(Klook) 딥링크.
  3. 스트리밍·스켈레톤·사전계산으로 생성 레이턴시 가리기.
  4. 분위기 피드백(긍정 4종 + 부정 "관심 없어요") + 취향 시드 저장 스키마 — 긍정 선호↑·부정 노출↓, 다음 방문 개인화 연료.

## 단계 4 — 재방문 (이력 있음, 진짜 개인화 aha)

이력이 생겨 후보③(개인화 aha)이 터진다. 「개인화」 상태와 연결. 첫 방문이 "이런 곳이 있었네"라면, 재방문은 **"나를 기억하네"**다.

- **트리거.** 저장·행동 이력이 있는 사용자의 재방문.
- **사용자가 보는 것.** 개인화 히어로 카드 "지난번 좋아하신 곡성과 분위기 비슷한데 덜 알려진 OO"(갔던 곳 기반·exploit). **우상단 X를 누르면 "안 가본 곳"(explore) 추천으로 전환** — 취향엔 맞되 아직 안 가본 테마·지역. + **soft prefill** "지난번엔 3월 2박3일로 보셨어요(수정 가능)".
- **시스템 동작.** 저장 소도시의 취향 벡터 → 유사 소도시 검색 → **신뢰 임계값** 통과 시에만 히어로 노출(미달이면 「기본」 상태로 강등). 온보딩 선언과 실제 행동이 다르면 행동 우선.
- **필요 데이터/전제.** 사용자 취향 벡터, 소도시 임베딩/유사도 인덱스, 임계값 정책, "좋아함"의 명시적 정의(저장·긍정 피드백·반복 클릭).
- **측정 이벤트.** 개인화 카드 클릭률, 안 가본 곳 전환율(X 클릭률), 재방문 → 추천 전환율, 개인화 적중/오추천 피드백.
- **할 일 체크리스트.**
  1. 저장·행동에서 취향 벡터 추출 로직.
  2. 소도시 유사도 인덱스(임베딩 기반).
  3. 신뢰 임계값 정책(미달 시 「기본」 상태 fallback).
  4. soft prefill UI — 날짜·일정은 hard 자동 기입 금지, 수정 가능한 제안만.
  5. **안 가본 곳(explore) 추천 로직 + X 토글** — 비슷한 곳 ↔ 안 가본 곳 전환, X 선택을 신호로 기록(비슷한 곳 비중 ↓).

## 공통 데이터 모델 — 취향 프로필/시드

단계 2(세션 신호) → 단계 3(저장 시드) → 단계 4(개인화)가 공유하는 데이터다. **선언(온보딩) / 행동(누적·감쇠) / 세션(비영속)** 3계층으로 나누고, 점수와 그 근거 신호 원장을 분리한다.

```
tasteProfile {
  userId; schemaVersion

  declared: { themes:[1~3], countryPref: 한|일 }          // 온보딩 기준선(불변)

  themeAffinity: { <themeId>: { score, lastUpdated } }    // 6 base + 확장 테마, 시간 감쇠 적용
  vibeAffinity:  { <vibeTag>: signedScore }               // 긍정/부정 통합(부호로)
  negatives:     [ { target: theme|vibe|city, source: X|관심없어요, ts } ]

  signals: [ { type: save|click|expand|fb_pos|fb_neg|explore_x,
               target, weight, ts } ]                      // 위 점수의 근거 원장(신호강도·출처·시점)
  counts:  { saves, clicks, feedbacks }                    // 「기본」↔「개인화」 게이팅·신뢰도

  savedCityIds: [...]            // explore 제외 집합 + "비슷한 곳" 임베딩 유사도 기준
  tasteVector?: [embedding]      // (선택) 저장 도시 임베딩 집계, 유사 소도시 검색용

  lastTrip: { country, month, nights }   // soft prefill(수정 가능)
  updatedAt
}

sessionTaste { signals[], 만료 }   // 단계 2 비영속 → 세션 종료 시 tasteProfile에 병합
```

설계 원칙:

- **신호 강도·출처 차등.** 저장 > 반복 클릭 > 펼침, 명시(좋아요/관심없어요) > 암묵(클릭). `signals[].weight`로 구분.
- **시간 감쇠.** 취향은 변하므로 점수에 recency decay(타임스탬프 기반)를 적용한다.
- **선언 vs 행동.** 온보딩 선언과 행동 점수를 따로 들고 블렌딩한다(행동 신호가 충분하면 선언을 덮어씀, 단계 4).
- **"갔던 곳"의 의미.** 실제 방문은 알 수 없으므로(GPS 없음) `savedCityIds`는 **저장·긍정한 곳**이다 — explore 제외 집합과 유사도 기준은 이 집합으로 잡는다.
- **계층 분리.** 세션 임시(`sessionTaste`, 비영속)와 계정 영속(`tasteProfile`)을 나누고 세션 종료 시 병합한다.
- **프라이버시.** 대화 원문·자연어 입력은 저장하지 않고 **파생 신호만** 저장한다(06_기술명세). 사용자 리셋·보존 기준을 둔다.

이벤트(트래킹) 명세는 `tracking-plan.md` 참고.

## 전체를 한 장으로

| 단계 | 화면 | 이력 | 터지는 aha | 핵심 장치 | 대표 할 일 |
| --- | --- | --- | --- | --- | --- |
| 0 첫 진입 | 메인 카드 슬롯(「기본」) | 없음 | 발견 | 맛보기 카드·혼잡 대조(월 기준) | 테마별 추천 사전계산 캐시 |
| 1 추천 기준 | THEME DETAIL | 없음 | 납득 | 통찰형 이유·타이밍(추천 적기/현재 달, 월 미선택) | 트레이드오프 프롬프트·CTA 전환 |
| 2 후속 질의 | 채팅 | 세션 내 | 응답성+예열 | "알아듣네"·세션 취향 벡터 | 간판 follow-up 5개 튜닝 |
| 3 일정·저장 | PLAN DETAIL | 씨앗 심음 | 실행가능·peak-end | 구체 일정·동선·날씨·피드백 1톨 | 동선·날씨 연결·취향 시드 저장 |
| 4 재방문 | 메인 카드 슬롯(「개인화」) | 있음 | 개인화 | 갔던 곳 기반·soft prefill | 취향 벡터·유사도 인덱스·임계값 |

## 요점

1. **cold start는 "타이밍은 누구에게나 있다 + 세션 내 신호로 예열한다"로 푼다**(단계 0~2).
2. **갔던 곳 기반 추천·자동 기입은 단계 4의 무기지 첫 인상용이 아니다.** 그래서 첫 방문 aha와 분리해 funnel 단계별로 둔다.
3. 어디부터 착수할지 막막하면 **ROI 순으로 단계 1(통찰형 이유) → 단계 2(간판 follow-up) → 단계 0(맛보기 캐시)** 순서를 권한다. 단계 3~4는 저장·재방문 데이터가 쌓인 뒤 효과가 난다.

1. 받은 피드백을 제품 질문으로 바꾸기

중간발표에서 문제 정의와 AI 적용 방식은 설득력이 있었지만, 그다음 질문이 남았다.

문제 정의와 AI 적용까지는 좋다. 그런데 사용자가 실제로 쓸 만큼 강한 kick이 있는가? 그리고 그 사용이 ROI로 이어지는가?

현재 설계는 소도시를 발견하고, 추천 이유를 납득하고, 일정으로 실행 가능성을 확인하는 흐름이 좋다. 다만 AI가 만든 낯선 소도시 일정만으로는 사용자가 마지막 선택을 미룰 수 있다. "추천은 이해했지만 정말 이대로 가도 되는가?"라는 불확실성이 남기 때문이다.

따라서 kick은 AI 추천을 하나 더 만드는 기능이 아니라, AI 추천을 실제 선택으로 바꾸는 증거층이어야 한다.

2. 핵심 kick 정의

로브의 kick을 다음 한 문장으로 정의한다.

"나와 취향이 비슷한 사용자가 선택한 공개 일정을 보고, 한 번에 내 일정으로 가져온다."

사용자 화면의 더 짧은 표현은 다음과 같다.

"나와 취향이 비슷한 사람들이 고른 일정, 그대로 시작하기"

이 kick은 세 요소가 동시에 있어야 성립한다.

요소사용자의 생각제품 장치
유사성"나와 취향이 비슷한 사람이 골랐다"연령대·성별·선호 국가·선택 테마의 일치 근거 표시
사회적 증거"실제로 선택·저장된 일정이다"공개 일정, 저장·복제 수, 일정 장소의 출처·운영 정보 확인 상태
즉시성"처음부터 짤 필요가 없다"이 일정으로 시작 1탭 복제 후 공개 일정 컨텍스트 챗봇에서 부분 조정

공개 일정을 구경만 하게 만들면 콘텐츠 피드에 그친다. 복제하고 내 조건에 맞게 수정할 수 있어야 kick이다.

3. Aha Moment와 Kick의 관계

95_aha_moment가 "이런 곳이 있었고, 나에게 맞으며, 갈 수 있겠다"를 만드는 여정이라면 kick은 그 여정의 끝에서 "그래서 이걸 고르겠다"를 만드는 전환 장치다.

발견(Aha) → 추천 이유 납득(Aha) → 비슷한 사람의 실제 일정 확인(Kick)
          → 1탭 복제 → 챗봇 부분 조정·조건 확인 → 저장·외부 링크 행동(ROI)
기존 단계기존 ahakick 보강
메인 카드 슬롯몰랐던 소도시 발견나와 취향이 비슷한 사용자가 고른 공개 일정 레일을 함께 노출
THEME DETAIL왜 이 테마인지 납득같은 테마를 선택한 유사 여행자 일정 2~3개 제시
PLAN DETAIL실제로 갈 수 있는 일정공개 일정 원본·복제 수·수정 포인트를 보여주고 1탭 복제
재방문나를 기억하는 개인화이전 저장·복제 행동을 다음 공개 일정 랭킹에 반영

공개 일정은 기존 Aha 흐름을 대체하지 않는다. AI는 후보를 찾고 이유를 설명하며, 공개 일정은 선택을 안심시키고 행동을 단축한다.

4. 핵심 사용자 경험

화면 흐름과 상태별 배치는 kick-public-plan-wireframe.html에서 확인한다. 한 파일 안에서 공개 일정 목록, 일정 상세, 공개 일정을 가져온 챗봇 조정 화면의 세 상태를 전환해 볼 수 있다.

4.1 진입

메인 카드 슬롯 아래 또는 PLAN DETAIL 진입 전후에 다음 섹션을 노출한다.

섹션 제목

수아 님과 취향이 비슷한 사람들이 고른 일정

보조 문구

연령대·성별·선호 국가·선택 테마를 기준으로 찾았어요.

4.2 공개 일정 카드

카드 한 장만 보고도 "나와 왜 비슷한지"와 "바로 쓸 수 있는지"를 판단할 수 있어야 한다.

[20대 · 여성]  공개 동의 범위만 표시
일본 선호 · 온천·휴양 테마

10월 유후인 2박 3일 · 여유로운 일정

나와 맞는 이유
같은 연령대 · 같은 성별 · 같은 선호 국가 · 같은 선택 테마

8개 장소 중 7개 정보 확인 · 42명이 저장 · 18명이 일정으로 시작

[일정 보기]  [이 일정으로 시작]

필수 표시 정보는 다음과 같다.

  • 유사 취향 근거: 연령대, 성별, 선호 국가, 선택 테마 중 실제로 일치한 항목.
  • 일정 정보: 목적지, 여행 월, 기간, 일정 속도, 주요 장소. 이 값들은 공개 일정의 내용이며, 현재 온보딩 유사성 점수에는 사용하지 않는다.
  • 나와 맞는 이유 2~4개. 단순히 "AI 추천"이라고 하지 않고 실제로 일치한 온보딩 조건만 보여준다.
  • 일정 품질 상태: 장소·운영 정보 확인 비율, 마지막 갱신일. 실제 방문 인증을 별도로 도입한 경우에만 인증 상태를 추가한다.
  • 사회적 증거: 저장 수와 일정 복제 수. 조회 수는 선택 증거가 약하므로 우선순위를 낮춘다.
  • CTA: 일정 보기보다 이 일정으로 시작을 더 강하게 둔다.

4.3 1탭 복제 후 챗봇 조정

이 일정으로 시작을 누르면 원본을 수정하지 않고 사용자 계정에 사본을 만든 뒤 기존 AI 챗봇 화면으로 이동한다. 챗봇 상단에는 가져온 공개 일정 카드를 컨텍스트로 고정하고, AI는 긴 질문을 다시 하지 않고 한 번만 묻는다.

이 일정을 수아 님 일정으로 가져왔어요.
이 일정을 수아 님에게 어떻게 맞춰드릴까요?

[11월로 변경] [조금 덜 걷게] [카페를 더 많이] [그대로 저장]

자유 입력: "온천은 유지하고 둘째 날 이동을 줄여줘"

챗봇 화면은 다음 네 요소를 함께 유지한다.

  • 가져온 공개 일정 카드 — 원본 제목·도시·기간·테마와 공개 원본임을 표시한다.
  • 빠른 수정 칩 + 자연어 입력 — 빈 채팅창만 주지 않고 대표 수정으로 시작을 돕되 자유 입력도 받는다.
  • 고정 조건 미리보기 — 원본 조건과 현재 조정 조건을 나란히 보여준다. 대화가 길어져도 화면에서 사라지지 않는다.
  • 변경안 확인 — AI가 무엇을 바꿨는지 요약한 뒤 변경 적용, 다시 조정, 원본으로 되돌리기를 제공한다.

핵심 원칙은 재생성보다 대화형 부분 수정이다. 사용자가 사회적 증거로 선택한 일정의 뼈대를 AI가 전부 다시 만들면 kick이 사라진다. 챗봇은 원본 일정 전체를 새로 만드는 것이 아니라 사용자가 말한 차이만 수정한다.

5. 현재 온보딩 기반 유사 취향 매칭 원칙

현재 온보딩에서 받는 값은 연령대·성별·선호 국가·선택 테마 네 가지다. MVP의 유사성은 이 네 값만으로 계산한다. 동행 유형·여행 목적·여행 기간·여행 월·여행 속도를 유사성 입력으로 새로 요구하지 않는다.

네 값만으로도 "나와 취향이 비슷한 사람이 선택했다"는 첫 사회적 증거를 만들 수 있다. 다만 이를 "나 같은 사람이 실제로 다녀왔다"거나 "나와 여행 조건이 완전히 같다"고 과장하지 않는다. 현재 데이터가 설명할 수 있는 범위는 인구통계 일부와 선언 취향의 유사성이다.

5.1 초기 랭킹 가설

아래 가중치는 제품 가설이며 운영 데이터로 조정한다.

매칭 신호MVP 역할이유
선택 테마핵심 매칭사용자가 선언한 여행 취향과 공개 일정의 성격을 직접 연결
선호 국가핵심 매칭한국·일본 중 사용자가 우선 탐색하려는 국가와 일정 국가를 연결
연령대보조 매칭정확한 나이가 아닌 온보딩 연령 구간의 일치 여부 사용
성별선택적 보조 매칭입력·공개에 동의한 경우에만 일치 근거로 사용하며 추론하지 않음

기본 랭킹은 선택 테마와 선호 국가를 핵심, 연령대와 성별을 보조 신호로 사용한다. 연령대·성별만 같다는 이유로 취향이 비슷하다고 단정하지 않는다.

5.2 매칭 수준과 화면 문구

일치 항목 수에 따라 유사성 문구의 강도를 다르게 한다.

매칭 수준조건화면 표현
A · 높은 유사성네 항목 모두 일치연령대·성별·선호 국가·테마가 모두 비슷해요
B · 취향 유사선택 테마·선호 국가 일치 + 연령대 또는 성별 일부 일치선호 국가와 선택 테마가 같은 사용자가 고른 일정
C · 테마 유사선택 테마만 일치같은 테마를 고른 사용자에게 인기 있는 일정
매칭 부족선택 테마도 불일치개인화 표현을 쓰지 않고 일반 추천·Lovv 큐레이션으로 노출

후보가 적다고 일치하지 않은 조건을 맞는 것처럼 표시하지 않는다. 카드에는 점수보다 같은 테마 2개, 같은 선호 국가처럼 실제 일치 근거를 우선 노출한다.

5.3 추천 점수 개념

공개 일정 점수
= 온보딩 유사도(선택 테마 + 선호 국가 + 연령대 + 성별)
+ 일정 품질(검증 상태·완성도)
+ 선택 증거(복제·저장)
+ 최신성
- 중복·오래된 정보·품질 위험 패널티

복제 수가 많다는 이유만으로 인기 일정만 계속 노출하지 않는다. 소도시 발견이라는 서비스 목적을 지키기 위해 테마·국가 유사성과 품질을 먼저 통과한 후보 안에서 사회적 증거를 사용한다.

6. 공개·개인정보 원칙

공개 일정은 신뢰 장치인 동시에 개인정보 위험이 가장 큰 지점이다.

  • 일정은 기본 비공개다. 사용자가 명시적으로 공개를 선택해야 한다.
  • 정확한 생년월일·나이는 공개하지 않고 20대, 30대처럼 구간화한다.
  • 성별은 선택 입력이며, 공개 여부도 별도로 동의받는다. 추론으로 채우지 않는다.
  • 닉네임·프로필 사진·동행자 정보·예약번호·숙소 상세 주소는 공개 일정에서 제거한다.
  • 일정 카드에 연령대·성별을 함께 표시할 때는 동일 조합의 공개 일정이 충분한 경우에만 노출한다. 표본이 적으면 같은 테마를 고른 사용자처럼 실제로 확인 가능한 더 넓은 표현으로 대체한다.
  • 원작자는 공개 해제할 수 있다. 이미 복제된 사본은 개인정보 연결을 끊고 독립 일정으로 유지한다.
  • 허위 사회적 증거를 만들지 않는다. 초기 데이터가 부족하면 가짜 사용자 일정을 생성하지 않고 Lovv 큐레이션으로 명확히 표시한다.

7. Cold Start 대응

공개 일정이 충분히 쌓이기 전에는 kick이 빈 화면이 될 수 있다. MVP에서는 다음 순서로 채운다.

  • 팀이 검수한 Lovv 큐레이션 일정을 공개한다. 사용자 일정처럼 위장하지 않는다.
  • 기존 PLAN DETAIL 저장 사용자에게 공개 전환을 선택적으로 제안한다.
  • 선호 국가·테마 조합별 최소 대표 일정을 확보한다.
  • 후보가 부족하면 성별 → 연령대 순으로 보조 조건을 완화한다. 그래도 부족하면 같은 테마를 고른 사용자에게 인기처럼 근거 수준에 맞게 표현한다.
  • 적합 일정이 없으면 낮은 품질 후보를 억지로 노출하지 않고 AI 신규 생성 흐름으로 보낸다.

초기 콘텐츠 확보 목표 수치는 데이터 분포를 확인한 뒤 정한다. 검증 전 임의 숫자를 확정 KPI로 사용하지 않는다.

8. ROI 연결

kick이 사업적으로 의미 있으려면 "좋아 보였다"가 아니라 선택·복제·외부 행동으로 이어져야 한다.

Lovv추천 소비를 계정 내 일정 자산과 재방문으로 전환세션 내 복제율, 저장률, 7/30일 재방문율
제휴 파트너일정 안의 지도·체험·입장권으로 자연스럽게 이동딥링크 CTR, 예약 가능 항목 전환율
지역 파트너어떤 취향군이 어떤 소도시 일정을 선택하는지 파악선호 국가·테마·연령대 집계별 노출→복제, 지역별 순증 방문 의향

사업 가설은 다음과 같다.

공개 일정이 AI 추천의 신뢰와 실행 속도를 높이면, 일정 복제·저장·외부 딥링크 행동이 증가하고 재방문 및 제휴 전환 가능성이 함께 커진다.

이는 아직 검증 전 가설이다. "공개 일정 = 매출"로 바로 단정하지 않고 아래 퍼널로 검증한다.

공개 일정 노출 → 상세 열람 → 일정 복제 → 조건 수정/저장
               → 지도·체험 딥링크 → 재방문

9. Kick 이벤트와 성공 기준

9.1 핵심 이벤트

kick의 대표 이벤트는 공개 일정 복제다.

이벤트의미

public_plan_open일정 상세 확인
public_plan_clone이 일정으로 시작 실행 — 핵심 kick 이벤트

9.2 우선 검증 지표

  • Kick Conversion: 공개 일정 노출 세션 중 복제까지 간 비율.
  • Time-to-Kick: 공개 일정 첫 노출부터 복제까지 걸린 시간.
  • Clone-to-Save: 복제한 일정이 저장까지 이어진 비율.
  • Action Lift: 공개 일정이 없는 흐름 대비 지도·외부 링크 클릭 상승.
  • Quality Guardrail: 복제 직후 대규모 삭제·이탈·신고 비율.
  • Supply Loop: 일정을 복제한 사용자가 여행 후 자신의 버전을 공개하는 비율.

목표 수치는 베이스라인 수집 후 정한다. 초기 문서에서 임의의 퍼센트를 성공 기준으로 확정하지 않는다.

10. MVP 범위

P0 — kick 성립에 필수

  • 저장 일정의 공개/비공개 선택과 익명화.
  • 연령대·성별·선호 국가·선택 테마 매칭 메타데이터.
  • 유사 취향 공개 일정 카드와 실제 일치 항목 기반 나와 맞는 이유 표시.
  • 일정 상세와 이 일정으로 시작 사본 생성.
  • 공개 일정 컨텍스트가 붙은 챗봇 진입과 3~4개 빠른 수정 칩·자연어 입력.
  • 원본 조건·현재 조정 조건의 고정 미리보기와 변경 적용·재조정·원본 복원.
  • 노출→열람→복제→조정→저장 이벤트 계측.

P1 — kick 강화

  • 일정 장소의 출처·운영 정보 확인 상태. 화면에는 일정 장소 9곳 중 8곳 출처·운영 정보 확인처럼 분모·분자를 함께 표시한다. 실제 방문 인증을 도입하기 전에는 다녀온 일정으로 표현하지 않는다.
  • 저장·복제 수 기반 사회적 증거.
  • 세션 행동과 기존 저장 이력을 반영한 개인화 랭킹.
  • 지도·체험·입장권 딥링크 성과 연결.

P2 — 검증 후 확장

  • 팔로우, 댓글, 좋아요 피드 등 커뮤니티 기능.
  • 공개 일정 작성자 보상·배지.
  • 지자체 큐레이션과 캠페인 성과 대시보드.

MVP에서 커뮤니티 피드를 먼저 만들지 않는다. kick의 본질은 관계망이 아니라 비슷한 일정 발견과 즉시 복제다.

11. 개발 설계 고도화

이 장은 화면 기획을 실제 구현으로 내릴 때 필요한 데이터·API·검증·운영 계약을 정의한다. 핵심 원칙은 공개 원본은 불변 스냅샷으로 보존하고, 사용자의 사본에는 검증된 변경 patch만 적용하는 것이다.

11.1 공개 일정 생명주기와 원본·사본 관계

공개 일정은 사용자의 현재 편집본을 그대로 노출하지 않고, 공개한 시점의 불변 스냅샷을 만든다. 공개 후 원작자가 자신의 일정을 수정해도 기존 공개 버전과 이미 생성된 사본이 예고 없이 바뀌지 않아야 한다.

상태의미허용 동작
DRAFT사용자 개인 일정, 공개 전편집·삭제·공개 요청
PUBLISHED공개 스냅샷 노출 중열람·복제·신고·새 버전 발행
UNPUBLISHED원작자가 공개 해제신규 노출·복제 중지, 기존 사본 유지
ARCHIVED운영 정책·오래된 정보로 보관검색 제외, 감사·통계용 최소 메타데이터 유지

복제 시 다음 계보(lineage)를 남긴다.

source_itinerary_id        원작자의 일정 ID
source_public_snapshot_id  사용자가 실제로 본 공개 스냅샷 ID
source_version             복제 기준 버전
cloned_itinerary_id        사용자 계정에 생성된 독립 사본 ID
cloned_at                  복제 완료 시각
  • 공개 일정 수정은 기존 스냅샷 덮어쓰기가 아니라 새 source_version 발행으로 처리한다.
  • 원작자가 공개 해제·탈퇴해도 이미 복제된 일정은 개인정보 연결을 제거한 독립 사본으로 유지한다.
  • public_plan_clone은 사본 DB 저장이 성공한 뒤에만 기록한다. 버튼 클릭만으로 복제 수를 올리지 않는다.
  • 동일 사용자가 같은 공개 버전을 반복 복제한 수치는 사회적 증거 집계에서 중복 제거한다.

11.3 유사 취향 조회와 privacy fallback

조회는 아래 순서로 완화한다. 화면 문구도 실제 매칭 단계에 맞춰 함께 낮춘다.

1. 테마 + 국가 + 연령대 + 성별
2. 테마 + 국가 + 연령대
3. 테마 + 국가
4. 같은 테마
5. Lovv 큐레이션
  • 선택 테마는 최소 1개 이상 일치해야 나와 취향이 비슷한 표현을 사용할 수 있다.
  • 성별이 비공개·미입력이면 매칭 점수와 화면 근거에서 모두 제외한다.
  • 특정 조합의 공개 일정 수가 너무 적으면 연령대·성별을 표시하지 않는다. 최소 집단 크기는 코드에 하드코딩하지 않고 MIN_PUBLIC_COHORT_SIZE 운영 설정으로 관리하며, 실제 데이터 분포를 본 뒤 확정한다.
  • fallback 단계, 실제 일치 필드, 제외된 민감 필드를 응답에 함께 담아 UI가 과장 없이 카피를 선택하게 한다.
  • 인기 일정만 고착되지 않도록 유사도·장소 품질을 통과한 후보 안에서 복제·저장 수를 보조 점수로 사용한다.

11.4 챗봇 변경 patch 계약

LLM이 복제 일정 전체를 다시 생성하지 않는다. 챗봇은 사용자 발화를 허용된 변경 명령의 배열로 구조화하고, 서버가 결정적으로 검증한 뒤 미리보기와 적용을 담당한다.

{
  "baseItineraryVersion": 3,
  "idempotencyKey": "adjust-req-uuid",
  "operations": [
    {
      "type": "change_month",
      "from": 10,
      "to": 11
    },
    {
      "type": "reduce_walking",
      "targetDay": 2
    }
  ]
}

초기 허용 operation은 범위를 좁힌다.

operation역할검증
change_month여행 월 변경월별 날씨·축제·운영 정보 재검증
reduce_walking이동 강도 완화장소 간 이동시간·하루 총 동선 상한 검사
add_category카페·온천 등 장소 유형 추가출처 확인 장소만 후보 허용
replace_place특정 장소 교체국가·도시·테마·영업 상태 검증
remove_item일정 항목 제거일차별 최소 일정 수와 연결 동선 재검사
restore_original공개 원본 상태로 복원동일 source_version 스냅샷에서 복구

처리 순서는 다음과 같다.

사용자 발화
→ LLM이 patch 초안 생성
→ schema·권한·장소·동선·버전 충돌 검증
→ 변경 전/후 diff와 조건 미리보기 반환
→ 사용자 "변경 적용"
→ 트랜잭션 적용·새 사본 버전 저장
  • baseItineraryVersion이 현재 사본 버전과 다르면 적용하지 않고 최신 상태로 다시 preview한다.
  • 같은 idempotencyKey 요청은 한 번만 적용해 중복 장소·중복 저장을 막는다.
  • 검증 실패 시 원본 사본은 변경하지 않고, 실패한 조건과 가능한 대안을 챗봇으로 돌려준다.
  • 다시 조정은 직전 patch를 적용하지 않은 채 새 preview를 만들고, 원본으로 되돌리기는 공개 스냅샷 기준 복원 patch를 사용한다.

11.5 장소 출처·운영 정보 확인 기준

일정 장소 한 곳은 다음 필수 기준을 모두 통과해야 확인으로 집계한다.

기준확인 내용
장소 식별내부 place_id와 국가·도시·장소명이 연결됨
위치주소 또는 좌표가 존재하고 대상 도시 범위와 일치
출처공식·공공·신뢰 가능한 원천의 source_url 존재
운영 상태폐업·운영 중단·장기 휴업으로 확인되지 않음
최신성verified_at이 출처 유형별 유효기간 안에 있음

화면의 일정 장소 9곳 중 8곳 출처·운영 정보 확인은 다음처럼 계산한다.

분자 = 필수 기준을 모두 통과한 실제 장소 항목 수
분모 = 일정에 포함된 실제 장소 항목 수
제외 = 이동 구간, 자유 식사, 설명용 항목, 장소 ID가 없는 placeholder
  • 영업시간·입장료처럼 자주 변하는 정보는 장소 존재 여부보다 짧은 검증 유효기간을 적용한다.
  • 검증이 만료되면 자동으로 stale 처리하고 확인 분자에서 제외한다.
  • 확인되지 않은 장소가 있어도 일정을 숨길지는 운영 정책으로 분리한다. 단, 챗봇이 새 장소를 추가할 때는 확인된 후보만 사용한다.

API책임LLM 호출

clone API는 LLM을 기다리지 않는다. 사본과 챗봇 컨텍스트를 먼저 반환해 화면을 즉시 열고, LLM 비용은 사용자가 실제 조정 요청을 보낼 때만 발생시킨다.

11.7 집계·이벤트 무결성

  • 저장·복제 수는 사용자·공개 스냅샷 조합별 unique 집계로 계산한다.
  • 동일 요청 재시도는 event_id·idempotencyKey로 중복 제거한다.
  • 탈퇴·차단·봇·운영 테스트 계정은 공개 사회적 증거 집계에서 제외한다.
  • public_plan_clone은 사본 생성 성공, cloned_plan_adjust는 검증된 preview 생성, cloned_plan_save는 적용·저장 트랜잭션 성공을 기준으로 기록한다.
  • 집계 수치가 너무 작거나 품질 기준을 충족하지 않으면 숫자를 숨기고 최근 저장된 일정처럼 정성 표현으로 대체한다.
  • 운영 지표에는 복제 API 실패율, patch 검증 실패율, 원본 복원율, 복제 직후 이탈률, 미리보기와 저장 결과 불일치율을 추가한다.

11.8 성능·비용·실패 대응

동기 경로: 공개 일정 조회 → 사본 생성 → 챗봇 화면·조건 미리보기 표시
추론 경로: 사용자 수정 입력 → patch 생성 → 검증 → 변경 preview
적용 경로: 사용자 승인 → 트랜잭션 적용 → 새 사본 버전 저장
  • 공개 일정 목록과 사회적 증거는 사전 집계·캐시해 첫 화면에서 LLM을 호출하지 않는다.
  • 챗봇 진입 시 원본 스냅샷과 현재 조건 미리보기를 먼저 표시한다. AI 응답 대기 때문에 화면 진입을 막지 않는다.
  • 자연어 해석에는 필요한 일정 요약과 변경 가능 필드만 전달하고 전체 대화·원문 장소 데이터를 반복 전송하지 않는다.
  • patch 생성 실패 시 빠른 수정 칩 기반 결정적 operation으로 fallback한다.
  • 장소·동선 검증 실패 시 자동 완화하지 않고 어떤 조건을 적용할 수 없는지와 대안을 보여준다.
  • 적용 중 충돌·오류가 나면 기존 사본 버전을 유지하고 재시도 가능한 preview_id를 반환한다.
  • P50·P95 응답시간, 요청당 모델 비용, patch당 validation retry 수를 계측하되 목표 수치는 베이스라인 측정 후 확정한다.

11.9 공개 일정 운영·신고

  • 공개 전 예약번호·숙소 상세 주소·동행자 이름·자유 메모의 개인정보를 탐지·제거한다.
  • 원작자에게 실제 공개되는 연령대·성별·국가·테마 범위를 미리 보여주고 항목별 동의를 받는다.
  • 사용자는 개인정보 노출·허위 장소·유해 콘텐츠·광고성 일정으로 신고할 수 있다.
  • 신고 임계값 또는 운영 검토로 PUBLISHED → UNPUBLISHED/ARCHIVED 전환할 수 있다.
  • 운영자가 노출을 중단해도 기존 사용자 사본을 임의로 삭제하지 않고 원본 연결만 익명화한다.

12. 실험 설계

실험 A — 공개 일정의 존재 효과

  • 대조군: 기존 AI 추천 → PLAN DETAIL.
  • 실험군: AI 추천 + 온보딩 유사 취향 공개 일정 2~3개 + 1탭 복제.
  • 비교: 저장률, 첫 저장까지 시간, 외부 링크 클릭, 재방문.

실험 B — 유사성 설명 효과

  • 대조군: 인기 공개 일정.
  • 실험군: 같은 선호 국가·같은 선택 테마·같은 연령대처럼 실제 온보딩 일치 이유를 표시한 일정.
  • 비교: 카드 상세 열람률과 복제율.

실험 C — AI 재생성 vs 챗봇 부분 수정

  • 대조군: 복제 후 일정 전체 재생성.
  • 실험군: 공개 일정 카드를 챗봇 컨텍스트로 가져와 뼈대를 유지하고, 사용자가 말한 변경 조건만 부분 수정.
  • 비교: 수정 완료율, 저장률, 원본 유지 비율, 이탈률.

13. 발표에서 보여줄 한 장

최종발표에서는 기능 목록보다 다음 Before/After가 kick을 가장 빠르게 설명한다.

BeforeKickAfter
AI가 낯선 소도시를 추천"같은 20대·여성·일본 선호·온천 테마 사용자가 고른 일정"이 일정으로 시작 → 내 일정으로 복제·수정·저장

발표 핵심 문장:

"Lovv는 AI가 여행지를 추천하는 데서 끝나지 않습니다. 나와 연령대·성별·선호 국가·테마가 비슷한 사용자가 고른 공개 일정을 보여주고, 한 번에 내 일정으로 바꿉니다."

14. 의사결정 요약

  • kick의 대표 기능은 온보딩 유사 취향 공개 일정 + 1탭 복제로 잡는다.
  • MVP 유사성은 현재 온보딩 값인 연령대·성별·선호 국가·선택 테마만 사용한다.
  • 공개 일정은 기존 Aha Moment 흐름 위에 사회적 증거와 실행 단축을 더한다.
  • 핵심 제품 이벤트는 조회나 좋아요가 아니라 public_plan_clone이다.
  • ROI는 복제→저장→외부 링크→재방문 퍼널로 검증한다.
  • 초기 공급 부족은 가짜 사용자 일정이 아니라 명시적인 Lovv 큐레이션으로 해결한다.
  • MVP는 피드·댓글보다 공개 동의, 매칭, 복제, 공개 일정 컨텍스트 챗봇·조건 미리보기·부분 수정, 계측에 집중한다.
  • 공개 일정은 불변 스냅샷과 원본→사본 계보로 관리하고, 공개 해제 후에도 기존 사본은 익명화해 유지한다.
  • 챗봇은 일정 전체를 재생성하지 않고 허용된 patch만 만들며, 서버가 검증·diff·버전 충돌·rollback을 담당한다.
  • clone은 LLM 없이 즉시 처리하고, 모델 호출은 사용자가 실제 조정을 요청할 때만 수행한다.

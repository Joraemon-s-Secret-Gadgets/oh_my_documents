# 자연어 처리 흐름

## **1. 기본 전제**

우리 서비스는 온보딩 설문 기반 추천과 챗봇 자연어 입력을 함께 사용한다.

기본 입력값:

- 선호 여행 테마 1-3개
- 여행국: 한국/일본
- 여행일수
- 축제 포함 여부
- 사용자 위치 또는 출발지

챗봇에서는 사용자가 추가 조건을 자연어로 말할 수 있다.

예:

> 자연이 좋고 맛집도 있었으면 좋겠어요. 조용한 분위기면 더 좋아요.
> 

이 자연어는 그대로 RAG에 넘기지 않고, 먼저 **판별 agent가 정형 조건으로 변환**한다.

---

## **2. 전체 자연어 처리 흐름**

```
사용자 자연어 입력
→ 판별 agent가 의미 분석
→ 정형 조건 추출
→ required / soft_preferences / unsupported로 분류
→ required는 DB/그래프DB 필터링
→ soft_preferences는 장소별 임베딩 유사도 조회와 랭킹에 반영
→ unsupported는 예외 안내 또는 대체 처리
→ 장소 검색 결과를 city_id 기준으로 도시 단위 집계
→ 도시 랭킹
→ 소도시 추천
→ 일정 생성
→ 숙박 검색 링크 제공
```

핵심은 **자연어를 우리 데이터에서 검색 가능한 조건으로 번역한 뒤 RAG와 랭킹에 넘기는 것**이다.

---

## **3. 정형 조건 설정 근거**

정형 조건은 반드시 우리 데이터 컬럼 또는 파생 가능한 feature를 근거로 설정해야 한다.

| 기준 | 데이터 근거 | 활용 |
| --- | --- | --- |
| 테마 | `theme` | 자연·트레킹, 미식·노포 등 |
| 장소 유형 | `content_type_id` | 관광지, 문화시설, 음식점, 쇼핑 등 |
| 도시 | `city_id` | 도시 단위 집계 |
| 위치 | `latitude`, `longitude`, `address` | 거리, 동선 계산 |
| 설명/키워드 | `name`, `description` | 조용한, 전망, 산책, 시장 등 유사도 검색 |
| 운영 정보 | `opening_hours`, `opening_period` | 일정 배치 신뢰도 |
| 데이터 신뢰도 | `data_confidence` | 검증 필요 여부 |

즉, LLM이 임의로 새로운 추천 기준을 만드는 것이 아니라, 우리가 가진 데이터로 매칭 가능한 범위 안에서만 정형화한다.

---

## **4. RAG로 넘길 조건과 넘기지 않을 조건: 3단계**

### **1단계: required**

명확하게 DB에서 필터링 가능한 조건이다.

RAG에 바로 던지기보다 먼저 후보군을 줄이는 데 사용한다.

예:

```json
{
  "country": "KR",
  "duration_days": 2,
  "required_themes": ["자연·트레킹", "미식·노포"],
  "festival_included": false
}
```

활용:

- 여행국 필터링
- 여행일수에 맞는 일정 슬롯 구성
- 선택 테마 전체 충족 여부 확인
- 축제 포함 여부 분기
- 사용자 위치 기반 거리 필터링

---

### **2단계: soft_preferences**

직접 필터링하기는 어렵지만, 장소 설명과 임베딩 유사도 검색으로 반영 가능한 조건이다.

후보 탈락 조건이 아니라 **랭킹 가산점**으로 사용한다.

예:

```json
{
  "soft_preferences": ["quiet", "walkable", "scenic_view", "shopping_nearby"]
}
```

매핑 예시:

| 자연어 표현 | 정형 조건 |
| --- | --- |
| 조용한, 한적한 | `quiet` |
| 활기찬, 북적이는 | `lively` |
| 전망 좋은, 뷰 좋은 | `scenic_view` |
| 산책하기 좋은 | `walkable` |
| 로컬 느낌 | `local` |
| 여유로운 | `slow_pace` |
| 쇼핑 가능한 곳 근처 | `shopping_nearby` |

soft preference는 설명에 특정 단어가 없더라도 유사 장소가 있을 수 있으므로, 필터링보다 유사도 점수와 랭킹에 반영하는 것이 안전하다.

---

### **3단계: unsupported**

현재 추천 기준에서 직접 검증하기 어렵거나 검색 조건으로 쓰면 위험한 조건이다.

RAG 검색 조건으로 넘기지 않고, 사용자에게 짧게 안내하거나 대체 가능한 조건만 반영한다.

예:

- 숙박 시설 자체의 품질
- 숙소 전망
- 숙소 가격
- 예약 가능 여부
- 실시간 혼잡도
- 의류 재고 여부
- 실시간 운영 여부

숙박 관련 요청은 일정 추천 기준에 직접 넣지 않고, **추천된 소도시 기준 숙박 검색 링크 제공**으로 처리한다.

답변 예시:

> 숙박 조건은 직접 판단하기 어려워요. 대신 여행 동선과 주변 장소를 고려하고, 마지막에 숙박 검색 링크를 제공할게요.
> 

---

## **5. 분위기 조건 처리**

`조용한 분위기`, `활기찬 분위기` 같은 조건은 온보딩 필수 설문으로 빼기보다 챗봇 자연어에서 선택적으로 받는 것이 좋다.

이유:

- 온보딩 입력값이 이미 많음
- 추가 설문이 늘면 사용자 피로도가 커짐
- 분위기는 이번 여행 맥락에서 달라질 수 있음

챗봇 진입 문구 예:

> 원하는 분위기나 꼭 포함하고 싶은 조건이 있으면 자유롭게 말해주세요.
> 

---

## **6. 온보딩 테마와 자연어 입력의 관계**

온보딩 테마는 **기본 취향**, 자연어 입력은 **이번 여행의 현재 의도**로 본다.

```
온보딩 테마 = 장기 선호
자연어 입력 = 현재 여행 의도
```

기본 원칙:

```
자연어 입력이 명확하면 자연어를 우선한다.
자연어가 분위기만 말하면 온보딩 테마를 유지한다.
```

다만 온보딩 테마를 버리는 것이 아니라, 자연어와 조합해서 최종 추천에 사용할 `active_required_themes`를 만든다.

---

## **7. 온보딩 vs 자연어 상황별 처리**

### **A. 자연어에 테마가 없고 분위기만 있는 경우**

온보딩 테마를 유지하고, 자연어는 soft preference로 반영한다.

```json
{
  "active_required_themes": ["자연·트레킹", "미식·노포"],
  "soft_preferences": ["quiet", "slow_pace"]
}
```

---

### **B. 자연어에 온보딩과 같은 테마가 다시 등장하는 경우**

해당 테마의 중요도를 강화한다.

예:

> 자연이 제일 중요하고 맛집도 있으면 좋겠어요.
> 

```json
{
  "active_required_themes": ["자연·트레킹", "미식·노포"],
  "theme_priority": {
    "자연·트레킹": "high",
    "미식·노포": "normal"
  }
}
```

---

### **C. 자연어에 새로운 테마가 추가된 경우**

새 테마를 현재 여행 조건에 추가한다.

```json
{
  "active_required_themes": ["자연·트레킹", "미식·노포", "역사·전통"],
  "theme_sources": {
    "자연·트레킹": "onboarding",
    "미식·노포": "onboarding",
    "역사·전통": "chat"
  }
}
```

---

### **D. 테마가 3개를 초과하는 경우**

자연어 테마를 우선하고, 일부 온보딩 테마는 보조 선호로 낮춘다.

처리 원칙:

- “이번엔”, “위주로”, “중심으로”가 나오면 자연어 테마 우선
- active theme은 최대 3개 정도 유지
- 밀려난 온보딩 테마는 `backup_themes` 또는 랭킹 보조로 사용

---

### **E. 자연어가 온보딩 테마를 제외하는 경우**

명시적으로 제외된 테마는 필수 조건에서 제거하거나 우선순위를 낮춘다.

예:

> 이번엔 맛집 말고 조용한 자연 여행을 하고 싶어요.
> 

```json
{
  "active_required_themes": ["자연·트레킹"],
  "excluded_themes": ["미식·노포"],
  "soft_preferences": ["quiet"]
}
```

---

### **F. 자연어가 온보딩과 완전히 다른 테마를 강하게 요구하는 경우**

자연어 테마를 현재 여행의 핵심 조건으로 사용하고, 온보딩은 보조 선호로 낮춘다.

예:

> 이번 여행은 예술적인 곳 중심으로 추천해주세요.
> 

```json
{
  "active_required_themes": ["예술·감성"],
  "backup_themes": ["자연·트레킹", "미식·노포"]
}
```

---

## **8. 우선순위 표현 처리**

사용자가 자연어로 우선순위를 말하면 랭킹에 반영한다.

| 표현 | 처리 |
| --- | --- |
| 제일, 가장, 꼭, 무조건 | 높은 우선순위 |
| 중심으로, 위주로 | 현재 여행 핵심 테마 |
| 도, 함께, 있으면 좋겠어요 | 보조 테마 또는 soft preference |
| 말고, 빼고, 제외 | 제외 조건 |
| 이번엔, 이번 여행은 | 자연어 의도 우선 |

예:

> 바다가 제일 중요하고, 맛집도 있으면 좋아요.
> 

```json
{
  "theme_priority": {
    "바다·해안": "high",
    "미식·노포": "normal"
  }
}
```

---

## **9. 선택 테마 전체 충족 원칙**

사용자가 선택한 테마는 단순 가산점이 아니라, 기본적으로 모두 충족해야 하는 조건으로 본다.

예를 들어 사용자가 `자연·트레킹`, `미식·노포`, `예술·감성`을 선택했다면 해당 도시는 최소한 다음 조건을 만족해야 한다.

- 자연·트레킹 장소 1개 이상
- 미식·노포 장소 1개 이상
- 예술·감성 장소 1개 이상

다만 후보가 지나치게 줄어들 수 있으므로 기준을 나눈다.

- 선택 테마 1개: 해당 테마가 도시의 핵심 특성인지 확인
- 선택 테마 2개: 두 테마 모두 최소 충족
- 선택 테마 3개: 세 테마 모두 최소 충족하되, 희소 테마는 기준 완화

---

## **10. soft_preferences 검색 방식**

`soft_preferences`는 DB 필터가 아니라 벡터 유사도 조회와 랭킹에 사용한다.

```
soft_preferences
→ 검색 쿼리 생성
→ 쿼리 임베딩
→ 장소별 임베딩 벡터와 비교
→ 유사도 점수 산출
→ 도시 랭킹에 반영
```

예:

```json
{
  "soft_preferences": ["quiet", "walkable", "scenic_view"]
}
```

검색 쿼리:

```
조용한 한적한 여유로운 산책하기 좋은 전망 좋은 자연 여행지
```

---

## **11. 임베딩 대상**

현재 프로젝트 단계에서는 **장소별 임베딩만 사용하는 것이 적절하다.**

도시별 프로필 임베딩은 제외하는 이유:

- 개발 범위가 커짐
- 벡터 생성/관리 비용 증가
- 검색 구조가 복잡해짐
- 장소 검색 결과를 도시 단위로 집계하면 도시 추천이 가능함
- 장소 검색 결과를 그대로 일정 생성에 활용 가능함

최종 비교 구조:

```
사용자 테마별 쿼리 벡터 ↔ 장소별 임베딩 벡터
사용자 soft preference 쿼리 벡터 ↔ 장소별 임베딩 벡터
```

---

## **12. 장소별 임베딩 텍스트**

장소별 임베딩 텍스트는 장소의 의미를 잘 담도록 구성한다.

예:

```
장소명: 고석사
도시: 포항
테마: 역사·전통
장소유형: 관광지
주소: 경상북도 포항시 남구 장기면
설명: 묘봉산에 있는 사찰, 불교문화, 역사 유적, 산 주변 경관
파생태그: 조용한, 전통적인, 산책, 역사, 사찰
```

임베딩에 넣기 좋은 것:

- 장소명
- 도시명
- 테마
- 장소 유형
- 주소 일부
- 설명
- 파생 태그

구조화 계산에 쓰는 것:

- 위도/경도
- 운영시간
- 여행일수
- 거리
- `content_type_id` 기반 일정 균형

---

## **13. 사용자 쿼리는 테마별로 여러 개 생성**

사용자 테마가 여러 개인 경우 하나의 쿼리로 합치면 특정 테마에 치우칠 수 있다.

따라서 테마별 쿼리와 soft preference 쿼리를 따로 만든다.

예:

> 자연이 좋고 맛집도 있었으면 좋겠어요. 조용한 분위기면 더 좋아요.
> 

```
쿼리 1: 자연·트레킹 숲 산책 트레킹 계곡 한적한 자연
쿼리 2: 미식·노포 지역 맛집 로컬 음식 오래된 식당
쿼리 3: 조용한 한적한 여유로운 휴식 분위기
```

비교:

```
자연·트레킹 쿼리 벡터 ↔ 장소별 임베딩 벡터
미식·노포 쿼리 벡터 ↔ 장소별 임베딩 벡터
soft preference 쿼리 벡터 ↔ 장소별 임베딩 벡터
```

이후 `city_id` 기준으로 검색 결과를 집계한다.

## **14. 숙박 관련 처리**

숙박 관련 세부 조건은 일정 추천의 직접 기준으로 사용하지 않는다.

처리 방식:

- 숙박 시설 자체를 추천 점수에 넣지 않음
- 숙소 전망, 가격, 예약 가능 여부는 unsupported로 분류
- “쇼핑 근처”, “전망 좋은 장소”처럼 일정 장소로 반영 가능한 조건만 soft preference로 활용
- 당일치기가 아닌 경우에는 일정 추천 마지막에 숙박 링크 반드시 제공

예:

> 근처에 옷을 살 수 있는 쇼핑 가능한 곳이면서 뷰가 좋은 숙박 원해요.
> 

```json
{
  "supported_conditions": ["쇼핑 가능한 장소", "전망 좋은 관광지"],
  "unsupported_conditions": ["숙박 추천", "숙소 전망 검증"],
  "ranking_preferences": ["shopping_nearby", "scenic_view"],
}
```

답변 예시:

> 숙박 조건은 직접 판단하기 어려워요. 대신 쇼핑 장소와 전망 좋은 장소를 포함한 일정을 추천하고, 마지막에 숙박 검색 링크를 제공할게요.
>

## **15. 축제 처리**

축제는 기본 테마 추천에 포함하지 않는다.

처리 방식:

- 사용자에게 축제 포함 여부를 별도로 받음
- 포함 원함: 여행 날짜에 맞는 축제를 일정에 우선 반영
- 포함 원하지 않음: 일반 관광지/음식점/문화시설 중심으로 구성
- 축제는 일반 장소와 섞지 않고 조건부 반영 요소로 둠

즉, 축제는 필수 추천 근거가 아니라 **사용자 선택에 따라 반영되는 일정 요소**다.

## **16. 판별 agent의 역할 및 출력 예시**

판별 agent는 추천을 직접 수행하지 않고, 자연어 입력을 추천 가능한 구조로 정리한다.

역할:

- 자연어에서 테마 추출
- 분위기/세부 선호 추출
- 제외 조건 인식
- 우선순위 표현 인식
- required / soft_preferences / unsupported 분류
- 온보딩 테마와 자연어 조건을 합쳐 active theme 생성
- RAG 검색용 쿼리 생성
- 예외 안내 문장 생성

출력 예시:

```json
{
  "can_recommend": true,
  "onboarding_themes": ["자연·트레킹", "미식·노포"],
  "chat_extracted_themes": [],
  "active_required_themes": ["자연·트레킹", "미식·노포"],
  "theme_priority": {
    "자연·트레킹": "normal",
    "미식·노포": "normal"
  },
  "soft_preferences": ["quiet"],
  "unsupported_conditions": [],
  "excluded_themes": [],
  "backup_themes": [],
  "theme_queries": {
    "자연·트레킹": "숲 산책 트레킹 계곡 자연 한적한",
    "미식·노포": "지역 맛집 로컬 음식 오래된 식당 노포"
  },
  "soft_preference_query": "조용한 한적한 여유로운 휴식 분위기",
  "accommodation_link_required": false,
  "user_notice": null
}
```

## **17. 우리가 정해야 하는 것과 LLM이 판단하는 것**

우리가 정해야 하는 것:

| 항목 | 설명 |
| --- | --- |
| 테마 목록 | 자연·트레킹, 바다·해안, 역사·전통, 미식·노포 등 |
| 장소 유형 목록 | `content_type_id` 기반 관광지, 문화시설, 음식점, 쇼핑 등 |
| soft preference 목록 | quiet, lively, scenic_view, walkable 등 |
| unsupported 조건 목록 | 숙박 세부 조건, 실시간 정보 등 |
| 온보딩 vs 자연어 우선순위 규칙 | 자연어가 현재 여행 의도면 우선 |
| active theme 최대 개수 | 후보 부족 방지를 위해 최대 3개 권장 |
| 추천 점수 계산 방식 | 필터, 유사도, 보정, 랭킹 기준 |
| 축제 처리 방식 | 기본 테마와 분리, 사용자 선택 시 반영 |
| 숙박 처리 방식 | 일정 후 검색 링크 제공 |

LLM이 판단하는 것:

| 항목 | 설명 |
| --- | --- |
| 자연어에서 테마 추출 | “자연이 좋아요” → 자연·트레킹 |
| 분위기 추출 | “조용한 곳” → quiet |
| 우선순위 인식 | “바다가 제일 중요” → 바다·해안 high |
| 제외 표현 인식 | “맛집 말고” → 미식·노포 제외 |
| 조건 가능 여부 분류 | supported / soft / unsupported |
| 검색 쿼리 생성 | 정형 조건을 벡터 검색용 문장으로 변환 |
| 사용자 안내 문장 생성 | 반영 어려운 조건을 짧게 설명 |

핵심:

> 우리는 판단 기준과 허용 범위를 설계하고, LLM은 사용자 문장을 그 기준 안에 맞게 분류한다.
>

## **18. 최종 추천 구조**

```
1. 온보딩 입력 수집
   - 테마 1-3개, 여행국, 여행일수, 축제 포함 여부

2. 챗봇 자연어 입력 수집
   - 분위기, 세부 조건, 제외 조건, 우선순위 등

3. 판별 agent 정형화
   - required / soft_preferences / unsupported 분류
   - active_required_themes 생성
   - 테마별 검색 쿼리 생성

4. 후보 도시 필터링
   - 국가, 거리, 선택 테마 전체 충족, 일정 구성 가능성

5. 장소별 임베딩 유사도 검색
   - 테마별 쿼리와 장소 임베딩 비교
   - soft preference 쿼리와 장소 임베딩 비교

6. 도시 단위 집계
   - city_id 기준으로 장소 검색 결과 집계
   - 테마별 매칭 장소 수
   - 테마별 최고 유사도
   - soft preference 적합도 계산

7. 랭킹 보정
   - 희소 테마 보정
   - 도시 규모/데이터 과다 보정
   - 콘텐츠 타입 균형
   - 접근성
   - 축제 포함 선택 시 조건부 반영

8. 소도시 추천
   - 추천 근거와 함께 제공

9. 일정 생성
   - 선택 테마 전체 반영
   - 장소 유형 균형
   - 동선 고려
   - 운영정보 참고

10. 숙박 검색 링크 제공
   - 추천된 소도시 기준으로 마지막에 연결
```

## **19. 예외 안내 문장**

예외는 100자 내외로 짧게 처리한다.

일반 예외:

> 해당 조건은 직접 확인이 어려워요. 대신 반영 가능한 장소 특성과 동선을 기준으로 추천드릴게요.
> 

숙박 관련:

> 숙박 조건은 직접 판단하기 어려워요. 대신 일정 추천 후 해당 소도시 숙박 검색 링크를 제공할게요.

## **최종 한 문장 정리**

챗봇 자연어 입력은 판별 agent가 우리 데이터에서 검색 가능한 정형 조건으로 변환하고, 온보딩 테마는 기본 취향, 자연어는 현재 여행 의도로 해석한다. `required`는 필터링에, `soft_preferences`는 장소별 임베딩 유사도 검색과 랭킹에, `unsupported`는 예외 안내나 대체 처리에 사용한다. 최종적으로 장소 검색 결과를 도시 단위로 집계해 소도시 추천과 일정 생성으로 연결하고, 숙박은 일정 추천 이후 검색 링크 제공으로 분리한다.

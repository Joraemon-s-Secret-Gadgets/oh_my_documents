# 추천 근거 및 흐름

## **전체 추천 흐름 근거 논리**

### **1. 사용자 입력 수집**

사용자는 온보딩 단계에서 선호 여행 테마를 1-3개 선택한다.

예시 테마:

- 자연·트레킹
- 바다·해안
- 역사·전통
- 미식·노포
- 온천·휴양
- 예술·감성

추가로 추천 전에 축제 포함 여부를 묻는다.

- 축제 포함 원함
- 축제 포함 원하지 않음

축제는 기본 테마 추천에 섞지 않고, 사용자가 원할 때만 일정 구성에 반영한다.

사용자는 챗봇 진입 후 자연어로 추가 조건을 입력할 수도 있다.

예:

> 자연이 좋고 맛집도 있었으면 좋겠어요. 조용한 분위기면 더 좋아요.
> 

또는:

> 이번엔 역사적인 곳 위주로 가고 싶어요.
> 

자연어 입력은 그대로 검색에 넘기지 않고, 먼저 판별 agent가 추천에 사용할 수 있는 정형 조건으로 변환한다.

---

### **1-1. 자연어 입력 처리 방식**

자연어 입력이 있을 경우, 판별 agent는 사용자 문장을 다음과 같이 분류한다.

- `active_required_themes`
- `soft_preferences`
- `cleaned_raw_query`
- `unsupported_conditions`
- `excluded_themes`
- `backup_themes`

예시:

```json
{
  "onboarding_themes": ["자연·트레킹", "미식·노포"],
  "chat_extracted_themes": [],
  "active_required_themes": ["자연·트레킹", "미식·노포"],
  "soft_preferences": ["quiet"],
  "cleaned_raw_query": "자연이 좋고 맛집도 있었으면 좋겠어요. 조용한 분위기면 더 좋아요.",
  "unsupported_conditions": [],
  "excluded_themes": [],
  "backup_themes": []
}
```

`chat_extracted_themes`는 자연어에서 추가로 뽑힌 테마이고, `excluded_themes`는 사용자가 이번 추천에서 빼달라고 한 테마이다.

두 값을 온보딩 테마와 조합해 최종 추천 조건인 `active_required_themes`를 만든다.

온보딩 테마는 사용자의 기본 취향이고, 자연어 입력은 이번 여행의 현재 의도로 본다.

```
온보딩 테마 = 장기 선호
자연어 입력 = 현재 여행 의도
```

따라서 자연어가 명확한 현재 여행 의도를 담고 있다면 자연어를 우선한다.

다만 온보딩 테마를 버리는 것이 아니라, 자연어와 조합해 최종 추천에 사용할 `active_required_themes`를 만든다.

상황별 처리 기준:

- 자연어가 분위기만 말함 → 온보딩 테마 유지 + soft preference 반영
- 자연어가 온보딩과 같은 테마를 다시 말함 → 해당 테마 우선순위 강화
- 자연어가 새 테마를 말함 → active theme에 추가
- 자연어가 기존 테마를 제외함 → 해당 테마 제외 또는 우선순위 하향
- 자연어가 완전히 다른 테마를 강하게 요구함 → 자연어 테마를 현재 여행 핵심 조건으로 사용
- active theme이 3개를 초과함 → 자연어 테마 우선, 일부 온보딩 테마는 backup으로 이동

### **1-2. required / soft_preferences / unsupported 분류**

자연어에서 추출한 조건은 세 단계로 나눈다.

### **required**

명확하게 필터링 가능한 조건이다.

예:

- 여행국
- 여행일수
- 사용자 위치
- 축제 포함 여부
- `active_required_themes`

이 조건들은 DB/그래프DB에서 후보 도시를 줄이는 데 사용한다.

### **soft_preferences**

직접 필터링하기는 어렵지만, 장소 설명과 임베딩 유사도 검색으로 반영 가능한 조건이다.

후보 탈락 조건이 아니라 랭킹 가산점으로 사용한다.

예:

- 조용한
- 활기찬
- 전망 좋은
- 산책하기 좋은
- 로컬 느낌
- 여유로운
- 쇼핑 가능한 곳 근처

예시 매핑:

| 자연어 표현 | 정형 조건 |
| --- | --- |
| 조용한, 한적한 | `quiet` |
| 활기찬, 북적이는 | `lively` |
| 전망 좋은, 뷰 좋은 | `scenic_view` |
| 산책하기 좋은 | `walkable` |
| 로컬 느낌 | `local` |
| 여유로운 | `slow_pace` |
| 쇼핑 가능한 곳 근처 | `shopping_nearby` |

### **unsupported**

현재 추천 기준에서 직접 판단하기 어렵거나 검색 조건으로 쓰면 위험한 조건이다.

이 조건은 RAG 검색 조건으로 넘기지 않고, 안내 또는 대체 처리한다.

예:

- 숙박 시설 자체의 품질
- 숙소 전망
- 숙소 가격
- 예약 가능 여부
- 실시간 혼잡도
- 의류 재고 여부
- 실시간 운영 여부

숙박 관련 세부 조건은 일정 추천 기준에 직접 넣지 않고, 일정 추천 후 해당 소도시 기준 숙박 검색 링크 제공으로 분리한다.

사용자 안내 예시:

> 숙박 조건은 직접 판단하기 어려워요. 대신 여행 동선과 주변 장소를 고려하고, 마지막에 숙박 검색 링크를 제공할게요.
>

### **1-3. cleaned raw query 사용**

자연어 입력이 있을 경우, 판별 agent는 `soft_preferences`만 뽑는 것이 아니라 `cleaned_raw_query`도 함께 만든다.

`cleaned_raw_query`는 사용자 원문에서 반영하기 어려운 조건만 제거하고, 사용자의 표현과 맥락은 최대한 보존한 검색용 문장이다.

예:

사용자 원문:

> 자연이 좋고 맛집도 있었으면 좋겠어요. 조용한 분위기면 더 좋아요. 뷰 좋은 숙박도 있으면 좋겠어요.
> 

분석 결과:

```json
{
  "active_required_themes": ["자연·트레킹", "미식·노포"],
  "soft_preferences": ["quiet"],
  "unsupported_conditions": ["숙소 전망"]
}
```

cleaned raw query:

```
자연이 좋고 맛집도 있었으면 좋겠어요. 조용한 분위기면 더 좋아요.
```

cleaned raw query를 사용하는 이유:

- soft preference로 정형화하며 사라지는 뉘앙스를 보존하기 위해
- 사용자 문장의 맥락을 장소 설명 임베딩과 직접 비교하기 위해
- LLM이 soft preference를 일부 놓쳤을 때 보완하기 위해
- 반영하기 어려운 조건이 검색에 섞이는 것을 막기 위해

즉:

```
soft_preferences = 설명 가능한 정형 랭킹 조건
cleaned_raw_query = 사용자 표현의 의미를 보존한 유사도 검색 조건
```

### **2. 거리 기반 1차 후보 필터링**

사용자 위치를 기준으로 접근 가능한 소도시를 먼저 필터링한다.

이유:

- 실제 여행 가능성이 낮은 지역을 먼저 제외하기 위해
- 추천 결과가 사용자의 현실적인 이동 범위 안에 들어오도록 하기 위해
- 일정 생성 시 이동 부담이 과도하게 커지는 것을 방지하기 위해

예시 기준:

- 당일치기: 가까운 거리 우선
- 1박 2일: 중거리까지 허용
- 2박 이상: 더 넓은 범위 허용

초기 버전에서는 사용자 위치와 소도시 중심 좌표 또는 도시 내 장소 평균 좌표 간 거리를 활용할 수 있다.

### **3. 선택 테마 전체 충족 여부 확인**

거리 조건을 통과한 소도시 중에서, 사용자가 선택한 테마를 모두 만족할 수 있는 도시만 후보로 남긴다.

핵심 원칙:

> 사용자가 선택한 테마는 모두 충족해야 하는 필수 조건으로 본다.
> 

예를 들어 사용자가 `자연·트레킹`, `미식·노포`, `예술·감성`을 선택했다면 해당 도시는 최소한 다음 조건을 만족해야 한다.

- 자연·트레킹 장소 1개 이상
- 미식·노포 장소 1개 이상
- 예술·감성 장소 1개 이상

자연어 입력이 없는 경우에는 온보딩에서 선택한 테마가 그대로 `active_required_themes`가 된다.

이때 충족 여부는 장소 데이터의 `city_id`와 `theme` 컬럼을 기준으로 도시별 장소 수를 집계해 확인한다.

예:

```
자연어 없음:
온보딩 테마 → active_required_themes
theme 컬럼 기반 count → 도시별 충족 여부 확인
```

자연어 입력이 있는 경우에는 판별 agent가 온보딩 테마와 자연어 조건을 병합해 `active_required_themes`를 만든다.

이 경우에도 필수 테마 충족 여부는 벡터 검색이 아니라 `theme` 컬럼 기반 집계로 판단한다.

이유:

- 테마는 이미 정형화되어 있음
- `theme` 컬럼으로 확인하는 것이 빠르고 명확함
- 벡터 검색보다 필수 조건 판단에 안정적임

```
자연어 있음:
온보딩 테마 + 자연어 조건 → active_required_themes
theme 컬럼 기반 count → 도시별 충족 여부 확인
soft/raw query 유사도 → 랭킹 보정
```

다만 사용자가 선택한 테마 개수에 따라 후보 도시가 지나치게 줄어들 수 있으므로, 선택 개수별로 충족 기준을 다르게 적용한다.

- **선택 테마 1개**
    - 해당 테마가 도시의 핵심 특성인지 확인
    - 도시 내 해당 테마 장소 수, 비중, 전체 평균 대비 특화도를 함께 확인
- **선택 테마 2개**
    - 두 테마 모두 최소 충족해야 함
    - 각 테마별 장소가 최소 1개 이상 있어야 하며, 일정 안에 두 테마가 모두 반영될 수 있어야 함
- **선택 테마 3개**
    - 세 테마 모두 최소 충족해야 함
    - 다만 희소 테마의 경우 장소 수 기준을 완화해 후보 도시가 과도하게 줄어드는 것을 방지
    - 예: `온천·휴양`, `예술·감성`, `바다·해안`은 1개만 있어도 최소 충족으로 인정 가능

이 기준을 넣는 이유:

- 사용자가 선택한 테마가 추천 결과와 일정에 실제로 반영되도록 하기 위해
- 단일 테마만 강한 도시가 모든 선호를 만족하는 것처럼 추천되는 문제를 막기 위해
- 테마를 3개 선택했을 때 추천 후보가 지나치게 줄어드는 문제를 방지하기 위해
- 사용자 선택을 존중하면서도 현실적으로 추천 가능한 후보군을 유지하기 위해

### **3-1. 자연어 세부 조건의 임베딩 유사도 반영**

필수 테마 충족 여부는 `theme` 컬럼 기반으로 확인하고, 자연어의 세부 취향은 장소별 임베딩 유사도 검색으로 반영한다.

장소별 임베딩 텍스트 예:

```
장소명: 고석사
도시: 포항
테마: 역사·전통
장소유형: 관광지
주소: 경상북도 포항시 남구 장기면
설명: 묘봉산에 있는 사찰, 불교문화, 역사 유적, 산 주변 경관
파생태그: 조용한, 전통적인, 산책, 역사, 사찰
```

임베딩에 넣기 좋은 항목:

- 장소명
- 도시명
- 테마
- 장소 유형
- 주소 일부
- 설명
- 파생 태그

구조화 계산에 쓰는 항목:

- 위도/경도
- 운영시간
- 여행일수
- 거리
- `content_type_id` 기반 일정 균형

자연어 입력이 있을 때는 다음 두 가지 쿼리를 장소별 임베딩과 비교한다.

```
soft query 벡터 ↔ 장소별 임베딩 벡터
cleaned raw query 벡터 ↔ 장소별 임베딩 벡터
```

이 점수는 후보 도시를 탈락시키는 기준이 아니라, 후보 도시와 장소를 재랭킹하는 기준으로 사용한다.

예:

```
soft query:
조용한 한적한 여유로운 산책하기 좋은 분위기

cleaned raw query:
자연이 좋고 맛집도 있었으면 좋겠어요. 조용한 분위기면 더 좋아요.
```

도시별 집계 기준:

- soft query와 유사한 장소 수
- soft query 최고 유사도
- cleaned raw query와 유사한 장소 수
- cleaned raw query 최고 유사도
- 일정에 넣을 수 있는 장소 여부

### **4. 희소 테마 보정**

현재 수집 데이터는 테마별 개수 차이가 크다.

예시:

| 테마 | 총 개수 | 특징 |
| --- | --- | --- |
| 미식·노포 | 많음 | 대부분 도시에서 쉽게 발견될 가능성 높음 |
| 역사·전통 | 많음 | 비교적 안정적으로 추천 가능 |
| 자연·트레킹 | 중간 | 도시별 특성 반영 가능 |
| 바다·해안 | 적음 | 해안 도시 특성으로 강하게 작용 가능 |
| 예술·감성 | 적음 | 존재 자체가 도시 매력 요소가 될 수 있음 |
| 온천·휴양 | 매우 적음 | 1-2개만 있어도 강한 추천 근거 가능 |

따라서 모든 테마에 동일한 기준을 적용하지 않는다.

예를 들어:

- `미식·노포`는 단순히 1개 있다고 해서 강한 추천 근거로 보지 않음
- `온천·휴양`은 1개만 있어도 희소한 테마 특성으로 인정 가능
- `바다·해안`은 도시 내 비중이 20% 미만이어도 전체 평균보다 높으면 특화 도시로 볼 수 있음
- `예술·감성`은 장소 수가 적어도 축제, 문화시설, 전시 공간이 있으면 의미 있게 반영 가능

이 기준을 넣는 이유:

- 데이터가 많은 테마만 추천에 유리해지는 문제를 막기 위해
- 희소하지만 사용자 취향에 중요한 테마가 과소평가되지 않도록 하기 위해
- 테마별 데이터 분포 차이를 추천 알고리즘이 반영하도록 하기 위해

---

### **5. 테마 특화도 계산**

단순히 도시 내 해당 테마 비중이 20% 이상인지 보는 것이 아니라, 전체 데이터 평균과 비교한다.

핵심 기준:

> 해당 도시의 특정 테마 비중이 전체 평균보다 얼마나 높은가?
> 

예시:

- 전체 데이터에서 `바다·해안` 비중이 5%인데 어떤 도시에서 15%라면, 이 도시는 바다·해안 특화 도시로 볼 수 있음
- 전체 데이터에서 `미식·노포` 비중이 53%인데 어떤 도시에서 20%라면, 미식 특화 도시라고 보기 어려움

이 기준을 넣는 이유:

- 20%라는 고정 기준의 한계를 보완하기 위해
- 테마별 데이터 불균형을 반영하기 위해
- 각 도시가 어떤 테마에서 상대적으로 강한지를 판단하기 위해

---

### **6. 도시 규모/데이터 과다 보정**

장소 수가 많은 도시가 무조건 높은 점수를 받지 않도록 보정한다.

예를 들어 강릉처럼 수집된 장소가 많은 도시는 단순 개수 기준에서 계속 유리해질 수 있다.

하지만 서비스 목적이 소도시 추천이라면, 장소 수가 많은 도시만 반복 추천되는 것은 적절하지 않다.

따라서 장소 개수는 그대로 더하지 않고 구간화한다.

예시:

| 선호 테마 장소 수 | 해석 |
| --- | --- |
| 0개 | 후보 제외 |
| 1-2개 | 최소 충족 |
| 3-5개 | 일정 구성 가능 |
| 6-10개 | 충분히 풍부함 |
| 11개 이상 | 상한 점수 적용 |

이 기준을 넣는 이유:

- 데이터가 많은 도시가 과도하게 유리해지는 문제를 막기 위해
- 작은 도시도 테마 특성이 뚜렷하다면 추천될 수 있도록 하기 위해
- 추천 결과의 다양성을 확보하기 위해

---

### **7. 콘텐츠 타입 균형 확인**

소도시 추천은 일정 생성까지 이어져야 하므로, 장소의 테마뿐 아니라 장소 유형도 확인해야 한다.

활용 가능한 기준:

- 관광지
- 문화시설
- 축제/공연/행사
- 레포츠
- 숙박
- 쇼핑
- 음식점

단, 숙박은 일정 장소 추천 점수에 직접 반영하기보다, 일정 추천 후 해당 소도시 기준 숙박 검색 링크 제공으로 분리한다.

일정 생성에 필요한 최소 구성 예시:

- 관광/체험 장소 2개 이상
- 음식점 1개 이상
- 문화/휴식/산책 장소 1개 이상
- 축제 포함 선택 시 축제 1개 우선 반영

이 기준을 넣는 이유:

- 테마는 맞지만 실제 일정이 빈약한 도시를 제외하기 위해
- 하루 종일 같은 유형의 장소만 방문하는 부자연스러운 일정을 막기 위해
- 사용자의 여행 경험이 관광, 식사, 휴식, 체험으로 균형 있게 구성되도록 하기 위해

---

### **8. 축제 포함 여부 반영**

축제는 기본 테마 추천에 포함하지 않는다.

대신 사용자에게 소도시 추천 전 또는 일정 생성 전에 축제를 포함할지 묻는다.

사용자가 축제 포함을 원하지만 해당 월에 사용자 선호 테마에 해당하는 축제가 없을 경우에는 일정에 포함하지 않고 일정 생성 시 마지막에 안내 문구를 출력한다.

사용자가 축제 포함을 원하면:

- 여행 예정일에 열리는 축제가 있는지 확인
- 선택한 도시 또는 후보 도시 안의 축제를 일정에 우선 반영
- 축제 일정이 있는 도시에 조건부 가산점 부여

사용자가 축제 포함을 원하지 않으면:

- 축제는 추천 점수에서 제외
- 일반 관광지, 음식점, 문화시설, 자연 장소 중심으로 일정 구성

이 기준을 넣는 이유:

- 축제는 기간성이 강해 일반 관광지와 같은 기준으로 비교하기 어렵기 때문에
- 사용자가 축제성 여행을 원하는지 여부가 만족도에 큰 영향을 주기 때문에
- 축제 데이터 부족이나 기간 불일치가 전체 추천 품질을 흔들지 않도록 하기 위해

---

### **9. 최종 소도시 점수화**

위 조건들을 통과한 후보 도시를 점수화한다.

예시 점수 구조:

| 기준 | 역할 |
| --- | --- |
| 접근성 | 사용자 위치에서 이동 가능한지 |
| 선택 테마 충족도 | 사용자가 고른 모든 테마를 만족하는지 |
| 테마 특화도 | 전체 평균 대비 해당 도시가 특정 테마에 강한지 |
| 희소 테마 보정 | 적은 데이터 테마가 과소평가되지 않도록 보정 |
| 도시 규모 보정 | 데이터가 많은 도시의 과도한 우위 방지 |
| 콘텐츠 타입 균형 | 실제 일정 구성이 가능한지 |
| soft preference 적합도 | 자연어의 분위기/세부 조건과 맞는 장소가 있는지 |
| cleaned raw query 적합도 | 사용자의 원문 맥락과 장소 설명이 얼마나 유사한지 |
| 축제 조건부 반영 | 사용자가 원할 경우에만 축제 반영 |

자연어 입력이 없는 경우에는 `soft preference 적합도`와 `cleaned raw query 적합도`를 제외하고 구조화 데이터 기반으로 점수화한다.

자연어 입력이 있는 경우에는 필수 테마 충족은 `theme` 컬럼으로 확인하고, 자연어의 세부 취향은 soft query와 cleaned raw query 유사도 점수로 랭킹에 반영한다.

최종 추천은 단순히 점수가 높은 도시를 보여주는 것이 아니라, 추천 근거도 함께 제공한다.

---

### **10. 소도시 추천 및 추천 근거 제공**

최종 추천은 단순히 점수가 높은 도시를 보여주는 것이 아니라, 사용자가 납득할 수 있는 추천 근거를 함께 제공한다.

추천 근거는 다음 세 가지를 중심으로 구성한다.

- **필수 조건 충족**
    - 사용자가 선택한 테마를 모두 충족했는지
    - 여행국, 거리, 여행일수 조건에 맞는지
- **도시 특성**
    - 전체 평균 대비 어떤 테마에 특화되어 있는지
    - 희소 테마가 의미 있게 존재하는지
    - 도시 규모 보정 이후에도 추천할 만한 근거가 있는지
- **일정 구성 가능성**
    - 실제 일정에 넣을 수 있는 장소가 충분한지
    - 관광/식사/문화/휴식 장소가 균형 있게 있는지
    - 자연어 입력이 있는 경우, soft preference와 cleaned raw query에 맞는 장소가 있는지
    - 축제 포함 선택 시 일정에 반영 가능한 축제가 있는지

추천 근거에는 가능하면 실제 장소 후보를 함께 포함한다.

예를 들어 단순히 “자연 테마가 강하다”라고 설명하기보다, 자연 산책 장소, 지역 음식점, 문화시설 등 실제 일정에 들어갈 수 있는 장소를 근거로 제시한다.

#### **추천 근거 기본 템플릿**

```
[도시명]은 선택하신 [테마 A], [테마 B]를 모두 충족하는 소도시입니다.
특히 [강한 테마/특화 기준]이 두드러지고, [장소 유형] 후보가 충분해 [여행일수] 일정 구성이 가능합니다.
[자연어 조건]도 [반영 방식]으로 고려했습니다.
```

#### **자연어 입력이 없는 경우 템플릿**

```
[도시명]은 선택하신 [테마 A], [테마 B]를 모두 충족합니다.
도시 내 [테마 A] 장소 비중이 전체 평균보다 높고, [테마 B] 장소도 충분해 [여행일수] 일정 구성이 가능합니다.
관광/식사/휴식 장소가 균형 있게 있어 실제 일정으로 연결하기 좋습니다.
```

#### **자연어 입력이 있는 경우 템플릿**

```
[도시명]은 선택하신 [테마 A], [테마 B]를 모두 충족합니다.
또한 말씀하신 “[자연어 조건]”과 유사한 장소가 있어, [분위기/동선/장소 유형]을 반영한 일정 구성이 가능합니다.
관광/식사/휴식 장소가 균형 있게 있어 [여행일수] 일정으로 추천드립니다.
```

#### **축제 포함 선택 시 템플릿**

```
[도시명]은 선택하신 [테마 A], [테마 B]를 충족하고, 여행 기간에 반영 가능한 축제 후보가 있어 추천드립니다.
축제 일정과 주변 관광/식사 장소를 함께 구성할 수 있어 [여행일수] 일정에 적합합니다.
```

#### **일부 조건 반영이 어려운 경우 템플릿**

```
[도시명]은 선택하신 [테마]를 충족하고, 반영 가능한 [조건]을 기준으로 추천되었습니다.
일부 조건은 직접 판단하기 어려워, 일정 마지막에 관련 검색 링크를 함께 제공하겠습니다.
```

#### **추천 근거 예시**

```
[도시명]은 선택하신 자연·트레킹, 미식·노포 테마를 모두 충족합니다.
자연 테마 장소 비중이 전체 평균보다 높고, 지역 음식점 후보도 충분해 산책 코스와 식사 일정을 함께 구성하기 좋습니다.
```

자연어 입력이 있는 경우:

```
[도시명]은 선택하신 자연·트레킹, 미식·노포 테마를 모두 충족하고, 말씀하신 “조용한 분위기”와 유사한 산책 장소가 있어 추천드립니다.
관광/식사/휴식 장소가 균형 있게 있어 1박 2일 일정 구성이 가능합니다.
```

축제 포함 선택 시:

```
[도시명]은 선택하신 역사·전통, 미식·노포 테마를 충족하고, 여행 기간에 반영 가능한 지역 축제가 있어 추천드립니다.
축제 방문 후 주변 음식점과 문화 장소를 함께 구성할 수 있어 일정 흐름이 자연스럽습니다.
```

일부 조건 반영이 어려운 경우:

```
[도시명]은 선택하신 자연·트레킹 테마와 조용한 분위기를 기준으로 추천되었습니다.
일부 조건은 직접 판단하기 어려워, 일정 마지막에 관련 검색 링크를 함께 제공하겠습니다.
```

### **11. 도시 내 일정 장소 추천**

소도시가 선택되면, 해당 도시 안에서 장소를 골라 일정을 생성한다.

장소 선정 기준:

- 사용자가 선택한 테마와 일치하는 장소 우선
- 선택한 모든 테마가 일정 안에 포함되도록 구성
- 콘텐츠 타입이 한쪽으로 치우치지 않도록 조정
- 장소 간 이동거리가 너무 멀지 않도록 동선 고려
- 운영시간이 확인되는 장소는 일정 배치 우선
- 운영정보가 부족한 장소는 안내 문구와 함께 보조 후보로 활용
- 축제 포함 선택 시 축제를 고정 일정으로 먼저 배치
- 자연어 입력이 있는 경우 soft preference와 cleaned raw query에 잘 맞는 장소를 우선 배치
- 일정 생성 후 해당 소도시 기준 숙박 검색 링크 제공

이 기준을 넣는 이유:

- 도시 추천이 실제 여행 일정으로 자연스럽게 이어지도록 하기 위해
- 추천된 도시 안에서도 사용자 취향이 일정 전체에 반영되도록 하기 위해
- 장소 간 이동 부담을 줄이고 현실적인 여행 코스를 만들기 위해

---

## **자연어 입력이 없는 경우 추천 흐름 요약**

```
1. 사용자가 선호 여행 테마 1-3개 선택
2. 여행국, 여행일수, 축제 포함 여부, 사용자 위치 입력
3. 온보딩 테마를 active_required_themes로 사용
4. 사용자 위치 기준 접근 가능한 소도시 1차 필터링
5. city_id + theme 기준으로 도시별 테마 장소 수 집계
6. 선택한 모든 테마를 충족할 수 있는 도시만 후보화
7. 선택 테마 개수별 기준 적용
8. 희소 테마 보정 적용
9. 전체 평균 대비 도시별 테마 특화도 계산
10. 도시 규모/데이터 과다 보정 적용
11. 콘텐츠 타입 균형으로 일정 구성 가능성 확인
12. 축제 포함 선택 시 조건부 반영
13. 최종 소도시 추천 및 추천 근거 제공
14. 추천된 도시 안에서 테마, 장소 유형, 동선, 운영시간을 고려해 일정 생성
15. 일정 추천 후 해당 소도시 기준 숙박 검색 링크 제공
```

## **자연어 입력이 있는 경우 추천 흐름 요약**

```
1. 사용자가 선호 여행 테마 1-3개 선택
2. 여행국, 여행일수, 축제 포함 여부, 사용자 위치 입력
3. 챗봇에서 사용자가 자연어 추가 조건 입력
4. 판별 agent가 자연어를 분석해 active_required_themes, soft_preferences, cleaned_raw_query, unsupported_conditions 추출
5. 온보딩 테마와 자연어 조건을 병합해 최종 active_required_themes 생성
6. 사용자 위치 기준 접근 가능한 소도시 1차 필터링
7. city_id + theme 기준으로 active_required_themes 전체 충족 여부 확인
8. 선택 테마 개수별 기준 적용
9. soft query와 cleaned raw query를 장소별 임베딩 벡터와 비교
10. 유사도 결과를 city_id 기준으로 집계해 자연어 적합도 계산
11. 희소 테마 보정 적용
12. 전체 평균 대비 도시별 테마 특화도 계산
13. 도시 규모/데이터 과다 보정 적용
14. 콘텐츠 타입 균형으로 일정 구성 가능성 확인
15. 축제 포함 선택 시 조건부 반영
16. 최종 소도시 추천 및 추천 근거 제공
17. 추천된 도시 안에서 테마, 장소 유형, 동선, 운영시간, 자연어 세부 취향을 고려해 일정 생성
18. 일정 추천 후 해당 소도시 기준 숙박 검색 링크 제공
```

## **두 흐름의 핵심 차이**

| 구분 | 자연어 입력 없음 | 자연어 입력 있음 |
| --- | --- | --- |
| 핵심 입력 | 온보딩 테마 | 온보딩 테마 + 자연어 |
| 판별 agent | 필수 아님 | 필요 |
| active theme | 온보딩 테마 그대로 사용 | 온보딩 테마와 자연어 조건 병합 |
| 테마 충족 확인 | `theme` 컬럼 기반 count | `theme` 컬럼 기반 count |
| soft preference | 없음 | 있음 |
| cleaned raw query | 없음 | 있음 |
| 임베딩 유사도 검색 | 필수 아님 | soft query / cleaned raw query 랭킹에 사용 |
| 도시 후보화 | 구조화 데이터 기반 | 구조화 데이터 기반 + 자연어 적합도 보정 |
| 랭킹 | 테마 수, 비중, 특화도 중심 | 테마 기준 + 자연어 뉘앙스 추가 |
| 예외 안내 | 거의 없음 | unsupported 조건 있을 때 필요 |

## **한 문장 정리**

> 본 서비스는 사용자가 선택한 1-3개의 여행 테마를 모두 충족할 수 있는 소도시를 우선 후보로 선정하고, 테마별 데이터 불균형을 보정한 뒤, 접근성·테마 특화도·도시 규모 보정·콘텐츠 타입 균형·일정 구성 가능성을 종합적으로 판단해 최종 소도시와 여행 일정을 추천한다. 자연어 입력이 있는 경우에는 판별 agent가 온보딩 테마와 현재 여행 의도를 병합하고, 필수 테마는 `theme` 컬럼 기반으로 확인하며, soft preference와 cleaned raw query는 장소별 임베딩 유사도 검색을 통해 랭킹과 일정 장소 선정에 반영한다.

- 서브:
    
    **“지도에서 흐릿한 곳, 여행에선 가장 빛나는 곳”**
    

## 1. 목적지 포커스 차별점 + 보완 방향

## 차별점 요약

- **트리플**
    - 메인 타깃은 도쿄·오사카·파리 같은 메이저 도시이며, 그 도시 안에서의 일정·동선 최적화가 핵심입니다.[triple]
    - AI 일정 역시 “사용자가 이미 선택한 도시” 안에서 일별 코스와 동선을 짜주는 형태입니다.[fnnews]
- **Lovv**
    - 1순위 목표가 “유명 관광지를 대체 가능한 소도시를 추천”하는 것이며, 오버투어리즘 회피와 소도시 발견이 문제 정의의 중심입니다.[ppl-ai-file-upload.s3.amazonaws]
    - 도시 선택이 아니라 **국가·여행 시기**를 먼저 고정하고, 그 조건에서 혼잡 회피 대안이 되는 소도시를 찾는 구조라, 목적지 레벨에서 문제를 푸는 서비스입니다.[ppl-ai-file-upload.s3.amazonaws]

→ 그래서 트리플이 “서울·오사카를 더 잘 돌아다니게 해주는 일정 도구”라면, Lovv는 “서울·오사카와 유사한 소도시 어디를 가야 덜 붐비고 감성 좋은 도시를 갈 수 있는지”를 알려주는 **목적지 탐색 엔진**으로 차별화할 수 있습니다.[fnnews][ppl-ai-file-upload.s3.amazonaws]

## 보완 포인트

- **PoC 단계에서 소도시를 ‘넓게’가 아니라 ‘깊게’**: 한국·일본 각각 2~3개 소도시를 선정해, 그 도시들에 대해 축제·기상·교통·체험까지 고밀도로 채워 “Lovv다운 목적지 추천 경험”을 보여주기.[ppl-ai-file-upload.s3.amazonaws]
- **브랜딩/카피에 ‘대체 여행지’ 정체성을 전면 배치**: 첫 화면·랜딩 문구에서 “**서울/오사카 대신 어디?**”·“덜 붐비는 소도시 찾기” 같은 메시지를 써서, 트리플과 출발점부터 다른 서비스라는 인식을 심어주기.[ppl-ai-file-upload.s3.amazonaws]

## 2. 추천 축/방식 차별점 + 보완 방향

## 차별점 요약

- **트리플**
    - 항공·숙소 예약, 일정, 가계부까지 포함한 여행 슈퍼앱을 지향하며, AI 일정은 “갈 곳 리스트 + 동선 최적화”에 집중되어 있습니다.[triple]
    - 추천 이유나 데이터 출처는 유저 경험에서 크게 드러나지 않고, 내부 추천/루팅 엔진이 블랙박스처럼 동작합니다.[fnnews]
- **Lovv**
    - 기획서에서 추천 축을
        - 혼잡 회피
        - 축제 기반
        - 날씨 기반
        - 미디어 촬영지
        - 철도 경로
        - 지역 문화 체험
        - 스키 액티비티
            
            로 명시적으로 분리해 두었습니다.[ppl-ai-file-upload.s3.amazonaws]
            
    - RAG 기반 검색과 멀티스텝 에이전트를 사용해, 각 추천에 대해 **추천 이유·날씨·지도·외부 링크**를 함께 제공하는 것을 목표로 합니다.[ppl-ai-file-upload.s3.amazonaws]

→ 따라서 Lovv는 “어디를 가라”에서 끝나는 것이 아니라, **왜 이 소도시인지, 어떤 데이터에 근거했는지**를 설명하는 **설명 가능한 추천(Explainable Recommendation)**으로 포지셔닝할 수 있습니다.[ppl-ai-file-upload.s3.amazonaws]

## 보완 포인트

- **Explainable 부분을 PoC에서 과장 없이라도 보여주기**
    - 출처·링크를 클릭하면 바로 공공/외부 사이트로 이동하도록 해서, “LLM이 만든 이야기”가 아니라 “데이터 기반 큐레이션”이라는 인상을 강화하기. → 크롤링한 사이트 링크 첨부

## 3. 입력·흐름 차별점 + 보완 방향

## 차별점 요약

- **트리플**
    - 도시나 여행지를 먼저 선택한 뒤, 그 도시 안에서 AI 일정 생성으로 들어가는 흐름입니다.[valuecreatorai]
    - 일정 단위·동선 설계에 최적화된 UX라, “도시는 이미 정했다”는 전제를 둡니다.[fnnews]
- **Lovv**
    - 1단계: 국가 + 여행 시기 입력 → 추천 대상 범위를 고정.
    - 2단계: 자연어 취향 또는 지도 마커로 테마를 설정 → 후보 소도시를 비교.[ppl-ai-file-upload.s3.amazonaws]
    - 3단계: 방문객 수·계절·축제·기상·피드백 기준으로 소도시 1곳을 선정한 뒤, 그 도시에 대한 일정과 추천 이유를 제공하는 **집중형 구조**입니다.[ppl-ai-file-upload.s3.amazonaws]

→ UX 관점에서, Lovv는 “**어디를 갈지조차 모르는 사람에게 한 곳을 골라주는 탐색기**”이고, 트리플은 “이미 정한 도시를 어떻게 돌아다닐지 최적화하는 도구”라는 분업 구조로 차별화됩니다.[valuecreatorai][ppl-ai-file-upload.s3.amazonaws]

## 보완 포인트

- **Lovv의 첫 질문을 ‘국가/시기’ 중심으로 아주 명확하게**
    - 예: 첫 화면에 “어디로 갈까요?” 대신 “**언제, 어느 나라로 떠나세요?**”를 우선 질문해 Lovv의 출발점을 트리플과 다르게 만들기.[ppl-ai-file-upload.s3.amazonaws]
- **소도시 1곳 집중 UX를 강조**
    - “소도시 후보 리스트 → 1곳을 선택하면 깊게 들어가는 상세 페이지” 흐름을 설계해, 여러 도시를 얕게 훑기보다 **한 곳에 몰입하는 느낌**을 주기.[ppl-ai-file-upload.s3.amazonaws]

- **Lovv**
    - 기획서에서 지자체·관광 운영자·공공 데이터 제공 기관을 명시적인 이해관계자로 설정하고,
        - 데이터 제안
        - 관리자 검토·승인·반려
        - 추천에 반영
        - 검증 이력 관리
            
            흐름까지 설계하고 있습니다.[ppl-ai-file-upload.s3.amazonaws]
            
    - 공공 관광 데이터, 축제 정보, 철도·날씨 등 다양한 외부 데이터를 RAG로 엮어 소도시를 추천하려는 구조입니다.[ppl-ai-file-upload.s3.amazonaws]

→ 이 때문에 Lovv는 “**지자체·공공 데이터·현지 운영자와 함께 만드는 소도시 추천 플랫폼**”이라는, 트리플과 다른 파트너 구조/브랜딩을 가져갈 수 있습니다.[ppl-ai-file-upload.s3.amazonaws]

## 5. 기술·AI 관점 차별점 + 보완 방향

## 차별점 요약

- **트리플**
    - AI 일정 기능은 추천+동선에 강점을 가지지만, 내부 로직·데이터 출처·추천 근거는 사용자에게 거의 드러나지 않습니다.[fnnews]
    - 도메인 특화 추천/루팅 엔진 위에 일부 AI 설명 기능이 덧붙은 형태로 추정됩니다.[fnnews]
- **Lovv**
    - 입력 → 조건 분류 → RAG 기반 후보 검색 → 후보 필터링·선정 → 일정 구성 → 추천 이유·외부 정보 보강 → 후속 Q&A 까지, 전체 파이프라인을 명시적으로 설계하고 있습니다.[ppl-ai-file-upload.s3.amazonaws]
    - “추천 결과는 근거 데이터와 함께 제공”, “데이터 출처·갱신 시각·검증 상태 표시” 등을 시스템 원칙으로 잡아, LLM/RAG 아키텍처 자체를 서비스의 차별 포인트로 삼고 있습니다.[ppl-ai-file-upload.s3.amazonaws]

→ LLM 엔지니어링 관점에서, Lovv는 **“Explainable RAG 기반 소도시 추천 엔진”**이라는 태그라인으로, 트리플의 블랙박스형 AI 일정과 기술 구조/철학에서 차이를 만들 수 있습니다.[ppl-ai-file-upload.s3.amazonaws]

## 보완 포인트

- **후속 Q&A를 ‘에이전트답게’ 설계**
    - 추천 결과를 받은 뒤, 사용자가 “이 도시 말고 더 조용한 곳?” “축제 없는 곳으로 바꿔줘” 같은 자연어 질의를 하면, 동일 파이프라인을 다시 돌리되 기준을 바꿔 재추천하는 흐름을 구현.[ppl-ai-file-upload.s3.amazonaws]

## 2. 섹션별 수정안

> 장 번호 표기 기준: 아래 수정안의 장 번호는 **현재 원본(00_project_plan.md) 기준**으로 위치를 가리킨다. "신규 비즈니스 모델"은 7장(기능 구성)과 8장(데이터·외부 연동) 사이에 삽입하며, 이에 따른 8~14장 재번호는 적용 시점에 일괄 수행한다(§3·실행 순서 참조).

### 1.3 기획 배경 (보강)

- 1차 타깃이 **한국인 재방일객의 일본 지역 여행**임을 명시.
- 시장 비대칭(2025년 한국→일본 약 945.96만 명, 일본→한국 약 365만 명, 약 2.6배)을 한 문장으로 추가.
- "소도시 선호는 전체 대중시장이 아니라, 재방문·근거리·목적형 조건에서 커지는 **고의도 하위시장**"이라는 냉정한 톤 반영(92_B2C 1장).

### 3. 프로젝트 목표·성공 기준 (보강)

- **3.1 핵심 목표**: 기존 "P1 구현, P2~P3 설계"에 KICK(P2)을 반영하고, 단일 소도시 원칙과 단계적 데이터 확장을 목표에 명시.
  - "추천 1건 = 소도시 1곳, 그 도시 내부에서만 일정 완성"을 핵심 목표 항목으로 추가.
  - "현재 PoC는 한국 강원·경북·일본 관동에 집중, 최종 한일 전국 도시로 확장"을 범위 목표로 추가.
- **3.2 성공 기준**: 표에 행 추가/보강.
  - 추천 경험 행에 "조건 입력 후 소도시 1곳·일정 생성"은 유지하되, KICK 성공 기준("유사 취향 공개 일정 노출 → 1탭 복제 → 저장")을 별도 행으로 추가.
  - 범위 통제 행에 "P2(KICK 포함) 설계 산출물 확보"를 반영.

### 4.1 핵심 서비스 (대폭 보강)

- **제품 불변 규칙** 명문화(92_B2C 8.2):
  - 추천 1건의 최종 목적지는 소도시 1곳.
  - 생성 일정의 모든 관광지·숙소·체험은 추천 소도시 서비스 범위 안에 있어야 함.
  - 공항·역·관문도시는 교통 안내에만 사용, 일정에 포함하지 않음.
  - 적합 소도시를 확정할 수 없으면 여러 도시를 섞지 않고 추가 조건을 요청.
- **소도시 운영적 정의** 추가: 관문 대도시(서울·도쿄·오사카·후쿠오카 등) 제외, 지역 거점도시 분리, 인접 소도시·농산어촌을 핵심으로(92_B2C 2.2).
- **KICK** 추가: "나와 취향이 비슷한 사용자의 공개 일정 → 1탭 복제 → 챗봇 부분 조정"을 발견·납득·실행 다음의 **선택 전환 장치**로 설명(94_kick 2·3장).
- **진입 정책**: 진입 시 **회원가입·로그인 필수**(95_aha v0.18 기준 확정). 96_travel_ai의 "로그인 전 탐색" 최소기능은 미채택. 4.1·4.2·7장에 로그인 필수를 일관 기술.
- **진입·흐름 정합**: 온보딩에서 테마·기본 국가 선택 → 카드/지도 마커/AI 챗봇으로 진입 → **챗봇이 국가·여행 시기(월)·여행 일정 길이(당일치기~2박3일)를 가장 먼저 질문** → 소도시 1곳 선정 → PLAN DETAIL로 수렴(진입점 3개)(95_aha 화면 흐름).
- **추천 경험의 설계 철학(Aha Moment) 명시**: 흐름뿐 아니라 "왜 그렇게 설계했는가"를 기획서에 개념 수준으로 한 단락 추가(95_aha).
  - **핵심 aha 정의 — 3성분**: "이런 곳이 있는 줄 몰랐는데, 지금 내 조건에 맞고, 진짜 갈 수 있겠다" = ① 발견(놀람) ② 납득(왜 여기) ③ 실행가능성. 추천 경험은 이 세 축을 더 자주·더 빨리·여정의 더 많은 지점에서 끌어내는 것을 목표로 한다.
  - **추천 여정 funnel(단계 0~4)**: 0 첫 진입(발견) → 1 추천 기준(납득) → 2 후속 질의(응답성) → 3 일정·저장(실행가능·peak-end) → 4 재방문(개인화). 개인화는 별도 기능이 아니라 방문이 쌓이며 자라는 곡선으로 다룬다.
  - **Cold start 전략**: 행동 이력이 없어도 "타이밍(날씨·축제·혼잡)은 누구에게나 지금 존재"한다는 점과 "세션 내 신호로 예열"로 첫 인상을 책임진다(단계 0~2). 갔던 곳 기반 개인화는 단계 4의 무기로 분리한다.

### 4.2 사용자 흐름 (표 갱신)

현재 5단계 표를 아래 흐름으로 갱신:

1. 회원가입·로그인(진입 필수) → 온보딩(테마·기본 국가 선호 입력)
2. 카드 슬롯 / 지도 마커 / AI 챗봇으로 일정 생성 진입
3. 챗봇이 국가·여행 시기(월)·여행 일정 길이(당일치기~2박3일) 선질문 → 조건 고정
4. 사용자가 자연어로 추가 취향을 자유롭게 입력(분위기·관심사·동행·이동 강도 등) → 취향을 테마/추천 진입점으로 해석
5. RAG 후보 검색 → 소도시 1곳 선정 → day-by-day 일정·추천 이유·지도·날씨·외부 링크
6. 유사 취향 공개 일정 노출 → 1탭 복제 → 챗봇 부분 수정·저장
7. 후속 질의("더 조용한 곳" 등) → 동일 파이프라인 재추천

### 5. 주요 사용자 및 이해관계자 (보강)

- **B2C 타깃을 코호트로 구분 명시**(단일 타깃이 아님):
  - **1차**: 한국인 재방일객의 일본 지역 여행(일본 2회 이상 방문, 2~4일 자유여행).
  - **2차**: 일본인 재방한객(한국 4회 이상 방문)의 한국 지역 여행 — 서울 재방문층을 부산·경주·안동·강원권으로 확장하는 별도 코호트로 작게 검증.
  - **공통 우선 세그먼트**: 짧은 휴가의 직장인·커플(이동 실패 민감), 테마형 여행자(온천·미식·축제). 연령·성별보다 방문 횟수·여행 기간·교통 허용도·테마·예약 시점으로 구분.
  - 초행 여행자는 무료 정보로 충분해 후순위(3순위)(92_B2C 7장).
- 예약 제휴사(클룩·라쿠텐 트래블·KKday)를 외부 연동/B2B 파트너로 추가.
- 지자체·공공기관 협력을 **B2G 비과금 공익 협력**으로 명확화("이번 달 여행지" 월간 노출).

### 6. 프로젝트 범위 (우선순위 표 수정)

- 현재 P-표는 추천 축(P1~P5) 중심. 여기에 **KICK을 P2로 추가**:
  - 예) `P2 | 미디어 촬영지·성지순례 추천, KICK(유사 취향 공개 일정·1탭 복제) | 설계 | 구현 | 선택 구현`
  - 또는 KICK을 별도 행으로 분리해 "PoC: 설계 / Production: 구현"으로 표기.
- KICK이 추천 축과 다른 축(전환 장치)임을 한 줄 주석으로 부기.

### 7. 기능 구성 (보강)

- **7.1 사용자 기능**: 회원가입·로그인(진입 필수), 온보딩(테마·기본 국가), 공개 일정 탐색·1탭 복제·챗봇 부분 조정, 저장·피드백·개인화 추가.
- **7.2 추천 기능**: 유사 취향 공개 일정(사회적 증거), 타이밍/시즌성 넛지(한일 시즌성 역전 근거) 행 추가.
- **7.3 운영·관리 기능**: 공개 일정 운영·신고·익명화(불변 스냅샷·사본 계보), B2G 월간 큐레이션("이번 달 여행지") 추가.

### (신규) 비즈니스 모델 섹션 — 7장과 8장 사이 삽입

기존 8장(데이터·외부 연동) 앞에 신규 장을 삽입하고 이후 장 번호를 한 칸씩 밀어 정리한다. 초안:

> **B2C (1차 수요·가치 검증 트랙)** — 1차 타깃은 한국인 재방일객의 일본 지역 여행. 사용자 가치는 정보 열람이 아니라 제한된 시간 내 실행 가능한 일정·이동 실패 방지·예약 가능한 다음 행동에 둔다. 핵심 KPI는 일정 저장률, 공유·내보내기율, 예약 파트너 클릭률, 재방문율이다. (근거: 92_B2C) ※ 과금·가격 모델은 기획서 범위에서 제외하고 상세는 92_B2C 참조.
>
> **B2B (성과형 제휴 트랙)** — 취향·일정 맥락을 예약 가능한 상품으로 연결하고 이용 완료된 거래에 성과형 수수료를 받는 추천 중개 모델. 파트너는 클룩·라쿠텐 트래블·KKday이며 국적별 우선순위를 다르게 둔다(한국인: KKday·클룩 액티비티 병렬 + 라쿠텐 숙박 조건부 / 일본인: 라쿠텐 숙박 + KKday 액티비티). 총 예약액이 아니라 승인 매출을 매출로 인식한다. (근거: 93_B2B)
>
> **B2G (비과금 공익 협력 트랙)** — 지자체·공공기관과 검증된 관광 정보를 "이번 달 여행지" 월간 큐레이션으로 노출하는 비과금 협력. 성과는 재정 수치가 아니라 노출·클릭·일정 생성/저장·공식 정보 이동·방문 의향 등 비재정 지표로 측정한다. 영리화 시나리오는 본 기획서 범위에서 제외한다. (근거: 96 b2g_public_sector)

### 8. 데이터 및 외부 연동 (구체화)

- **필요 데이터(8.1)**: 실제 수집 범위로 갱신 — 한국 강원·경북 40개 도시, TourAPI 4.0 관광지 3,709건·축제 106건, DataLab 월별 방문통계 / 일본 관동 지역 우선 수집 / VisitorStatistics(JNTO·관광청) / 기후 정합성(ComfortScore·recommended_months).
  - **범위 단계 명시**: 위 40개 도시(강원·경북)·일본 관동은 **현재(PoC) 우선 수집 범위**이며, **최종 목표는 한국·일본 전체 도시로 확장**한다. 기획서에는 "현재 PoC 범위 → 한일 전국 확장"의 단계적 구조로 기술하고, 같은 공통 데이터 모델(AdministrativeArea→City→Attraction/Festival/VisitorStatistics)을 전국에 동일 적용한다(03 1.2·2.2).
- **외부 연동(8.2)**: 기존(Google Maps·Kakao·Yahoo Japan·WeatherAPI·공공 관광 데이터)에 추가 — 예약 제휴(클룩·라쿠텐 트래블·KKday 딥링크), TourAPI/DataLab, JNTO/JTA, 일본기상청(JMA), e-Stat/Statistical LOD.
  - **날씨 데이터 역할 구분 명시**: WeatherAPI = 목적지 상세 화면의 **실시간 현재 날씨 표시**, 기상청 API허브·일본기상청(JMA) = **월별 기후 경향·정합성 검증**(ComfortScore·recommended_months 근거). 둘은 용도가 달라 함께 사용한다(03 2.6·4.1).

### 9. 기술 방향 (보강)

- **데이터 파이프라인** 추가: 자동 수집 → 로컬 검증 산출물(`data/KR`, `data/JP`) → S3 Raw Bucket → Lambda 배치 전처리 → DynamoDB 정규화 적재(03 3장).
- **취향 프로필 3계층**(선언/행동/세션)과 시간 감쇠·프라이버시 원칙(95_aha 공통 데이터 모델).
- **사전계산·캐시**로 첫 화면 LLM 비호출, 이유 문구는 템플릿/정형(95_aha 단계 0).

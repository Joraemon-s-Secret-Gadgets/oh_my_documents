## 정리 원칙

- 대표 기획서와 이 보조 문서는 동일한 Explainable RAG 추천 파이프라인과 시스템 원칙을 유지한다.
- 공개 일정 스냅샷, API Key 관리, 캐시·프록시 같은 구현 상세는 대표 기획서 본문과 상세 명세서에서 함께 추적한다.
- 상세 명세서와 충돌하면 상세 명세서의 결정을 우선한다.

## 추천 파이프라인

1. 사용자 입력 수집
2. 국가·여행 시기·일정 길이·테마·진입점 조건 분류
3. RAG 기반 후보 검색
4. 후보 필터링 및 선정
5. 일정 구성
6. 추천 이유와 외부 정보 보강
7. 출처·갱신 시점·검증 상태 표시
8. 공개 일정 복제 또는 후속 질의응답
9. 조건 변경 기반 재추천

## 시스템 원칙

- 추천 결과는 근거 데이터와 함께 제공한다.
- 추천은 단순 생성형 답변이 아니라 검색된 데이터와 검증 상태를 기반으로 구성한다.
- API Key는 개발·운영 담당자가 서버 환경 변수 또는 백엔드 프록시에서 관리한다.
- 클라이언트에서 직접 호출하기 어려운 API는 딥링크 또는 백엔드 프록시로 대체한다.
- 데이터 출처, 최신성, 검증 상태를 표시한다.
- 사용자가 후속 질문에서 조건을 바꾸면 동일한 추천 파이프라인을 재실행해 대체 소도시 또는 수정 일정을 제안한다.
- 첫 화면은 가능한 한 사전계산·캐시를 활용하고, 이유 문구는 템플릿 또는 정형 근거를 우선 사용한다.
- 공개 일정은 불변 스냅샷과 사본 계보로 관리하고, LLM은 전체 재생성 대신 허용된 수정 operation만 제안한다.
- 서버는 공개 일정 복제·수정의 diff, 버전 충돌, rollback을 검증한다.
- PoC에서는 빠른 검증을 위해 핵심 흐름을 우선 구현한다.

## 데이터 파이프라인 연결

데이터 파이프라인은 자동 수집, 로컬 검증 산출물, S3 Raw Bucket 보존, Lambda 배치 전처리, DynamoDB 정규화 적재 흐름을 기준으로 한다. 원본 데이터는 재처리 가능성을 위해 보존하고, 추천·검색에 필요한 정규화 데이터는 운영 테이블과 벡터 검색 계층으로 분리한다.

상세 데이터 범위와 운영 리스크는 `data_ops_scope.md`에서 관리한다.

# 1. 프로젝트 개요

## 1.1 프로젝트명

로브 (Lovv)

Lovv는 Local(소도시), Vibe(분위기·감성), Voyage(여행)의 의미를 결합한 소도시 여행 추천 서비스다.

## 1.2 한 줄 정의

로브는 여행 시기와 국내 지역 조건부터 시작해 오버투어리즘을 피할 수 있는 소도시 대체 여행지를 발견하게 해주는 설명 가능한 여행 추천 에이전트다.

**서울·오사카 말고, 지금은 이곳**

**지도에서 흐릿한 곳, 여행에선 가장 빛나는 곳**

## 1.3 기획 배경

국내외 여행 수요는 회복되고 있지만, 인기 관광지로의 집중은 여행 만족도 저하와 지역 수용력 문제를 동시에 만든다. 여행자는 덜 붐비는 목적지를 원하지만, 방문객 수가 낮은 관광지와 그 주변 소도시 정보는 흩어져 있고 이동성, 계절성, 축제, 날씨, 주변 관광지 정보를 한 번에 비교하기 어렵다.

로브의 1차 검증 타깃은 한국 강원·경북 소도시 여행자다. PoC는 일본 데이터 수집 실패 리스크를 반영해 한국 강원·경북의 실제 수집·검증 가능한 데이터를 기준으로 추천 경험을 먼저 검증하고, 한국 전국 확장은 P2로 둔다. 일본 데이터 기반 추천과 일본 현지 데이터 연동은 데이터 수집 경로가 복구된 뒤 P5에서 재검토한다.

로브는 오버투어리즘 회피와 소도시 발견이라는 문제를 여행자 관점의 추천 경험으로 해결한다. 사용자가 여행 시기와 국내 지역 조건을 먼저 입력하면, 서비스는 그 조건을 기준으로 사용자의 자연어 취향과 Wikipedia 기반 월별 날씨 경향을 해석하고 추천 근거가 있는 소도시 여정과 주변 정보를 함께 제공한다.

일반 여행 일정 서비스가 사용자가 이미 선택한 대도시 안에서 동선과 일정을 최적화하는 데 강점이 있다면, 로브는 어디를 갈지부터 정하지 못한 사용자에게 국내 지역과 시기 조건에 맞는 대체 소도시를 발견하게 하는 목적지 탐색 엔진으로 차별화한다.

PoC 단계에서는 많은 지역을 얕게 나열하기보다 한국 강원·경북의 제한된 소도시를 깊게 다룬다. 축제, 기상, 교통, 체험, 외부 링크를 고밀도로 채워 Lovv다운 목적지 추천 경험을 먼저 검증한다.

## 1.4 시장 포지셔닝과 차별점

트리플이 인기 도시 안에서 여행 일정을 최적화하는 슈퍼앱이라면, Lovv는 여행 시기와 국내 지역 조건부터 시작해 오버투어리즘을 피할 수 있는 소도시 대체 여행지를 발견하게 해주는 설명 가능한 추천 에이전트다.

Lovv는 “어디를 가라”에서 끝나는 서비스가 아니라, 왜 이 소도시인지, 어떤 데이터에 근거했는지, 지금 실제로 갈 수 있는지를 함께 보여주는 목적지 탐색 엔진이다.

| 축 | 기존 일정 서비스 | Lovv |
| --- | --- | --- |
| 목적지 | 사용자가 이미 선택한 도시 내부 일정 최적화 | 국내 지역·시기 조건에서 대체 소도시 1곳 발견 |
| 추천 방식 | 장소 목록과 동선 중심 | RAG 기반 후보 검색, 추천 이유, 출처·검증 상태 제공 |
| 입력 흐름 | 도시 선택 후 일정 생성 | 로그인·온보딩 후 국내 지역·시기·일정 길이 선질문 |
| 데이터 운영 | 내부 DB·상품 중심 | 공공·지자체·외부 API·제휴 링크를 함께 관리 |
| AI 철학 | 블랙박스형 일정 생성에 가까움 | Explainable RAG 기반 소도시 추천 엔진 |

PoC 보완 방향은 다음과 같다.

- 소도시를 넓게 나열하기보다 제한된 후보 도시를 깊게 구성한다.
- 출처와 외부 링크를 UI에서 적극적으로 노출해 “LLM이 만든 이야기”가 아니라 데이터 기반 큐레이션임을 드러낸다.
- 첫 질문은 “어디로 갈까요?”보다 “언제, 어느 지역으로 떠나세요?”로 설계한다.
- 소도시 후보 리스트보다 소도시 1곳을 깊게 이해하는 상세 흐름을 우선한다.
- 지자체·공공 파트너가 볼 수 있는 추천 노출, 클릭, 관심도 지표를 장기 확장 방향으로 둔다.

# 3. 프로젝트 목표

## 3.1 핵심 목표

- 방문객 수가 많은 혼잡 관광지 대신 대체 가능한 소도시 여행지를 추천한다.
- 추천 1건은 소도시 1곳을 최종 목적지로 삼고, 생성 일정은 그 도시 서비스 범위 안에서 완성한다.
- PoC는 한국 강원·경북 소도시를 기준으로 검증하고, 한국 전국 확장은 P2 범위로 둔다.
- 일본 데이터 기반 추천과 일본 현지 데이터 연동은 수집 경로가 복구된 뒤 P5에서 재검토한다.
- 여행 시기, 축제, Wikipedia 기반 월별 날씨 경향, 이동 경로, 주변 관광 정보를 결합한 여정을 제안한다.
- RAG 기반 검색과 멀티스텝 에이전트로 추천 근거와 안정성을 확보한다.
- 추천 이유, 출처, 검증 상태를 함께 보여주는 설명 가능한 추천 경험을 제공한다.
- KICK(유사 취향 공개 일정·1탭 복제)을 P2 설계 범위에 포함해 선택 불안을 줄이고 저장·외부 링크 전환 가능성을 검증한다.

## 3.2 성공 기준

| 구분 | 기준 |
| --- | --- |
| 추천 경험 | 사용자가 국내 지역과 여행 시기, 여행 일정 길이, 자연어 취향 또는 지도 마커를 입력하면 조건에 맞는 소도시 1곳과 day-by-day 일정이 생성된다. |
| 근거 제공 | 추천 결과에는 추천 이유, 지도 링크, 월별 날씨 경향, 외부 탐색 링크, 출처·검증 상태 중 핵심 정보가 함께 제공된다. |
| KICK 전환 | 유사 취향 공개 일정이 노출되고 사용자가 1탭 복제 후 저장 또는 부분 수정으로 이어지는 흐름을 설계 산출물로 확보한다. |
| 범위 통제 | PoC에서는 P1 한국 강원·경북 데이터 기반 추천을 구현하고, P2 한국 전국 확장과 KICK은 설계 중심으로 정리하며, 일본 데이터 기반 추천은 P5 재검토 범위로 관리한다. |
| 확장성 | 신규 지역, 축제, 촬영지, 체험 정보, 외부 데이터 소스를 추가할 수 있는 공통 데이터 모델과 운영 구조를 유지한다. |

# 4. 서비스 개념

## 4.1 핵심 서비스

로브는 공개 추천 흐름에서는 선택 로그인 또는 온보딩을 통해 테마와 기본 국내 지역 선호를 입력받고, 카드 슬롯·지도 마커·AI 챗봇으로 일정 생성에 진입하게 한다. 챗봇은 국내 지역, 여행 시기(월), 여행 일정 길이(당일치기~2박 3일)를 먼저 확인한 뒤 사용자의 자연어 취향을 테마 또는 추천 진입점으로 해석한다.

소도시는 행정구역의 인구수만으로 판단하지 않는다. 관문 대도시(서울·부산 등)는 교통 안내에만 사용하고, 지역 거점도시와 인접 소도시·농산어촌을 분리해 관광객 집중도가 낮고 실제 여행 경험을 만들 수 있는 목적지와 주변 지역을 중심으로 본다.

PoC 기준 핵심 설계 원칙은 다음과 같다.

- 추천 1건의 최종 목적지는 소도시 1곳이다.
- 생성 일정의 주요 관광지와 체험은 추천 소도시 서비스 범위 안에서 구성한다. 숙소는 직접 추천보다 검색 링크로 연결한다.
- 공항·역·관문도시는 교통 안내에만 사용하고 일정의 목적지로 섞지 않는다.
- 적합 소도시를 확정할 수 없으면 여러 도시를 섞지 않고 추가 조건을 요청한다.

초기 일정 구성 원칙은 다음과 같다.

- 소도시를 정할 때 근거가 된 관광지 후보들만으로 전체 일정을 만들지 않는다.
- 각 날은 1개의 메인 관광지와 복수의 서브 관광지로 구성한다.
- 메인 관광지는 소도시 선정 근거가 되었던 관광지 후보에서 우선 채운다.
- 서브 관광지는 추천 소도시의 DynamoDB 관광지 데이터에서 다시 인출한다.
- 같은 장소 타입만 반복되지 않도록 균형을 둔다. 예를 들어 바다·해안 테마라도 해변만 연속으로 채우지 않는다.
- 식당은 관광지 추천과 분리한 식사 슬롯으로 제공하고, 사용자는 프론트엔드 목록에서 선택한다.

일정 수정 원칙은 다음과 같다.

- 초기 일정 이후의 세부 조정은 사용자가 주도한다.
- 사용자는 자연어로 변경을 요청하거나 DB에서 조회된 관광지 목록을 직접 보며 교체할 수 있다.
- 메인 관광지는 테마를 유지하는 범위에서 교체할 수 있다.
- 서브 관광지는 테마 고정 없이 사용자가 원하는 장소로 자유롭게 지정할 수 있다.

로브의 첫 질문은 “어디로 갈까요?”보다 “언제, 어느 국내 지역으로 떠나세요?”에 가깝다. 사용자가 도시를 이미 정했다고 가정하지 않고, 지역과 여행 시기부터 고정한 뒤 조건에 맞는 소도시를 좁혀 간다.

추천 경험의 핵심 aha는 “이런 곳이 있는 줄 몰랐는데, 지금 내 조건에 맞고, 진짜 갈 수 있겠다”이다. 이 문장은 세 요소로 나뉜다.

| 요소 | 의미 |
| --- | --- |
| 발견 | 사용자가 몰랐던 소도시를 만난다. |
| 납득 | 왜 이 도시가 지금 내 조건에 맞는지 이해한다. |
| 실행가능성 | 일정, 지도, 월별 날씨 경향, 외부 링크로 실제 행동이 가능해진다. |

추천 여정 funnel은 다음과 같다.

| 단계 | 목표 |
| --- | --- |
| 0. 첫 진입 | 온보딩 직후 행동 이력이 없어도 타이밍·축제·혼잡 정보로 첫 발견을 만든다. |
| 1. 추천 기준 | 왜 이 도시인지 설명해 납득을 만든다. |
| 2. 후속 질의 | 세션 내 신호로 미니 개인화를 제공한다. |
| 3. 일정 생성·저장 | day-by-day 일정과 저장으로 실행가능성을 만든다. |
| 4. 재방문 | 선언·행동·세션 신호를 누적해 개인화 정확도를 높인다. |

행동 이력이 없어도 타이밍, Wikipedia 기반 월별 날씨 경향, 축제, 혼잡은 누구에게나 적용 가능한 공통 신호다. Lovv는 초기 추천에서 이 공통 신호를 활용하고, 사용자의 세션 내 선택과 후속 질의를 빠르게 반영한다. 갔던 곳 기반 개인화는 재방문 단계의 무기로 분리한다.

취향 프로필은 세 계층으로 다룬다.

- 선언 신호: 온보딩에서 입력한 테마·국내 지역·여행 스타일
- 행동 신호: 저장, 클릭, 복제, 외부 링크 이동
- 세션 신호: 현재 대화에서 나온 여행 시기, 일정 길이, 동행, 이동 강도

KICK은 발견·납득·실행 다음의 선택 전환 장치다. 사용자는 나와 취향이 비슷한 공개 일정을 참고하고, 필요한 경우 1탭 복제와 챗봇 부분 조정으로 내 일정에 가깝게 바꿀 수 있다. 대표 기획서에서는 KICK을 추천 축 자체가 아니라 선택 불안을 줄이는 P2 설계 검토 범위로 다룬다.

KICK의 핵심 후보 흐름은 다음과 같다.

1. 로그인·온보딩으로 기본 취향을 확보한다.
2. 공개 일정 카드에서 유사 취향 근거를 보여준다.
3. 사용자는 공개 일정을 1탭으로 복제하거나 참고한다.
4. 챗봇이 일정 길이, 이동 강도, 동행 등 일부 조건을 조정한다.
5. 조정된 일정은 저장 또는 재추천 흐름으로 이어진다.

KICK 운영 원칙은 상세 설계에서 다음 방향으로 검토한다.

- 공개 일정 부족 시 가짜 사용자 일정을 만들지 않는다.
- 초기에는 명시적인 `Lovv 큐레이션`으로 노출한다.
- 유사성 표현은 실제 일치 근거만 표시하고 과장하지 않는다.
- 공개 일정 복제와 부분 수정은 개인정보, 공개 범위, 버전 관리 기준을 먼저 확정한 뒤 설계한다.
- 개인정보는 선택 공개, 익명화, 공개 범위 제한을 기본 방향으로 검토한다.

KICK 성공 기준은 다음과 같다.

- 공개 일정 노출
- 유사 취향 근거 확인
- 1탭 복제
- 챗봇 부분 수정
- 일정 저장
- 외부 링크 클릭
- 재방문

# 6. 프로젝트 범위

## 6.1 PoC 범위

PoC는 P1 구현을 핵심 범위로 한다. P2와 P3는 설계 산출물을 우선 만들고, 일정 여유가 있을 경우 일부 구현한다. KICK은 추천 축 자체가 아니라 선택 전환 장치이므로 P2 설계 범위에 별도 포함한다. 일본 데이터 기반 추천과 일본 현지 데이터 연동은 현재 수집 실패 리스크를 반영해 P5 재검토 범위로 내린다.

| 우선순위 | 기능 범위 | PoC | Production | 상태 |
| --- | --- | --- | --- | --- |
| P1 | 한국 강원·경북 데이터 기반 혼잡 회피 대체 여행지 추천, 국내 축제 기반 여정 추천, 당일치기~2박3일 일정 구성, Wikipedia 월별 날씨 경향 기반 추천 | 구현 | 고도화 | 핵심 |
| P2 | 한국 전국 확장, KICK(유사 취향 공개 일정·1탭 복제) | 설계 | 구현 검토 | 선택 구현 |
| P3 | 미디어 촬영지 추천, 철도 경로 기반 주변 관광지 안내, 지역 문화 체험 추천 | 설계 | 구현 검토 | 설계 |
| P4 | 스키 액티비티 여행 추천 | 정의 | Production 구현 검토 | Production |
| P5 | 일본 데이터 기반 추천, 일본 현지 데이터/API 연동, 별 관측 조건 기반 여행지 추천, 치유·웰니스 여정 추천, 예산 역산 추천 | 제외 | 검토 | 제외 |

PoC 데이터는 한국 강원·경북을 우선 수집 범위로 삼는다. 전국을 선제 구축하지 않고 3~5개 검증 후보 도시부터 깊게 구성한 뒤, 같은 공통 데이터 모델을 한국 전체 도시로 확장한다. 일본 데이터는 수집 경로와 검증 산출물이 확보될 때까지 P5로 보류한다.

## 6.2 제외 범위

다음 기능은 PoC 상세 요구기능에서 제외하고, 우선순위 및 단계별 추진 범위에만 기록한다.

- 일본 데이터 기반 추천

- B2C 과금·가격 모델

## 7.4 비즈니스 모델

B2C·B2B·B2G는 서로 다른 수익화 단계가 아니라 수요 검증, 성과형 제휴, 공익 협력을 분리해 관리하는 사업 트랙이다. 가격 실험, 수익성 시뮬레이션, 파트너별 제안 전문은 상세 문서에 둔다.

B2C 과금·가격 모델과 B2G 영리화 시나리오는 대표 기획서 범위에서 제외한다. 비즈니스 모델 세부 내용은 보조 문서 `supplemental/business_model.md`에서 관리한다.

### B2C: 수요·가치 검증 트랙

1차 타깃은 한국 강원·경북 소도시 여행자다. 서비스 가치는 정보 열람이 아니라 제한된 시간 안에서 실행 가능한 일정, 이동 실패 방지, 예약 가능한 다음 행동에 둔다.

핵심 KPI는 다음과 같다.

- 일정 저장률
- 공유·내보내기율
- 예약 파트너 클릭률
- 재방문율

### B2B: 성과형 제휴 트랙

B2B는 취향·일정 맥락을 예약 가능한 상품 탐색으로 연결하고, 성과형 제휴 가능성을 검토하는 추천 중개 트랙이다.

| 사용자 | 우선 검토 제휴 |
| --- | --- |
| 한국 강원·경북 여행자 | 국내 숙박·액티비티·지도·탐색 플랫폼 |
| 한국 전국 확장 여행자 | 지역별 숙박·액티비티·교통 탐색 플랫폼 |

수익 인식 기준은 총 예약액보다 승인 매출 기준으로 검토한다. 성과 구간, CPA, 추천 모듈·API, 화이트라벨, 공동브랜드 같은 장기 확장 후보는 보조 문서에서 관리한다.

### B2G: 비과금 공익 협력 트랙

B2G는 지자체·공공기관과 검증된 관광 정보를 “이번 달 여행지” 월간 큐레이션으로 노출하는 비과금 협력이다.

성과 지표는 재정 수치보다 다음 비재정 지표를 우선한다.

- 노출
- 클릭
- 일정 생성
- 일정 저장
- 공식 정보 이동
- 방문 의향

공식 정보 링크와 제휴 링크는 분리한다. 관문도시 방문은 소도시 성과로 보지 않고, 지역 숙박·지역 상품 클릭·공식 정보 이동을 별도로 집계한다.

# 8. 데이터 및 외부 연동

## 8.1 필요 데이터

우선 데이터는 한국 강원·경북 소도시 기본 정보, 축제·행사, 혼잡도·방문 통계, Wikipedia에서 추출한 월별 날씨 경향, 지도·장소, 출처·검증 정보로 구성한다. 일정 구성을 위해 소도시 선정 근거 관광지 후보와 DynamoDB에서 재인출할 서브 관광지 후보, 장소 타입, 식사 슬롯 후보를 함께 관리한다. 한국 전국 데이터는 P2 확장 데이터로 검토한다. 미디어 촬영지, 철도·교통, 지역 문화 체험, 공개 일정 데이터는 P3 이후 확장 데이터로 검토한다. 일본 데이터 기반 추천과 일본 현지 데이터 연동은 수집 경로가 복구될 때까지 P5 재검토 범위로 둔다.

| 구분 | 데이터 | 활용 |
| --- | --- | --- |
| 우선 | 소도시 기본 정보 | 목적지 설명, 지역, 태그, 추천 이유 구성 |
| 우선 | 축제·행사 정보 | 시기별 추천과 주변 여정 구성 |
| 우선 | 혼잡도·방문 통계 | 대체 여행지 추천, 수요 분산 근거, 관문도시 성과 오인 방지 |

| P2 확장 | 한국 전국 소도시 데이터 | 전국 City, Attraction, Festival, VisitorStatistics 확장 |

| P3 확장 | 공개 일정 데이터 | 익명화된 일정, 복제·저장 전환 측정 |

## 8.2 외부 연동

외부 연동은 지도·장소, 마커 간 이동 동선, 관광·방문 통계, 예약 딥링크, 공공 관광 데이터로 나뉜다. 월별 날씨 경향은 Wikipedia 원문에서 추출·전처리한 정적 추천 데이터로 관리하며, 월별 경향 외 날씨 데이터는 호출하거나 화면에 표시하지 않는다. 클라이언트에서 직접 호출하기 어려운 API는 딥링크 또는 백엔드 프록시로 대체한다.

| 출처 | 역할 |
| --- | --- |
| TourAPI / DataLab | 한국 관광지·축제·방문 통계 |
| Google Maps Platform / Kakao Maps | 지도, 장소, 외부 탐색 딥링크 |
| OpenRouteService | 일정 마커 간 이동 경로 폴리라인, 이동 시간·거리 계산, 후보 장소 간 Matrix 비교 |
| 국내 숙박·액티비티 플랫폼 | 예약·숙박·액티비티 탐색 링크 연결 |
| 공공 관광 데이터 | 관광지, 축제, 교통, 지역 공식 정보 |
| JNTO / 일본 관광청(JTA) / e-Stat / Yahoo Japan | P5 재검토: 일본 방문 통계, 지역 검증, 현지 탐색 딥링크 |

월별 날씨 경향, 추천 적기, 날씨 정합성 검증은 PoC에서 Wikipedia의 지역별 기후·날씨 표와 설명을 추출한 데이터셋을 기준으로 한다. 비·폭설·장마·태풍 등 해당 월의 여행 영향이 감지되면 추천 결과에서 대체 일정 추가 여부를 알림 팝업으로 확인하고, 사용자가 동의할 때만 실내·우천 적합 대체 일정을 일정에 추가한다.

OpenRouteService는 지도 SDK를 대체하지 않고, 일정에 배치된 관광지 마커 사이의 이동 경로와 시간·거리를 계산하는 보조 연동으로 사용한다. Directions API는 선택된 일정 순서의 경로 표시, Matrix API는 서브 관광지 후보 간 이동 부담 비교에 우선 활용한다. API Key는 백엔드 프록시 또는 서버 환경 변수에서 관리하고, 호출 실패 시 직선 연결 또는 장소 목록 중심 UI로 폴백한다.

# 9. 기술 방향

## 9.1 추천 구조

Lovv의 기술 방향은 Explainable RAG 기반 소도시 추천 엔진이다. 추천 파이프라인은 다음 단계로 구성한다.

1. 사용자 입력 수집
2. 국내 지역·여행 시기·일정 길이·테마·진입점 조건 분류
3. RAG 기반 후보 검색
4. 후보 필터링 및 선정
5. 메인 관광지·서브 관광지·식사 슬롯 기반 일정 구성
6. 추천 이유와 외부 정보 보강
7. 출처·갱신 시점·검증 상태 표시
8. P2 KICK 흐름 또는 후속 질의응답
9. 조건 변경 기반 재추천

## 9.2 시스템 원칙

- 추천 결과는 근거 데이터와 함께 제공한다.
- 추천은 단순 생성형 답변이 아니라 검색된 데이터와 검증 상태를 기반으로 구성한다.
- API Key는 개발·운영 담당자가 서버 환경 변수 또는 백엔드 프록시에서 관리한다.
- 클라이언트에서 직접 호출하기 어려운 API는 딥링크 또는 백엔드 프록시로 대체한다.
- 데이터 출처, 최신성, 검증 상태를 표시한다.
- 사용자가 후속 질문에서 조건을 바꾸면 동일한 추천 파이프라인을 재실행해 대체 소도시 또는 수정 일정을 제안한다.
- 초기 일정은 소도시 선정 근거 관광지만으로 채우지 않고, 선정 근거 관광지 기반 메인 관광지와 DynamoDB 재인출 기반 서브 관광지를 조합한다.
- 장소 타입이 과도하게 중복되지 않도록 균형을 두고, 식당은 별도 식사 슬롯에서 사용자가 선택하게 한다.
- 일정 수정에서는 메인 관광지는 테마 유지 조건에서 교체하고, 서브 관광지는 테마 고정 없이 사용자가 자유롭게 바꿀 수 있게 한다.
- 첫 화면은 가능한 한 사전계산·캐시를 활용하고, 이유 문구는 템플릿 또는 정형 근거를 우선 사용한다.
- 공개 일정 복제와 부분 수정은 P2 상세 설계에서 권한, 개인정보, 버전 관리 기준을 정리한다.
- PoC에서는 빠른 검증을 위해 핵심 흐름을 우선 구현한다.

## 9.3 데이터 파이프라인

데이터 파이프라인은 수집, 로컬 검증, 원본 보존, 정규화, 추천·검색 활용 계층으로 나눈다. S3 Raw Bucket, Lambda, DynamoDB 같은 구체 구현 방식은 데이터 수집 계획서, 데이터베이스 설계 명세서, 기술 명세서에서 관리한다.

1. 자동 수집
2. 로컬 검증 산출물 생성
3. 원본 데이터 보존
4. 배치 전처리
5. 정규화 데이터 적재
6. 추천·검색 계층에서 RAG와 운영 조회에 활용

# 11. 기대 효과

로브를 통해 여행자는 유명 관광지 중심의 반복적인 여행 탐색에서 벗어나, 덜 붐비면서도 자신의 취향과 여행 시기에 맞는 소도시 대체 여행지를 더 쉽게 발견할 수 있다. 단순히 장소 목록을 보여주는 것이 아니라 추천 이유, Wikipedia 기반 월별 날씨 경향, 지도, 외부 링크, 출처·검증 상태를 함께 제공하므로 실제 여행 계획으로 이어지는 의사결정 부담도 줄어든다.

유사 취향 공개 일정과 1탭 복제는 낯선 소도시 선택의 불안을 낮추고, 일정 저장, 외부 링크 클릭, 재방문으로 이어지는 전환 경로를 만든다.

지역 관광 운영자는 상대적으로 노출이 부족했던 소도시 관광 자원과 축제, 지역 문화 체험을 여행자에게 소개할 수 있다. 이를 통해 특정 대도시나 유명 관광지에 집중된 수요를 주변 지역으로 분산시키고, 지역 관광 자원의 활용도를 높이는 기회를 확보할 수 있다.

B2B 파트너는 고의도 맥락형 트래픽과 증분 거래를 확보할 수 있고, 지자체·공공기관은 B2G “이번 달 여행지” 노출을 통해 비재정 성과를 측정할 수 있다.

서비스 운영자는 RAG 기반 추천 근거, 사용자 피드백, 공개 일정 복제 퍼널, 데이터 검증 상태를 바탕으로 추천 정책을 점진적으로 개선할 수 있다.

| 관문도시 방문을 소도시 실적으로 오인 | 잘못된 시장 판단 | 지역 숙박·지역 상품 클릭·공식 정보 이동 별도 집계 |
| 공개 일정 개인정보 노출 | 신뢰·법적 위험 | 선택 공개, 익명화, 공개 범위 제한 기준 검토 |
| 제휴·교통 조건 의존 | 정책 변경 또는 지방 항공 노선 중단 시 수요 급락 | 파트너 다변화, 철도 대체 가능 소도시 우선, 노선 상태 재검증 |

상세 리스크는 보조 문서 `supplemental/business_model.md`, `supplemental/data_ops_scope.md`, `supplemental/kick_summary.md`에서 관리한다.

운영 리스크 해석 원칙은 다음과 같다.

- 입국자 수는 고유 사용자 수가 아니므로 여행 횟수 흐름 지표로 해석한다.
- 관문도시 방문을 소도시 성과로 보지 않는다.
- 축제·교통 데이터는 갱신 시각과 출처를 함께 표시하고, Wikipedia 월별 날씨 경향 데이터는 추출 기준일과 출처 URL을 함께 표시한다.
- 외부 API 제약이 생기면 딥링크, 캐시, 백엔드 프록시를 조합한다.
- 지방 항공 노선 중단 가능성을 고려해 철도 대체 가능 국내 소도시를 우선한다.
- 일본 데이터는 로컬 검증 산출물과 출처 정합성 기준이 확보되기 전까지 추천 근거로 사용하지 않는다.

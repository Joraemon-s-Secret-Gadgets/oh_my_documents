## 1. 무엇을, 왜 테스트했나 (목적·범위)

프론트엔드와 에이전트는 **V2 계약**이라는 한 지점에서 맞물린다. 화면이 규칙대로 요청을 만들어도 AI가 그 규칙을 안 지키면 서비스가 깨지고, 반대도 마찬가지다. 그래서 이 약속을 양쪽에서 확인하는 것이 이번 테스트의 목적이다.

- **화면 쪽 질문**: 클라이언트가 계약대로 요청을 만들고, 돌아온 응답을 제대로 화면에 그리고 저장·재조회하는가?
- **AI 쪽 질문**: 실제 배포된 AI가 그 계약을 실제로 이행해 올바른 응답·상태 전환·예외 처리를 하는가?

**검증 범위**: `일정 생성 → 추가 질문 답변 → 일부 수정 → 저장 → 저장 목록 → 저장 상세 재진입`이라는 전체 흐름과, 그 과정에서 대화가 끊기지 않도록 유지되는 표식(세션·스레드), 그리고 응답에 담기는 정보(상황 코드·일정 항목 형식).

## 2. 어떻게 나눠 테스트했나 (전략)

통합 지점(계약)을 두 축으로 나눠 확인하고, 두 결과를 계약 항목별로 맞대어 봤다.

| 축 | 무엇을 증명하나 | 방식 (쉬운 설명) |
|---|---|---|
| **화면 축 — E2E** | 화면이 규칙대로 요청을 만들고, 응답을 그리고·저장·재조회한다 | 브라우저 자동화(Playwright)로 실제 클릭·화면 흐름을 재현. 이때 서버는 **약속된 가짜 응답(모킹)**으로 대체해, 화면 로직이 규칙을 지키는지만 흔들림 없이 확인 |
| **AI 축 — 로컬 + 배포** | 실제 AI가 그 규칙을 지켜 동작한다 | 개발 PC에서 코드를 직접 실행(로컬)하고, **실제 배포된 AI를 진짜로 호출(배포)**해 응답·상태 전환·예외를 확인 |

> **왜 화면은 가짜 응답을 썼나?** 실제 AI 호출은 비용이 들고 응답 시간이 들쭉날쭉해, 화면 로직의 옳고 그름을 반복 검증하기엔 부적합하다. 그래서 화면 쪽은 **약속(계약)면**을 결정론적으로 검증하고, AI 쪽은 **실제 이행면**을 진짜 호출로 검증하는 역할 분담을 택했다. 로컬과 배포 시간 측정은 서로 섞지 않고 따로 기록했다.

## 3. 테스트 환경과 구성

### 3.1 화면(프론트엔드)

- 도구: **Playwright 1.61.1**(브라우저 자동화 도구), Chromium 브라우저
- 테스트 파일: `Lovv-pg/frontend/e2e/lovv-v2-itinerary.e2e.ts`
- 방식: 화면–서버 사이 V2 요청·응답 규칙을 **가짜 응답(모킹)**으로 검증. 실행 중 실제 AWS(API Gateway·Lambda·AgentCore)나 지도·이미지·폰트 서버를 호출하지 않도록 외부 통신을 차단(격리)했다.

### 3.2 AI(에이전트)

- 위치(Region): `us-east-1`
- 사용 도구(쉬운 설명): **Bedrock**(AI 언어모델 호출), **벡터 검색**(S3 Vectors, 의미 기반 장소·도시 검색), **데이터베이스**(DynamoDB, 장소·축제 상세 조회), **경로 계산**(ORS, 이동 시간)
- 로컬: 개발 PC 안에서 코드를 그대로 실행. AI 기억(Memory) 기능은 비용·과거 오류 이력 때문에 꺼 두고(`LOVV_MEMORY_ENABLED=false`) 실행
- 배포: 실제 올라간 AI를 **직접 호출**. 이때 "같은 작업을 이어간다"는 표식(세션·스레드·사용자 식별자)을 함께 넣어야 하며, 안 넣으면 AI가 대화 상태를 못 찾아 오류(500)를 낸다
- 모든 실제 호출은 각 **60초 제한**으로 실행
- 증빙 자료: 로컬 `docs/tasks/results/v2_agent_test_local/20260713T120000Z/`, 배포 `docs/tasks/results/v2_agent_test_deployed/20260713T120000Z/`

### 3.3 화면과 AI를 잇는 약속(V2 계약)

이 항목들이 곧 "통합"의 접점이며, 뒤에서 양쪽이 이걸 지키는지 확인한다.

- **요청 순서**: 생성 → 추가 질문 → 수정 → 저장 → 저장 목록 → 저장 상세
- **세션 이어가기**: 생성·추가 질문·수정이 같은 세션 표식을 사용 (같은 사용자의 이어지는 작업)
- **스레드 이어가기**: 추가 질문·수정이 생성 때 받은 같은 대화 표식을 재사용
- **현재 일정 전달**: 수정 요청에 "지금 화면의 일정과 순서"를 함께 보내, AI가 그 위에서 고침
- **상황 코드**: 날씨 위험 대체 제안 / 후보 부족 되물음 / 축제 없음 / 지원 밖 지역 등
- **일정 항목 형식**: 각 장소에 출처 종류, 실내·실외 구분, AI가 쓴 소개·추천 이유가 담김

## 4. 테스트 방법

### 4.1 화면 — 11단계 사용자 흐름

로그인·초기 설정을 마친 사용자가 여행 플래너에서 일정을 만들고, 추가 질문에 답하고, 일부를 수정·저장한 뒤, 마이페이지에서 다시 여는 **전체 흐름을 하나의 시나리오로** 재현했다. 각 단계마다 "규칙을 지켰는지"를 자동으로 확인(assert)한다.

### 4.2 AI — 4개 경로

| 경로 | 목적 (쉬운 설명) | 실행 |
|---|---|---|
| **경로 A — 규칙·회귀** | 코드를 바꾼 뒤에도 기본 규칙과 예전 동작이 그대로인지 자동 확인 | `pytest`(파이썬 테스트)로 실행 |
| **경로 B — 로컬 실행** | 개발 PC에서 생성·수정·재개를 실제로 돌려 봄 | 코드 직접 실행, 기억 기능 끔 |
| **경로 C — 배포 실행** | 실제 서비스에 올라간 AI를 진짜로 호출 | 60초 제한, 대표 사례 16개 + 반복 측정 |
| **이전 버전(V1)과 비교** | 같은 뜻의 요청을 옛 버전과 새 버전에 넣어 속도 비교 | 대표 4개(P01~P04) 각 1회 |

### 4.3 테스트 커버리지 — 왜 이 사례들인가

"많이"가 아니라 **에이전트가 실제로 갈라지는 경로를 한 번씩 덮는 최소 구성**을 목표로 사례를 골랐다.

- **생성 경로**: 단일/복수 테마, '조용함/활기' 혼잡 조건, 장소 직접 지정(anchor), 후보가 얕은 희소 테마(온천·휴양), 축제 포함/미포함 — 도시 선택·일정 생성이 조합마다 다른 경로를 타므로 대표 조합을 고루 포함.
- **수정 경로**: 한 장소 교체(검색 유무), 대표(Seed) 장소 교체, 같은 날/다른 날 여러 장소 교체, 하루 전체 재구성, 명시/자동 도시 변경 — 부분 수정의 각 분기를 개별 검증.
- **되물음·재개 경로**: 날씨 위험 대체, 후보 부족 조건 완화, 축제 없음 이어가기 — 멈췄다 이어가는 상태 유지를 검증.
- **예외·경계**: 지원하지 않는 지역 입력 — 안전 거부·경계 처리를 검증.

화면 쪽은 이 계약을 관통하는 대표 흐름 하나를 11단계로 세분해, 요청 형성부터 저장·재조회까지 한 번에 덮었다.

### 4.4 검증 기준 (합격 조건)

| 검증 항목 | 합격 조건 |
|---|---|
| 요청 순서 | 생성→추가질문→수정→저장→목록→상세가 설계 순서대로 전송·처리 |
| 세션·스레드 연속성 | 이어지는 요청이 같은 세션·대화 표식 유지, 재개가 이전 결과 위에서 이뤄짐 |
| 현재 일정 전달 | 수정 요청의 현재 일정·순서를 AI가 복원해 그 위에서 교체 |
| 상황 코드 | 날씨·후보 부족·축제 없음·지원 밖을 정직하게 코드로 반환·재개 |
| 일정 항목 형식 | 응답 항목에 출처·실내외·소개·이유 필드 존재 |
| 예외 처리 | 잘못된/지원 밖 입력을 오류 전파 없이 안전 처리 |
| 안정성 | 정상 시나리오 오류 0, 무한 반복·시간초과 0 |

### 5.2 약속(계약)을 지켰는가 — 확인 결과

- 요청 순서(생성 → 추가 질문 → 수정 → 저장 → 목록 → 상세)를 그대로 따름
- 세션 이어가기: 생성·추가 질문·수정이 같은 세션 표식 사용
- 스레드 이어가기: 추가 질문·수정이 생성 때 받은 같은 대화 표식 사용
- 현재 일정 전달: 수정 요청에 지금 화면의 일정·순서가 담김
- 저장 데이터: 최종 수정 일정·선호 조건·목적지·기간·테마가 모두 포함
- 재조회 일관성: 저장 목록·상세에서 최종 수정 결과가 유지됨
- 외부 통신 차단: 실제 AWS·지도·이미지·폰트 호출 없이 실행

### 5.3 실행이 안정적인가 — 반복·회귀

- 브라우저 자동화 테스트: 따로 실행 1회 통과, 전체 실행 1회 통과, 보강 실행 1회 통과 — **서로 다른 브라우저 실행에서 반복 통과**해 우연한 성공이 아님을 확인

### 6.5 실제로 어떤 도구를 썼나 (관측)

로컬 실행 로그에서 아래 도구 호출을 실제로 확인했다 — 즉 AI가 말로만 답한 게 아니라 진짜 검색·계산을 거쳤다는 증거다.

- **AI 언어모델 호출(Bedrock)**: 요청 해석·설명 문구 생성
- **의미 검색(벡터)**: 도시·장소 후보 찾기
- **데이터베이스 조회**: 장소·축제 상세 채우기
- **경로 계산(ORS)**: 이동 가능성 확인

| 흐름 | 내부 처리시간 | 의미 검색 | DB 조회 | AI 모델 호출 | 경로 계산 |
|---|---:|---:|---:|---:|---:|
| 단일 해안 | 7.446초 | 3회 / 261ms | 2회 / 53ms | 2회 / 4.146초 | 1회 / 481ms |
| 자연+해안 | 8.883초 | 4회 / 331ms | 2회 / 58ms | 2회 / 5.553초 | 1회 / 424ms |
| 장소 직접 지정 | 7.093초 | 1회 / 106ms | 1회 / 43ms | 2회 / 4.155초 | 1회 / 435ms |
| 축제 포함 | 17.031초 | 4회 / 282ms | 2회 / 22ms + 1회 / 180ms | 2회 / 13.977초 | 1회 / 432ms |

정리하면 — 의미 검색은 100~330ms, DB 조회는 20~60ms로 **1초를 한참 밑돌고**, 시간은 사실상 전부 **AI 언어모델 호출(4~14초)**에 쏠린다. 즉 최적화 대상은 검색·DB가 아니라 LLM 호출이다.

### 6.6 이전 버전(V1)과 속도 비교

같은 뜻의 요청을 옛 버전과 새 버전에 각각 넣어 전체 소요 시간을 비교했다.

| 흐름 | V1(옛) | V2(새) | 해석 |
|---|---:|---:|---|
| 단일 해안 | 22.949초 | 15.131초 | V2가 7.818초 빠름 |
| 자연+해안 | 37.634초 | 16.040초 | V2가 21.594초 빠름 |
| 장소 직접 지정 | 12.730초 | 14.294초 | V2가 1.564초 느림 |
| 축제 포함 | 21.613초 | 23.700초 | V2가 2.087초 느림 |

새 버전은 의미 검색을 **병렬로**, 상세 조회를 **한 번에 묶어**(옛 버전은 하나씩 여러 번) 처리하도록 바꿨다. 그래서 검색·DB 왕복이 줄었고, 남은 최적화 여지는 AI 모델 호출 쪽이다.

### 6.7 대표 테스트 케이스 (정식)

각 경로의 대표 케이스를 **전제 → 입력 → 실행 단계 → 기대 → 실측 → 판정** 양식으로 기술한다.

**TC-A — 생성 + 축제 (핵심 복합 경로 · 배포 실행)**

| 구분 | 내용 |
|---|---|
| 전제 | 확정 축제가 있는 도시·달이 데이터에 적재, 배포 런타임 직접 호출 |
| 입력 | 축제 포함 생성 요청(테마 + 축제 포함 + 여행 월), 세션·스레드 표식 주입 |
| 실행 단계 | ① Intent 요청 정리 → ② Festival 확정 축제 판별 → ③ City Select 도시 선택 → ④ Planner 일정 생성·축제 배치 → ⑤ 응답 |
| 기대 | HTTP 200, 확정 축제 포함 일정, 날씨 대체 제안 |
| 실측 | HTTP 200 · **17.955초**, 축제 포함 city/month 경로, 날씨 대체 제안 동반 |
| 판정 | **Pass** — 축제 판별·배치가 한 요청에서 정상 |

**TC-B — 되물음 → 이어가기 (상태 유지 · 로컬 실행)**

| 구분 | 내용 |
|---|---|
| 전제 | 야외 비중이 높은 일정에서 날씨 위험이 감지됨 |
| 입력 | ① 생성 요청 → '대체 볼까요?' 되물음 → ② 같은 세션에서 '대체 보기' 선택으로 재개 |
| 실행 단계 | 생성 → `weather_alternative_available` 반환(대기) → 사용자 선택 → 중단 지점부터 재개 → 실내 위주 대체 일정 |
| 기대 | 같은 세션에서 대체 일정으로 정상 재개, 무한 반복·오류 0 |
| 실측 | 생성 **35.991초**(`weather_alternative_available`) → 재개 **8.978초**, 동일 session 대체 일정 정상. 기억 off 기록, 무한 반복·시간초과 0 |
| 판정 | **Pass** — 되물음 후 상태를 잃지 않고 이어감 |

**TC-C — 명시 도시 변경 (별도 경로 + 상태 유지 · 로컬 실행)**

| 구분 | 내용 |
|---|---|
| 전제 | 생성으로 동해시 일정이 만들어진 상태(선행 상태 필요) |
| 입력 | 같은 세션에서 "강릉시로 바꿔줘" |
| 실행 단계 | 도시 변경은 부분 수정이 아니라 **City Select부터 다시 도는 별도 경로**로 처리 |
| 기대 | 도시 식별자 교체, 항목 수 유지 |
| 실측 | 생성 **27.525초**(동해시) → 변경 **13.080초**(강릉시), `KR-51-170 → KR-51-150`, **10개 항목 유지** |
| 판정 | **Pass** — 별도 경로로 정상, 상태 이어감 |
| 비고 | 이 요청을 **단독**(선행 생성 없이)으로 돌리면 선행 상태가 없어 실패 — 요청 특성상 정상(§6.1 참조) |

**TC-D — 지원 밖 지역 (경계·예외 · 배포 실행)**

| 구분 | 내용 |
|---|---|
| 전제 | 지원하지 않는 지역을 요청 |
| 입력 | 지원 밖 지역 입력, 배포 직접 호출 |
| 실행 단계 | 경계 감지 → 예외 처리 응답 |
| 기대 | 오류 전파 없이 안전한 경계 처리 |
| 실측 | HTTP **500 · 0.998초** — `unsupported_region` 코드가 응답 형식 허용 목록에 없어 실패. **코드에 허용 목록 추가 반영**, 규칙 회귀 통과, 다음 배포에서 재확인 예정 |
| 판정 | **조건부** — 경계는 감지했으나 응답 형식 계약 불일치로 500. 소스 보완 완료, 배포 재확인 필요 |

## 7. 통합 정합성 — 화면과 AI가 같은 약속을 지키는가 (핵심)

화면이 만든 약속을, 실제 배포된 AI가 똑같이 이행하는지 **항목별로 맞대어** 확인했다.

| 약속(계약) 항목 | 화면 쪽에서 확인 | AI 쪽에서 확인 | 일치 |
|---|---|---|---|
| 요청 순서(생성→추가질문→수정→저장) | 11단계 흐름에서 순서·조건 통과 | 로컬·배포에서 생성→수정·재개 이어짐 | ✅ |
| 세션 이어가기 | 세 요청이 같은 세션 표식 사용 | 세션 표식을 넣자 같은 작업으로 재개 | ✅ |
| 스레드(대화) 이어가기 | 생성 때 받은 대화 표식을 재사용 | 재개 사례가 같은 대화로 이어짐 | ✅ |
| 현재 일정 전달 | 수정 요청에 현재 일정·순서 포함 | 전달된 상태를 복원해 그 위에서 교체 | ✅ |
| 추가 질문·상황 코드 | 선택지·도움말 정상 표시 | 날씨 대체·후보 부족·축제 없음 코드 반환·재개 | ✅ |
| 일정 항목 형식 | 저장·재조회에서 수정 결과 유지 | 출처·실내외·소개·이유가 응답에 담김 | ✅ |
| 예외·경계 처리 | 잘못된 요청 차단·기록 | '지원 밖 지역' 경계 처리(목록 반영) | ✅ |

**요약**: 7개 약속 항목 모두, 화면이 기대한 것과 AI가 실제로 한 것이 일치했다.

## 8. 종합 판정

- **화면**: 핵심 흐름(만들기·추가 질문·수정·저장·목록·상세 재진입) 11단계 전부 통과, 약속·세션·대화 연속성 일치, 회귀 323건 통과.
- **AI**: 규칙·회귀 46/46, 로컬 실행 36/35(시간초과 0), 배포 실행 16/15(정상 응답), 이전 버전 비교 8/8. 실제 AI 호출·의미 검색·DB 조회·경로 계산까지 관측.
- **통합**: 화면이 만든 V2 약속을 실제 배포 AI가 똑같이 이행함을 7개 항목에서 확인했다. **통합 약속 정합성 조건부 합격** — 주요 경로는 일치했고, '지원 밖 지역' 1건만 다음 배포에서 재확인 예정이다.

## 9. 부록 — 요청·응답 형식 예시

> **주의**: 아래는 캡처된 실측 dump가 **아니라**, 문서에 기재된 필드와 실제 사례값으로 **재구성한 형식 예시(스키마)**다. 요청·응답에 계약 항목이 어떻게 담기는지 보여주기 위한 것이며, 실제 필드명·구조는 구현에 따라 다를 수 있다. 실제 캡처 원본은 §3.2의 증빙 자료 경로에 있다.

**① 생성 요청 (화면 → AI)**

```json
{
  "requestType": "create",
  "entryType": "chat",
  "destination": "동해시",
  "tripDuration": "1박 2일",
  "themes": ["sea_coast"],
  "includeFestivals": true,
  "sessionId": "<고정 세션 표식>"
}
```

**② 추가 질문 응답 (AI → 화면) — 상황 코드 동반**

```json
{
  "responseStatus": "clarification",
  "reasonCode": "weather_alternative_available",
  "question": "이 시기 야외 일정에 날씨 위험이 있어요. 대체 일정을 볼까요?",
  "options": [
    { "optionId": "see_alternative", "label": "대체 일정 보기", "helperText": "실내 위주로 다시 구성합니다" },
    { "optionId": "keep_current",    "label": "지금 일정 유지", "helperText": "현재 일정을 그대로 사용합니다" }
  ],
  "sessionId": "<고정 세션 표식>",
  "threadId": "<생성 응답에서 받은 대화 표식>"
}
```

**③ 수정 요청 (화면 → AI) — 현재 일정 동봉**

```json
{
  "requestType": "modify",
  "message": "1일차 첫 장소 바꿔줘. 더 조용한 곳을 원해.",
  "currentItinerary": {
    "day1": [
      { "order": 1, "placeName": "묵호등대" },
      { "order": 2, "placeName": "..." }
    ]
  },
  "sessionId": "<고정 세션 표식>",
  "threadId": "<생성 응답에서 받은 대화 표식>"
}
```

**④ 수정 응답 항목 (AI → 화면) — 교체된 항목의 형식**

```json
{
  "responseStatus": "completed",
  "changedItem": {
    "day": 1,
    "order": 1,
    "placeName": "한섬감성바닷길",
    "sourceType": "reserve_pool",
    "indoorOutdoor": "outdoor",
    "body": "한섬해변을 따라 걷는 조용한 산책로로, 붐빔이 적어 여유로운 아침에 어울립니다.",
    "reason": "더 조용한 곳을 원하셔서, 붐빔이 덜한 장소로 교체했습니다."
  },
  "unchangedPreserved": true
}
```

각 예시에서 확인되는 계약 항목: **세션·스레드 표식**(①~③), **상황 코드**(②), **현재 일정 전달**(③), **일정 항목 형식**(④: `sourceType`·`indoorOutdoor`·`body`·`reason`), **나머지 보존**(④: `unchangedPreserved`)

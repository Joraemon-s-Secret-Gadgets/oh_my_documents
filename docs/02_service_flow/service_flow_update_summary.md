# 로브 서비스 흐름 추가 요청 요약

> 문서 성격: 보조 Markdown
> 대표 문서: `02_service_flow.md`
> 정리 기준: 이 문서는 서비스 흐름 보강 요청을 정리한 참고 문서이며, 공개 정본은 `02_service_flow.md`로 유지한다.

## 반영 요청

Lovv 서비스 기획 문서에 아래 `User Flow`와 `Service Flow` 내용을 추가한다.

## User Flow

| 단계 | 화면 / 상태 | 사용자 행동 | 시스템 반응 | 사용자에게 보이는 결과 | 완료 조건 |
| --- | --- | --- | --- | --- | --- |
| **1** | **진입 및 인증** | 사용자가 Lovv에 접속 후 소셜/자체 로그인 진행 | 백엔드에서 OAuth 인증 후 JWT(Access/Refresh Token) 발급, 유저 상태 및 기존 취향/저장 일정 로드 | 로그인 세션 유지 상태로 맞춤형 메인 대시보드 노출 | 로그인 완료 및 메인 화면 진입 |
| **2** | **목적지 탐색 (1단계)** | "언제, 어느 나라로 떠나세요?"에 국가 및 시기 입력 | 입력된 시기의 공공 관광 데이터(혼잡도, 계절성)를 기반으로 후보군 필터링 준비 | 국가/시기 선택 완료 상태 UI 반영 | 국가 및 시기 확정 |
| **3** | **테마 설정 (2단계)** | 자연어(예: "드라마 촬영지 중심의 조용한 곳") 혹은 테마 태그 칩 선택 | RAG 엔진이 벡터 DB에서 조건에 맞는 소도시 후보 검색 및 스코어링 수행 | 오사카/서울 등 대도시와 유사하지만 덜 붐비는 '소도시 후보군 비교 카드' 노출 | 추천 소도시 리스트 확인 |
| **4** | **소도시 확정 및 챗봇** | 추천된 소도시 중 1곳을 선택하여 "AI 일정 짜기" 진입 | 대화형 워크스페이스가 열리며, 선택한 소도시의 날씨 경향 및 축제 정보 팝업 | "○○ 도시로 결정하셨군요! 현지 축제 정보를 일정에 포함할까요?" 안내 메시지 노출 | 대화 시작 상태 진입 |
| **5** | **동적 일정 생성 (3단계)** | 사용자가 세부 요구사항 입력 및 축제 포함 여부 선택 | 멀티스텝 에이전트가 외부 API(Tabelog 평점, 카카오맵, 축제 데이터) 실시간 조회 후 동선 최적화 | 추천 이유(Explainable 근거)와 데이터 출처 뱃지가 포함된 시간대별 타임라인 UI 생성 | 초안 일정 및 근거 확인 |
| **6** | **실시간 피드백 및 조정** | "이 날은 덜 걷게 해줘", "카페 추가해줘" 등 추가 자연어 입력 | 에이전트 파이프라인 재가동, 특정 일정 블록 실시간 업데이트 및 기상 악화 대비 대체 일정 토글 반영 | 실시간 갱신된 타임라인 및 실내 중심 대체 일정 확인 | 만족하는 일정 확정 |
| **7** | **일정 저장 및 연동** | 일정을 마이페이지에 저장하고 맛집/숙소 상세 정보 클릭 | 중앙 RDB에 대화 로그(Transcript) 및 완성된 일정(Plan) 저장, 외부 플랫폼(Tabelog, Jalan 등) 딥링크 생성 | 내 저장 목록(마이페이지) 반영 및 현지 신뢰 플랫폼으로의 클린 딥링크 이동 | 외부 예약 이동 혹은 마이페이지 저장 완료 |

## Service Flow

| 단계 | Trigger | Frontend 처리 | 상태 / 데이터 관리 (Backend & DB) | 프로덕션 비즈니스 로직 (AI & Infra) | 결과 |
| --- | --- | --- | --- | --- | --- |
| **1** | **앱 로드 및 세션 검증** | 브라우저 쿠키에 저장된 Refresh Token 유무 확인 및 백엔드 전송 | `Cookie: Refresh Token` / `Memory: Access Token` | 백엔드 서버에서 토큰 유효성 검증 후 유저 고유 ID 식별. 만료 시 재발급 처리 | 유효 시 메인 화면으로 진입, 토큰이 없거나 만료 시 로그인 화면 리다이렉트 |
| **2** | **소셜/자체 로그인 완료** | OAuth 인가 코드(Authorization Code) 또는 로그인 폼 데이터를 백엔드로 전송 | `POST /v1/auth/login`<br>`Server RDB (User Table)` | 1. 자체 로그인의 경우 비밀번호 단방향 암호화(Bcrypt) 검증<br>2. 소셜 로그인의 경우 카카오/구글 API를 통해 유저 정보 식별 후 회원가입 또는 기존 회원 매핑<br>3. 유저 취향 정보 조회 | JWT 토큰 발급 및 보안 쿠키 설정 완료, 사용자 개인화 메인 대시보드 출력 준비 |
| **3** | **목적지 및 테마 전송** | 사용자가 입력한 국가, 시기, 자연어 취향 데이터를 백엔드로 요청 | `POST /v1/recommendations/cities`<br>`State: searchCriteria` | 1. 유저 자연어 쿼리를 임베딩하여 벡터 DB(소도시 특징, 미디어 성지, 테마 정보) 유사도 검색<br>2. 공공 데이터 API 기반으로 해당 시기의 혼잡도 및 계절성 필터링 적용 | 오사카/서울 등 메이저 대도시를 대체할 수 있는 신뢰성 높은 소도시 후보군 데이터 반환 |
| **4** | **소도시 선택 및 대화 시작** | 특정 소도시 확정 후 AI 일정 생성 챗봇 진입 버튼 클릭 | `POST /v1/chat/sessions`<br>`Server RDB (ChatSession Table)` | 1. 새로운 대화 세션 ID 생성 및 RDB 기록<br>2. 선택된 소도시의 축제 정보, 기상 경향 컨텍스트를 챗봇 초기 프롬프트에 주입 | 초기 어시스턴트 안내 메시지와 소도시 맞춤형 상태 칩(진행률) UI 렌더링 |
| **5** | **동적 AI 일정 생성 요청** | 사용자가 세부 조건(기간, 이동 성향 등) 입력 후 메시지 전송 | `POST /v1/chat/messages` (Stream 응답 형태 권장)<br>`Server RDB (Message Table)` | 1. Intent 분류를 통해 동선 생성/수정 요청 인식<br>2. 멀티스텝 에이전트 구동: 외부 API(Tabelog 평점 데이터 파싱, 국내 카카오맵 API 등) 실시간 연동 및 동선 최적화<br>3. LLM Structured Outputs 기능을 활용하여 정형화된 JSON 포맷으로 빌드 | 데이터 출처 및 검증 상태 뱃지가 포함된 시간대별 타임라인(오전/오후/저녁) 및 챗봇 응답 출력 |
| **6** | **실시간 일정 피드백** | 사용자가 "카페 추가", "동선 최적화" 등 추가 수정 요구사항 입력 | `PUT /v1/plans/{planId}/draft`<br>`Server RDB (PlanDraft Table)` | 1. 기존 생성된 `PlanDraft` 객체와 신규 요구사항을 에이전트 파이프라인에 재투입<br>2. 특정 타임라인 블록만 수정하고 기상 상황에 대비한 '실내 중심 대체 일정' 데이터 동적 결합 | 수정된 타임라인 레이아웃 및 대체 일정 토글 UI 실시간 반영 |
| **7** | **최종 일정 저장 및 외부 연동** | 사용자가 '마이페이지에 일정 저장' 또는 '맛집/숙소 상세 정보' 클릭 | `POST /v1/plans/{planId}/save`<br>`Server RDB (Plan Table: status=SAVED)` | 1. `PlanDraft` 상태를 `SAVED`로 변경하고 유저 ID와 영구 매핑<br>2. 저장 시 발생한 피드백 로그를 축적하여 향후 RAG 가중치 업데이트 데이터로 활용<br>3. 파트너 플랫폼(Tabelog, Jalan, 라쿠텐 등) 연동을 위한 클린 딥링크 생성 | 마이페이지 내 일정 보존 및 신뢰할 수 있는 현지 예약 플랫폼으로의 외부 이동 완료 |
| **8** | **지자체 데이터 검증 및 업데이트** | (Admin/B2B 영역) 지자체나 로컬 파트너가 축제/체험 정보 업데이트 제안 | `POST /v1/admin/contents`<br>`Vector DB / Server RDB` | 1. 관리자 대시보드를 통해 데이터 접수 후 승인 프로세스 진행<br>2. 승인 완료 시 벡터 DB에 임베딩 및 업데이트<br>3. 유저 화면 내 "○○관광공사 검증 완료" 뱃지 로직 활성화 | 일반 유저가 검색할 때 최신화된 신뢰성 기반의 로컬 소도시 데이터가 추천 풀에 즉시 반영 |

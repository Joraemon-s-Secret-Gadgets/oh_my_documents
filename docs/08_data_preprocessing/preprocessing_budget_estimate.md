# 1. 목적

본 문서는 S3 Raw 적재, Lambda 전처리, DynamoDB 적재, Bedrock 기반 LLM 태깅을 1주일 동안 PoC로 운영할 때 필요한 예상 비용을 임시로 기록한다.

비용은 실제 AWS 사용량, 모델 선택, 재처리 횟수, 로그 보존량, 환율에 따라 달라질 수 있다. 본 문서는 예산 검토를 위한 1차 산정값으로 사용한다.

# 5. LLM 태깅 비용 주의사항

LLM 태깅은 모델 선택에 따라 비용 차이가 크다.

저가 태깅 모델을 사용하면 20,000건 기준 LLM 비용은 약 `$2.76`, 원화 약 4,179원 수준으로 예상한다.

반면 Claude Sonnet급 모델로 전건 태깅을 수행하면 동일한 20,000건 기준으로 LLM 비용만 약 `$168`, 원화 약 254,352원 수준까지 증가할 수 있다.

따라서 PoC 기본 정책은 다음과 같이 둔다.

- 규칙 기반 태깅을 1차로 수행한다.
- 카테고리·설명·국가별 표현 차이 때문에 애매한 데이터만 LLM으로 보강한다.
- LLM 결과에는 `tag_source`, `tag_confidence`, `tag_status`를 남긴다.
- 운영 전환 시 Lambda 내부 직접 호출보다 비동기 태깅 Job으로 분리한다.

# 6. 예산 관리 기준

| 관리 항목 | 기준 |
| --- | --- |
| 리전 | `us-east-1` 단일 리전 기준 |
| S3 | Raw, processed, failed Prefix를 분리하고 불필요한 중복 원본 저장을 줄인다. |
| Lambda | Raw Prefix 또는 Manifest 기준 배치로 처리하되, 타임아웃 위험이 있으면 작은 배치로 나눈다. |
| DynamoDB | PoC는 On-Demand로 시작하고, 사용 패턴이 고정되면 Provisioned 전환을 검토한다. |
| Bedrock | 저가 모델 또는 Batch/Flex 옵션을 우선 검토한다. |
| CloudWatch | 디버그 로그는 PoC 기간에만 상세하게 남기고 보존 기간을 제한한다. |

# 본론 — 시스템 흐름 슬라이드 구성 (1장 / 2~3장 버전)

> 목적: 본론 진입부 **B1 "시스템 흐름"** 슬라이드를 두 가지 분량으로 제안한다.
> 근거 문서: `02_service_flow.md`(§4 Service Flow), `sooa/user_system_flow.md`(§3 시스템 흐름도 — 전체 구성/시퀀스/상태), `06_technical_spec.md`(§2 시스템 5계층), `body_revision_proposal.md`(B1).
> 단계: 중간발표 (PoC). 큰 흐름 위주 — 디테일은 발표자 노트로, 에이전트 내부는 B3에서.
> 색상: `color-map.md` 토큰(Main Orange `#F26518`, Card Peach `#FCE7DB`, Soft Beige `#FFF3E7`, Dark Charcoal `#222222`, Sub Text Gray `#666666`).

> **결정: A. 1장 버전으로 진행한다.** (2026-06-10)
> 이유: 본론은 **B1 시스템 흐름 → B2 유저 플로우 → B3 에이전트 플로우** 순으로 좁혀지는 깔때기 구조다. 동적 시퀀스(검색→검증→선정→일정)는 **B3가 확대해서 받으므로**, B1은 5계층 정적 큰 그림 1장으로 위치만 잡는다. 2~3장 버전의 B1-2(추천 생성 흐름)는 B3 노드와 중복되어 채택하지 않는다.
> 아래 **B. 2~3장 버전은 대안으로 보존**한다(신뢰 설계를 본론 초반에 강조하고 싶을 때는 B1-2를 빼고 **B1-1 + B1-3 = 2장** 조합 사용).

## 0. 두 버전 한눈에

| 버전 | 슬라이드 | 언제 쓰나 | 핵심 |
| --- | --- | --- | --- |
| **A. 1장 버전 ✅ 채택** | 1장 | 기본 진행안(서론+본론 12장 유지) | 5계층 파이프라인 큰 그림 1개. 가이드 원안(S6=B1) 그대로. |
| **B. 2~3장 버전 (대안 보존)** | 2~3장 | 신뢰 설계를 초반에 강조하고 싶을 때 | ① 5계층 구성도 → ② 추천 생성 시퀀스(B3와 중복 → 채택 안 함) → ③ 설계 원칙. 채택 시 **B1-1+B1-3 2장** 권장. |

> 공통 원칙: **슬라이드당 핵심 메시지 1개.** 화면엔 도식과 한 줄 카피만, 설득 논리·수치·예외는 발표자 노트로. 에이전트 노드 세부는 여기서 풀지 않고 **B3 에이전트 플로우**로 넘긴다(중복 금지).

# A. 1장 버전 (= 대표문서 S6 / 본론 진입)

**핵심 메시지 (화면 카피)**
> **Explainable RAG — 입력부터 근거 있는 추천·일정까지, 한 흐름으로**

**화면 구성 (5계층 파이프라인 도식 1개)**

좌→우 흐름 위에 5계층 레이어를 얹는다.

```
[사용자 입력]        [Backend API]        [Recommendation Agent]      [Data Store]            [근거 포함]
 국가·시기·테마   →   인증·검증·프록시  →   조건분류→검색→선정→일정  →  목적지·축제·기상경향  →  추천 + 일정
 (자연어/지도)        ·추천 실행                                      ↕ External APIs(지도·날씨·딥링크)
```

| 계층 | 한 줄 역할 (화면) |
| --- | --- |
| Client Web | 온보딩·지도·챗봇·결과·마이페이지 |
| Backend API | 인증·검증·외부 프록시·추천 실행 |
| Recommendation Agent | 조건 분류 → 검색 → 선정 → 일정·이유 생성 |
| Data Store | 목적지·축제·월별 기상 경향·저장·검토 이력 |
| External APIs | Google/Kakao Maps, WeatherAPI, Yahoo Japan 딥링크 |

**UX 모멘트 ↔ 시스템 동작 (도식 위 주석 3개)** — "이 사용자 경험은 시스템의 이 부분이 만든다"를 한 장에 보이게 한다. 파이프라인은 풀지 않고(=B3), 경험과 컴포넌트만 잇는다.

| 사용자가 겪는 경험 | 만들어내는 시스템 동작 | 근거 |
| --- | --- | --- |
| ① "왜 이곳인지" + 출처·검증 뱃지 | Agent(검색→축제 검증→Validator) + Data Store(확정 원장·출처) | `02_service_flow` §7·§8.1 |
| ② "덜 걷게/카페 추가" 실시간 수정 | Agent가 기존 PlanDraft 재평가 → 특정 일정 블록만 갱신 | `02_service_flow` §8.3 |
| ③ "지도·숙소로 바로 이동" | External Proxy가 국가별 딥링크 생성 | `02_service_flow` §4 (7단계) |

> 화면 표현: 5계층 도식 위에 ①②③ 말풍선/콜아웃만 얹는다(텍스트 3줄). 상세 동작은 노트·B3로.

**시각 도식 (1장 버전 미리보기)**

![Lovv 시스템 흐름 B1 — 5계층 파이프라인 + UX 모멘트 매핑](system_flow_b1.svg)

> 위 도식이 화면에 들어갈 1장의 예시. 아래 5계층(사용자 입력 → Backend → 추천 에이전트 → 데이터 스토어 → 추천+일정, External은 프록시)이 시스템 동작, 위 코랄 카드 3개가 그 동작이 만드는 사용자 경험(①근거 ②실시간 수정 ③외부 이동)이다. 점선이 "경험 ↔ 동작"을 잇는다. 색: `color-map.md` 토큰.

**화면용 / 노트용**
- *화면*: 5계층 파이프라인 도식 1개 + "Explainable RAG" 한 줄. 디테일 표기 최소화.
- *노트*: 핵심 원칙 3가지 —
  1. 외부 API 응답은 **경계에서 검증**하고 신뢰하지 않는다(`06_technical_spec` §1.2).
  2. 추천은 실시간 예보가 아니라 **월별 기상 경향 통계** 기반(`02_service_flow` §8.2).
  3. **대화 원문 미저장** — 구조화 선호·저장 일정·피드백만 보관(`user_system_flow` §5).

**서론 복선 — 예고만 (완전 회수 X)**
- S5 컨셉① 발견 / ② 일정 / ③ 근거가 **이 한 흐름 위에 모두 올라간다**는 점만 가볍게 — "흩어진 걸 한 흐름으로".
- **컨셉↔에이전트 1:1 매핑은 여기서 하지 않는다.** 에이전트 노드가 화면에 뜨는 **B3에서 실제 회수**하고, 데모(B6)가 눈으로 최종 확인. 여기는 B3로 넘기는 **다리(예고)** 역할.

**발표자 노트 (큐)**
- "입력 → 검색 → 선정 → 일정, **한 흐름**" · *(큰 그림만, 빠르게)*
- "핵심은 **근거가 보이는** 추천 = Explainable RAG" · *(차별점 한 번 짚기)*
- "우리가 약속한 **3가지(발견·일정·근거)가 이 흐름 위에** 다 올라간다 — **어디서 어떻게 되는지는 잠시 뒤**" → B3 예고 *(여기선 매핑 X)*
- "지금은 PoC — 정적 시드/JSON 기반, 추천 API 최소 구현" *(노트, 정직한 톤)*

# B. 2~3장 버전

1장의 큰 그림을 **B1-1 정적 구조(무엇으로 이뤄졌나)** 와 **B1-2 동적 흐름(어떻게 도는가)** 으로 쪼갠다. (선택) **B1-3 설계 원칙** 한 장을 더 둘 수 있다.

## B1-1 — 시스템 구성도 (5계층, 정적 구조)

**핵심 메시지 (화면 카피)**
> **5개 계층이 각자의 책임만 — 느슨하게 연결된 한 시스템**

**화면 구성 (계층 블록 다이어그램, `user_system_flow` §3.1 축약)**

```
┌ Client Web ──────────────┐     온보딩·지도·챗봇·결과·마이페이지 / PoC Local Storage
│            ↓ ↑           │
├ Backend API ─────────────┤     Auth · Preference · Destination · Recommendation · External Proxy · Admin
│            ↓ ↑           │
├ Recommendation Agent ────┤     Intent → Supervisor → Retriever → Festival Verifier → Ranker → Writer → Validator
│            ↓ ↑           │
├ Data Store ──────────────┤     MySQL(원장) · VectorDB/RAG Index · DynamoDB(로그)
└ External APIs ───────────┘     Google/Kakao Maps · WeatherAPI · Yahoo Japan 딥링크 · 공식 관광/축제 출처
```

| 계층 | 책임 | 중간발표에서 강조할 점 |
| --- | --- | --- |
| Client Web | 입력·지도·결과·로컬 저장 | PoC는 로컬 스토리지로 선호/저장 대체 |
| Backend API | 인증·검증·프록시·추천 실행 | **외부 응답을 경계에서 검증**(신뢰 X) |
| Recommendation Agent | 조건분류→검색→선정→일정·이유 | 단일 LLM 아님, 멀티스텝(상세는 B3) |
| Data Store | 확정 원장 + 파생 인덱스 분리 | MySQL=확정 근거, Vector=의미검색 |
| External APIs | 지도·날씨·딥링크 | 날씨는 **표시용**, 추천 점수엔 미사용 |

**발표자 노트 (큐)**
- "관심사를 5개로 나눴다 — 화면은 화면, 추천은 추천, 데이터는 데이터" · *(책임 분리)*
- "데이터는 둘로 나뉜다 — **확정 원장(MySQL)** 과 재생성 가능한 **파생 인덱스(Vector)**" *(B4 하이브리드 검색 복선)*
- "외부 API는 안 믿고 경계에서 검증" *(신뢰성 톤)*

## B1-2 — 추천 생성 흐름 (동적, 입력→근거 있는 결과)

**핵심 메시지 (화면 카피)**
> **조건만 입력하면 — 검색·검증을 거쳐 근거 붙은 일정까지**

**화면 구성 (좌→우 시퀀스 5스텝, `user_system_flow` §3.2 / `02_service_flow` §4 축약)**

```
① 입력            ② 추천 실행          ③ 검색·검증            ④ 선정·생성           ⑤ 결과 표시
국가·시기·테마  →  POST /recommendations → Retriever가 후보 검색  → Ranker가 소도시 1곳  → 일정 카드 + 추천 이유
(자연어/지도)      개인화 컨텍스트 로드    Festival Verifier 개최일  Itinerary Writer 일정  + 출처·검증 뱃지
                                        검증(confirmed만)         Output Validator 검증   + 지도·날씨·딥링크
```

| 스텝 | 무슨 일이 일어나나 | 사용자가 보는 것 |
| --- | --- | --- |
| ① 입력 | 국가·월·테마·자연어 또는 지도 마커로 조건 구성 | 입력 조건 칩 |
| ② 추천 실행 | `POST /recommendations` → 선호·피드백 등 개인화 컨텍스트 로드 | 생성 진행 상태 |
| ③ 검색·검증 | Retriever가 목적지/축제 검색, Festival Verifier가 개최일 검증 | 출처·검증 뱃지 |
| ④ 선정·생성 | Ranker가 혼잡도·계절·개인화로 **소도시 1곳** 선정, Writer가 일정·이유 생성 | 소도시 1곳 + 타임라인 |
| ⑤ 결과 표시 | Output Validator 통과분만 반환, 외부 링크 생성 | 일정 카드·추천 이유·딥링크 |

**화면용 / 노트용**
- *화면*: 5스텝 시퀀스 1줄 + 각 스텝 한 단어. 에이전트 노드 전부 나열하지 말 것(B3에서).
- *노트*: 필수 조건(국가·월·일정 유형·축제 포함 여부) 부족 시 **추가 질문 루프**, `confirmed` 축제만 일정 직접 배치, 검증 실패 시 폴백/재시도(상한 2회) — `user_system_flow` §3.2/§4.

**서론 회수**
- ③·⑤의 **출처·검증 뱃지**가 곧 "못 믿음→근거"(S5 컨셉③)의 시스템적 증거.

**발표자 노트 (큐)**
- "사용자는 조건만 — 검색·검증·일정은 시스템이" · *(S5 핵심 메시지 회수)*
- "각 결과엔 **왜 이곳인지 + 출처·검증 뱃지**가 붙는다" · *(근거성)*
- "통과 못 한 추천은 **사용자에게 안 나간다**" → Output Validator(상세 B3)

## (선택) B1-3 — 설계 원칙 3가지

> 2장으로 충분하면 생략. 시간 여유가 있고 "왜 이렇게 만들었나"를 본론 초반에 한 번 못박고 싶을 때만.

**핵심 메시지 (화면 카피)**
> **믿을 수 있게 만든 3가지 원칙**

| 원칙 | 내용 | 근거 |
| --- | --- | --- |
| 경계 검증 | 외부 API 응답은 신뢰하지 않고 경계에서 검증 | `06_technical_spec` §1.2 |
| 표시용/추천용 분리 | WeatherAPI 현재 날씨는 표시용, 추천은 월별 기상 경향 | `02_service_flow` §8.2 |
| 최소 저장 | 대화 원문 미저장 — 구조화 선호·일정·피드백만 보관 | `user_system_flow` §5 |

**발표자 노트 (큐)**
- "신뢰는 기능이 아니라 **설계 원칙**에서 나온다" · *(B4 딥다이브로 연결)*

## 적용 체크리스트

- [ ] 1장 버전을 쓸 경우, 대표문서 **S6와 동일 슬라이드**로 둘 것(중복 제작 금지).
- [ ] (대안 채택 시) 2~3장은 **B1-1(구성도) + B1-3(설계 원칙) = 2장**. B1-2(동적 흐름)는 B3와 중복이라 제외.
- [ ] 두 버전 모두 **에이전트 노드 세부는 B3로** 미룬다(여기선 "멀티스텝"까지만).
- [ ] 화면엔 도식 1개 + 한 줄 카피. 수치(추천 8초·RAG 1초)·예외·재시도는 노트로.
- [ ] 날씨 "표시용 / 추천은 월별 경향" 구분을 노트에 반드시 포함(설계 일관성).
- [ ] 시퀀스 다이어그램 화살표 방향·API 경로(`POST /recommendations`)를 원본과 일치시킬 것.

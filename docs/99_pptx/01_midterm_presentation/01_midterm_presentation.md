## 발표 흐름 한눈에

소개(S1) → 오버투어리즘이란·현상(S2) → 왜 문제: 4가지 비용(S3) → 한일 양방향 수요 + 프로젝트 동기(S4) → 서론 마무리: 의미 + 핵심 컨셉 3가지(S5) → **본론 진입**: 시스템 흐름(S6)

> 구버전(5장) 대비 핵심 변경:
> 1. **S3**를 단일 축("유명지 집중↔소외") → **4가지 비용 4축(주민 생활·환경·도시 기능·지역 문화)**으로 확장.
> 2. **기존 S4(인바운드)+S5(아웃바운드)를 한 장(S4)으로 통합** — 좌/우 양방향 + 하단 한국 국내 보조 + 프로젝트 동기. *일본인→일본 국내*는 근거가 약해 화면에서 빼고 본론 확장으로.
> 3. **S5(서론 마무리·핵심 컨셉 3가지) 신설** — 문제(서론)와 해결(본론)을 잇는 다리.
> 4. S2를 후킹·반전 중심으로 강화(현상 인지), 비용 단어는 S3로 분리.

서론 키워드 줄기(끝까지 1:1로 관통): **쏠림→분산 / 못 찾음→발견 / 흩어짐→일정 / 못 믿음→근거**.

## 슬라이드 2 — 오버투어리즘이란? (현상)

**이 슬라이드의 역할**: ① 오버투어리즘을 쉽게 소개, ② 워드클라우드로 현상을 *체감*, ③ "왜 문제인지"(S3)로 넘긴다.
**역할 분담(S2↔S3)**: S2 = "현상 인지", S3 = "비용 입증". *비용성 단어*(소음·쓰레기·주민 이탈·환경 훼손·교통난·문화 변형)는 S2에 쓰지 않고 **S3로 넘긴다.**

**① 후킹 (화면 상단 또는 말로)**
**"다들 아는 그곳, 막상 가보니 사람만 보였던 적 없나요?"** → (한 박자 뒤) **"이게 바로 오버투어리즘입니다."**

**② 정의 (화면 한 줄)**
오버투어리즘 = over + tourism. 특정 관광지·시기에 *수용 능력을 넘는* 관광객이 몰리는 현상.

**③ 워드클라우드 (현상·체감 키워드만)**
- 키워드: `오버투어리즘` · `인파` · `포토스팟 웨이팅` · `성수기 쏠림` · `핫플 집중` · `수용 한계 초과` · `유명지 편중` · `SNS 성지` · `한 곳 쏠림`
- 화면 중앙 워드클라우드 1개, 핵심어 `Main Orange`.

**④ 반전 (흥미 포인트 — '쏠림의 원인'으로 착지)**
> **"붐비는 걸 좋아해서가 아니다 — 안 붐비는 곳을 *못 찾아서*, 결국 한 곳에 *쏠린다*."**
- 소도시 한 곳 가려면 축제/관광지 일정·교통·숙소·후기를 앱 여러 개 띄워 비교 → 게다가 낯선 곳이라 *믿을 수 있는지도 확신이 안 서* → 검색 잘 되고 익숙한 유명지로 회귀. (이 둘 = S5 컨셉 ②·③의 짝)
- **작은 흩어진-앱 컷**(아이콘/탭 미니 컷, PPT 애니메이션)으로 곁들임. 메인 시각은 워드클라우드 유지. *풀 전후 대비(흩어진 앱→Lovv 한 화면)는 본론 데모에서 콜백.*
- 재프레이밍: '무개념 관광객 탓'이 아니라 **정보·발견의 실패**(비난조 X).

**⑤ 하단 전환 (화면 마지막 줄 = S3행)**
**"문제는, 이 쏠림이 그냥 불편으로 끝나지 않는다는 것."** → (S3) "그래서 어떤 *비용*을 남기는가."

**발표자 노트 (키워드·뉘앙스 큐 · ~30~40초)**
1. **후킹** — "도착하니 사람만, 줄서기·포토스팟" · *(공감 질문, 던지고 한 박자 쉼)*
2. **정의** — over+tourism, 수용 넘는 쏠림 · *(가볍게)*
3. **워드클라우드** — "이게 그 모습" / 인파·웨이팅·성수기 · *(화면 가리키며)*
4. **반전** — 붐벼서가 아니라 *못 찾아서* + 낯설어 *못 믿어서* → 유명지 회귀 · *(한 박자 쉬고 반전 톤, 흥미↑)*
5. **착지** — "그래서 더 쏠린다 → 비용을 남긴다(다음 장)" · *(넘기는 텐션, 솔루션은 X)*
- 3원칙: ① 탓하지 않기 · ② 솔루션 아직 말 X(복선만) · ③ 마지막 말은 '비용'으로.

> 화면용: 워드클라우드(메인) + 정의 1줄 + 반전 1줄 + 작은 앱 컷 + 전환 1줄. 후킹은 말로(또는 상단 짧게). [확인 필요] 워드클라우드 수치는 출처 확보 후에만.

## 슬라이드 3 — 왜 문제인가: 오버투어리즘의 4가지 비용

**핵심 메시지 (화면 카피)**
오버투어리즘은 4가지 비용을 남긴다

**화면 구성 — 2×2 카드 (축당 🇰🇷·🇯🇵 대표 수치 1개씩)**

| 축 | 🇰🇷 대표 수치 | 🇯🇵 대표 수치 |
| --- | --- | --- |
| **주민 생활** | 북촌: 주민 1,500명 vs 연 664만 명 | 교토 기온: 사유지 골목 출입·촬영 통제 |
| **환경** | 제주: 생활폐기물 **+74.1%** | 후지산: 하루 4,000명 제한·2,000엔 예약제 |
| **도시 기능** | 감천마을: 주민 **40% 감소** | 가마쿠라 에노덴: 파크 앤 라이드·24h CCTV |
| **지역 문화** | 흰여울: 주민 이탈·문화 소비공간화 | 교토: 전통 거리의 관광 상품화 |

- 화면엔 카드당 대표 1개만. 상세 수치·인용·신뢰도 주석은 발표자 노트로(상세: `s3_problem_axes_revision.md`).
- 🇰🇷+🇯🇵를 나란히 = "오버투어리즘은 한일 공통 문제" → S4 양방향 예고.

**하단 전환 문구 (→ S4)**
**"이 비용은 소수 유명지에 쏠린다 — 답은 수요의 분산이다."**

**비즈니스 포인트 (기회 수준으로만 암시)**
같은 비용이 소외 소도시엔 *유치 기회*다(양면성). 구체 수익모델(B2G·송객)은 단정하지 않고 본론 확장/데모의 지자체 파트너 대시보드에서 드러낸다.

**발표자 노트**
- 4축은 사회·환경 *비용*. 경제 축은 제외(시장·비즈니스 근거는 S4가 담당, 중복 방지).
- 전환 멘트: "4가지 비용은 결국 소수 유명지에 수요가 몰려 생긴다. 소도시로 나누면 부담은 줄고 소외 지역엔 기회가 생긴다. 그렇다면 그 수요는 정말 있는가? → 다음 장."
- 상세 근거·수치·신뢰도(블로그 출처는 1차 확인 전 비수치 처리)는 `s3_problem_axes_revision.md` §2~§3, 출처는 §7.

## 슬라이드 4 — (통합) 한일 양방향 시장 근거 + 프로젝트 동기

> 기존 S4(인바운드)+S5(아웃바운드)를 한 장으로 통합. 좌/우 양방향(둘 다 근거 강함) + 하단 한국 국내 보조. **일본인→일본 국내는 화면에서 제외**(점유율·정책 위주라 세기 안 맞음) — 제품은 지원하되 본론 **확장 방향**에서 다룬다.

**핵심 메시지 (화면 카피)**
일본인은 한국 소도시로, 한국인은 일본 소도시로 — 소도시 수요는 이미 양방향으로 움직인다 (국내에서도 마찬가지)

**좌/우 — 한일 양방향 (메인, 칸당 대표 수치 1개)**

| 좌 — 인바운드 (일본 → 한국) | 우 — 아웃바운드 (한국 → 일본) |
| --- | --- |
| **"서울 다음은 어디?"** (라쿠텐 서태석, 국가관광전략회의) | 일본 소도시 예약 **+35%** (모두투어) |
| 외국인 3,000만 목표(2029)·FIT 90%+ | (노트) 구마모토 +261%·여객 2,746만·노선 31개 |

- 인물 사진 미사용 — 텍스트 인용 카드(`국가관광전략회의 / 대통령 주재` 라벨로 권위). 출처는 하단 작게.
- (대안) 아웃바운드 대표를 임팩트로 가려면 "+35%" 대신 **구마모토 +261%**(라쿠텐).

**하단 — 한국 국내 보조 (짧게 한 줄)**
국내에서도 같은 흐름 — 묵호·영월 등 **소도시 언급 +184%**(썸트렌드). *(보조 신호 한 줄로만; 메인은 양방향)*

**하단 — 프로젝트 동기 (양방향+국내를 묶는 한 줄)**
> **수요도, 정보도 있다 — 흩어져 있어 '나에게 맞는 소도시'를 못 고를 뿐이다.**
> *(멘트) 그 흩어진 걸 모아 찾아주는 게 Lovv다.* → 다음 장 S5(핵심 컨셉)로 연결.

**발표자 노트 (신뢰도·범위 — 중요)**
- **[직함 혼동 금지]** 서태석(라쿠텐트래블 *한국지사장*)과 김태광(라쿠텐 트래블 *동북아시아 비즈니스 총괄 이사*)은 다른 인물.
- 아웃바운드 노트: 구마모토 +261%·홋카이도 +204%(라쿠텐)는 *한국 Z세대*가 견인 → 일본 국내가 아닌 *아웃바운드* 근거. 사가 +611%. LCC·FSC 지방공항 취항은 "구조적 확대" 방향성만 화면, 개별 노선·단항/휴항은 노트로.
- 한국 국내 노트: 묵호·영월 +184%는 *언급/관심* 지표(예약 아님), 영월은 영화 흥행 효과 일부 → "관심 신호"로 표기.
- **[일본 국내는? Q&A 대비]** 제품은 일본인→일본 국내도 지원. 근거(관광청: 지방부 숙박 90%가 일본인 / 同一현내 숙박 ~20%→35% 근거리화·마이크로투어리즘)는 *점유율·정책* 위주라 근거 슬라이드에선 뺐고, 본론 **확장 방향**에서 다룬다.
- 동기 멘트: "찾아주는 길이 없다"가 아니라 "정보도 있는데 *흩어져* 못 고른다"로 — S2의 '앱 여러 개 비교' 장면을 떠올리게 해 즉시 납득.

> 근거 세기: 아웃바운드 ★★★ / 인바운드 ★★★ / 한국 국내 ★★(보조). 상세·출처는 §출처, `external_evidence_slides_2articles.md`.

## 슬라이드 5 — 서론 마무리: 우리의 의미 + 핵심 컨셉 3가지 (신설)

> 문제(서론)와 해결(본론)을 잇는 다리. (A) 서론 문제 정리 → ① 프로젝트 의미 → ② 핵심 컨셉 3가지(=강점, 문제와 1:1) → bookend.

**(A) 서론 문제 한 줄 정리 (멘트, 화면 X)**
"여기까지 — 유명지 쏠림은 4가지 비용을 남기고, 그럼에도 소도시 수요는 한일 양방향으로(+국내) 움직인다. 막는 건 셋뿐: ① 대안을 *모르고* · ② 정보는 *흩어졌고* · ③ 낯선 곳은 *못 믿는다*. **한마디로, 오버투어리즘을 '분산'으로 푼다.**"

**핵심 메시지 (화면 카피)**
조건만 말하면 — 나에게 맞는 소도시부터 여행 일정까지, 근거 있는 한 번의 대화로

**① 프로젝트의 의미 (화면 한 줄)**
> **남들 다 가는 곳 말고, 나에게 맞는 소도시 — 더 나은 여행이 곧 분산이 된다.**
- (의미 풀이) 여행자는 한적하고 더 나은 여행을 얻고, 그 선택이 모이면 과밀은 풀리고 소외 소도시엔 기회가 된다. **개인에게 좋은 선택 = 사회적으로도 좋은 결과**(win-win). S3 "비용=양면 기회" 회수.

**② 핵심 컨셉 3가지 = 강점 (화면: 카드 3개, 발견 → 일정 → 근거)**

| 문제 태그 | 컨셉 (화면) | 왜 강점·차별점인가 (노트) |
| --- | --- | --- |
| `[대안을 모름]` | ① **대체 소도시 발견** — "어디로?" 대신 "언제·어느 나라로?" | 도시를 정해놓고 검색하는 기존 여행앱과 *반대* — 목적지를 제안받는 탐색 |
| `[정보 흩어짐]` | ② **일정까지 짜는 Agent** — 소도시 추천 + 여행 일정(축제·날씨·동선) | 추천에서 끝나지 않고 **일정까지** — 앱 여러 개 띄울 필요 없음 |
| `[못 믿음]` | ③ **Explainable RAG** — 근거·출처가 보이는 추천 | 일반 생성형 추천과 달리 '왜 이곳인지' 근거가 보여 처음 가는 곳도 신뢰 |

**(B) 컨셉 ↔ 의미 연결 (멘트)**
"이 셋으로 개인의 '더 나은 발견 + 일정'을 돕고, 그 선택이 모이면 곧 분산이 된다." — 미시(개인)와 거시(분산)를 잇는다.

**마무리 bookend (S1 태그라인 원문 회수)**
> **"서울·오사카 말고, 지금은 이곳 — 그게 Lovv입니다."** (S1과 똑같은 문장으로 닫아 수미상관)

**발표자 노트**
- 화면엔 **메시지 1 + 컨셉 카드 3 + 의미 1줄**만. recap·강점 설명·연결은 멘트로(글자 폭탄 방지). 카드마다 *문제 태그*를 남겨 "문제→컨셉 1:1"이 화면에 보이게.
- 복선 회수: S2의 "못 찾는 거라면?"을 여기서 컨셉①로 1차 회수(완전 회수는 본론 데모). 같은 단어(찾아준다·모아준다·믿고)로 받기.

## 슬라이드 6 — 본론 진입: Lovv 시스템 흐름 (Explainable RAG)

**핵심 메시지 (화면 카피)**
Explainable RAG — 추천에 근거·출처·검증을 함께

**화면 구성**
- 좌→우 파이프라인 1개: 사용자 입력(국가·시기·취향) → RAG 검색 → 멀티스텝 Agent → 근거 포함 추천 + 일정
- (선택) Agent 내부 확대: Intent → Supervisor → Retriever/Festival Verifier → Ranker → Itinerary Writer → Output Validator

**발표자 노트**
- 국가·시기 입력 → 조건에 맞는 대체 소도시 1곳을 RAG로 검색·선정 → 멀티스텝 Agent가 일정·축제·날씨·동선 구성 + 추천 이유·출처·검증 함께 제시.
- 시스템 5계층: Client Web / Backend API / Recommendation Agent / Data Store / External APIs.
- 여기서부터 본론(유저 플로우 → 시스템 흐름 → 에이전트 플로우 → 구현 딥다이브 → 활용 데이터 → 확장 → 데모)으로 전개. 시스템 아키텍처는 큰 그림만 보여주고, 에이전트 흐름·핵심 구현은 본론에서 자세히.

**마무리 (PoC → Production)**
- 중간발표(PoC): P1 핵심 기능이 좁은 범위에서 실제 동작함을 보여준다(범위: P1 구현, P2~P3 설계, P5 제외).
- 최종발표(Production): 라이브 데모 + 추천 품질·응답속도(추천 8초·RAG 1초 목표) 검증 + 지자체 파트너 확장.

## 슬라이드 2 — 오버투어리즘이란? (현상)

**이 슬라이드의 역할**: ① 오버투어리즘을 쉽게 소개, ② 워드클라우드로 현상을 *체감*, ③ "왜 문제인지"(S3)로 넘긴다.
**역할 분담(S2↔S3)**: S2 = "현상 인지", S3 = "비용 입증". *비용성 단어*(소음·쓰레기·주민 이탈·환경 훼손·교통난·문화 변형)는 S2에 쓰지 않고 **S3로 넘긴다.**

**① 후킹 (화면 상단 또는 말로)**
**"다들 아는 그곳, 막상 가보니 사람만 보였던 적 없나요?"** → (한 박자 뒤) **"그 순간 여행지는 지역의 부담이 됩니다."**

**② 정의 (화면 한 줄)**
**오버투어리즘은 수용 한계를 넘은 관광의 압박이다.**

**③ 워드클라우드 (오버투어리즘 설명 + 문제)**
- 방향: 워드클라우드를 단어 장식이 아니라 **정의와 문제를 동시에 설명하는 시각자료**로 쓴다. 중앙에는 `오버투어리즘`과 핵심 문제 단어를 크게 두고, 주변에는 뉴스 장면에서 나온 보조 단어를 작게 배치한다.
- 화면 카피: **오버투어리즘은 수용 한계를 넘은 관광의 압박이다.**
- 화면 중앙 전체: 워드 클라우드 1개. 별도 설명 카드 없이 워드클라우드가 화면의 70% 이상을 차지한다.
- 중심 단어(크게): `오버투어리즘` · `수용 한계 초과` · `인파` · `웨이팅` · `쓰레기` · `주민 불편` · `환경 부담` · `교통 혼잡` · `지역 문화 훼손` · `통제`
- 주변 보조 단어(작게): `SNS 성지` · `사유지 출입 금지` · `성수기 쏠림` · `무단 촬영` · `핫플 집중` · `생활폐기물` · `좁은 거리` · `플라스틱` · `관광객 급증`
- 실제 기사 단어 근거:

| 출처 | 기사 장면에서 뽑은 단어 |
| --- | --- |
| AP | `mass tourism`, `crowds`, `infrastructure pressure`, `housing costs`, `short-term rentals`, `water scarcity` |
| Guardian | `Instagrammers`, `litter`, `millions of visitors`, `narrow streets`, `local customs` |
| 뉴시스 | `관광객`, `출입 통제`, `무단 촬영`, `신체 접촉`, `비매너`, `벌금`, `좁은 거리` |
| 제주일보 | `생활폐기물`, `쓰레기 급증`, `소각장 처리 한계`, `관광객 증가`, `플라스틱`, `환경오염` |

- 워드클라우드 하단: **"결국 남는 문제: 인파 · 웨이팅 · 통제 · 쓰레기 · 주민 불편 · 수용 한계"** 한 줄을 넣는다.
- 출처 칩 4개만 작게 배치한다. `AP`, `Guardian`, `뉴시스`, `제주일보`.
- 우측 별도 카드/문장 카드 제거. 화면은 **집중된 워드클라우드 + 문제 요약 한 줄 + 하단 출처**로만 구성한다.

**④ 반전 (흥미 포인트 — '쏠림의 원인'으로 착지)**
> **"붐비는 걸 좋아해서가 아니다 — 안 붐비는 곳을 *못 찾아서*, 결국 한 곳에 *쏠린다*."**
- 소도시 한 곳 가려면 축제/관광지 일정·교통·숙소·후기를 앱 여러 개 띄워 비교 → 게다가 낯선 곳이라 *믿을 수 있는지도 확신이 안 서* → 검색 잘 되고 익숙한 유명지로 회귀. (이 둘 = S5 컨셉 ②·③의 짝)
- **작은 흩어진-앱 컷**(아이콘/탭 미니 컷, PPT 애니메이션)으로 곁들임. 메인 시각은 워드클라우드 유지. *풀 전후 대비(흩어진 앱→Lovv 한 화면)는 본론 데모에서 콜백.*
- 재프레이밍: '무개념 관광객 탓'이 아니라 **정보·발견의 실패**(비난조 X).

**⑤ 하단 전환 (화면 마지막 줄 = S3행)**
**"문제는, 이 쏠림이 그냥 불편으로 끝나지 않는다는 것."** → (S3) "그래서 어떤 *비용*을 남기는가."

**발표자 노트 (키워드·뉘앙스 큐 · ~30~40초)**
1. **후킹** — "도착하니 사람만, 줄서기·포토스팟" · *(공감 질문, 던지고 한 박자 쉼)*
2. **정의** — over+tourism, 수용 넘는 쏠림 · *(가볍게)*
3. **워드클라우드** — "이게 그 모습" / 인파·웨이팅·성수기 · *(화면 가리키며)*
4. **반전** — 붐벼서가 아니라 *못 찾아서* + 낯설어 *못 믿어서* → 유명지 회귀 · *(한 박자 쉬고 반전 톤, 흥미↑)*
5. **착지** — "그래서 더 쏠린다 → 비용을 남긴다(다음 장)" · *(넘기는 텐션, 솔루션은 X)*
- 3원칙: ① 탓하지 않기 · ② 솔루션 아직 말 X(복선만) · ③ 마지막 말은 '비용'으로.

> 화면용: 워드클라우드(메인) + 정의 1줄 + 반전 1줄 + 작은 앱 컷 + 전환 1줄. 후킹은 말로(또는 상단 짧게). [확인 필요] 워드클라우드 수치는 출처 확보 후에만.

## 슬라이드 3 — 왜 문제인가: 오버투어리즘의 4가지 비용

**핵심 메시지 (화면 카피)**
오버투어리즘은 4가지 비용을 남긴다

**화면 구성 — 2×2 카드 (축당 대표 사례 1개씩)**

| 축 | 화면 대표 사례 | 화면 수치/문구 |
| --- | --- | --- |
| **주민 생활** | 🇰🇷 **북촌** | 주민 **1,500명 vs 연 664만 명** |
| **환경** | 🇰🇷 **제주** | 생활폐기물 **1,330t(+74.1%)** |
| **도시 기능** | 🇯🇵 **가마쿠라 에노덴**(슬램덩크 배경지) | 촬영 인파가 건널목·차도 주변으로 집중 |
| **지역 문화** | 🇯🇵 **교토 기온** | 전통 거리의 관광 상품화·사유지 골목 출입 통제 |

- 화면엔 카드당 대표 1개만. 감천·흰여울·후지산 등 보조 사례는 노트/부록으로 내린다(상세: `s3_problem_axes_revision.md`).
- 🇰🇷 2개 + 🇯🇵 2개 = "오버투어리즘은 한일 공통 문제" → S4 양방향 예고.
- 각 카드의 `PHOTO` placeholder는 실제 뉴스/현장 사진으로 대체한다.
  - 북촌: 관광객이 한옥마을 골목에 몰린 사진(AJU PRESS)
  - 제주: 쓰레기로 넘친 클린하우스 사진(경향신문)
  - 가마쿠라 에노덴: 슬램덩크 성지 건널목 촬영 인파 사진(Nippon.com)
  - 교토 기온: 게이샤/마이코를 촬영하는 관광객 사진(Business Insider/Getty Images)
- 화면은 숫자보다 사진이 먼저 보이게 구성한다. 텍스트는 축·사례·핵심 비용만 남긴다.

**하단 전환 문구 (→ S4)**
**"사람이 한곳에만 몰리면 모두가 힘들어진다 — 여행지를 나눠 가면 부담이 줄어든다."**

**비즈니스 포인트 (기회 수준으로만 암시)**
같은 비용이 소외 소도시엔 *유치 기회*다(양면성). 구체 수익모델(B2G·송객)은 단정하지 않고 본론 확장/데모의 지자체 파트너 대시보드에서 드러낸다.

**발표자 노트**
- 4축은 사회·환경 *비용*. 경제 축은 제외(시장·비즈니스 근거는 S4가 담당, 중복 방지).
- 전환 멘트: "4가지 비용은 결국 소수 유명지에 수요가 몰려 생긴다. 소도시로 나누면 부담은 줄고 소외 지역엔 기회가 생긴다. 그렇다면 그 수요는 정말 있는가? → 다음 장."
- 상세 근거·수치·신뢰도(블로그 출처는 1차 확인 전 비수치 처리)는 `s3_problem_axes_revision.md` §2~§3, 출처는 §7.

## 슬라이드 4 — (통합) 한일 양방향 시장 근거 + 프로젝트 동기

> 기존 S4(인바운드)+S5(아웃바운드)를 한 장으로 통합. 좌/우 양방향(둘 다 근거 강함) + 하단 한국 국내 보조. **일본인→일본 국내는 화면에서 제외**(점유율·정책 위주라 세기 안 맞음) — 제품은 지원하되 본론 **확장 방향**에서 다룬다.

**핵심 메시지 (화면 카피)**
일본인은 한국 소도시로, 한국인은 일본 소도시로 — 소도시 수요는 이미 양방향으로 움직인다 (국내에서도 마찬가지)

**좌/우 — 한일 양방향 (메인, 칸당 대표 수치 1개)**

**하단 — 한국 국내 보조 (짧게 한 줄)**

## 슬라이드 5 — 서론 마무리: 우리의 의미 + 핵심 컨셉 3가지 (신설)

> 문제(서론)와 해결(본론)을 잇는 다리. (A) 서론 문제 정리 → ① 프로젝트 의미 → ② 핵심 컨셉 3가지(=강점, 문제와 1:1) → bookend.

**(A) 서론 문제 한 줄 정리 (멘트, 화면 X)**
"여기까지 — 유명지 쏠림은 4가지 비용을 남기고, 그럼에도 소도시 수요는 한일 양방향으로(+국내) 움직인다. 막는 건 셋뿐: ① 대안을 *모르고* · ② 정보는 *흩어졌고* · ③ 낯선 곳은 *못 믿는다*. **한마디로, 오버투어리즘을 '분산'으로 푼다.**"

**핵심 메시지 (화면 카피)**
조건만 말하면 — 나에게 맞는 소도시부터 여행 일정까지, 근거 있는 한 번의 대화로

**① 프로젝트의 의미 (화면 한 줄)**
> **남들 다 가는 곳 말고, 나에게 맞는 소도시 — 더 나은 여행이 곧 분산이 된다.**
- (의미 풀이) 여행자는 한적하고 더 나은 여행을 얻고, 그 선택이 모이면 과밀은 풀리고 소외 소도시엔 기회가 된다. **개인에게 좋은 선택 = 사회적으로도 좋은 결과**(win-win). S3 "비용=양면 기회" 회수.

**② 핵심 컨셉 3가지 = 강점 (화면: 카드 3개, 발견 → 일정 → 근거)**

| 문제 태그 | 컨셉 (화면) | 왜 강점·차별점인가 (노트) |
| --- | --- | --- |
| `[대안을 모름]` | ① **대체 소도시 발견** — "어디로?" 대신 "언제·어느 나라로?" | 도시를 정해놓고 검색하는 기존 여행앱과 *반대* — 목적지를 제안받는 탐색 |
| `[정보 흩어짐]` | ② **일정까지 짜는 Agent** — 소도시 추천 + 여행 일정(축제·날씨·동선) | 추천에서 끝나지 않고 **일정까지** — 앱 여러 개 띄울 필요 없음 |
| `[못 믿음]` | ③ **Explainable RAG** — 근거·출처가 보이는 추천 | 일반 생성형 추천과 달리 '왜 이곳인지' 근거가 보여 처음 가는 곳도 신뢰 |

**(B) 컨셉 ↔ 의미 연결 (멘트)**
"이 셋으로 개인의 '더 나은 발견 + 일정'을 돕고, 그 선택이 모이면 곧 분산이 된다." — 미시(개인)와 거시(분산)를 잇는다.

**마무리 bookend (S1 태그라인 원문 회수)**
> **"서울·오사카 말고, 지금은 이곳 — 그게 Lovv입니다."** (S1과 똑같은 문장으로 닫아 수미상관)

**발표자 노트**
- 화면엔 **메시지 1 + 컨셉 카드 3 + 의미 1줄**만. recap·강점 설명·연결은 멘트로(글자 폭탄 방지). 카드마다 *문제 태그*를 남겨 "문제→컨셉 1:1"이 화면에 보이게.
- 복선 회수: S2의 "못 찾는 거라면?"을 여기서 컨셉①로 1차 회수(완전 회수는 본론 데모). 같은 단어(찾아준다·모아준다·믿고)로 받기.

## 슬라이드 B1 — 본론 진입: Lovv 시스템 흐름 (Explainable RAG)

**핵심 메시지 (화면 카피)**
Explainable RAG — 입력부터 근거 있는 추천·일정까지, 한 흐름으로

**화면 구성**
- 좌→우 5계층 파이프라인 1개: 사용자 입력(국가·시기·테마) → Backend API → Recommendation Agent → Data Store ↔ External APIs → 근거 포함 추천 + 일정
- 5계층 역할:

| 계층 | 한 줄 역할 |
| --- | --- |
| Client Web | 온보딩·지도·챗봇·결과·마이페이지 |
| Backend API | 인증·검증·외부 프록시·추천 실행 |
| Recommendation Agent | 조건 분류 → 검색 → 선정 → 일정·이유 생성 |
| Data Store | 목적지·축제·월별 기상 경향·저장·검토 이력 |
| External APIs | Google/Kakao Maps, WeatherAPI, Yahoo Japan 딥링크 |

- 도식 위 UX 주석 3개: ① 이유·출처·검증 뱃지 ② "덜 걷게/카페 추가" 실시간 수정 ③ 지도·숙소 딥링크

**발표자 노트**
- "입력 → 검색 → 선정 → 일정, 한 흐름"만 빠르게 설명한다.
- 핵심 차별점은 **근거가 보이는 추천 = Explainable RAG**.
- 세부 에이전트 동작은 B3에서 확대한다.
- PoC는 정적 시드/JSON 기반과 최소 추천 API로 핵심 흐름을 검증한다. 추천은 실시간 예보가 아니라 월별 기상 경향 기반이고, WeatherAPI 현재 날씨는 표시용이다.

## 슬라이드 B2 — 유저 플로우

**핵심 메시지 (화면 카피)**
"어디로?"가 아니라 "언제·어느 나라로?" — 묻는 순간부터 다르다

**서브 카피**
들어오는 길은 둘 — 말로 묻거나(AI 챗봇), 지도에서 고르거나(마커). 일정부터는 한 흐름.

**화면 구성**
- 흐름 도식 1개:

```text
                ┌─ (A) AI 챗봇 ── 국가·시기·취향 자연어 입력 ── 시스템이 소도시 1곳 내부 선정 ─┐
① 온보딩 ─ 진입 ┤                                                                              ├─ AI 일정 ─ 저장·연동
                └─ (B) 지도 마커 ─ 국가·월·테마 필터 ── 마커 직접 클릭(사용자가 소도시 선택) ──┘
```

- 결과 카드 히어로 컷 1개: 추천 소도시 1곳 + 시간대별 타임라인 + 추천 이유·출처·검증 뱃지.
- 목업 4~5컷을 한 장에 모두 깔지 않는다. 전체 워크스루는 B6 데모가 받는다.

**발표자 노트**
- 챗봇 트랙: 도시를 정하지 않은 사용자가 국가·시기·취향만 말하면 시스템이 소도시 1곳을 내부 선정한다.
- 지도 트랙: 이미 가고 싶은 곳이 있거나 둘러보고 싶은 사용자는 마커를 직접 선택한다.
- 어느 길이든 일정 생성부터 합류한다. 결과에는 "왜 이곳인지"와 출처·검증 뱃지가 붙는다.
- "덜 걷게", "카페 추가"처럼 말로 일정 일부를 고치는 흐름을 B6 데모에서 보여준다.

## 슬라이드 B3 — 에이전트 플로우

**핵심 메시지 (화면 카피)**
Tool은 계산하고, Agent는 판단한다

**서브 카피**
검색·점수·검증은 Tool 절차로 분리하고, Agent는 그 결과를 보고 다음 행동을 정한다.

**화면 구성 — 7단계 Tool/Agent 파이프라인**

| 단계 | 노드 | Tool이 하는 일 | Agent가 판단하는 일 |
| --- | --- | --- | --- |
| 1 | 의도 분류 | 입력 스키마 검증 | 자연어를 필수/선호/미지원으로 구조화 |
| 2 | 라우팅 | 상태·완료 플래그 확인 | 다음 노드·재시도·fallback 선택 |
| 3 | 검색(다형) | 게이트·벡터검색·후보 추출 | 검색 모드 선택, 후보 부족 시 fallback |
| 4 | 선정 | 도시별 점수 분해 | 최종 소도시 선정 + 추천 근거 생성 |
| 5 | 축제 검증 | 개최일·출처 확인 | 확정/잠정/제외에 따라 일정 반영 판단 |
| 6 | 일정 작성 | 동선·유형 제약 확인 | 읽히는 일정·대체안·설명 생성 |
| 7 | 출력 검증 | 조건·근거·환각 체크 | 실패 유형별 재시도 또는 안내문 |

**서론 회수 띠(화면 하단)**
- `[3·4 검색·선정] → 발견`
- `[6 일정 작성] → 일정`
- `[5·7 검증] → 근거`

**발표자 노트**
- LLM이 점수를 상상하거나 없는 장소를 만들 위험을 줄이기 위해 계산·검색은 Tool 절차로 분리한다.
- Agent는 Tool 결과가 충분한지 판단해 진행, 재검색, 조건 완화, 안내문 전환을 선택한다.
- 미검증 축제, 국가 혼합, 근거 없는 장소는 최종 결과로 내보내지 않는다.

## 슬라이드 B4-1 — 구현 딥다이브 ① Explainable RAG

**핵심 메시지 (화면 카피)**
벡터 검색만으로는 "왜 여기"를 말할 수 없다

**서브 카피**
의미는 vector, 사실은 원장 — 역할을 나눠 근거가 보이는 추천을 만든다.

**화면 구성 — 문제 → 설계 → 효과**

| 갈래 | 무엇을 찾나 | 역할 | 구현 단계 |
| --- | --- | --- | --- |
| 사실 원장 | 장소·축제·좌표 등 확정 사실 | 화면·API가 신뢰하는 최종 기준 | PoC: RDS |
| vector RAG Index | 의미·분위기로 유사한 장소 | 자연어·soft 조건 매칭, 유사도 재랭킹 | PoC 구현·검증 |
| 관계 탐색 | 도시–테마–축제–인접도시 관계 | 관계 기반 후보 확장 | PoC: RDS JOIN, 규모 확장 시 그래프DB 검토 |

**하단 각주**
현재 PoC는 RDS와 vector 기반으로 구현한다. 그래프DB(Neptune)는 관계가 깊어지는 Production 단계의 미래 검토 항목으로만 둔다.

**발표자 노트**
- 벡터 검색은 분위기·의미 유사도는 잘 잡지만, 장소 존재 여부·올해 운영 여부·축제 개최일은 확정하지 못한다.
- 확정 사실은 원장(RDS), 의미 매칭은 vector index로 분리해 추천 근거와 출처를 붙인다.
- 그래프DB는 현재 핵심 근거가 아니다. 다도시 일정, 사용자 행동 기반 발견, 3홉 이상 관계 순회가 필요해질 때 검토한다.

## 슬라이드 B4-2 — 구현 딥다이브 ② Ranker

**핵심 메시지 (화면 카피)**
가장 비슷한 장소가 있는 도시 ≠ 가장 좋은 여행지

**서브 카피**
점수를 항목으로 분해해 최종 소도시를 고르고, 왜 골랐는지 남긴다.

**화면 구성 — 점수 구성요소 6개**

| 점수 항목 | 역할 |
| --- | --- |
| 테마 충족도 | 선택한 모든 테마를 만족하는지 `theme` 컬럼 집계로 확정 |
| 테마 특화도 | 고정 비율이 아니라 전체 평균 대비 강한 테마인지 비교 |
| 희소 테마 보정 | 온천처럼 희소한 테마는 적은 개수도 강한 근거로 반영 |
| 도시 규모 보정 | 장소 수를 구간화해 큰 도시의 물량 우위를 제한 |
| soft/raw 적합도 | 분위기·원문 맥락은 임베딩 유사도로 가산 |
| audit log | 어떤 항목이 선택에 기여했는지 기록 |

**화면 작은 캡션**
실측: 미식 43% vs 온천 2%, 도시별 장소 수 21~440개. 단순 개수·평균은 큰 도시·흔한 테마로 쏠린다.

**발표자 노트**
- Ranker는 최적 공식을 완성했다는 주장이 아니라, 추천 이유를 분해·기록해 검증 가능하게 만드는 scoring framework다.
- 테마 충족은 벡터가 아니라 구조화 컬럼으로 확정하고, 분위기 같은 soft 조건만 임베딩으로 보정한다.
- 이후 추천 품질 평가와 ablation으로 항목별 효과를 분리 검증할 수 있다.

## 슬라이드 B6 — 데모 영상

**핵심 메시지 (화면 카피)**
흩어진 앱 비교가 — 한 번의 대화와 일정으로 끝난다

**구성(30~60초)**

**발표자 노트**

- 영상 준비 전까지는 Before/After 도식 자리만 유지한다.

## 슬라이드 B7 — 이슈/트러블슈팅: Neptune 도입 판단

**핵심 메시지 (화면 카피)**
Neptune은 지금 넣는 기술이 아니라, 조건이 맞을 때 승격하는 선택지다

**서브 카피**
그래프DB가 멋있어서 쓰는 것이 아니라, 다단계 관계 탐색이 실제 병목이 될 때만 도입한다.

**화면 구성 — 이슈 → 판단 → 대응 → 승격 기준**

| 구분 | 발표 화면 문구 |
| --- | --- |
| 이슈 | 도시·테마·축제·인접도시 관계를 그래프로 풀 수 있지만, PoC 관계는 대부분 1~2-hop 수준 |
| 비용 | Neptune Serverless도 최소 1 NCU부터 시작해 월 30만원 예산에서 고정비 비중이 큼 |
| 판단 | PoC/Production 1차는 Neptune 미도입. RDS JOIN, DynamoDB 인접 리스트, 사전계산 후보 테이블로 대체 |
| 트러블슈팅 | 그래프를 원본 저장소가 아닌 재생성 가능한 파생 인덱스로 정의해 vendor lock-in과 비용 리스크를 낮춤 |
| 승격 기준 | 3-hop 이상 임의 경로 탐색, 대규모 실시간 그래프 쓰기, 복잡한 그래프 알고리즘이 필요해질 때 Neptune 도입 |

**화면 도식**

```text
문제 인식      비용 검토        대체 구현                 승격 기준
관계 탐색 필요 → Neptune 고정비 → RDS/DynamoDB/사전계산 → 3-hop+·실시간 그래프
```

**하단 출처**
내부: `04_database_design.md`, `neptune_alternative.md` / 외부: AWS Neptune Pricing

**발표자 노트**
- “Neptune을 못 쓴다”가 아니라 “지금 쓰면 오버엔지니어링이라 보류했다”로 말한다.
- PoC의 목적은 추천 흐름과 근거 제시 검증이다. 현재 규모에서는 RDS JOIN과 사전계산으로 충분하다.
- 그래프DB는 다도시 일정, 사용자 행동 기반 co-visitation, 3-hop 이상 관계 순회가 실제 요구가 될 때 승격한다.
- 질문 대비: “왜 처음부터 Neptune을 안 쓰나?” → 비용 대비 효용이 낮고, 현재 관계 깊이는 RDS/DynamoDB로 방어 가능하다.

## 슬라이드 B8 — 확장 방향 + 남은 태스크

**핵심 메시지 (화면 카피)**
PoC로 핵심 가설을 좁게 검증하고, Production에서 전국·운영·비즈니스로 확장한다

**화면 구성 — PoC → Production 2열**

| 영역 | 지금(PoC) | 다음(Production) |
| --- | --- | --- |
| 데이터 범위 | 강원·경북 40도시 + 일본 관동 일부 | 한국 전국·일본 전역으로 도시·관광지·축제 확장 |
| 데이터 운영 | 1회 수집, 출처·수집시각·신뢰도 기록 | 시변 데이터 정기 갱신 파이프라인, `needs_review` 검수, 관리형 DB 적재 |
| 추천 모드 | Discovery(목적지 미정 → 소도시 발견) 흐름 검증 | 지도 선택·일정 보충 모드로 확장 |
| 추천 기준 | 강원·경북 도시 전체 적재, 소도시 별도 추출 전 | 월별 방문객 수 기반 혼잡도 필터링, 비수기 유명지까지 같은 기준으로 포함 |
| 성능·품질 | 핵심 흐름 동작 검증 | 추천 8초·RAG 1초 목표, 추천 품질 평가·모니터링 하네스 |
| 비즈니스·운영 | 문제·시장 근거 제시 | 지자체 파트너 대시보드(B2G), 송객·콘텐츠 제휴, 역할 기반 운영/검토 |

**마무리 한 줄**
PoC로 핵심 가설이 좁은 범위에서 작동함을 확인 — Production에서 전국·운영·비즈니스로 확장한다.

**발표자 노트**
- 범위는 정직하게 말한다: P1 구현, P2~P3 설계, P5 제외.
- 핵심 축은 "소도시냐 대도시냐"가 아니라 **혼잡도**다. 서울·오사카 말고는 성수기 혼잡 회피의 메시지이며, 비수기 유명지도 같은 로직으로 확장 가능하다.
- 다음 우선순위: ① 데이터 범위 확장 ② 추천 품질 평가·모니터링 하네스 ③ 시변 데이터 갱신 파이프라인 ④ Production 인증·운영 API·지자체 대시보드.

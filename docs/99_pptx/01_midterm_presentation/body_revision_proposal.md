> 목적: 서론(S1~S5) 다음 **본론**의 슬라이드 틀을 제안한다. 실제 설계 문서(`06_technical_spec`·`05_agent_spec`·`02_service_flow`·`03_data_collect_plan`)에 근거.
> 짝 문서: `intro_revision_proposal.md`(서론), `01_midterm_presentation.md`(대표 문서, S1~S6).
> 단계: 중간발표 (PoC). 큰 흐름 위주 — 세부 기능 전부 설명 X, "어떤 문제를 정의→어떤 핵심 흐름을 구현→무엇을 발전시킬지"를 보여준다.

## 0. 본론 흐름 한눈에

```
B1 시스템 흐름      → 시스템이 어떻게 동작하는가 (큰 그림, S6에서 이어받음)
B2 유저 플로우      → 사용자가 어떻게 쓰는가
B3 에이전트 플로우  → 각 에이전트의 역할과 연결
B4 구현 딥다이브    → 왜 그렇게 설계했나 (1~2개 깊게)
B5 활용 데이터      → 무엇을 어디서, 어떤 목적으로 (표 1장)
B6 데모 영상        → 실제 동작 (하이라이트)
B7 확장 방향        → PoC 범위 + Production 확장 + 남은 태스크
```

- **번호 연결**: 대표 문서 `01_midterm_presentation.md`의 **S6(본론 진입: 시스템 흐름)**이 곧 **B1**이다. 즉 본론은 S6=B1에서 시작해 B7까지. 전체 슬라이드 = 서론 5장 + 본론 7장 ≈ **12장**(가이드 10~20장 내).
- **순서 메모**: 가이드 원안은 *유저 플로우 → 시스템 흐름* 순이었으나, 여기서는 사용자 지시대로 **시스템 흐름을 먼저** 둔다(서론 마지막 S6가 시스템 흐름이라 자연스럽게 이어짐, 큰 그림→사용자 경험→내부 동작 순으로 좁혀짐). "사용자 경험을 먼저 보여주고 싶다"면 B1↔B2를 교체해도 된다.
- **PoC 톤**: 아키텍처·파이프라인은 *큰 그림*만, 디테일은 B4 딥다이브 1~2개로 압축. 데모(B6)가 본론의 클라이맥스.
- **서론 회수 지점**: 서론에서 깐 복선·키워드를 본론에서 받는다(아래 §0.5 표가 전체 매핑).

## 0.5 서론 복선 → 본론 회수 매핑 (intro `§2.9` 대응)

서론에서 던진 모든 복선·키워드가 본론에서 받쳐지는지 1:1로 점검한 표다. 본론 슬라이드 제작 시 *같은 단어*로 받아야 콜백이 산다(못 찾는다→찾아준다, 흩어졌다→모아준다, 못 믿는다→근거가 보인다).

| 서론에서 던진 복선/키워드 | 깐 위치 | 본론 회수 | 상태 |
| --- | --- | --- | --- |
| "안 가는 게 아니라, *못 찾는* 거라면?" (대안 소도시 발견) | S2 반전 / S5 컨셉① | **B2** 첫 화면 "언제·어느 나라로?" → **B6** 데모에서 실제 발견 | ✅ |
| 날씨·축제·교통·숙소·후기가 *흩어져* 비교 어렵다 | S2 앱 비교 장면 / S5 컨셉② | **B2** 한 흐름으로 모임 → **B6 "흩어진 앱 → Lovv 한 화면" 전후 대비**(완전 회수) | ✅ |
| 낯선 곳은 *못 믿어* 유명지로 회귀 | S2 반전 / S5 컨셉③ | **B2·B6** 추천 이유·출처·검증 뱃지(Explainable RAG) / **B3** Festival Verifier·Output Validator | ✅ |
| "쏠림 → 분산" (거시 루프) | S2·S3·S4 / S5 의미 | **B7** 분산이 곧 사업 의미·확장(지자체 대시보드) + 전반 톤 | ✅ |
| "조건만 말하면 소도시부터 *일정까지*" | S5 핵심 메시지 | **B6** 데모로 그대로 시연 / **B2·B3** Itinerary_Writer | ✅ |
| 비용 = 양면 기회 (소외 소도시엔 기회) | S3 비즈니스 포인트 | **B7** 지자체 파트너 대시보드(B2G) | ✅ |
| "서울·오사카 말고, 지금은 이곳" | S1 태그라인 | **S5 bookend에서 이미 닫힘**(본론 회수 아님 — 참고용) | — |

**시각 회수 규칙(가장 강한 콜백)**: S2의 *흩어진-앱 작은 컷*과 B6 데모의 *흩어진 앱 → 한 화면* 컷의 **아이콘·레이아웃을 일치**시킨다. 같은 그림을 두 번 써야 "아, 아까 그거"가 완성된다.

## B1 — 시스템 흐름 (= 대표문서 S6)

**목적**: 사용자 경험을 구현하기 위해 시스템이 어떻게 동작하는지 *큰 그림*으로.

**핵심 메시지 (화면 카피)**
Explainable RAG — 입력부터 근거 있는 추천·일정까지, 한 흐름으로

**화면 구성 (5계층 큰 그림 1개)**
좌→우 파이프라인 + 5계층 레이어:
`사용자 입력(국가·시기·테마)` → `Backend API` → `Recommendation Agent(멀티스텝)` → `Data Store(목적지·축제·기상경향)` ↔ `External APIs(지도·날씨·딥링크)` → `근거 포함 추천 + 일정`

| 계층 | 한 줄 역할 |
| --- | --- |
| Client Web | 온보딩·지도·챗봇·결과·마이페이지 |
| Backend API | 인증·검증·외부 프록시·추천 실행 |
| Recommendation Agent | 조건 분류 → 검색 → 선정 → 일정·이유 생성 |
| Data Store | 목적지·축제·월별 기상 경향·저장·검토 이력 |
| External APIs | Google/Kakao Maps, WeatherAPI, Yahoo Japan 딥링크 |

**화면용 / 노트용**
- *화면*: 5계층 파이프라인 도식 1개 + "Explainable RAG" 한 줄. (디테일 X)
- *노트*: 핵심 원칙 — 외부 API 응답은 경계에서 검증(신뢰 X), 추천은 실시간 예보가 아닌 *월별 기상 경향* 기반, 대화 원문 미저장. 내부 에이전트 디테일은 B3에서.

**발표자 노트 (키워드·뉘앙스 큐)**
- "입력 → 검색 → 선정 → 일정, 한 흐름" · *(큰 그림만, 빠르게)*
- "핵심은 *근거가 보이는* 추천 = Explainable RAG" · *(차별점 한 번 짚기)*
- "자세한 에이전트 동작은 잠시 뒤" → B3 예고
- PoC: 정적 시드/JSON 기반, 추천 API 최소 구현(노트).

## B2 — 유저 플로우

**목적**: 사용자가 우리 서비스를 어떻게 활용하는지(여정).

**핵심 메시지 (화면 카피)**
"어디로?"가 아니라 "언제·어느 나라로?" — 묻는 순간부터 다르다

**화면 구성 (스크린 플로우, 7단계 → 화면은 4~5컷)**
온보딩(대도시 스타일→테마) → 국가·시기 입력 → 테마/자연어 → **소도시 후보 비교 카드**(추천 이유·혼잡도·출처 뱃지) → 소도시 확정 + 챗봇 → **일정 타임라인**(추천 이유·검증 뱃지·외부 링크) → 저장/피드백

**화면용 / 노트용**
- *화면*: 실제 화면 흐름 스크린샷/목업 4~5컷 + 단계 라벨. 핵심은 *입력 질문의 전환*("언제·어느 나라로?")과 *결과物*(이유·출처·검증이 붙은 일정).
- *노트*: 7단계 상세(진입·인증 → 탐색 → 후보 비교 → 확정 → 동적 일정 → 실시간 피드백 → 저장·딥링크), 자연어 예시("6월에 비 덜 오는 일본 소도시"), 대체 일정 토글(기상 악화 시 실내 중심).

**서론 회수**
- **S5 컨셉① 회수**: "어디로? 대신 언제·어느 나라로?"가 여기서 실제 첫 화면으로 등장.
- **S2 고통 장면 복선 → 여기서 한 흐름으로**: 흩어진 앱 비교 대신 한 대화로(완전 회수는 B6 데모).

**발표자 노트 (큐)**
- "도시를 정해놓고 시작하지 않는다 — 조건만" · *(S1·S5 톤 이어받기)*
- "결과엔 *왜 이곳인지* + 출처·검증 뱃지가 붙는다" · *(근거성 강조)*
- 실시간 수정("덜 걷게", "카페 추가") 한 번 시연 언급.

## B3 — 에이전트 플로우

**목적**: 각 에이전트의 역할과 전체 흐름 안에서의 연결.

**핵심 메시지 (화면 카피)**
단일 LLM이 아니라 — 역할을 나눈 멀티스텝 Agent + 결정적 Skill

**화면 구성 (좌→우 파이프라인 다이어그램, 노드별 한 줄)**

| 단계 | 에이전트/Skill | 한 줄 역할 |
| --- | --- | --- |
| 1 | `Intent_Agent` | 자연어·조건 구조화, 멀티턴 정리 |
| 2 | `Supervisor_Router` | 상태(`fulfilled_matrix`) 기반 라우팅·재시도·폴백 |
| 3 | `Polymorphic_Retriever` | 소도시·관광지·축제 후보 검색(전국구/지역 제한) |
| 4 | `Ranker`(1차) | 구조화 점수화, 축제 검증 Top-K 선별 |
| 5 | `Festival_Verifier` | Top-K 축제의 해당 연도 개최일 웹 검증 |
| 6 | `Ranker`(최종) | 검증 반영해 **소도시 1곳** 선정 |
| 7 | `Itinerary_Writer` | 일정·대체 일정·추천 이유·동선 설명 생성 |
| 8 | `Validation Skill` → `Output_Validator` | 결정적 검증 → 근거성·환각·설명 의미 검증 |

**화면용 / 노트용**
- *화면*: 파이프라인 노드 도식 1개 + 노드명만(또는 한 줄). `fulfilled_matrix`(retrieval·ranking·festival·generation·validation) 5단계 상태 표시는 *작게*.
- *노트*: Supervisor는 raw 대화·웹 원문·RAG 원문을 안 들고 *압축 상태·참조 키*만 라우팅(I/O 허브). Sub-Agent(해석·생성)와 결정적 Skill(계산·검증·링크·상태전이) 분리. 검증 실패 시 카테고리별 재시도(상한 2회).

**서론 회수 (S5 컨셉 3개 ↔ 에이전트)**
- ① 발견 → Retriever + Ranker / ② 일정 → Itinerary_Writer / ③ 근거 → Festival_Verifier + Output_Validator(Explainable).

**발표자 노트 (큐)**
- "각자 역할이 다른 전문가 팀" · *(비유로 가볍게)*
- "검증 우선 — 미검증 축제·국가 혼합·근거 없는 장소는 *최종에 안 내보낸다*" · *(신뢰 강조)*
- 디테일(노드 10단계 전부)은 화면 X, 흐름만.

## B4 — 구현 딥다이브 (왜 이렇게 설계했나)

**목적**: 주요 설계의 *왜(문제)·어떻게(설계)·개선 과정*. PoC답게 **1~2개만 깊게**(코드 나열 X, 그림으로).

**추천 딥다이브 후보 (2개 선택 권장)**

**① Explainable RAG = 하이브리드 검색 (왜 vector 단독이 아닌가)**
- 문제: 벡터 검색만으로는 장소 존재·축제 일정·운영 여부를 *확정*할 수 없다(환각 위험).
- 설계: **S3 vector(의미 검색) + Neptune(관계 탐색) + MySQL 원장(확정 근거)** 3단 하이브리드. vector/Neptune은 재생성 가능한 *파생 인덱스*, 확정은 원장.
- 효과: "왜 이곳인지" 근거·출처를 붙일 수 있음(= S5 컨셉③ 회수).

**② Festival Verifier — Top-K 검증 + 캐시 (왜 전수 검증을 안 하나)**
- 문제: 축제 날짜는 매년 바뀌어 환각·오배치 위험이 큰데, 후보 전수를 매번 웹 검증하면 비용·지연 폭증.
- 설계: Ranker 1차 점수 **Top-K(2~3곳)만** 웹 검증, `festival_id+travelYear` 캐시(TTL: confirmed 30일/tentative 7일). `confirmed`만 일정에 직접 배치, `tentative`는 안내로만.
- 개선 과정: 검증을 랭킹 *뒤*가 아니라 Top-K 사이에 끼워 비용↓.

**(대안) ③ 멀티스텝 통합·단순화 (개선 스토리)**
- `Condition_Parser`를 `Intent_Agent`에 통합, `Itinerary Planner + Explanation Writer`를 `Itinerary_Writer`로 통합 → "이유–일정 불일치"↓, LLM 호출 횟수↓.
- Supervisor I/O 허브 + `fulfilled_matrix`로 루프·재시도·폴백을 결정적으로 제어.

**화면용 / 노트용**
- *화면*: 선택한 1~2개를 **문제 → 설계 → 효과** 3박자 도식으로. before/after 또는 분기 그림 1장씩.
- *노트*: 수치 목표(추천 8초·RAG 1초), retry 상한 2, 실패 카테고리(grounding/hallucination/condition/explanation/fallback) 등은 노트로.

**발표자 노트 (큐)**
- "왜 이렇게? → *환각을 막고 근거를 보이려고*" · *(설계 의도 한 문장)*
- "개선: 노드를 합쳐 불일치·비용↓" · *(개선 스토리 1개)*

**화면용 / 노트용**
- *화면*: 위 표(또는 축약 — 구분·출처·목적 3열). 핵심 출처 로고/이름만.
- *노트*: 출처 추적 원칙(`source_url`·`collected_at`·`data_confidence`), PoC는 정적 JSON/시드(`data/KR/*.json`, `data/JP/*.json`)로 검증 후 S3 Raw 적재. 표시용 날씨는 추천 스코어링에 미사용.

**발표자 노트 (큐)**
- "공식·공공 데이터 우선 — 출처를 함께 저장" · *(신뢰)*
- "날씨는 *표시용*, 추천은 *월별 경향*으로" · *(설계 일관성)*

## B6 — 데모 영상 (하이라이트)

**목적**: 실제 서비스가 어떻게 동작하는지 보여준다. 본론의 클라이맥스.

**구성 (30~60초, 핵심 흐름 1개)**
국가·시기·테마 입력 → 소도시 추천(추천 이유·출처·검증 뱃지) → 일정 타임라인 → "덜 걷게/카페 추가" 실시간 수정 → 저장.

**서론 완전 회수 (가장 강력한 콜백)**
- **S2 고통 장면 → 여기서 해소**: "흩어진 앱 12개 비교"(S2 작은 컷)를 **그대로 다시 띄웠다가 → Lovv 한 화면**으로 전환하는 전후 대비. *(S2 컷과 데모 컷의 비주얼을 일치시킬 것 — §2.9)*
- **S5 핵심 메시지 회수**: "조건만 말하면 소도시부터 일정까지" 가 실제로 그렇게 되는지 눈으로.

**화면용 / 노트용**
- *화면*: 영상 임베드 + 핵심 순간 캡션 2~3개.
- *노트*: 영상 길이·로딩 대비(실패 시 캡처 백업), 데모 환경은 PoC(P1) 범위임을 한 줄 고지.

**발표자 노트 (큐)**
- "아까 그 흩어진 앱들 — 이제 한 화면" · *(S2 콜백 한 마디)*
- 영상 틀기 전 "보실 건 딱 하나의 흐름" · *(기대 정렬)*

## B7 — 확장 방향 + 남은 태스크

**목적**: PoC에서 구현한 범위와 앞으로 확장할 방향·남은 태스크.

**화면 구성 (PoC → Production 2열 대비)**

| 영역 | 지금 (PoC) | 다음 (Production) |
| --- | --- | --- |
| 데이터 | 강원·경북 + 관동 일부, 정적 JSON/시드 | 전국·전역 확장, **관리형 DB + 갱신 파이프라인** |
| 흐름 | 핵심 추천 흐름(P1) 동작 | 인증·역할 기반 운영/검토, 감사 이력 |
| 성능·품질 | 동작 검증 | 추천 8초·RAG 1초, 평가·모니터링(AgentCore Evaluations) |
| 비즈니스 | 문제·시장 근거 | **지자체 파트너 대시보드(B2G)**, 송객·콘텐츠 제휴 |

**남은 태스크 (예시)**
- 일본 데이터 확충(관동 → 확대), 일본 국내 수요 근거 1차 출처 확정
- 평가 하네스(L1/L2/L4 + nightly trajectory)
- Production 인증·운영 API, 데이터 갱신 파이프라인

**서론 회수**
- **S4에서 뺀 "일본인→일본 국내"를 여기서 회수**(확장으로) — Q&A 대비도 됨.
- **S3 비즈니스 양면 기회 → 지자체 대시보드(B2G)**로 구체화.

**발표자 노트 (큐)**
- "PoC로 *핵심 가설이 작동*함을 보였고 — 다음은 확장" · *(PoC→Production 프레이밍)*
- "범위를 솔직하게: P1 구현 / P2~P3 설계" · *(정직한 톤)*

# Lovv 중간발표 PPT — 내용 초안 (5장)

> 용도: 디자인 도구(Claude / Gemini)에 넣어 슬라이드를 생성하기 위한 **내용 초안**
> 디자인(색상·레이아웃·폰트)은 미정 — 본 문서는 텍스트/구성/발표 노트만 정의
> 스타일 원칙: AWS/테크 컨퍼런스식 — 슬라이드당 핵심 메시지 1개 + 시각자료 1개, 본문 글 최소화. 상세 설명은 발표자 노트로 전달
> 비즈니스 관점: 전체의 약 15% 반영

## 슬라이드 1 — 프로젝트 소개 (Lovv)

**핵심 메시지 (화면 카피)**
서울·오사카 말고, 지금은 이곳

**서브 카피**
지도에서 흐릿한 곳, 여행에선 가장 빛나는 곳

**화면 구성 (디자인 지시)**
- 중앙: 로고 타이포 "Lovv"
- 하단: Local(소도시) · Vibe(분위기·감성) · Voyage(여행) 3개 아이콘 도식
- 본문 텍스트 없음

**비즈니스 포인트 (작게 표기)**
한일 소도시 여행 추천 / inbound 여행 시장 타깃

**발표자 노트**
- Lovv = Local + Vibe + Voyage. 혼잡한 유명 관광지 대신 덜 알려진 소도시·지역 경험을 추천하는 대화형 여행 추천 서비스.
- 한 줄 정의: 한일 여행자에게 대도시 대신 대체 소도시를 발견하게 하는 "목적지 탐색 엔진".

---

## 슬라이드 2 — 문제: 오버투어리즘이란 (현상)

**핵심 메시지 (화면 카피)**
특정 관광지·시기에 수요가 과도하게 집중된다

**화면 구성 (디자인 지시)**
- 화면 전체: 워드클라우드 1개
- 워드클라우드 키워드: 혼잡 · 주민 불편 · 환경 훼손 · 만족도 저하 · 수요 집중 · 성수기 · 유명 관광지 편중 · 소도시 정보 부족
- 본문 텍스트 없음 (키워드로만 문제를 "느끼게" 함)

**발표자 노트**
- 오버투어리즘: 특정 관광지·시기에 방문 수요가 과도하게 몰리면서 혼잡, 주민 불편, 환경 훼손, 여행 만족도 저하를 유발하는 문제.
- 여행자는 덜 붐비는 곳을 원하지만, 소도시 정보(계절성·축제·이동·주변 관광)는 흩어져 있어 비교가 어렵다.

> [확인 필요] 워드클라우드에 실제 통계 수치를 넣으려면 출처 데이터가 별도로 필요. 현재 문서에는 키워드만 존재.

---

## 슬라이드 3 — 문제의 영향 (왜 중요한가)

**핵심 메시지 (화면 카피)**
유명지 집중 ↔ 소도시 소외

**화면 구성 (디자인 지시)**
- 좌우 대비 도식 1개: (좌) 유명 관광지에 인파 집중 / (우) 소도시는 노출 기회 부족
- 하단 한 줄 강조

**비즈니스 포인트 (하단 강조 한 줄)**
수요 불균형 = 지역 관광 경제 손실

**발표자 노트**
- 일부 지역에 관광객이 집중되는 동안 다른 지역은 노출 기회를 잃는다 → 지역 관광 수요 불균형.
- 데이터(관광지·축제·교통·날씨·지도)는 여러 플랫폼에 분산되어 있어 여행자가 대안을 찾기 어렵다.
- 즉, 이건 여행자 불편이자 동시에 지역 관광 경제의 기회 손실 문제다.

## 슬라이드 4 — 시장은 이미 움직인다 (외부 근거)

**핵심 메시지 (인용)**
> "최근 여행 트렌드는 대도시뿐만 아니라 지방 소도시로까지 다양화되는 흐름을 보이고 있다."

**출처·화자 (정확 표기)**
김태광 — 라쿠텐 트래블 동북아시아 비즈니스 총괄 이사
서울경제TV, 2025.7.22

> [주의] "라쿠텐 트래블 코리아 지사장"이 아님. 이 발언의 정확한 직함은 위와 같으므로 슬라이드에 "지사장"으로 표기하지 말 것 (사실 오류 방지).

**보조 데이터 (인용 뒷받침)**
- 전년 동기 대비 홋카이도 예약 +204%
- 규슈 소도시 구마모토 예약 +261% (가장 가파른 상승)
- → 수요가 실제로 대도시에서 소도시로 분산되고 있음

**화면 구성 (디자인 지시)**
- 인물 직함 + 큰따옴표 안 핵심 문장 1개 (AWS 키노트 "고객 인용" 형식)
- 보조 데이터는 작은 숫자 강조 그래픽으로

**비즈니스 포인트**
업계 1위 플랫폼도 인정하는 흐름 → "소도시 분산 수요"는 가설이 아니라 이미 형성된 시장.

**발표자 노트**
- 우리가 정의한 문제(소도시로의 수요 분산)는 우리만의 주장이 아니라, 일본 최대 여행 플랫폼의 데이터로도 확인된다.
- 즉 시장이 이미 그 방향으로 움직이고 있고, Lovv는 그 흐름에 필요한 도구다.

## 슬라이드 5 — 해결: Lovv 시스템 흐름

**핵심 메시지 (화면 카피)**
Explainable RAG — 추천에 근거·출처·검증을 함께

**화면 구성 (디자인 지시)**
- 좌→우 파이프라인 다이어그램 1개:
  사용자 입력(국가·시기·취향) → RAG 검색 → 멀티스텝 Agent → 근거 포함 추천 결과
- (선택) Agent 내부 단계는 강조 시 별도 확대 슬라이드로 분리:
  Intent → Condition Parser → Retriever/Festival Verifier → Ranker → Itinerary Planner → Output Validator

**발표자 노트**
- 사용자가 국가와 시기를 먼저 입력 → 조건에 맞는 대체 소도시 1곳을 RAG로 검색·선정.
- 멀티스텝 Agent가 일정·축제·날씨·동선을 구성하고, 추천 이유·출처·검증 상태를 함께 제시.
- 차별점: 단순 생성형 답변이 아니라 "근거가 보이는 추천" (Explainable RAG).
- 시스템 5계층: Client Web / Backend API / Recommendation Agent / Data Store / External APIs.

# 중간발표 PPT 작성 가이드라인

> 적용 대상: `../01_midterm_presentation.md`, `../pages/*.md`, `../html_export/` 발표 산출물
> 목적: 중간발표 자료를 전문적인 기술 발표 형태로 유지하고, 대표 문서와 페이지별 문서가 서로 어긋나지 않게 관리한다.

---

## 1. 문서 운영 원칙

- `../01_midterm_presentation.md`를 대표 문서로 둔다.
- `../pages/*.md`는 슬라이드별 작업·검토를 위한 분리 문서다.
- 큰 흐름, 발표 순서, 전체 메시지를 바꾸면 대표 문서를 먼저 수정한다.
- 특정 슬라이드의 화면 구성, 발표자 노트, 근거만 다듬을 때는 해당 `../pages/*.md`에서 먼저 초안을 정리한 뒤 대표 문서에 반영한다.
- HTML/PPT 산출물은 생성물로 보고, 의미가 바뀌는 수정은 Markdown 원본에 먼저 남긴다.
- 구버전 초안과 중복 제안서는 삭제하지 않고 `../archive/` 또는 `../bak/`에 보관한다.

---

## 2. 발표 톤

- 감성적 표현보다 객관적 문제 제기, 데이터, 기술 설계 이유를 우선한다.
- 슬라이드 타이틀은 짧게 쓴다.
- 한 장에는 핵심 메시지 1개만 둔다.
- 장표 화면에는 도식, 표, 수치, 짧은 문장을 배치하고 상세 설명은 발표자 노트로 내린다.
- 전체 애니메이션은 제거한다. 필요한 경우에도 발표 흐름상 꼭 필요한 단계적 강조만 예외로 검토한다.
- 문제 제기는 기사, 통계, 공신력 있는 외부 근거로 보강한다.

---

## 3. 권장 발표 구조

| 구간 | 목적 | 포함 내용 |
| --- | --- | --- |
| 도입 | 무엇을 만들고 왜 필요한지 설명 | 프로젝트 소개, 목차, 문제 정의, 한일 타깃 |
| 문제·시장 | 서비스 필요성 입증 | 오버투어리즘, 기존 서비스 한계, 트리플 대비 차별점 |
| 기술 본론 | 어떻게 풀었는지 설명 | 시스템 흐름, 유저 플로우, 멀티 에이전트 구조, RAG 딥다이브 |
| 구현 현황 | PoC가 어디까지 왔는지 설명 | 데이터 구성, 데모, 주차별 진행 현황 |
| 사업·확장 | 서비스 가치와 이후 계획 설명 | 비즈니스 모델, 확장 방향, 출처, Q&A |

## 4. 슬라이드 작성 규칙

### 4.1 화면 구성

- 제목은 1줄을 원칙으로 한다.
- 본문 텍스트는 카드 3~4개 또는 표 1개 수준으로 제한한다.
- 복잡한 문장은 발표자 노트로 이동한다.
- 시스템·유저·에이전트 흐름은 Mermaid 또는 단순 도식으로 먼저 설계한다.
- 데이터는 가능한 한 표로 정리한다.
- 출처는 화면에는 짧게, 전체 URL은 출처 슬라이드 또는 하단 출처 섹션에 모은다.

### 4.2 기술 설명

- 기능 목록보다 설계 이유를 먼저 말한다.
- 멀티 에이전트는 "왜 단일 LLM 호출로 충분하지 않은가"에서 출발한다.
- RAG는 PoC의 핵심 기술 축으로 설명한다.
- 트러블슈팅은 별도 장표보다 각 설계 선택의 `어려움 -> 선택 -> 결과` 구조 안에 녹인다.
- 툴과 에이전트 구분 자체보다, 검색·검증·랭킹·일정 생성의 책임 분리를 강조한다.

### 4.3 비즈니스 설명

- 시장 가치가 있어야 서비스라는 전제를 명확히 둔다.
- 비즈니스 모델은 차별점과 연결해 설명한다.
- 가능하면 여행 수요, 시장 규모, 지역 관광 예산, 예약/제휴 수익 구조 같은 정량 근거를 붙인다.
- 트리플과의 비교는 비방이 아니라 포지셔닝 차이로 표현한다.

## 5. 페이지별 문서 규칙

`../pages/` 아래 슬라이드 문서는 다음 형식을 따른다.

```markdown
# 슬라이드 N — 제목

> 원본 위치: `../01_midterm_presentation.md`
> 상태: Draft
> 역할: 이 슬라이드가 발표 흐름에서 맡는 역할

## 핵심 메시지

## 화면 구성

## 발표자 노트

## 근거·출처

## 수정 체크리스트
```

페이지별 문서는 검토 편의를 위한 작업 단위다. 최종 반영 여부는 대표 문서와 HTML 발표본을 함께 확인한다.

---

## 6. 백업 규칙

- 큰 구조 변경 전에는 `../bak/YYYY-MM-DD_작업명/` 폴더를 만든다.
- 백업에는 최소한 대표 문서, 페이지별 문서, 주요 보조 Markdown을 포함한다.
- 백업 폴더는 되돌리기 근거이므로 직접 수정하지 않는다.
- 단순 구버전 초안은 `../archive/`에, 특정 작업 직전 스냅샷은 `../bak/`에 둔다.

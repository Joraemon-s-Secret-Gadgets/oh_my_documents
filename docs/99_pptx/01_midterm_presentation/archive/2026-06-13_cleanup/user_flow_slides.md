# 본론 — 유저 플로우 슬라이드 구성 (B2)

> 목적: 본론 **B2 "유저 플로우"** 슬라이드를 두 가지 분량(1장 채택 / 2장 대안)으로 제안한다. "사용자가 우리 서비스를 어떻게 활용하는가(여정)"를 보여준다.
> 근거 문서: `02_service_flow.md`(§3 User Flow · §5 온보딩 · §6 목적지 탐색 · §7 챗봇 · §8 추천 결과), `sooa/user_system_flow.md`(§2.1 일반 여행자 흐름 · §4.1 상태 흐름), `body_revision_proposal.md`(B2), `midterm_vs_final_presentation_guide.md`(§7.2-1).
> 짝 문서: `system_flow_slides.md`(B1 시스템 흐름), `agent_deep_dive_slides_condensed.md`(B3 에이전트 플로우).
> 단계: 중간발표 (PoC). 화면은 *사용자 여정*만 — 시스템 계층(B1)·에이전트 노드(B3)는 여기서 풀지 않는다.
> 색상: `color-map.md` 토큰(Main Orange `#F26518`, Muted Brown `#D44A14`, Card Peach `#FCE7DB`, Soft Beige `#FFF3E7`, Dark Charcoal `#222222`, Sub Text Gray `#666666`).

> **결정: A. 1장 버전으로 진행한다.** (2026-06-10)
> 이유: 본론은 **B1 시스템 흐름(정적 큰 그림) → B2 유저 플로우(사용자 경험) → B3 에이전트 플로우(내부 동작)** 로 좁혀지는 깔때기다. B2는 "사용자가 보는 화면 여정" 한 줄기만 책임지고, 동적 추천 시퀀스(검색→검증→선정)는 B3가 받는다. PoC 톤(슬라이드당 메시지 1개)에 맞춰 4~5컷 스크린 플로우 1장으로 위치를 잡는다.
> 아래 **B. 2장 버전은 대안으로 보존**한다(목업 화면을 크게 보여주고 싶거나, 진입 분기(지도/챗봇)와 결과物(이유·출처·검증)을 각각 강조하고 싶을 때 B2-1 + B2-2 사용).

# A. 1장 버전 (= 본론 B2 / 유저 플로우)

**핵심 메시지 (화면 카피)**
> **"어디로?"가 아니라 "언제·어느 나라로?" — 묻는 순간부터 다르다**

**서브 카피 (작게)**
> 들어오는 길은 둘 — 말로 묻거나(AI 챗봇), 지도에서 고르거나(마커). 일정부터는 한 흐름.

**화면 구성 (흐름 도식 1개 + 결과 카드 히어로 컷 1장)**

> 공간 원칙: 실제 목업을 4~5컷 깔지 않는다(슬라이드당 시각자료 1개 원칙·가독성). **분기→합류는 흐름 도식 1개**로, **실제 화면은 결과 카드 1컷만** 크게 보여주고, 전체 화면 워크스루는 **B6 데모 영상**이 받는다.

*(1) 흐름 도식 — 두 진입점으로 갈렸다가(A 챗봇 = 시스템이 소도시 내부 선정 / B 지도 = 사용자가 마커로 직접 선택) 소도시가 확정되면 **AI 일정~저장 공통 흐름으로 합류***

```
                ┌─ (A) AI 챗봇 ── 국가·시기·취향 자연어 입력 ── 시스템이 소도시 1곳 내부 선정 ─┐
① 온보딩 ─ 진입 ┤                                                                              ├─ AI 일정 ─ 저장·연동
                └─ (B) 지도 마커 ─ 국가·월·테마 필터 ── 마커 직접 클릭(사용자가 소도시 선택) ──┘
```

*(2) 히어로 목업 1컷 — 결과 카드* : 추천 소도시 1곳 + 시간대별 타임라인에 **추천 이유·출처·검증 뱃지**가 붙은 화면 한 장. "결과物에 근거가 보인다"를 이 한 컷으로 증명한다(Explainable RAG의 사용자측 증거).

각 단계가 화면에서 어떻게 보이는지는 **표가 아니라 발표자 노트로** 푼다(아래 *노트* 참조). 도식의 노드 라벨과 결과 카드 한 컷이면 화면은 충분하다.

> **두 진입점 핵심 차이**: 챗봇은 *소도시 선정을 시스템이* (사용자는 조건만), 지도는 *소도시 선정을 사용자가* (마커 직접 클릭). 일정 생성부터는 동일하다.
>
> **왜 둘 다 두는가**: **입력 질문의 전환("어디로?"→"언제·어느 나라로?")은 AI 챗봇이 해결**한다 — 도시를 안 정한 사용자가 조건만 말하면 시스템이 소도시를 발견·선정. 동시에 **이미 가고 싶은 소도시가 있거나, 지도에서 자유롭게 소도시를 둘러보며 탐색하고 싶은 사용자**를 위해 **지도 마커도 함께 구현**했다. 즉 발견(챗봇)과 직접 탐색·선택(지도)을 모두 지원한다.

**화면용 / 노트용**
- *화면*: **흐름 도식 1개(분기→합류) + 결과 카드 히어로 컷 1장**만. 목업 풀세트·단계별 스크린샷은 화면에 깔지 않는다(B6 데모로). 강조 셋 — (1) **입력 질문의 전환**("어디로?"→"언제·어느 나라로?"), (2) **두 진입점**(말로 묻기 / 지도에서 고르기), (3) **결과物에 이유·출처·검증이 붙는다**(히어로 컷으로 증명).
- *노트*: 두 진입 트랙을 구분해 설명한다.
  - **A. AI 챗봇 진입**: 자연어 조건 입력 → 필수 조건(국가·시기·일정 유형·축제 포함 여부) 추가 취향 질문 루프(선택) → 조건 확정. **소도시 1곳은 시스템이 내부 선정**(후보군 비교·랭킹 비노출, 선정 로직은 B3 에이전트 플로우). 자연어 예시: "바다가 있고 산책하기 좋은 곳으로 가고 싶어."
  - **B. 지도 마커 진입**: 국가·월·테마 필터로 마커 조회 → **사용자가 마커를 직접 클릭해 소도시 확정** → 일정 유형 선택. 여행할 소도시 직접 선택 수단.
  - **합류 이후 공통**: 축제 포함 여부 확인 → 일정 생성 → 결과(이유·출처·검증 뱃지) → 실시간 수정 → 저장·딥링크. 대체 일정 토글은 우천·폭설·장마·태풍 가능성 높을 때 실내 중심으로 전환. 후속 재추천은 진입 경로와 무관하게 챗봇 파이프라인으로 합류.

**서론 회수**
- **S5 컨셉① 발견 회수**: "어디로? 대신 언제·어느 나라로?"가 여기서 **실제 첫 화면**으로 등장(도시를 정해놓고 검색하는 기존 여행앱과 반대). 특히 **A 챗봇 트랙**이 이 컨셉을 그대로 체현(시스템이 발견·선정). **B 지도 트랙**은 둘러보며 직접 고르고 싶은 사용자를 위한 보조 진입. 완전 회수는 B6 데모.
- **S2 고통 장면 → 한 흐름으로**: 날씨·축제·교통·숙소·후기를 앱 여러 개 띄워 비교하던 장면이, 한 대화·한 화면 흐름으로 모인다(시각 완전 회수는 B6의 "흩어진 앱 → Lovv 한 화면" 전후 대비).
- **S5 컨셉③ 근거 예고**: ③·④의 **출처·검증 뱃지**가 "못 믿음→근거"(Explainable RAG)의 사용자측 증거. 시스템적 증거는 B3(Festival Verifier·Output Validator)에서.

**발표자 노트 (큐)**
- "들어오는 길은 둘 — **말로 묻거나(챗봇), 지도에서 고르거나(마커)**" · *(진입 분기 한 번 짚기)*
- "**'어디로?'에서 '언제·어느 나라로?'로 바꾼 건 챗봇** — 도시를 안 정한 사용자의 발견을 돕는다" · *(입력 질문 전환 = 챗봇이 해결)*
- "챗봇이면 **고르는 건 시스템** — 조건만 주면 맞는 소도시 1곳을 찾아 제시" · *(후보 비교 X, 선정은 내부)*
- "**이미 가고 싶은 곳이 있거나, 지도를 둘러보고 싶은** 사용자를 위해 **지도 마커도 구현** — 고르는 건 사용자" · *(지도 = 직접 탐색·선택)*
- "어느 길이든 **일정부터는 똑같다** — 결과엔 *왜 이곳인지 + 출처·검증 뱃지*" · *(합류·근거성 강조)*
- "일정도 **말로 고친다** — '덜 걷게', '카페 추가'" · *(실시간 수정 한 번 시연 언급)*
- "지금은 PoC — 선호·저장은 로컬 스토리지로 대체, 핵심 흐름만" · *(정직한 톤, 노트)*

# B. 2장 버전 (대안 보존)

1장의 여정을 **B2-1 진입~확정(두 진입점으로 소도시 1곳까지)** 과 **B2-2 일정~저장(근거 붙은 일정, 말로 고친다)** 로 쪼갠다. 목업을 크게 보여주거나 진입 분기와 결과物을 각각 강조하고 싶을 때.

## B2-1 — 진입부터 소도시 확정까지 (두 진입점)

**핵심 메시지 (화면 카피)**
> **말로 묻거나, 지도에서 고르거나 — 소도시 1곳으로 좁힌다**

**화면 구성 (목업 2~3컷, 온보딩 공통 → 진입 분기, `02_service_flow` §5~§6 / `user_system_flow` §2.1.1~§2.1.3)**

```
                       ┌─ (A) AI 챗봇 ── 국가·시기·취향 자연어 ── 시스템이 소도시 1곳 내부 선정 ┐
온보딩(스타일→6테마) ─ ┤                                                                        ├─ 소도시 1곳 확정
                       └─ (B) 지도 마커 ─ 국가·월·테마 필터 ── 마커 직접 클릭(사용자 선택) ─────┘   (이유·출처 뱃지)
```

| 트랙 | 단계 | 사용자 행동 | 시스템 반응 | 소도시 확정 주체 |
| --- | --- | --- | --- | --- |
| 공통 | 진입·온보딩 | 대도시 스타일 선택 | 6개 기본 테마로 선호 매핑 저장 | — |
| **A 챗봇** | 조건 입력 | 국가·시기 + 자연어 취향 | 조건 구조화, 부족 시 추가 질문 → 확정 후 **내부 선정** | **시스템** (후보 비교 비노출) |
| **B 지도** | 마커 선택 | 필터 조정 → 소도시 마커 직접 클릭 → 일정 유형 선택 | `destinationId` 고정 | **사용자** (직접 클릭) |

**발표자 노트 (큐)**
- "들어오는 길은 둘 — 첫 질문은 '어디로?'가 아니라 '**언제·어느 나라로?**'" · *(S5 컨셉① 회수)*
- "**챗봇이면 시스템이 1곳을 골라** 준다 — 후보를 늘어놓지 않는 집중형" · *(내부 선정)*
- "**지도면 직접 고른다** — 둘러보고 싶은 사용자용" · *(직접 선택, 대비)*
- "어느 쪽이든 그 1곳에 **왜 추천인지 + 출처**가 붙는다" · *(근거성 1차)*

## B2-2 — AI 일정부터 저장·연동까지

**핵심 메시지 (화면 카피)**
> **근거 붙은 일정까지 — 게다가 말로 고친다**

**화면 구성 (목업 2~3컷, `02_service_flow` §3 5~7단계 · §8 / `user_system_flow` §2.1 7~10단계)**

```
AI 일정 타임라인           →  실시간 수정                →  저장·외부 연동
오전/오후/저녁 +              "덜 걷게/카페 추가/축제 빼줘"   마이페이지 저장 + 지도·숙소
추천 이유·검증 뱃지           해당 블록만 갱신 · 대체 토글     맛집 딥링크 · 좋아요/싫어요
```

| 단계 | 사용자 행동 | 시스템 반응 | 사용자가 보는 것 |
| --- | --- | --- | --- |
| 동적 일정 | 기간·이동 성향·축제 포함 여부 입력 | 일정 초안 생성(Structured Output) | 시간대별 타임라인 + 추천 이유·검증 뱃지 |
| 실시간 수정 | "이 날 덜 걷게", "카페 추가" | 해당 블록만 갱신, 우천 시 실내 대체 | 갱신된 타임라인 + 대체 일정 토글 |
| 저장·연동 | 저장, 피드백, 외부 링크 클릭 | 명시 데이터만 저장, 딥링크 생성 | 마이페이지 저장 일정 + 지도·숙소 딥링크 |

**서론 회수**
- ④·⑤의 **출처·검증 뱃지**가 곧 "못 믿음→근거"(S5 컨셉③)의 사용자측 증거.
- "조건만 말하면 소도시부터 **일정까지**"(S5 핵심 메시지)가 여기서 화면으로.

**발표자 노트 (큐)**
- "추천에서 끝이 아니라 **일정까지**" · *(S5 컨셉② 회수)*
- "마음에 안 들면 **말로 고친다** — 한 줄이면 끝" · *(실시간 수정)*
- "숙소는 직접 추천 X — **검색 딥링크**만(신뢰 가능한 현지 플랫폼)" · *(설계 일관성, 노트)*

- [ ] 1장 채택안 화면 = **흐름 도식 1개(분기→합류) + 결과 카드 히어로 컷 1장**만. 목업 4~5컷 금지(공간·가독성), 전체 워크스루는 B6 데모로. 7단계 상세·자연어 예시·필수 조건·예외는 노트로.
- [ ] 첫 화면은 **"언제·어느 나라로?"** 강조(도시 선입력 가정 X).
- [ ] **두 진입점 구분**: A 챗봇 = *시스템이 소도시 내부 선정*(후보 비교 카드 UI 없음, 선정 과정 비노출, 로직은 B3) / B 지도 = *사용자가 마커 직접 클릭해 선택*. 일정 생성부터 합류.
- [ ] 결과物(추천 소도시·일정)에 **이유·출처·검증 뱃지**가 보이게 — Explainable RAG의 사용자측 증거.
- [ ] **에이전트 노드(Intent·Retriever·Ranker·Validator)는 화면 X** — B3로 미룬다(여기선 "추천 받는다"까지만).

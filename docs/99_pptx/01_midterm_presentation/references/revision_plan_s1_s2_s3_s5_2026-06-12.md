## 1. S1 — 프로젝트 소개

### 1-1. 의미 카드 `L` → `Lo` (확정)
- **현재**: 의미 카드가 `L / Local`, `V / Vibe`, `V / Voyage`. 브랜드는 `Lovv`인데 첫 글자가 `L`이라 `L-V-V`로 읽혀 철자가 어긋난다.
- **변경**: 첫 행 글자를 **`Lo`**(Local)로 바꿔 `Lo-V-V = Lovv`가 정확히 맞게 한다.
- **대상·위치**
  - native HTML: `<div class="letter">L</div><div><strong>Local</strong>...` → `Lo`
  - 대표 문서: S1 화면 구성 `**L / Local**` → `**Lo / Local**`
  - (체크) `.letter` 박스 폭(60px)에 `Lo` 두 글자가 들어가는지 확인. 넘치면 `font-size` 미세 축소.

### 3-2. 가마쿠라 "슬램덩크" 화면 표기 여부 〔확정: A〕
- **현재**: 화면에는 `가마쿠라 에노덴 / 촬영 인파가 건널목 주변으로 집중`만. "슬램덩크"는 대표 문서 노트·괄호에만 존재(화면 X).
- **확정 결정**: **A. 화면 미표기 + 발표 때 구두 언급.** 슬라이드는 미니멀 유지(AWS식)하고 저작권/IP 표기 부담을 피한다. 사진 자체가 성지 건널목이라 시각 후킹은 이미 있으므로, 발표자가 "슬램덩크 성지 건널목"이라고 말로 풀어 인지 후킹을 살린다.
- **반영**: 화면(native HTML) 변경 없음. 대표 문서 S3 발표자 노트에 "가마쿠라 에노덴 = 슬램덩크 성지 건널목 — 화면 미표기, 구두 언급" 한 줄 명시.

### 3-3. "나눠 가면 부담↓" 발표 멘트 보강 (확정 — 노트)
- **현재 화면**: `여행지를 나눠 가면 부담이 줄어든다` (유지)
- **변경**: 화면 문구는 두고, **발표자 노트에 논리 사슬을 명시**한다(이미 일부 있음 → 사용자 카피로 정련).

## 4. S5 — 서론 마무리(핵심 컨셉 3가지)

### 4-1. 문제 → 해결 카드 오버레이 애니메이션 (확정 — 구조 추가)
- **현재**: BRIDGE 슬라이드는 해결 카드 3개(`① 대체 소도시 발견 / ② 일정까지 짜는 Agent / ③ Explainable RAG`)를 정적으로 보여준다.
- **변경**: 같은 3칸에 먼저 **문제**를 깔고, 해결 카드를 하나씩 위에 얹는 빌드 애니메이션.
  - 초기(build-0): ① **발견 난항** · ② **정보 분산** · ③ **신뢰 부족**
  - 클릭마다(build-1·2·3): 각 칸 위에 해결 카드가 페이드/슬라이드로 덮임 → 문제 1:1 해결을 시각적으로 증명.

#### 〔확정〕 문제 레이어는 **격식 명사형**으로 (구어체 X)
- "못 찾음·흩어짐·못 믿음"은 키워드 줄기와는 맞지만 **발표 슬라이드에 띄우기엔 구어체**라 격이 떨어진다. → 해결 카드와 같은 **명사형(2글자 한자어)**으로 격식을 맞춘다.
- **①의 단어 선택 주의**: "대안 부재"는 *대안 자체가 없다*는 뜻이라 이 덱의 핵심 논지(S4 "수요도 정보도 **있다**, 흩어져 못 고를 뿐")와 **모순**된다. 문제는 대안이 없는 게 아니라 **스스로 찾아내지 못하는 것** → "못 찾음"의 의미는 **발견의 실패**로 담는다.
- 채택 표현(서론 키워드 줄기 못 찾음→발견 / 흩어짐→일정 / 못 믿음→근거를 명사형으로 승격):
  - **발견 난항 · 정보 분산 · 신뢰 부족**
- 이렇게 하면:
  - 해결 카드(발견·일정·근거)와 **1:1로 또렷하게 겹친다**(오버레이 효과 극대화). 특히 ① `발견 난항 → 대체 소도시 발견`은 **같은 단어 '발견'으로 문제→해결을 회수**해 메시지가 또렷하다.
  - 2글자+2글자 한자어로 **길이·형식이 균일**해 정렬이 안정적이다.
  - 화면 톤이 격식 있게 유지되면서도 키워드 줄기 회수는 그대로 살아 있다.
  - 매핑(문제 명사형 → 해결):
    | 칸 | 문제(초기·명사형) | 의미 | 해결(오버레이) |
    | --- | --- | --- | --- |
    | ① | **발견 난항** | 직접 찾아도 안 붐비는 대체 소도시를 못 찾는다(대안은 있으나 발견 실패) | 대체 소도시 발견 — "언제·어느 나라로?" |
    | ② | **정보 분산** | 날씨·축제·교통·숙소가 흩어져 비교가 어렵다 | 일정까지 짜는 Agent |
    | ③ | **신뢰 부족** | 낯선 곳이라 믿기 어려워 유명지로 회귀한다 | Explainable RAG(근거·출처) |
  - (보조 한 줄, 선택) 각 문제 명사 아래 작은 회색 부연을 둘 수 있음(위 "의미" 열). 화면이 비면 추가, 빡빡하면 생략(노트로).
  - (대안 표현) `발견 난항` 대신 `탐색 한계`도 가능(발견 단어 중복을 피하고 싶을 때). 더 무게감을 원하면 조사 `의`를 넣은 **`발견의 한계 · 정보의 분산 · 신뢰의 부재`**. → 기본은 `발견 난항`.
- **대상·위치**
  - native HTML: BRIDGE 섹션을 `costs-slide`/`market-slide`의 빌드 패턴과 동일하게 구현. 각 칸에 문제 레이어(단어형) + 해결 레이어를 겹쳐 두고 클래스 토글로 전환. JS의 `nextStep/prevStep`에 빌드 카운터(0→3) 추가.
  - 대표 문서: S5 ② 컨셉 표에 "초기 문제 단어 → 해결 카피" 2단 상태를 명시(현재는 문제 태그 `[대안을 모름]` 등 괄호로 있음 → 화면 초기 단어 `못 찾음·흩어짐·못 믿음`으로 승격).
- **연출 메모**: 클릭 3번이 부담되면 "문제 단어 3개 동시 표시 → 1클릭에 해결 3개 동시 오버레이"의 2단(0→1)으로 축약 가능. **권장: 3단(개별)으로 1:1 회수 강조** — 발견·일정·근거 메시지가 또렷해짐.

### 4-2. 월별 혼잡도 기반 추천을 S5에서 밝힐지 〔확정: A〕
- **맥락**: 현재 '혼잡도 필터'는 확장 방향(B10) 표에서 **Production 단계**로 잡혀 있고, PoC 추천 기준은 "도시 전체 적재"다. B1 노트엔 "월별 기상 경향 기반"만 언급.
- **확정 결정**: **A. S5에서는 밝히지 않는다.** S5는 컨셉 다리(발견·일정·근거)에 집중한다. PoC 범위상 "월별 혼잡도 필터"를 서론에서 단정하면 NEXT 표와 충돌하는 과대주장 위험이 있다. 혼잡도 메커니즘은 B1/B10이 받는다.
- **반영**: S5 화면·노트 모두 혼잡도 미언급. 필요 시 Q&A로만 대응. 혼잡도 본격 설명은 B10 "핵심 축은 혼잡도"가 담당.

### 4-3. "비수기의 유명지도 포함" 문구를 넣을지 〔확정: A+B〕
- **맥락**: Lovv는 "유명지 무조건 제외"가 아니라 **혼잡도가 낮으면(비수기) 유명지도 후보**라는 입장. B8 푸터가 이미 "핵심 축은 '소도시냐 대도시냐'가 아니라 혼잡도"로 이 논리를 깔고 있다.
- **확정 결정**: **A + B 병행.**
  - **A (S5)**: S5 화면은 건드리지 않고, 발표자 노트/Q&A 방어 포인트로만 둔다 — "유명지 배제 아님 — 비수기 유명지도 포함, 기준은 혼잡도". ("왜 유명지를 무시하나?" 반론 대비.)
  - **B (B10)**: 종론 B10 혼잡도 축 설명에 "비수기 유명지도 포함" 한 줄을 더해 제품 철학을 못박는다.
- **반영 범위 메모**: 이번 작업 범위는 S1~S5다. **B항(B10 수정)은 이번 범위 밖**이므로, B10 한 줄 추가는 본론/종론 수정 때 함께 처리하도록 별도 표시만 남긴다(아래 §7 후속).

- 전체 장수는 **S1~S5 + B1~B10 = 16장**을 기준으로 한다. B4가 `B4-1/B4-2` 두 장이므로 실제 페이지 수는 16장이다.

## 4C. B7~B10 재배치 확정 — 비교와 비즈니스 분리

> 배경: 기존 대표 문서에는 **B7 "기존 제품 비교 + 예상 수익"**이 한 장에 합쳐져 있었다. 발표 흐름상 비교와 수익이 한 장에 묶이면 정보량이 많고, 비즈니스가 본론 중간에 끼어 흐름이 끊긴다.

### 결정: 비교는 B8, 비즈니스는 B9

- **B7**: 이슈/트러블슈팅: Neptune 도입 판단
  - 진행 현황의 마지막 장으로 둔다.
  - "기술 판단을 했다"는 메시지로 본론을 닫는다.

- **B8**: 기존 서비스 대비 차별점
  - `lovv_vs_triple.md`의 5행 요약표를 사용한다.
  - 헤드라인: **"트리플은 '정한 도시를 잘' 짜고, Lovv는 '어디를'부터 찾는다"**
  - 제품 설명과 데모 이후에 배치해 비교의 설득력을 높인다.

- **B9**: 비즈니스 모델 / 예상 수익
  - 제목: **"비즈니스 모델 · 예상 수익"**
  - 헤드라인 예: **"경쟁이 아니라 선점 — 발견·분산 단계와 지자체 파트너를 가져간다."**

  1. **앞 B8로 한 줄 콜백** — "앞서 본 차별점이 곧 사업 영역."
  2. **수익 구조 대비 1줄** — 트리플=OTA 상품 / Lovv=지자체·B2G·송객·데이터(수익표로 가는 다리).
  3. **예상 수익표 유지**(지자체 캠페인·송객·리포트 → 월 2,000만원 / 연 2.4억, "가정치·검증 필요" 주석 유지).
  4. 파트너 선점 메시지·발표자 노트·Q&A("트리플이 소도시 기능 넣으면?") 유지.
- 4행 매트릭스(시작점·핵심가치·AI역할·수익구조)는 **B9에서 삭제**한다. 시작점·AI 차별점은 B8로 보내고, B9는 수익 구조에 집중한다.

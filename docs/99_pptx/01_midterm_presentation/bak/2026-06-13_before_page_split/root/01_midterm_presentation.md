## 슬라이드 2 — 오버투어리즘: 현상과 비용

**이 슬라이드의 역할**: ① 오버투어리즘을 한 단어로 소개, ② 워드클라우드로 현상을 체감, ③ 같은 워드클라우드 안에서 비용 메시지로 전환한다.

**① 후킹 (화면 상단 또는 말로)**
**"다들 아는 그곳, 막상 가보니 사람만 보였던 적 없나요?"** → (한 박자 뒤) **"그 순간 여행지는 지역의 부담이 됩니다."**

**② 정의 (화면 한 줄)**
**오버투어리즘**

**③ 워드클라우드 (오버투어리즘 설명 + 문제)**
- 방향: 워드클라우드를 단어 장식이 아니라 **문제 장면을 설명하는 시각자료**로 쓴다. 제목에는 `오버투어리즘`만 남기고, 워드클라우드에서는 중복되는 `오버투어리즘` 단어를 제거한다.
- 화면 카피: **오버투어리즘**
- 화면 중앙 전체: 워드 클라우드 1개. 별도 설명 카드 없이 워드클라우드가 화면의 70% 이상을 차지한다.
- 중심 단어(크게): `수용 한계 초과` · `인파` · `웨이팅` · `쓰레기` · `주민 불편` · `환경 부담` · `교통 혼잡` · `지역 문화 훼손` · `통제`
- 주변 보조 단어(작게): `SNS 성지` · `사유지 출입 금지` · `성수기 쏠림` · `무단 촬영` · `핫플 집중` · `생활폐기물` · `좁은 거리` · `플라스틱` · `관광객 급증`
- 실제 기사 단어 근거:

| 출처 | 기사 장면에서 뽑은 단어 |
| --- | --- |
| AP | `mass tourism`, `crowds`, `infrastructure pressure`, `housing costs`, `short-term rentals`, `water scarcity` |
| Guardian | `Instagrammers`, `litter`, `millions of visitors`, `narrow streets`, `local customs` |
| 뉴시스 | `관광객`, `출입 통제`, `무단 촬영`, `신체 접촉`, `비매너`, `벌금`, `좁은 거리` |
| 제주일보 | `생활폐기물`, `쓰레기 급증`, `소각장 처리 한계`, `관광객 증가`, `플라스틱`, `환경오염` |

- 워드클라우드 하단 캡션은 빌드 단계에 따라 바꾼다.
  - build-0: **"안 붐비는 곳을 못 찾아서, 결국 한 곳에 쏠린다"**를 3구간으로 나눠 순차 등장 처리한다.
  - build-1: 워드클라우드는 배경처럼 약하게 낮춘 뒤, 중앙 오버레이 박스에 **"이 쏠림은 그냥 불편으로 끝나지 않고, 비용을 남긴다"** 캡션을 띄운다.
- S2 화면에서는 출처 칩을 제거한다. 상세 출처는 마지막 출처 모음과 문서 하단 §출처에 둔다.
- 우측 별도 카드/문장 카드 제거. 화면은 **집중된 워드클라우드 + 중앙 오버레이 캡션**으로만 구성한다.

**④ 반전 (흥미 포인트 — '쏠림의 원인'으로 착지)**
> **"붐비는 걸 좋아해서가 아니다 — 안 붐비는 곳을 *못 찾아서*, 결국 한 곳에 *쏠린다*."**
- 소도시 한 곳 가려면 축제/관광지 일정·교통·숙소·후기를 앱 여러 개 띄워 비교 → 게다가 낯선 곳이라 *믿을 수 있는지도 확신이 안 서* → 검색 잘 되고 익숙한 유명지로 회귀. (이 둘 = S5 컨셉 ②·③의 짝)
- **작은 흩어진-앱 컷**(아이콘/탭 미니 컷, PPT 애니메이션)으로 곁들임. 메인 시각은 워드클라우드 유지. *풀 전후 대비(흩어진 앱→Lovv 한 화면)는 본론 데모에서 콜백.*
- 재프레이밍: '무개념 관광객 탓'이 아니라 **정보·발견의 실패**(비난조 X).

**⑤ 하단 전환**
워드클라우드에서 비용 메시지를 보여준 뒤 S3 비용 카드 슬라이드로 넘긴다.

**발표자 노트 (키워드·뉘앙스 큐 · ~30~40초)**
1. **후킹** — "도착하니 사람만, 줄서기·포토스팟" · *(공감 질문, 던지고 한 박자 쉼)*
2. **정의** — over+tourism, 수용 넘는 쏠림 · *(가볍게)*
3. **워드클라우드** — "이게 그 모습" / 인파·웨이팅·성수기 · *(화면 가리키며)*
4. **반전** — 붐벼서가 아니라 *못 찾아서* + 낯설어 *못 믿어서* → 유명지 회귀 · *(한 박자 쉬고 반전 톤, 흥미↑)*
5. **착지** — "그래서 더 쏠린다 → 비용을 남긴다" · 같은 워드클라우드에서 시장 근거로 전환.
- 3원칙: ① 탓하지 않기 · ② 솔루션 아직 말 X(복선만) · ③ 마지막 말은 '비용'으로.

> 화면용: 워드클라우드(메인) + 하단 반전 캡션 → 흐려진 워드클라우드 + 중앙 비용 캡션. 후킹은 말로(또는 상단 짧게).

> 기존 S4(인바운드)+S5(아웃바운드)를 한 장으로 통합. 좌/우 양방향(둘 다 근거 강함) + 하단 한국 국내 보조. **일본인→일본 국내는 화면에서 제외**(점유율·정책 위주라 세기 안 맞음) — 제품은 지원하되 본론 **확장 방향**에서 다룬다.

**핵심 메시지 (화면 카피)**
일본인은 한국 소도시로, 한국인은 일본 소도시로 — 소도시 수요는 이미 양방향으로 움직인다 (국내에서도 마찬가지)

**좌/우 — 한일 양방향 (메인, 칸당 대표 수치 1개)**

| 좌 — 인바운드 (일본 → 한국) | 우 — 아웃바운드 (한국 → 일본) |
| --- | --- |
| **"서울 다음은 어디?"** (라쿠텐 서태석, 국가관광전략회의) | 일본 소도시 예약 **+35%** (모두투어) |
| 외국인 3,000만 목표(2029)·FIT 90%+ | (노트) 구마모토 +261%·여객 2,746만·노선 31개 |

- 인물 사진 미사용 — 텍스트 인용 카드(`국가관광전략회의 / 대통령 주재` 라벨로 권위). 출처는 하단 작게.
- (대안) 아웃바운드 대표를 임팩트로 가려면 "+35%" 대신 **구마모토 +261%**(라쿠텐).

**하단 — 한국 국내 보조 (짧게 한 줄)**
국내에서도 같은 흐름 — 묵호·영월 등 **소도시 언급 +184%**(썸트렌드). *(보조 신호 한 줄로만; 메인은 양방향)*

**하단 — 프로젝트 동기 (양방향+국내를 묶는 한 줄)**
> **수요도, 정보도 있다 — 흩어져 있어 '나에게 맞는 소도시'를 못 고를 뿐이다.**
> 화면 연출: 하단 고정 문구가 아니라 슬라이드 중앙에 강조 박스로 나타나게 구성한다. 실제 PPT 애니메이션을 넣는 경우 `Appear/Fade in` 계열로 마지막 클릭에 등장시키고, 정적 산출물에서는 중앙 강조 상태로 배치한다.
> *(멘트) 그 흩어진 걸 모아 찾아주는 게 Lovv다.* → 다음 장 S5(핵심 컨셉)로 연결.

**발표자 노트 (신뢰도·범위 — 중요)**
- **[직함 혼동 금지]** 서태석(라쿠텐트래블 *한국지사장*)과 김태광(라쿠텐 트래블 *동북아시아 비즈니스 총괄 이사*)은 다른 인물.
- 아웃바운드 노트: 구마모토 +261%·홋카이도 +204%(라쿠텐)는 *한국 Z세대*가 견인 → 일본 국내가 아닌 *아웃바운드* 근거. 사가 +611%. LCC·FSC 지방공항 취항은 "구조적 확대" 방향성만 화면, 개별 노선·단항/휴항은 노트로.
- 한국 국내 노트: 묵호·영월 +184%는 *언급/관심* 지표(예약 아님), 영월은 영화 흥행 효과 일부 → "관심 신호"로 표기.
- **[일본 국내는? Q&A 대비]** 제품은 일본인→일본 국내도 지원. 근거(관광청: 지방부 숙박 90%가 일본인 / 同一현내 숙박 ~20%→35% 근거리화·마이크로투어리즘)는 *점유율·정책* 위주라 근거 슬라이드에선 뺐고, 본론 **확장 방향**에서 다룬다.
- 동기 멘트: "찾아주는 길이 없다"가 아니라 "정보도 있는데 *흩어져* 못 고른다"로 — S2의 '앱 여러 개 비교' 장면을 떠올리게 해 즉시 납득.

> 근거 세기: 아웃바운드 ★★★ / 인바운드 ★★★ / 한국 국내 ★★(보조). 상세·출처는 §출처, `external_evidence_slides_2articles.md`.

## 슬라이드 5 — 서론 마무리: 우리의 의미 + 핵심 컨셉 3가지

> 문제(서론)와 해결(본론)을 잇는 다리. (A) 서론 문제 정리 → ① 프로젝트 의미 → ② 핵심 컨셉 3가지(=강점, 문제와 1:1) → bookend.

**(A) 서론 문제 한 줄 정리 (멘트, 화면 X)**
"여기까지 — 유명지 쏠림은 4가지 비용을 남기고, 그럼에도 소도시 수요는 한일 양방향으로(+국내) 움직인다. 막는 건 셋뿐: ① 대안을 *모르고* · ② 정보는 *흩어졌고* · ③ 낯선 곳은 *못 믿는다*. **한마디로, 오버투어리즘을 '분산'으로 푼다.**"

**핵심 메시지 (화면 카피)**
조건만 말하면 — 나에게 맞는 소도시부터 여행 일정까지, 근거 있는 한 번의 대화로

**① 프로젝트의 의미 (화면 한 줄)**
> **남들 다 가는 곳 말고, 나에게 맞는 소도시 — 더 나은 여행이 곧 분산이 된다.**
- (의미 풀이) 여행자는 한적하고 더 나은 여행을 얻고, 그 선택이 모이면 과밀은 풀리고 소외 소도시엔 기회가 된다. **개인에게 좋은 선택 = 사회적으로도 좋은 결과**(win-win). S3 "비용=양면 기회" 회수.

**② 핵심 컨셉 3가지 = 강점 (화면: 카드 3개, 발견 → 일정 → 근거)**

| 단계 | 초기 문제 레이어 | 해결 오버레이(화면) | 왜 강점·차별점인가 (노트) |
| --- | --- | --- |
| build-1 | **발견 난항** | ① **대체 소도시 발견** — "어디로?" 대신 "언제·어느 나라로?" | 도시를 정해놓고 검색하는 기존 여행앱과 *반대* — 목적지를 제안받는 탐색 |
| build-2 | **정보 분산** | ② **일정까지 짜는 Agent** — 소도시 추천 + 여행 일정(축제·날씨·동선) | 추천에서 끝나지 않고 **일정까지** — 앱 여러 개 띄울 필요 없음 |
| build-3 | **신뢰 부족** | ③ **Explainable RAG** — 근거·출처가 보이는 추천 | 일반 생성형 추천과 달리 '왜 이곳인지' 근거가 보여 처음 가는 곳도 신뢰 |

**(B) 컨셉 ↔ 의미 연결 (멘트)**
"이 셋으로 개인의 '더 나은 발견 + 일정'을 돕고, 그 선택이 모이면 곧 분산이 된다." — 미시(개인)와 거시(분산)를 잇는다.

**마무리 bookend (S1 태그라인 원문 회수)**
> **"서울·오사카 말고, 지금은 이곳 — 그게 Lovv입니다."** (S1과 똑같은 문장으로 닫아 수미상관)

**발표자 노트**
- 화면엔 **메시지 1 + 컨셉 카드 3 + 의미 1줄**만. recap·강점 설명·연결은 멘트로(글자 폭탄 방지). 3단계 오버레이로 `발견 난항 → 대체 소도시 발견`, `정보 분산 → 일정까지 짜는 Agent`, `신뢰 부족 → Explainable RAG`를 하나씩 덮어 문제→해결 1:1을 보이게 한다.
- 복선 회수: S2의 "못 찾는 거라면?"을 여기서 컨셉①로 1차 회수(완전 회수는 본론 데모). 같은 단어(찾아준다·모아준다·믿고)로 받기.

## 슬라이드 B1 — 본론 진입: Lovv 시스템 흐름 (Explainable RAG)

**핵심 메시지 (화면 카피)**
Explainable RAG — 입력부터 근거 있는 추천·일정까지, 한 흐름으로

**화면 구성**
- 좌→우 5계층 파이프라인 1개: 사용자 입력(국가·시기·테마) → Backend API → Recommendation Agent → Data Store ↔ External APIs → 근거 포함 추천 + 일정
- 5계층 역할:

| 계층 | 한 줄 역할 |
| --- | --- |
| Client Web | 온보딩·지도·챗봇·결과·마이페이지 |
| Backend API | 인증·검증·외부 프록시·추천 실행 |
| Recommendation Agent | 조건 분류 → 검색 → 선정 → 일정·이유 생성 |
| Data Store | 목적지·축제·월별 기상 경향·저장·검토 이력 |
| External APIs | Google/Kakao Maps, WeatherAPI, Yahoo Japan 딥링크 |

- 도식 위 UX 주석 3개: ① 이유·출처·검증 뱃지 ② "덜 걷게/카페 추가" 실시간 수정 ③ 지도·숙소 딥링크

**발표자 노트**
- "입력 → 검색 → 선정 → 일정, 한 흐름"만 빠르게 설명한다.
- 핵심 차별점은 **근거가 보이는 추천 = Explainable RAG**.
- 세부 에이전트 동작은 B3에서 확대한다.
- PoC는 정적 시드/JSON 기반과 최소 추천 API로 핵심 흐름을 검증한다. 추천은 실시간 예보가 아니라 월별 기상 경향 기반이고, WeatherAPI 현재 날씨는 표시용이다.

## 슬라이드 B2 — 유저 플로우

**핵심 메시지 (화면 카피)**
"어디로?"가 아니라 "언제·어느 나라로?" — 묻는 순간부터 다르다

**서브 카피**
들어오는 길은 둘 — 말로 묻거나(AI 챗봇), 지도에서 고르거나(마커). 일정부터는 한 흐름.

**화면 구성**
- 흐름 도식 1개:

```text
                ┌─ (A) AI 챗봇 ── 국가·시기·취향 자연어 입력 ── 시스템이 소도시 1곳 내부 선정 ─┐
① 온보딩 ─ 진입 ┤                                                                              ├─ AI 일정 ─ 저장·연동
                └─ (B) 지도 마커 ─ 국가·월·테마 필터 ── 마커 직접 클릭(사용자가 소도시 선택) ──┘
```

- 결과 카드 히어로 컷 1개: 추천 소도시 1곳 + 시간대별 타임라인 + 추천 이유·출처·검증 뱃지.
- 목업 4~5컷을 한 장에 모두 깔지 않는다. 전체 워크스루는 B6 데모가 받는다.

**발표자 노트**
- 챗봇 트랙: 도시를 정하지 않은 사용자가 국가·시기·취향만 말하면 시스템이 소도시 1곳을 내부 선정한다.
- 지도 트랙: 이미 가고 싶은 곳이 있거나 둘러보고 싶은 사용자는 마커를 직접 선택한다.
- 어느 길이든 일정 생성부터 합류한다. 결과에는 "왜 이곳인지"와 출처·검증 뱃지가 붙는다.
- "덜 걷게", "카페 추가"처럼 말로 일정 일부를 고치는 흐름을 B6 데모에서 보여준다.

## 슬라이드 B3 — 에이전트 플로우

**핵심 메시지 (화면 카피)**
Tool은 계산하고, Agent는 판단한다

**서브 카피**
검색·점수·검증은 Tool 절차로 분리하고, Agent는 그 결과를 보고 다음 행동을 정한다.

**화면 구성 — 7단계 Tool/Agent 파이프라인**

| 단계 | 노드 | Tool이 하는 일 | Agent가 판단하는 일 |
| --- | --- | --- | --- |
| 1 | 의도 분류 | 입력 스키마 검증 | 자연어를 필수/선호/미지원으로 구조화 |
| 2 | 라우팅 | 상태·완료 플래그 확인 | 다음 노드·재시도·fallback 선택 |
| 3 | 검색(다형) | 게이트·벡터검색·후보 추출 | 검색 모드 선택, 후보 부족 시 fallback |
| 4 | 선정 | 도시별 점수 분해 | 최종 소도시 선정 + 추천 근거 생성 |
| 5 | 축제 검증 | 개최일·출처 확인 | 확정/잠정/제외에 따라 일정 반영 판단 |
| 6 | 일정 작성 | 동선·유형 제약 확인 | 읽히는 일정·대체안·설명 생성 |
| 7 | 출력 검증 | 조건·근거·환각 체크 | 실패 유형별 재시도 또는 안내문 |

**서론 회수 띠(화면 하단)**
- `[3·4 검색·선정] → 발견`
- `[6 일정 작성] → 일정`
- `[5·7 검증] → 근거`

**발표자 노트**
- LLM이 점수를 상상하거나 없는 장소를 만들 위험을 줄이기 위해 계산·검색은 Tool 절차로 분리한다.
- Agent는 Tool 결과가 충분한지 판단해 진행, 재검색, 조건 완화, 안내문 전환을 선택한다.
- 미검증 축제, 국가 혼합, 근거 없는 장소는 최종 결과로 내보내지 않는다.

## 슬라이드 B4-1 — 구현 딥다이브 ① Explainable RAG

**핵심 메시지 (화면 카피)**
벡터 검색만으로는 "왜 여기"를 말할 수 없다

**서브 카피**
의미는 vector, 사실은 원장 — 역할을 나눠 근거가 보이는 추천을 만든다.

**화면 구성 — 문제 → 설계 → 효과**

| 갈래 | 무엇을 찾나 | 역할 | 구현 단계 |
| --- | --- | --- | --- |
| 사실 원장 | 장소·축제·좌표 등 확정 사실 | 화면·API가 신뢰하는 최종 기준 | PoC: RDS |
| vector RAG Index | 의미·분위기로 유사한 장소 | 자연어·soft 조건 매칭, 유사도 재랭킹 | PoC 구현·검증 |
| 관계 탐색 | 도시–테마–축제–인접도시 관계 | 관계 기반 후보 확장 | 현재: Lambda 관계 탐색 보조 기능 구현 예정, 규모 확장 시 그래프DB 검토 |

**하단 각주**
현재 PoC는 RDS와 vector 기반으로 구현하고, 그래프DB와 유사한 관계 탐색 기능은 Lambda 보조 기능으로 구현 예정이다. 그래프DB(Neptune)는 관계가 깊어지는 Production 단계의 미래 검토 항목으로만 둔다.

**발표자 노트**
- 벡터 검색은 분위기·의미 유사도는 잘 잡지만, 장소 존재 여부·올해 운영 여부·축제 개최일은 확정하지 못한다.
- 확정 사실은 원장(RDS), 의미 매칭은 vector index로 분리해 추천 근거와 출처를 붙인다.
- 그래프DB는 현재 핵심 근거가 아니다. 현재는 Lambda로 유사 관계 탐색 기능을 구현하고, 다도시 일정, 사용자 행동 기반 발견, 3홉 이상 관계 순회가 필요해질 때 그래프DB를 검토한다.

## 슬라이드 B4-2 — 구현 딥다이브 ② Ranker

**핵심 메시지 (화면 카피)**
가장 비슷한 장소가 있는 도시 ≠ 가장 좋은 여행지

**서브 카피**
점수를 항목으로 분해해 최종 소도시를 고르고, 왜 골랐는지 남긴다.

**화면 구성 — 점수 구성요소 6개**

| 점수 항목 | 역할 |
| --- | --- |
| 테마 충족도 | 선택한 모든 테마를 만족하는지 `theme` 컬럼 집계로 확정 |
| 테마 특화도 | 고정 비율이 아니라 전체 평균 대비 강한 테마인지 비교 |
| 희소 테마 보정 | 온천처럼 희소한 테마는 적은 개수도 강한 근거로 반영 |
| 도시 규모 보정 | 장소 수를 구간화해 큰 도시의 물량 우위를 제한 |
| soft/raw 적합도 | 분위기·원문 맥락은 임베딩 유사도로 가산 |
| audit log | 어떤 항목이 선택에 기여했는지 기록 |

**화면 작은 캡션**
실측: 미식 43% vs 온천 2%, 도시별 장소 수 21~440개. 단순 개수·평균은 큰 도시·흔한 테마로 쏠린다.

**발표자 노트**
- Ranker는 최적 공식을 완성했다는 주장이 아니라, 추천 이유를 분해·기록해 검증 가능하게 만드는 scoring framework다.
- 테마 충족은 벡터가 아니라 구조화 컬럼으로 확정하고, 분위기 같은 soft 조건만 임베딩으로 보정한다.
- 이후 추천 품질 평가와 ablation으로 항목별 효과를 분리 검증할 수 있다.

## 슬라이드 B6 — 데모 영상

**핵심 메시지 (화면 카피)**
흩어진 앱 비교가 — 한 번의 대화와 일정으로 끝난다

**구성(30~60초)**
국가·시기·테마 입력 → 소도시 추천(추천 이유·출처·검증 뱃지) → 일정 타임라인 → "덜 걷게/카페 추가" 실시간 수정 → 저장.

## 슬라이드 B7 — 이슈/트러블슈팅: Neptune 도입 판단

**핵심 메시지 (화면 카피)**
Neptune은 지금 넣는 기술이 아니라, 조건이 맞을 때 승격하는 선택지다

**서브 카피**
그래프DB가 멋있어서 쓰는 것이 아니라, 다단계 관계 탐색이 실제 병목이 될 때만 도입한다.

**화면 구성 — 이슈 → 판단 → 대응 → 승격 기준**

| 구분 | 발표 화면 문구 |
| --- | --- |
| 이슈 | 도시·테마·축제·인접도시 관계를 그래프로 풀 수 있지만, PoC 관계는 대부분 1~2-hop 수준 |
| 비용 | Neptune Serverless도 최소 1 NCU부터 시작해 월 30만원 예산에서 고정비 비중이 큼 |
| 판단 | PoC/Production 1차는 Neptune 미도입. Lambda로 그래프DB 유사 기능을 구현하고 RDS JOIN, DynamoDB 인접 리스트, 사전계산 후보 테이블을 조합 |
| 트러블슈팅 | 관계 탐색을 원본 저장소가 아닌 Lambda 기반 재생성 가능 보조 기능으로 정의해 vendor lock-in과 비용 리스크를 낮춤 |
| 승격 기준 | 3-hop 이상 임의 경로 탐색, 대규모 실시간 그래프 쓰기, 복잡한 그래프 알고리즘이 필요해질 때 Neptune 도입 |

**화면 도식**

```text
문제 인식      비용 검토        대체 구현                 승격 기준
관계 탐색 필요 → Neptune 고정비 → Lambda/RDS/DynamoDB/사전계산 → 3-hop+·실시간 그래프
```

**하단 출처**
내부: `04_database_design.md`, `neptune_alternative.md` / 외부: AWS Neptune Pricing

**발표자 노트**
- “Neptune을 못 쓴다”가 아니라 “지금 쓰면 오버엔지니어링이라 보류했다”로 말한다.
- PoC의 목적은 추천 흐름과 근거 제시 검증이다. 현재 규모에서는 RDS JOIN, DynamoDB 인접 리스트, 사전계산, Lambda 인메모리 그래프로 충분하다.
- 그래프DB는 다도시 일정, 사용자 행동 기반 co-visitation, 3-hop 이상 관계 순회가 실제 요구가 될 때 승격한다.
- 질문 대비: “왜 처음부터 Neptune을 안 쓰나?” → 비용 대비 효용이 낮고, 현재 관계 깊이는 Lambda/RDS/DynamoDB로 방어 가능하다.

## 슬라이드 B8 — 기존 서비스 대비 차별점

**핵심 메시지 (화면 카피)**
트리플은 '정한 도시를 잘' 짜고, Lovv는 '어디를'부터 찾는다

**서브 카피**
여행 실행 앱과 경쟁하기보다, 목적지 발견 단계와 수요 분산 문제에 집중한다.

**화면 구성 — 완화형 비교표**

| 항목 | 트리플 (여행 실행·일정) | Lovv (추천 엔진) |
| --- | --- | --- |
| 시작점 | 도시·여행지를 고른 뒤 일정 생성·관리로 들어감 | 국가·시기·취향만 주면 대체 소도시 후보부터 추천 |
| 목적지 타깃 | 대도시·인기 관광지 일정 실행에 강점 | 한일 소도시와 혼잡 회피 목적지 발견에 집중 |
| 날씨·계절 | 일정 생성에서 날씨·계절 조건이 핵심 비교축으로 드러나지는 않음 | 월별 기상 경향과 실내 중심 대체 일정 토글을 추천 맥락에 반영 |
| 추천 근거 | 취향 기반 추천과 상품 연결 중심, 근거 노출은 제한적 | RAG로 후보를 비교하고 이유·출처를 함께 제시 |
| 외부 연동 | 예약·상품·혜택 연결이 강점 | 직접 판매보다 현지 플랫폼·지도 딥링크로 연결 |

**하단 주석**
비교는 기능 우열이 아니라 **출발점의 차이**를 보여주기 위한 것이다. 트리플은 여행 실행·예약 연결에 강하고, Lovv는 목적지 발견·근거 기반 추천·수요 분산에 집중한다. 작은 출처: `triple.guide`, `interparktriple`.

**발표자 노트**
- 화면 문구는 완화해서 쓰고, 말로 풀 때만 차이를 선명하게 설명한다.
- "트리플은 이미 도시를 정한 사람에게 강하다. Lovv는 아직 어디로 갈지 모르는 사람에게 소도시 후보를 먼저 찾아준다"로 설명한다.
- "날씨 미반영", "블랙박스"처럼 단정적으로 들리는 표현은 화면에서 쓰지 않는다. 대신 "핵심 비교축으로 드러나지는 않음", "근거 노출은 제한적"처럼 말한다.
- 질문 대비: "트리플이 소도시 기능을 넣으면?" → 트리플은 예약 상품 연결에 강하고, Lovv는 지역 데이터 검증·혼잡 분산 목적·B2G 리포트가 설계 중심이라 출발점과 파트너가 다르다.

## 슬라이드 B9 — 비즈니스 모델 · 예상 수익

**핵심 메시지 (화면 카피)**
경쟁이 아니라 선점 — 발견·분산 단계와 지자체 파트너를 가져간다

**서브 카피**
앞서 본 차별점이 곧 사업 영역이다. Lovv는 예약 경쟁보다 지역 발견·분산 성과·데이터 리포트에 초점을 둔다.

**화면 구성 — 수익 구조 + 예상 수익**

| 수익원 | 산식 | 월 예상 |
| --- | --- | --- |
| 지자체 캠페인·대시보드 | 5개 지역 × 월 300만원 | 1,500만원 |
| 송객·콘텐츠 제휴 | 제휴 GMV 월 5,000만원 × 5% | 250만원 |
| 지역 데이터 리포트 | 5개 기관 × 월 50만원 | 250만원 |
| **합계** |  | **월 2,000만원 / 연 2.4억원** |

**하단 주석**
숫자는 초기 Production 가정치다. PoC에서는 사용자 추천 흐름과 파트너 대시보드 수요를 검증하고, 실제 단가는 지자체 제안·제휴 계약에서 확정한다.

**발표자 노트**
- 비즈니스는 서론에서 말하지 않고, 제품 구조와 데모를 본 뒤 종론에서 말한다.
- 예상 수익은 확정 매출이 아니라 사업성 검토용 시나리오다.
- 파일럿에서는 1~2개 지역 무료/저가 실증 → Production에서 유료 캠페인·리포트로 전환한다.
- 수익 메시지는 "광고를 붙인다"가 아니라 "분산 성과와 지역 파트너 데이터를 판다"에 가깝게 설명한다.

## 슬라이드 B10 — 확장 방향 + 남은 태스크

**핵심 메시지 (화면 카피)**
PoC로 핵심 가설을 좁게 검증하고, Production에서 전국·운영·비즈니스로 확장한다

**화면 구성 — PoC → Production 2열**

| 영역 | 지금(PoC) | 다음(Production) |
| --- | --- | --- |
| 데이터 범위 | 강원·경북 40도시 + 일본 관동 일부 | 한국 전국·일본 전역으로 도시·관광지·축제 확장 |
| 데이터 운영 | 1회 수집, 출처·수집시각·신뢰도 기록 | 시변 데이터 정기 갱신 파이프라인, `needs_review` 검수, 관리형 DB 적재 |
| 추천 모드 | Discovery(목적지 미정 → 소도시 발견) 흐름 검증 | 지도 선택·일정 보충 모드로 확장 |
| 추천 기준 | 강원·경북 도시 전체 적재, 소도시 별도 추출 전 | 월별 방문객 수 기반 혼잡도 필터링, 비수기 유명지까지 같은 기준으로 포함 |
| 성능·품질 | 핵심 흐름 동작 검증 | 추천 8초·RAG 1초 목표, 추천 품질 평가·모니터링 하네스 |
| 비즈니스·운영 | 문제·시장 근거 제시 | 지자체 파트너 대시보드(B2G), 송객·콘텐츠 제휴, 역할 기반 운영/검토 |

**마무리 한 줄**
PoC로 핵심 가설이 좁은 범위에서 작동함을 확인 — Production에서 전국·운영·비즈니스로 확장한다.

**발표자 노트**
- 범위는 정직하게 말한다: P1 구현, P2~P3 설계, P5 제외.
- 핵심 축은 "소도시냐 대도시냐"가 아니라 **혼잡도**다. 서울·오사카 말고는 성수기 혼잡 회피의 메시지이며, 비수기 유명지도 같은 로직으로 확장 가능하다.
- 다음 우선순위: ① 데이터 범위 확장 ② 추천 품질 평가·모니터링 하네스 ③ 시변 데이터 갱신 파이프라인 ④ Production 인증·운영 API·지자체 대시보드.

## 디자인 도구에 넘기기 전 확인 사항

- [ ] 표지에는 단계("중간발표 / PoC") 표기를 넣지 않고, 날짜·발표자만 우측 하단에 배치
- [ ] S1 로고 아래 "무엇인가" 한 줄 정의 포함
- [ ] S2 워드클라우드는 *현상 키워드만*(비용 단어는 S3로), 수치 포함 시 출처 필요. 흩어진-앱 작은 컷(애니메이션)
- [ ] S3 2×2 카드 — 카드당 대표 수치 1개만, 블로그 출처 수치는 정성 표현 or 1차 확인 후
- [ ] S4 인물 사진 미사용(텍스트 인용 카드), 직함 혼동 금지, LCC "구조적 확대"만 화면, 한국 국내는 보조 한 줄
- [ ] S4 일본 국내 칸 없음(확장으로) — "일본 국내는?" Q&A 멘트 준비
- [ ] S5 화면 = 메시지 1 + 컨셉 카드 3(문제 태그) + 의미 1줄, 마무리 S1 태그라인 bookend
- [ ] B1 시스템 흐름은 5계층 큰 그림 1장으로 유지하고, 에이전트 상세는 B3로 넘김
- [ ] B2는 분기→합류 흐름 도식 + 결과 카드 히어로 컷 1장만 사용
- [ ] B3는 7단계 Tool/Agent 파이프라인과 발견·일정·근거 회수 띠를 포함
- [ ] B4-1은 vector+RDS grounding 중심, Neptune은 미래 각주로만 표기
- [ ] B4-2는 Ranker를 최적 공식이 아닌 검증 가능한 scoring framework로 설명
- [ ] B5는 한국/일본/표시·연동용 3카드로 출처와 목적을 같이 표기
- [ ] B6은 영상 준비 전까지 Before/After 도식 자리만 유지
- [ ] B7은 Neptune 이슈를 "도입 실패"가 아니라 "비용·범위 검토 후 대체 아키텍처로 트러블슈팅"으로 설명

## HTML 변환 후 감량 우선 후보

HTML/PPT 변환 뒤 글자 수가 많거나 시선 집중에 방해될 가능성이 높은 슬라이드는 아래 순서로 먼저 점검한다.

1. **S2 오버투어리즘 워드클라우드** — 단어가 많아지면 메인 단어 6~8개만 강하게 두고, 보조 단어는 흐리게 줄인다. 출처는 단어 묶음 옆 작은 칩으로만 둔다.
2. **S5 핵심 컨셉 3가지** — 3단계 오버레이가 핵심이므로 카드 안 설명문은 1줄 이하로 줄이고, 차별점 설명은 발표자 노트로 내린다.
3. **B3 에이전트 플로우** — 7단계 표가 길면 화면에는 `의도 분류 → 검색·선정 → 검증·일정 → 출력` 4묶음으로 압축하고, 세부 Tool/Agent 차이는 노트로 보낸다.
4. **B4-1 Explainable RAG** — 표의 `역할` 문장이 길면 `사실 원장 / vector / 관계 탐색` 3분할 구조만 남긴다.
5. **B7 Neptune 트러블슈팅** — 표 5행 + 도식이 함께 과밀하면 표를 `이슈·판단·대응·승격 기준` 4행으로 줄인다.
6. **B8 트리플 비교** — 5행 표가 빽빽하면 화면엔 `시작점 / 추천 근거 / 비즈니스 연결` 3행만 두고, 나머지는 발표자 노트로 설명한다.
7. **B10 확장 방향** — 6행 표가 과밀하면 `데이터 / 추천 / 운영·비즈니스` 3묶음으로 접는다.

## 0. 이 문서의 사용법

`archive/2026-06-13_cleanup/agent_deep_dive_revision_proposal.md`는 메시지·논리·방어 자료가 충분하지만 발표 슬라이드로 쓰기엔 밀도가 너무 높다. 이 문서는 그 내용을 **실제 슬라이드 3장**으로 압축하고, 모든 요소를 세 곳으로 분리한다.

```text
[화면]  슬라이드에 실제로 띄울 것 — 한글 한 줄 + 도식 1개 원칙, 영어 용어 최소화
[노트]  발표자가 말로 푸는 것 — 영어 용어·세부 로직은 여기로
[Q&A]   슬라이드에 없고, 질문 들어오면 꺼내는 백업 — §5 부록
```

**슬라이드 수 판단**: B3 파이프라인 1장 + **B4 딥다이브 2장(B4-1 하이브리드 RAG, B4-2 Ranker)** = **3장**. B4 딥다이브 주제는 **RAG와 Ranker로 확정**한다. 기존의 검증/Fallback 레일은 폐기하지 않고 **§3-대안**으로 아래에 남겨, 시간·청중에 따라 교체할 수 있게 둔다.

**서론 회수 원칙(전 슬라이드 공통)**: 서론 S5의 3컨셉을 *같은 단어*로 받는다.

```text
① 발견  → Polymorphic Retriever + Ranker
② 일정  → Planner / Itinerary Writer
③ 근거  → Explainable RAG(하이브리드) + Festival Verifier + Output Validator
```

## 1. 슬라이드 B3 — 에이전트 플로우

### [화면]

**제목 카피**

```text
Tool은 계산하고, Agent는 판단한다
```

**서브 카피**

```text
검색·점수·검증은 Tool 절차로 분리하고, Agent는 그 결과를 보고 다음 행동을 정한다
```

**도식 (좌→우 파이프라인 1개, 노드명 + Tool/Agent 한 줄씩)**

| 단계 | 노드 | Tool이 하는 일 | Agent가 판단하는 일 |
| --- | --- | --- | --- |
| 1 | 의도 분류 | 입력 스키마 검증 | 자연어를 필수/선호/미지원으로 구조화 |
| 2 | 라우팅 | 상태·완료 플래그 확인 | 다음 노드·재시도·fallback 선택 |
| 3 | 검색 (다형) | 게이트·벡터검색·후보 추출 | 검색 모드 선택, 후보 부족 시 fallback |
| 4 | 선정 | 도시별 점수 분해 | 최종 소도시 선정 + 추천 근거 생성 |
| 5 | 축제 검증 | 개최일·출처 확인 | 확정/잠정/제외에 따라 일정 반영 판단 |
| 6 | 일정 작성 | 동선·유형 제약 확인 | 읽히는 일정·대체안·설명 생성 |
| 7 | 출력 검증 | 조건·근거·환각 체크 | 실패 유형별 재시도 또는 안내문 |

> 화면에는 **노드명 7개 + 마지막 두 열만** 띄운다. Tool/Agent 구분이 핵심 메시지("계산 vs 판단")를 그대로 받친다.

**서론 회수 띠 (파이프라인 하단에 작게, 화면 요소)**

파이프라인 노드 아래에 서론 S5의 3컨셉을 색 태그로 깔아, 어느 단계가 어느 약속을 받는지 *눈으로* 묶는다. 멘트만으로는 B1·B2를 거치며 서론과 멀어져 콜백이 약해지므로, 화면에 같은 단어를 다시 띄워 시각 회수한다(원본 `archive/2026-06-13_cleanup/body_revision_proposal.md` §0.5 "가장 강한 콜백 = 시각 회수").

```text
[3·4 검색·선정] ──▶ 🔵 발견      "못 찾던 소도시를 찾아준다"
[6 일정 작성]   ──▶ 🟢 일정      "장소 목록이 아니라 움직일 수 있는 일정으로"
[5·7 검증]      ──▶ 🟠 근거      "왜 이곳인지 출처·검증과 함께"
```

- 색·아이콘은 서론 S5 컨셉 카드에서 쓴 것과 **동일하게** 맞춘다(같은 그림을 두 번 써야 "아, 아까 그거"가 산다).
- 태그는 노드 위에 얇게 얹는 라벨 수준 — 새 도식을 추가하지 말고 기존 파이프라인에 색만 입힌다.

### [노트]

- "LLM이 점수를 상상하거나 없는 장소를 만들 위험을 줄이려고, 계산·검색은 Tool 절차로 분리합니다."
- "Agent는 Tool 결과를 보고 다음 행동을 정합니다. 후보가 부족하면 조건을 완화하거나 backup theme을 쓰고, 축제가 불확실하면 일정에 직접 넣지 않고 안내로 돌립니다."
- "그래서 Lovv는 추천 결과뿐 아니라 '왜 이 추천이 나왔는지'까지 설명할 수 있습니다."
- 영어 용어(`Polymorphic_Retriever`, `Supervisor_Router`, `fulfilled_matrix`)는 화면 X, 필요 시 말로만.

### [서론 회수 멘트]

- (화면의 색 태그를 손으로 짚으며) "서론에서 약속한 발견·일정·근거 — 이 세 가지를 각각 어느 단계가 책임지는지 보입니다." (3·4=발견, 6=일정, 5·7=근거)
- 멘트는 **화면 회수 띠를 보조**하는 용도다. 띠가 1차 콜백, 멘트가 2차 확인.

### [원본 B3 → 이 슬라이드 변경 매핑]

| 항목 | 원본 B3 (`archive/2026-06-13_cleanup/body_revision_proposal.md`) | 이 슬라이드 (= `agent_deep_dive` 우선) |
| --- | --- | --- |
| 핵심 카피 | "단일 LLM이 아니라 — 멀티스텝 Agent + 결정적 Skill" | "Tool은 계산하고, Agent는 판단한다" |

## 2. 슬라이드 B4-1 — 딥다이브 ① Explainable RAG (하이브리드 검색)

> 근거 문서: `06_technical_spec.md` §6.1 추천용 데이터 / §8.1 Agent 런타임 경계 · `04_database_design.md` §원장 vs 검색 보조

### [화면]

**제목 카피**

```text
벡터 검색만으로는 "왜 여기"를 말할 수 없다
```

**서브 카피**

```text
의미는 vector, 사실은 원장, 관계는 graph — 역할을 나눈 하이브리드 검색
```

**도식 (문제 → 설계 → 효과, 3계층 1장)**

| 갈래 | 무엇을 찾나 | 역할 | 구현 단계 |
| --- | --- | --- | --- |
| 사실 원장 | 장소·축제·좌표 등 **확정 사실** | 화면·API가 신뢰하는 최종 기준 | **PoC: RDS** |
| vector RAG Index | 의미·분위기로 **유사한 장소** | 자연어·soft 조건 매칭, 유사도 재랭킹 | PoC 구현·검증 |
| 관계 탐색 | 도시–테마–축제–인접도시 **다단계 관계** | 관계 기반 후보 확장 | **PoC: RDS JOIN → 최종: 그래프DB(Neptune)** |

**하단 한 줄 (화면에 작게, 정직하게)**

```text
지금은 RDS로 구현 — 관계가 깊어지는 Production 단계에서 그래프DB로 전환한다
```

**문제 → 설계 → 효과 (화면 3박자)**

- 문제: 벡터 검색만으로는 장소의 존재·올해 운영·축제 개최를 *확정*할 수 없다(환각 위험).
- 설계: 의미 검색(vector) + 사실 원장(RDS) + 관계 탐색을 **역할로 분리**. 관계는 PoC는 RDS, 최종은 그래프DB.
- 효과: "왜 이곳인지" 근거·출처를 붙일 수 있다 (= 서론 S5 컨셉③ 회수).

### [노트]

- "벡터 검색은 비슷한 걸 잘 찾지만, 그게 실제로 존재하는지·올해 운영하는지·축제가 열리는지는 보장 못 합니다. 그래서 의미 검색은 vector, 확정 사실은 원장으로 분리했습니다."
- "사실의 단일 기준이 원장 하나라, vector 인덱스는 원장에서 언제든 재생성 가능한 **파생 인덱스**입니다. 그래서 추천에 출처(`source_url`)를 붙일 수 있습니다."
- **PoC 정직성 (핵심)**: "현재 PoC는 **RDS(관계형 DB)로 구현·검증**했습니다. 소수 도시 규모에선 RDS로 충분합니다."
- **왜 최종엔 그래프DB인가 (개선 스토리 = 딥다이브의 '왜')**: "도시–테마–축제–인접도시–이동처럼 **관계를 여러 단계 타고 들어가는 탐색**은 RDS에선 JOIN이 깊어질수록 비효율적입니다. 전국·관계 확장 단계에서는 그래프DB(Neptune)가 순회로 자연스럽게 풉니다. 그래서 PoC는 RDS, Production은 그래프로 가는 전환을 설계에 박아뒀습니다."
- 품질 목표(노트로만): RAG 검색 1초 이내, 추천 전체 8초 이내.
- 예상 질문 대비: "RDS로 됐는데 왜 그래프냐" → 위 '다단계 관계 탐색' 한 줄로 답한다.

### [서론 회수 멘트]

- "서론의 '낯선 곳은 못 믿어 유명지로 돌아간다' — 근거가 보이는 추천(Explainable RAG)이 그 불신을 받는 설계입니다." (③ 근거)

## 3. 슬라이드 B4-2 — 딥다이브 ② Ranker (설명 가능한 소도시 선정)

> 근거 문서: `05_agent_spec/recommendation_flow.md` §4~10 · `05_agent_spec/theme_analysis_report.md`(실측 분석)

### [화면]

**제목 카피**

```text
가장 비슷한 장소가 있는 도시 ≠ 가장 좋은 여행지
```

**서브 카피**

```text
점수를 항목으로 분해해 최종 소도시를 고르고, 왜 골랐는지 남긴다
```

**도식 (점수 구성요소 — 핵심 6개만)**

| 점수 항목 | 역할 |
| --- | --- |
| 테마 충족도 | 선택한 모든 테마를 만족하는가 (`theme` 컬럼 집계로 확정) |
| 테마 특화도 | 고정 20%가 아니라 **전체 평균 대비** 강한가 |
| 희소 테마 보정 | 온천 1개=강한 근거, 미식 1개=약함 |
| 도시 규모 보정 | 장소 수 **구간화** — 큰 도시의 물량 우위 제한 |
| soft / raw 적합도 | 분위기·원문 맥락을 임베딩 유사도로 가산 |
| audit log | 어떤 항목이 선택에 기여했는지 기록 |

**하단 한 줄**

```text
Ranker는 "정답 공식"이 아니라, 추천 이유를 분해해 검증·설명 가능하게 만드는 scoring framework
```

**문제는 데이터로 증명 (화면 작은 캡션 1~2줄)**

```text
실측: 미식 43% vs 온천 2%,  도시별 장소 수 21~440개 (편차 큼)
→ 단순 개수·평균은 큰 도시·흔한 테마로 쏠린다  → 특화도·희소 보정·규모 구간화가 필요
```

### [노트]

- **테마 충족도**: "선택한 테마를 모두 만족해야 후보입니다. 이 필수 판단은 벡터가 아니라 `theme` 컬럼 집계로 확정합니다 — 빠르고 안정적이라서요."
- **테마 특화도**: "전체에서 바다·해안이 5%인데 어떤 도시가 15%면 그 도시는 바다 특화로 봅니다. 고정 20% 기준의 한계를 평균 대비 비교로 보완합니다."
- **희소 테마 보정**: "온천은 1~2개만 있어도 강한 추천 근거지만, 미식은 흔해서 1개론 약합니다. 테마별 데이터 분포 차이를 반영합니다."
- **도시 규모 보정**: "장소 수를 그대로 더하면 강릉처럼 큰 도시만 반복 추천됩니다. 0 / 1-2 / 3-5 / 6-10 / 11+ 구간으로 끊어 물량 우위를 제한합니다."
- **soft/raw 적합도**: "분위기·자연어 뉘앙스만 임베딩 유사도로 가산점에 씁니다 — 후보를 탈락시키는 게 아니라 재랭킹 기준입니다."
- **추천 근거 생성**: "점수 breakdown과 대표 장소를 사용자 언어로 바꿔 '왜 이 도시인지'를 함께 냅니다."
- **PoC 정직성**: 피해야 할 말 "이 공식이 가장 정확한 추천을 보장합니다." → 권장 "각 보정 항목은 이후 ablation으로 효과를 분리 검증할 수 있게 설계했습니다." (현재 Ranker는 검증 완료된 최적 공식이 아님)

### [서론 회수 멘트]

- "서론에서 약속한 '발견' — 못 찾던 소도시를 '여행지로 성립하는 도시'로 골라주는 단계가 여기입니다." (① 발견)

### [원본 B4 → 이 두 슬라이드 변경 매핑]

| 항목 | 원본 B4 후보 | 이 슬라이드 (= 확정안) |
| --- | --- | --- |
| 딥다이브 구성 | ① Explainable RAG 하이브리드 ② Festival Verifier ③ 멀티스텝 통합 중 1~2개 | **① 하이브리드 RAG(B4-1) + ② Ranker(B4-2)** 로 확정 |
| Festival Verifier | 독립 딥다이브 후보 | **§3-대안**으로 이동 (시간·청중 따라 교체용) |
| Ranker | 원본 B3 노드 안에만 존재 | **독립 딥다이브로 승격**, 실측 데이터로 문제 증명 |

## 3-대안. (백업 슬라이드) 검증 & Fallback 레일

> RAG/Ranker 대신, 또는 시간이 남을 때 끼워 넣을 수 있는 대안 딥다이브. 기존 안을 폐기하지 않고 보존한다.

### [화면]

**제목 카피**

```text
부족하거나 불확실하면, 만들어내지 않는다
```

**서브 카피**

```text
Tool이 부족·불확실·실패를 감지하고, Agent가 완화·재검색·안내로 전환한다
```

**도식 ① Fallback 레일**

```text
Tool 결과
  ├─ 충분함        → Ranker / Planner 진행
  ├─ 후보 부족     → backup theme · 조건 완화
  ├─ 축제 불확실   → 일정 배치 대신 참고 안내
  └─ 검증 실패     → 재시도 · fallback 안내
```

**도식 ② 축제 검증 레일 (하단 callout)**

```text
축제 후보 → 개최일·출처 확인 → 확정만 일정 직접 배치
```

| 상태 | 의미 | 일정 반영 |
| --- | --- | --- |
| 확정(confirmed) | 해당 연도 개최일·출처 확인 | 일정에 직접 배치 |
| 잠정(tentative) | 정보는 있으나 날짜·출처 불확실 | 참고 안내로만 |
| 제외(rejected) | 조건 불일치·신뢰 불가 | 일정에서 제외 |

### [노트]

- "검색 결과는 항상 충분하지 않습니다. 특히 소도시는 데이터가 적어, 희소 테마나 축제 조건이 들어오면 후보가 부족할 수 있습니다. 이때 생성 모델이 빈칸을 상상하지 않도록 Agent가 fallback 경로를 선택합니다."
- "축제는 매년 날짜가 바뀝니다. 그래서 일반 관광지처럼 바로 일정에 넣지 않고, 해당 연도 날짜가 confirmed일 때만 직접 배치하고, 불확실하면 참고 안내로 분리합니다."
- 표현 주의: "항상 환각을 막는다"(X) → "환각 위험을 줄이기 위해 근거 부족 시 fallback한다"(O).

### [서론 회수 멘트]

- "서론의 '낯선 곳은 못 믿어 유명지로 돌아간다' — 그 불신을 근거·검증 뱃지로 받는 지점입니다." (③ 근거)

## 7. 미결 검토 메모 — B4-1 RAG의 그래프DB(Neptune) 정당성 (결정 보류)

> 상태: **결정 보류.** 슬라이드는 아직 고치지 않음. 개발 진행 상황을 보고 확정한다.
> 쟁점: B4-1에서 그래프DB(Neptune)를 딥다이브 근거로 들고 가는 것이 **오버엔지니어링**으로 보이지 않는가.

### 논의 요약

**그래프DB가 정당화되는 일반 기준** — 세 조건이 동시에 충족될 때만 값을 한다.

1. 쿼리가 관계를 **3홉 이상 가변 길이로 순회**한다 (A→B→C→D…).
2. 그걸 **대화형 지연** 안에 풀어야 한다.
3. 그 **관계 자체가 쿼리의 핵심**이다 (단순 필터가 아니라).

전형 사례: 소셜 그래프, 사기 탐지 링, 지식 그래프.

**Lovv의 실제 쿼리 점검** (`recommendation_flow.md` 기준)

- 추천 로직의 거의 전부가 관계형 집계다: 거리 필터, `city_id+theme` 집계, 테마 특화도(평균 대비), 도시 규모 구간화, 콘텐츠 균형.
- 의미 매칭은 vector가 별도로 담당.
- → 현재 스코프엔 **다단계 관계 순회 쿼리가 사실상 없다.** RDS가 "충분"을 넘어 **정답**이며, 지금 그래프DB 도입은 오히려 오버엔지니어링.

**그래프가 진짜 필요해지는 두 기능 (그리고 현재 상태)**

| 후보 기능 | 그래프 필요성 | 현재 상태 |
| --- | --- | --- |
| 다도시 일정 (인접 도시 묶어 2도시 코스) | 도시–인접–도시 순회 → 그래프 적합 | **로드맵에 없음** (사용자 확인) |
| 관계 기반 발견 (축제→테마→다른 도시, 유사 도시 등) | 대부분 **2홉** → RDS JOIN으로 처리 가능. 3홉+ 가변 순회라야 그래프가 값 | 현 스코프는 2홉 수준. 사용자 행동 기반(co-visitation)은 PoC에 데이터 없음 |

**잠정 결론 (보류)**

- 다도시가 로드맵에 없고 관계 기반 발견도 2홉 수준이면, **현재 그래프DB는 정당화되지 않는다.** 발표에서 핵심 근거로 들면 "오버엔지니어링" 지적을 방어하기 어렵다.
- 유력 대안: B4-1을 **실제 구현된 "vector(의미)+원장(RDS, 사실) 분리 = Explainable"** 축으로 앵커링하고, 그래프DB는 "사용자 행동 데이터 축적·다도시 확장 시 관계 순회용으로 검토"라는 **미래 각주 한 줄**로만 남긴다.
- 단, 이는 **확정이 아님.** 개발 진행 중 3홉+ 관계 순회가 실제로 필요한 기능이 생기면 그래프DB 근거가 살아난다.

### 확정 시 체크 포인트

- [ ] 개발 단계에서 3홉 이상 관계 순회가 필요한 실제 쿼리/기능이 생겼는가?
- [ ] 다도시 일정 또는 사용자 행동 기반 발견이 로드맵에 다시 들어왔는가?
- [ ] 둘 다 아니면 → B4-1을 vector+원장 grounding으로 재구성, 그래프DB는 각주로 강등.
- [ ] 하나라도 예면 → 그 **구체 기능**에 묶어 그래프DB를 정당화 (관계가 좋아서가 아니라 그 기능에 인접/관계 순회가 필요해서).

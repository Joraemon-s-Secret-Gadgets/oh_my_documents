# 중간발표 vs 최종발표 구성 가이드 (Lovv)

> 용도: 중간발표와 최종발표를 분리해 준비할 때의 역할 분담·구성 전략 정리
> 대상: 부트캠프 PoC 프로젝트(Lovv)
> 문서 성격: 보조 Markdown
> 대표 문서: `99_pptx.md`
> 핵심 원칙 한 줄: **중간 = "될 것 같다"를 설득(계획·근거·가능성), 최종 = "됐다"를 증명(결과·데모·증거)**

---

## 1. 중간발표와 최종발표의 본질적 차이

중간발표는 아직 결과가 없으므로 **방향성과 실현 가능성**을 파는 자리다. 문제가 진짜인지, 접근이 타당한지, 팀이 끝까지 갈 수 있는지를 보여준다.

최종발표는 **만든 결과와 그 증거**를 파는 자리다. 데모, 실제 동작, 검증 데이터로 "주장한 것을 해냈다"를 입증한다.

| 구분 | 중간발표 | 최종발표 |
| --- | --- | --- |
| 목적 | 방향·계획·실현 가능성 설득 | 결과·성과·증거 입증 |
| 무게중심 | 문제 정의, 시장 근거, 설계/아키텍처 | 구현 결과, 라이브 데모, 검증 결과 |
| 핵심 질문 | "이게 왜 필요하고, 될 것 같은가?" | "정말 됐고, 얼마나 잘 되는가?" |
| 기술 깊이 | 아키텍처·흐름도 수준(개념) | 구현 상세 1~2개 깊게 + 데모 |
| 데이터 | 시장/문제 근거 데이터 | 실제 동작·성능·테스트 결과 |
| 마무리 | 남은 일정·계획 | 회고(배운 점)·향후 방향 |

---

## 2. 슬라이드 구성 비교

### 2.1 중간발표 권장 구성 (현재 Lovv 5장 초안과 매핑)

1. 프로젝트 소개 — 무엇을 만드는가 (Lovv 초안 S1)
2. 문제: 오버투어리즘 현상 (S2)
3. 문제의 영향·비즈니스 임팩트 (S3)
4. 시장 근거 — 외부 인용/데이터 (S4, 라쿠텐)
5. 해결: 시스템 흐름/아키텍처 (S5)
   - (+선택) UI 목업/와이어프레임 — "이렇게 보일 것이다" (최종 결과 아님, 의도만)
   - (+선택) 남은 일정·역할 분담 — 끝까지 갈 수 있음을 보여주는 장

> 중간발표는 아직 구현물이 없어도 된다. UI는 **목업 수준**, 기술은 **아키텍처 다이어그램 수준**이면 충분하다.
>
> 단, Lovv는 중간발표를 **PoC 단계**로 잡으므로(§6) 한 단계 더 나아간다. 즉 유저 플로우·시스템 흐름·에이전트 플로우·핵심 구현 딥다이브·간단한 데모 영상까지 **큰 흐름 위주로** 보여준다. 다만 세부 기능을 모두 설명하기보다 "어떤 문제를 정의했고, 어떤 핵심 흐름을 구현했으며, 앞으로 무엇을 발전시킬지"를 중심에 둔다. 구체 슬라이드 구성은 §7 참고.

### 2.2 최종발표 권장 구성

1. (짧게) 문제·목표 재요약 — 중간발표 내용을 1~2장으로 압축 (청중 환기)
2. 솔루션 전체 개요 — 완성된 모습 한눈에
3. **라이브 데모 또는 데모 영상** — 가장 강력한 슬라이드(자리). 실제 동작이 핵심
4. 핵심 구현 1~2개 깊게 — 차별점(Explainable RAG, 멀티스텝 Agent) 중심. 소스코드 나열 X, 그림으로 설명
5. 검증/결과 — 추천 품질, 응답 시간(목표 8초 이내 등), 사용자 테스트. 양보다 질
6. 회고 + 향후 방향 — 배운 점, Production 확장 계획(P2~P4)

## 3. 분리 준비의 실전 전략

### 3.1 슬라이드 재사용(중간 → 최종 진화)
중간발표 자료를 버리지 말고 진화시킨다. 같은 마스터/디자인 시스템을 쓰고, 슬라이드를 "교체"가 아니라 "심화"한다.

| 중간발표 슬라이드 | 최종발표에서의 변화 |
| --- | --- |
| 문제·시장 근거(S2~S4) | 1~2장으로 **압축**해 도입부로 재사용 |
| 아키텍처 흐름도(S5) | 개념도 → **실제 구현 기준**으로 갱신, 핵심 노드 확대 |
| UI 목업 | **실제 화면 스크린샷/데모**로 교체 |
| 남은 일정 | **회고 + 향후 방향**으로 교체 |

### 3.2 디자인 일관성
- 중간·최종 공통 마스터 템플릿(색상·폰트·로고 위치) 하나로 통일 → 같은 프로젝트의 연속임을 시각적으로 전달.
- 표지에 단계 표시("중간발표 / 최종발표")와 날짜만 다르게.

### 3.3 스토리라인 일관성
중간에서 "이렇게 하겠다"라고 약속한 것을, 최종에서 "약속대로 했다 + 결과는 이렇다"로 받아치면 설득력이 가장 크다. 중간발표의 목표/성공 기준을 최종발표 검증 슬라이드에서 그대로 대응시킨다.

## 4. 공통 발표 원칙 (둘 다 적용)

- 구조는 항상 서론–본론–결론(시작·전개·마무리)을 갖춘다.
- 슬라이드당 핵심 메시지 1개. 글머리표는 최대 5개, 한 항목 5단어 내외(AWS식 미니멀과 동일 방향).
- PPT보다 **발표 원고(스크립트)를 먼저** 쓰고 슬라이드를 만든다.
- 코드 나열 대신 **그림 1장**으로 솔루션을 설명한다.
- 발표 전 **시간 측정 리허설** 필수. 중간발표는 **발표 20~25분 + Q&A** 기준, 슬라이드는 **10~20장 내외**로 구성한다(최종도 실제 배정 시간에 맞춰 조정).

---

## 5. Lovv 프로젝트에 적용하면

- **중간발표(지금)**: 현재 5장 초안 그대로 + (선택) UI 목업 1장, 남은 일정 1장. 핵심은 "오버투어리즘은 진짜 문제이고(시장 근거: 라쿠텐), Explainable RAG로 풀 수 있다(아키텍처)"를 설득하는 것.
- **최종발표(이후)**: 위 5장을 도입부로 압축하고, **라이브 데모 + P1 구현 결과 + 추천 품질/응답속도 검증 + 회고**를 중심에 둔다. 일정상 PoC 범위(P1 구현, P2~P3 설계)와 성공 기준을 결과로 증명하는 구성이 자연스럽다.

---

## 6. PoC = 중간발표 / Production = 최종발표 프레이밍 (Lovv 실제 적용)

Lovv는 중간발표를 **PoC 단계**, 최종발표를 **Production 단계**로 계획한다. 이 경우 표준 "중간=계획 / 최종=결과" 구도가 한 단계 올라가, **둘 다 결과물을 보여주되 검증 범위가 달라지는** 구조가 된다.

### 6.1 각 발표가 증명할 것

- **중간발표 = PoC → "핵심 가설이 실제로 작동한다"**
  단순 계획 발표가 아니라 이미 만든 PoC(P1 기능)를 데모할 수 있다. 메시지는 "오버투어리즘을 Explainable RAG로 푸는 것이 좁은 범위에서 실제 동작한다." 범위를 솔직하게 한정(P1 구현, P2~P3 설계, P5 제외)하고 그 안에서 "됐다"를 보여주는 것이 핵심.

- **최종발표 = Production → "실제 서비스로 확장된다/했다"**
  PoC 검증을 전제로 깔고 확장성·운영·안정성·비즈니스를 증명한다. P2~P4 구현, 역할 기반 인증·운영/검토 기능, 데이터 갱신 파이프라인, 성능(추천 8초·RAG 1초 목표), 지자체 파트너 대시보드 같은 비즈니스 확장이 중심.

### 6.2 비교표

| 구분 | 중간발표 (PoC) | 최종발표 (Production) |
| --- | --- | --- |
| 증명할 것 | 핵심 가설 작동(좁게) | 실서비스 확장성(넓게) |
| 데모 | P1 핵심 기능 데모 | 운영기능 포함 전체 서비스 |
| 데이터 | 시드/정적 데이터 기반 | 관리형 DB·갱신 파이프라인 |
| 검증 지표 | "동작하는가" | 성능·품질·안정성 수치 |
| 비즈니스 | 문제·시장 근거(라쿠텐) | 수익/파트너 확장 모델 |
| 마무리 | Production 전환 계획 | 실증 결과 + 사업화 방향 |

### 6.3 5장 초안에 미치는 영향

PoC/Production 프레이밍에서는 5장 초안 마지막의 "남은 일정" 자리를 **"PoC 데모 + Production 계획"**으로 바꾸는 것이 더 맞다. 중간발표 끝에서 "PoC로 여기까지 됐고, Production에서는 이렇게 확장한다"를 보여줘야 PoC → Production 흐름이 자연스럽게 이어진다. (필요 시 5장을 6장으로 확장)

## 7. 중간발표 슬라이드 구성 (상세)

> 전제: 발표 20~25분 / 슬라이드 10~20장 / **청중은 프로젝트를 처음 듣는다.** 따라서 처음부터 기술을 깊게 설명하지 말고, **문제 상황과 서비스의 필요성**을 쉽게 이해시키는 흐름으로 잡는다. 전체 구조는 **기-승-전-결**을 갖춘다.

### 7.1 서론 (도입 — 흥미와 공감)

청중을 기술이 아니라 **문제**로 끌어들이는 구간이다.

1. **프로젝트 소개** — 무엇을 만드는지 한 장으로 간단히. 자세한 설명은 뒤로 미루고 한 문장 인상만 남긴다.
2. **오버투어리즘이란?** — 현상을 흥미롭게 소개해 관심을 끈다. 숫자·사진·짧은 사례로 "이게 남 얘기가 아니다"를 느끼게 한다.
3. **왜 문제인가** — 오버투어리즘의 비용을 4개 축으로, 각 축에 객관 근거(한국·일본 각 1개)를 붙여 제시한다. 축: **주민 생활 · 환경 · 도시 기능 · 지역 문화**. 화면은 축당 대표 수치 1개(상세는 노트), 하단에 "비용은 유명지에 쏠린다 → 수요 분산이 답"이라는 전환 문구로 시장 근거·본론(대체 소도시)에 연결한다. 비즈니스 함의는 시장 근거 슬라이드가 받는다. (상세: `01_midterm_presentation/s3_problem_axes_revision.md`)
   - 기존 4·5페이지(문제 상황 ↔ 프로젝트 동기) 통합 구성은 현재 파일의 기존 방침을 그대로 따른다.
4. **서론 마무리 — 우리 프로젝트의 의미 + 해결 방식 + 핵심 컨셉 3가지**
   - 앞에서 제시한 문제를 **어떤 방식으로** 해결하는지 보여준다.
   - 서비스의 강점이자 핵심 컨셉을 **3가지**로 정리한다.
   - 이때 3가지 컨셉이 **앞서 제시한 문제와 1:1로 연결**되도록 배치한다(문제 → 컨셉이 해결책으로 이어지는 다리 역할).

### 7.2 본론 (전개 — 어떻게 만들었는가)

서론에서 약속한 해결책을 **사용자 경험 → 시스템 → 에이전트 → 구현 근거** 순으로 좁혀 들어간다.

0. **(본론 진입 전) 시스템 아키텍처 개요** — 전체 구조를 **간략히** 한 장으로 보여주는 용도. 여기서는 큰 그림만 잡고, 복잡한 설명은 하지 않는다. 실제 디테일은 아래 에이전트 흐름·구현 포인트에서 풀어낸다.
1. **유저 플로우** — 사용자가 우리 서비스를 어떻게 활용하는지(여정) 보여준다.
2. **시스템 흐름** — 그 사용자 경험을 구현하기 위해 시스템이 어떻게 동작하는지 설명한다.
3. **에이전트 플로우** — 각 에이전트가 어떤 역할을 하며, 전체 흐름 안에서 어떻게 연결되는지 보여준다. (아키텍처 개요보다 **여기를 더 자세히**)
4. **구현 딥다이브** — 각 주요 기능·에이전트를 **왜 그렇게 설계했는지**, 어떤 문제를 풀기 위해 만들었는지, 어떤 **개선 과정**을 거쳤는지 설명한다. 소스코드 나열이 아니라 의도와 개선 스토리 중심.
5. **활용 데이터** — 어떤 데이터를 **어디서** 가져왔고 **어떤 목적**으로 썼는지 **표 한 장**으로 정리한다.
6. **확장 방향** — 현재 PoC에서 구현한 범위와 앞으로 확장할 방향을 설명한다.
7. **(본론 마무리) 데모 영상** — 간단한 데모 영상으로 실제 서비스가 어떻게 동작하는지 보여주며 본론을 닫는다.

### 7.3 결론 (마무리)

- **확장 방향 + 남은 태스크** 정리. PoC에서 여기까지 됐고, 다음 단계에서 무엇을 더 발전시킬지로 마무리한다(§6의 PoC → Production 흐름과 연결).

### 7.4 작성 시 주의 (PoC 성격 유지)

- 중간발표는 완성된 최종 서비스가 아니라 **PoC**다. 세부 기능을 전부 설명하려 하지 말고, **① 어떤 문제를 정의했는가 ② 어떤 핵심 흐름을 구현했는가 ③ 앞으로 무엇을 발전시킬 것인가**를 **큰 흐름 위주**로 보여준다.
- 슬라이드당 핵심 메시지 1개 원칙(§4)을 지키면 10~20장 안에 위 구성이 들어간다.

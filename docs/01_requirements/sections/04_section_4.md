# 4. 기능 요구사항

본 섹션은 로브 서비스가 제공해야 하는 사용자 기능과 시스템 동작을 정의한다.
각 요구사항은 개발, 설계, 테스트에서 공통 기준으로 사용할 수 있도록 식별자, 적용 영역, 요구사항 내용, 비고로 구분한다.

기능 요구사항 표에서는 우선순위 컬럼을 제외한다.
실제 개발 우선순위는 별도 백로그에서 P1~P5 구간으로 산정한다.

| FR-STORE-003 | Production 회원가입 기반 저장 | Production 단계에서 사용자는 회원가입 후 자신의 여행 일정, 선호, 피드백, 북마크, 방문 기록, 방문 소도시 별점 데이터를 서버에 저장하고 마이페이지에서 조회할 수 있다. | Production |
| FR-ONBOARD-001 | 온보딩 질문 | 사용자는 서비스 첫 진입 시 선호 여행 스타일을 파악하기 위한 온보딩 질문에 응답해야 한다. 온보딩은 건너뛸 수 없으며 이후 마이페이지에서 재응답할 수 있어야 한다. | 필수 |

## 4.4 RAG·챗봇·에이전트 기능

| ID | 서비스/영역 | 요구사항 내용 | 비고 |
| --- | --- | --- | --- |
| FR-RAG-001 | RAG 검색 | 서비스는 실제 수집 데이터 또는 벡터화된 지식베이스를 근거로 추천과 답변을 생성해야 한다. | 환각 방지 |
| FR-RAG-002 | 멀티스텝 에이전트 | 추천 파이프라인은 조건 분류, 데이터 검색, 후보 선정, 일정 구성, 검증 단계를 순차적으로 처리해야 한다. | 다단계 처리 |

## 4.6 서비스별 세부 기능

### 우선순위 및 단계별 추진 범위

PoC 단계는 P1 구현을 핵심 범위로 한다.
P2는 PoC에서 설계까지 완료하고, 일정 여유가 있을 경우 일부 구현한다.
P3는 PoC에서 설계 산출물까지만 정리한다.
P4는 Production 단계에서 구현한다.
P5는 이번 프로젝트 대상에서 제외하고 향후 검토한다.
P5 항목은 우선순위 및 단계별 추진 범위에만 기록하고, 아래 상세 기능 요구사항 표에서는 제외한다.
P2 이상 기능은 핵심 AI 추천 경험(P1)이 완성된 이후 진입한다. P1 단계에서 recommendation flow, personalization, ranking, explainability가 충분히 검증되지 않으면 P2 구현을 시작하지 않는다.

## 4.8 추천 파이프라인 명세

각 단계별 입력·출력·판단 기준을 정의한다.

| 단계 | 입력 | 판단 기준 | 출력 |
| --- | --- | --- | --- |
| 1. 조건 수집 | 사용자 자연어, 국가, 시기(월), 온보딩 선호 프로필, 북마크, 방문 기록, 방문 소도시 별점, 추천 피드백, `여행 일정 추천 받기` 실행 이벤트 | 필수 조건 충족 여부, 부족 조건 존재 여부 | 구조화된 여행 조건 객체 또는 추가 질문 |
| 2. 테마·진입점 분류 | 구조화된 조건 | LLM 분류 + 룰 기반 fallback | 테마 태그 + 진입점 유형 |
| 3. 후보 검색 | 테마, 시기(월), 국가 | RAG 유사도 점수, 계절 적합도 | 후보 소도시 목록 |
| 4. 후보 필터링·순위 | 후보 목록 + 혼잡도·월별 기상 경향·북마크·방문 기록·방문 소도시 별점·개인화 피드백 + 직전 추천 결과 | 가중치 기반 스코어링, 사용자별 소도시 적합도 점수, PoC 경량 ML/유사도 랭킹 실험, `추천 다시 받기` 요청 시 직전 후보 제외 또는 감점 | 상위 1곳 선정 |
| 5. 일정 구성 | 선정 소도시 1곳 + 일정 유형 + 축제 포함 여부 | 일별 방문지 수·동선 흐름·이동 효율 | 1곳 일별 세부 일정 카드 |
| 6. Explainability 생성 | 일정 + 사용자 조건 매칭 근거 | FR-REC-003 4요소 기준 | 추천 이유 텍스트 |
| 7. 외부 정보 보강 | 일정 카드 | 지도·날씨 표시·숙소 링크 | 최종 추천 결과 |

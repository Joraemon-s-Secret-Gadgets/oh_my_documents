# 8. 제약사항 및 가정

## 8.1 기술적 제약사항

**일본 데이터 수집 실패:**

일본 소도시·축제·방문 통계·월별 날씨 경향 데이터의 수집 경로 확보와 로컬 검증에 실패하여, 일본 추천 트랙은 후보·축제·방문 통계·날씨 검증 근거가 부족하다.

이에 따라 PoC/P1은 한국 강원·경북 데이터 기반 추천으로 고정하고, 한국 전국 확장은 P2로 둔다. 일본 데이터 기반 추천과 일본 현지 데이터/API 연동(Yahoo Japan·じゃらん 등 현지 플랫폼, 일본어 검색 키워드, 애니메이션 성지순례)은 최하위 우선순위인 P5 재검토 범위로 보류한다.

일본 데이터는 수집 경로와 출처 정합성 기준, 로컬 검증 산출물이 확보되기 전까지 추천 근거로 사용하지 않는다. 수집 경로가 복구되면 한국 트랙과 동일한 나라별 독립 트랙 기준으로 재승격을 검토한다.

**Yahoo Japan Maps SDK 지원 중단 (일본 트랙 P5 재검토 시 적용):**

Yahoo Japan Maps JavaScript SDK(Yjans)가 지원 중단되어, 향후 일본 트랙을 재개할 경우 일본 소도시 지도는 Google Maps API로 대체 제공한다.

**Yahoo Japan REST API CORS:**

Local Search 등 REST API는 CORS 제한이 있다.

향후 백엔드 프록시 서버를 통해 연동한다.

App ID는 향후 사용을 위해 입력 항목만 준비한다.

**Kakao REST API 도메인 제약:**

로컬 파일 환경(

`file://`

)에서는 CORS 정책으로 Kakao Local REST API 호출이 제한될 수 있다.

등록된 도메인 환경에서 정상 동작한다.

**Google Places API 과금:**

Places API는 무료 한도 초과 시 과금이 발생한다.

서비스 규모에 따른 비용 산정이 필요하다.

## 8.2 가정사항

- 서비스 초기에는 단일 HTML 파일 기반 데모로 운영하며, API 키는 사용자가 직접 입력한다.
- 초기 데이터는 정적 파일 또는 외부 API 기반으로 관리하며, 저장 구조는 별도 데이터 설계 단계에서 결정한다. 월별 날씨 경향은 Wikipedia에서 추출·전처리한 정적 데이터로 관리한다.
- 추천 엔진은 RAG(검색증강) + 멀티스텝 에이전트 구조로 구현하며, LLM API를 활용한다.
- 자체 모델 학습(sLLM)은 본 범위에서 제외하며, 기성 LLM API + RAG + 에이전트 설계로 품질을 확보한다.
- 축제·계절 정보는 월·계절 단위로만 관리하여 연도별 갱신 부담과 환각 위험을 최소화한다.
- 이동 동선은 방문 순서 제안 수준으로 한정하며, 실시간 교통·항공편 최적화는 제외한다.
- 추천 후보 스코어링과 대체 일정 알림 팝업 분기에는 Wikipedia 기반 월별 날씨 경향 데이터를 사용한다. 월별 경향 외 날씨 데이터는 호출하거나 화면에 표시하지 않는다.
- PoC 단계의 개인화 데이터는 브라우저 로컬 스토리지 중심으로 관리하며, Production 단계에서는 로그인 사용자 계정 기반으로 누적·조회·삭제할 수 있도록 확장한다.
- 위치 반경 추천과 여행 중 즉흥 일정 조정은 PoC 상세 범위에서 제외하고 향후 확장 후보로 관리한다.
- 관리자 권한의 최종 판단 기준은 로브 서비스 DB의 활성 역할·지역 할당이며, Cognito Group 등 외부 인증 그룹은 보조 신호로만 사용한다. 관리자 프론트의 Mock SSO·역할 선택은 개발 모드 전용이며 운영 빌드에서는 사용하지 않는다.
- 데이터 제안 승인과 월간 여행지 게시는 분리해 운영하며, 공익(B2G) 성과 리포트와 제휴(B2B) 성과 지표는 같은 사용자 여정 안에서도 분리해 기록·표시한다.
- 일본 데이터 기반 추천, 일본 현지 데이터/API 연동, 별 관측, 치유·웰니스, 예산 역산 추천은 P5 향후 검토 후보이며, 대표 문서에는 데이터 후보와 검토 조건만 유지한다. 상세 기능 요구사항으로 승격하려면 별도 우선순위 재검토와 데이터 확보 기준을 먼저 확정한다. 특히 일본 항목은 일본 데이터 수집 경로 복구와 로컬 검증 산출물 확보를 재승격 전제로 한다.

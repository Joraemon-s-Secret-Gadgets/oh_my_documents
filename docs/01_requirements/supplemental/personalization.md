# 개인화 요구사항 정리

> 문서 성격: 보조 Markdown
> 대표 문서: `../01_requirements.md`

## 1. 목적

본 문서는 로브(Lovv) 서비스의 개인화 기능을 PoC와 Production 단계로 구분해 정리한다.
개인화 기능은 사용자의 저장 행동, 실제 방문 기록, 방문 소도시 별점을 기반으로 추천 품질을 개선하고, Production 단계에서 계정 기반 개인화 데이터 누적, 개인화 랭킹 운영, 위치 기반 및 여행 중 실시간 일정 조정 기능으로 확장하는 것을 목표로 한다.

## 2. PoC 개인화 범위

PoC 단계에서는 사용자의 명시적 행동 데이터를 수집하고, 이를 추천 결과 개선의 기초 데이터로 활용한다.
복잡한 모델 학습보다는 개인화에 필요한 입력 구조와 피드백 흐름을 검증하는 데 집중한다.

### 2.1 사용자 행동 데이터 수집

| 항목 | 설명 | 활용 방향 |
| --- | --- | --- |
| 소도시 북마크 | 사용자가 관심 있는 소도시를 저장한다. | 선호 지역, 테마, 계절 조건 추론 |
| 대도시 북마크 | 사용자가 관심 있거나 다시 확인하고 싶은 대도시 자체를 저장한다. | 대도시 기준 주변 소도시 탐색, 비교에 활용 |
| 추천 일정 좋아요/싫어요 | 추천받은 일정에 대해 긍정 또는 부정 피드백을 남긴다. | 다음 추천의 가중치 조정 |
| 실제 방문 기록 | 사용자가 실제로 다녀온 소도시 또는 여행 루트를 기록한다. | cold start 완화 및 추천 신뢰도 개선 |
| 방문 소도시 별점 | 사용자가 실제로 방문한 소도시에 한해 별점을 부여한다. | 개인별 만족도 및 목적지 품질 신호로 활용 |

### 2.2 Cold Start 보완

사용자 개인 데이터가 충분하지 않은 초기 상태에서는 외부 레퍼런스 기반의 여행 루트를 활용한다.
예시는 다음과 같다.

- 여행 유튜버의 소도시 여행 루트
- 관광공사 또는 지자체가 제공하는 추천 여행 루트
- 기존 인기 일정 또는 검증된 테마별 여행 코스

위 데이터는 신규 사용자의 초기 추천 후보를 구성하거나, 사용자의 첫 북마크·피드백이 쌓이기 전 기본 추천 품질을 보완하는 데 활용한다.

### 2.3 소도시 랭킹 및 PoC 머신러닝

PoC 단계에서는 소도시 랭킹의 기본 구조를 정의한다.
랭킹은 사용자의 개인 경험과 다른 사용자의 집단 행동 데이터를 함께 고려하는 방향으로 설계하고, 제한된 범위의 머신러닝 기반 랭킹 실험을 수행한다.

| 랭킹 기준 | 설명 |
| --- | --- |
| 내가 갔던 곳 | 사용자의 실제 방문 기록, 방문 소도시 별점, 좋아요/싫어요를 기준으로 개인 선호를 반영한다. |
| 남이 갔던 곳 | 다른 사용자의 방문 기록, 저장, 평가 데이터를 기반으로 목적지의 일반 선호도를 반영한다. |
| PoC 머신러닝 랭킹 | 북마크, 방문 기록, 방문 소도시 별점, 좋아요/싫어요, 목적지 태그를 특징값으로 사용해 간단한 랭킹 모델을 실험한다. |

PoC에서 적용 가능한 머신러닝 범위는 다음과 같다.

- 사용자 행동 데이터와 목적지 메타데이터를 기반으로 사용자별 소도시 적합도 점수를 계산한다.
- 실제 학습 데이터가 부족한 초기에는 관광공사 추천 루트, 여행 유튜버 루트, 팀 내부 평가 데이터를 seed 데이터로 사용한다.
- 모델은 복잡한 딥러닝보다 로지스틱 회귀, 의사결정나무, LightGBM, k-NN, 콘텐츠 기반 유사도 등 설명 가능한 경량 모델을 우선 검토한다.
- PoC 결과는 추천 품질 검증과 feature 중요도 확인에 사용한다.

## 3. Production 개인화 범위

Production 단계에서는 PoC에서 검증한 행동 데이터 기반 개인화를 사용자 계정 기반으로 누적하고, 실제 추천 순위에 반영하는 1차 운영으로 확장한다.
이후 누적 행동 데이터를 기반으로 개인화 랭킹 모델을 고도화하고, 사용자의 현재 상황과 위치를 반영한 추천으로 확장한다.

### 3.1 사용자 계정 기반 개인화 데이터 누적

Production 단계에서는 사용자가 로그인한 계정을 기준으로 북마크, 방문 기록, 방문 소도시 별점, 좋아요/싫어요, 저장 일정 데이터를 누적하고 마이페이지에서 관리할 수 있어야 한다.

| 데이터 | 설명 |
| --- | --- |
| 북마크 | 소도시·대도시 저장 이력 |
| 방문 기록 | 실제 방문한 소도시와 여행 루트 |
| 방문 소도시 별점 | 실제 방문한 소도시에 대한 사용자 평가 |
| 추천 피드백 | 추천 일정 또는 소도시에 대한 좋아요/싫어요 |
| 저장 일정 | 추천 결과로 생성된 여행 일정 저장 이력 |

### 3.2 개인화 랭킹 운영

Production 1차 운영에서는 PoC에서 검증한 규칙 기반 점수 또는 사용자별 소도시 적합도 점수를 실제 추천 순위에 반영한다.
복잡한 머신러닝 운영보다 안정적인 점수식 반영을 우선한다.

Production 고도화 단계에서는 누적 행동 데이터를 기반으로 모델 재학습, A/B 테스트, 협업 필터링, 세션 기반 추천, 모델 모니터링을 검토한다.

### 3.3 현재 위치 기반 반경 추천

서비스는 사용자의 현재 위치를 기준으로 추천받고 싶은 탐색 반경을 입력받을 수 있어야 한다.
사용자는 현재 위치에서 몇 km 이내의 소도시를 추천받을지 직접 설정하며, 시스템은 해당 반경 안에서 이동 가능한 목적지를 우선 추천한다.

| 입력 항목 | 설명 |
| --- | --- |
| 현재 위치 | 사용자의 현재 좌표 또는 사용자가 직접 입력한 기준 위치 |
| 추천 반경 | 사용자가 설정한 거리 범위. 예: 10km, 30km, 50km, 100km |
| 이동 가능 시간 | 선택 입력 항목. 거리뿐 아니라 실제 이동 시간을 고려할 때 사용한다. |

위치 기반 추천은 국가별 데이터 범위를 유지하되, 핵심 목적은 국가 자동 선택이 아니라 사용자가 지정한 현재 위치와 반경에 맞는 소도시 후보를 좁히는 것이다.
국가 간 이동을 전제로 한 추천은 별도 여행 계획 흐름에서 다룬다.

### 3.4 즉흥적 일정 변동 대응

Production 단계에서는 사용자가 현재 여행 중인 상황에서 예산, 남은 일정, 이동 가능 시간에 따라 일정을 즉흥적으로 조정할 수 있어야 한다.

| 변동 조건 | 설명 |
| --- | --- |
| 금액 변화 | 남은 예산 또는 추가 지출 가능 금액에 맞춰 일정, 장소, 식사, 숙박 수준을 조정한다. 전체 예산을 기준으로 여행지를 역산 추천하는 기능과 구분한다. |
| 일정 변화 | 남은 여행 시간, 숙박일 수, 당일 이동 가능 시간에 맞춰 코스를 재구성한다. |
| 현재 여행 상태 | 사용자가 이미 여행 중인 경우 현재 위치와 남은 일정을 기준으로 다음 방문지를 추천한다. |

## 4. 단계별 요구사항 요약

| 단계 | 요구사항 | 비고 |
| --- | --- | --- |
| PoC | 북마크, 좋아요/싫어요, 실제 방문 기록, 방문 소도시 별점 수집 | 개인화 입력 데이터 확보 |
| PoC | 유튜버·관광공사 여행 루트 기반 cold start 보완 | 초기 추천 품질 확보 |
| PoC | 내가 갔던 곳과 남이 갔던 곳을 구분한 소도시 랭킹 및 경량 머신러닝 실험 | 북마크, 방문 기록, 방문 소도시 별점, 피드백 기반 |
| Production | 사용자 계정 기반 개인화 데이터 누적 | 북마크, 방문 기록, 방문 소도시 별점, 피드백, 저장 일정 |
| Production | 개인화 랭킹 1차 운영 | 규칙/점수 기반 랭킹을 실제 추천 순위에 반영 |
| Production | 개인화 랭킹 모델 고도화 | 모델 재학습, A/B 테스트, 협업 필터링 등 |
| Production | 현재 위치와 사용자 지정 반경 기반 추천 제공 | 몇 km 이내 추천받을지 사용자가 설정 |
| Production | 현재 여행 중 예산·일정 변화에 따른 즉흥 일정 조정 | 실시간 개인화 확장 |

## 5. 메인 요구사항 문서 반영 후보

- `4.2 추천·탐색 기능`에 개인화 피드백, 실제 방문 기록, 방문 소도시 별점 요구사항 추가
- `7.1 필요 데이터 항목`에 북마크, 방문 기록, 방문 소도시 별점, 위치 기반 추천 데이터 추가
- `8.2 가정사항`에 PoC 단계의 개인화 데이터 수집 범위와 Production 위치 기반 추천 확장 조건 추가
- 향후 백로그에서 PoC 개인화와 Production 실시간 개인화를 분리해 우선순위 산정

## 1. 문서 목적

하나투어의 여행상품 기획 에이전트 사례와 Lovv 설계를 비교해 다음 세 가지를 기록한다.

1. Lovv가 상대적으로 앞서 있는 설계 영역
2. 추후 개발 과정에서 반영할 후보와 우선순위
3. 사례를 따라가지 않기로 한 항목과 그 이유

본 문서는 참고 자료이며 대표 기술 명세의 결정을 바꾸지 않는다. 반영이 확정된 항목은 각 정본 문서(`../sections/*.md`, Agent 명세, 테스트 계획)에 별도로 반영한다.

해석 한계: 비교 근거는 공개 블로그 한 편이다. 본문에서 "하나투어에 없다"는 "블로그에 언급되지 않았다"로 읽어야 한다. 또한 하나투어는 도입 성과(초안 기획 2~3일 → 2~3분)가 공개된 운영 시스템이고, Lovv는 설계·PoC 단계다. 이 성숙도 차이를 전제로 우위를 판단한다.

## 2. 사례 요약

하나투어는 MD(상품기획자)의 패키지 일정 초안 작업을 자동화하는 내부 도구를 구축했다.

- 데이터: 패키지 상품을 16종 노드(국가·도시·호텔·관광지·항공 구간·트렌드 등)와 20종 엣지로 분해한 Amazon Neptune 지식 그래프. 일반 벡터 RAG의 "관계 평탄화"를 근본 한계로 보고 GraphRAG로 전환.
- 실행: AgentCore Runtime(서버리스 호스팅) + AgentCore Gateway(MCP 변환·IAM 인증) + 단일 openCypher 쿼리 책임의 Lambda + DynamoDB 저장 + ElastiCache(Valkey) 캐시 + LangGraph.
- 생성: 2-Phase. 도시 묶기·동선·시간창·식당 배분 같은 구조 계산은 결정론 알고리즘이 담당하고, 날짜별 상세는 Claude Opus가 병렬 생성, 입력 해석은 Claude Haiku가 담당. 결정론 검증 → 의미 검증 → 문제 날짜만 부분 재생성(최대 5회).
- 그라운딩: "그래프에서 찾은 데이터만 사용" 원칙. 트렌드 노드에는 `virality_score × (1 - decay_rate) ^ 경과 개월` 신선도 감쇠와 hot/steady/seasonal 3등급을 적용.
- 성과: 8일 일정 생성 약 750초 → 약 75초, Phase별 컨텍스트 필터링 + Bedrock Prompt Caching으로 LLM 입력 토큰 약 76% 절감.

## 3. 전제 차이

두 시스템은 "여행 에이전트"라는 표면만 같고 푸는 문제가 다르다. 우위·반영 판단은 이 전제 위에서만 유효하다.

| 축 | 하나투어 | Lovv |
| --- | --- | --- |
| 사용자 | MD(내부 전문가), B2B 내부 도구 | B2C 최종 사용자 |
| 핵심 문제 | 목적지가 주어진 상태에서 패키지 일정 초안 대량 생산 | 목적지 선정부터 시작하는 소도시 1곳 발견과 일정 구성 |
| 데이터 규모 | 전 세계 상품, 대규모 그래프 | 강원·경북 3~5개 검증 도시 고밀도 (P2에서 전국 확장) |
| HITL 위치 | 생성 전 폼 조정, 생성 완료 후 MD 검토 | 생성 도중 사용자 개입(대화형 빌더 반경 루프) |
| 속도 | 8일 일정 약 75초 | 멀티스텝 전체 8초 목표 (재검토 대상, 6장 참조) |

## 4. Lovv가 앞서 있는 영역

아래는 설계 정본 기준의 우위다. 하나투어 항목은 운영 검증이 끝났고 Lovv 항목은 구현·검증이 남아 있으므로, 구현 완료 전까지는 "설계상 우위"로만 취급한다.

| 영역 | Lovv | 하나투어 (블로그 기준) |
| --- | --- | --- |
| 목적지 선정(city discovery) | Candidate Evidence Agent가 여러 도시의 장소 evidence 검색과 scoring으로 소도시 1곳을 선정. 탐색 엔진의 핵심 단계 | 목적지는 사용자 입력으로 주어짐. 해당 단계 자체가 없음 |
| 생성 중 HITL | 대화형 빌더: PICK·REPLACE_PLACE·EXCLUDE·FILTER·DONE action 어휘, interrupt/resume checkpointer로 생성 루프 안에 사용자가 개입 | 생성 전 슬라이더 조정과 생성 후 검토만 존재 |
| 외부 실시간 검증 | Festival Verifier가 공식 웹 출처로 개최일을 검증(confirmed/tentative/unknown/outdated)하고 TTL 캐시로 재사용. 하이브리드 그라운딩이 설계 정본에 포함 | 실시간 정보(날씨·현지 상황·이벤트)의 외부 그라운딩은 "향후 검토" 단계로 명시 |
| 불확실성 표면화 | `confidence`, `user_notice`(반영 불가 조건 안내), 출처·갱신 시점·검증 상태 노출, 안전 폴백 템플릿 | 추천 사유 생성은 있으나 결과 한계를 고지하는 계약은 없음. 전문가 검토 전제라 불필요했던 축 |
| 개인화·멀티턴 | 온보딩 프로필, 피드백 이력, `conversation_summary`, `fulfilled_matrix` 세션 상태 | 언급 없음. 내부 도구 특성상 불필요 |

주의: 평가 하네스(Baseline/Ours 비교, 회귀 평가)와 관측성 명세는 블로그에 없지만 하나투어 내부에 존재할 가능성이 높다. 대외적으로 Lovv의 차별점으로 주장하지 않는다.

## 5. 수렴 항목: 기존 설계의 프로덕션 방증

아래 Lovv 결정은 하나투어가 프로덕션에서 검증한 패턴과 독립적으로 수렴했다. 변경 없이 유지하고, 설계 근거를 물을 때 이 사례를 인용한다.

| Lovv 결정 | 하나투어 대응 패턴 |
| --- | --- |
| DB 근거 없는 장소 노출 금지, S3 vector 결과 단독 확정 금지, MySQL 원장 재검증 | "그래프에서 찾은 데이터만 사용" 그라운딩 원칙 |
| 결정적 Skill / Sub-Agent 분리, 결정론 코드 Supervisor | 구조 계산은 결정론 알고리즘, 추론만 LLM |
| `validation_retry_count ≤ 2` 재시도 상한 | 1차 시도 실패 원인이 무한 생성·평가 루프(4박5일 20~30분)였음 |
| 실패 카테고리별 분기 재시도 | 문제 날짜만 부분 재생성(최대 5회) |
| 스텝별 병렬 fan-out (Candidate∥지도API∥축제) | 날짜별 상세 병렬 생성 |
| 단일 책임 Skill/Lambda 분리 | Gateway + 단일 openCypher 쿼리 Lambda |

## 6. 추후 개발 반영 후보

우선순위 기준은 구현 비용 대비 효과와 현 일정(P1 Production·P2 개발, ~2026-07-13) 내 반영 가능성이다.

| 우선순위 | 항목 | 내용 | 반영 지점 | 근거·기대 효과 |
| --- | --- | --- | --- | --- |
| 1 | Prompt Caching + 컨텍스트 필터링 강화 | 반복되는 시스템 프롬프트·후보 패키지 공통부에 Bedrock Prompt Caching 적용. 노드별 필요 컨텍스트만 전달하는 기존 원칙을 캐시 경계에 맞춰 재점검 | Bedrock Converse adapter, `AgentCore-Function` | 하나투어 입력 토큰 약 76% 절감 사례. 구현 비용이 가장 낮고 비용·지연에 즉효 |
| 2 | 속도 목표 스텝별 분해 | "멀티스텝 전체 8초"를 도시 선정 / 반경 Provider 루프 / 일정 생성 / 검증 스텝별 목표로 재정의 | `../sections/` 8장 품질 기준, 테스트 계획 | 하나투어는 결정론+병렬+tier 분리 후에도 8일 75초. 단일 8초 목표는 측정·달성 기준이 모호해 검증 불가 위험 |
| 3 | 콘텐츠 신선도 감쇠 | 축제·트렌드성 콘텐츠 스코어에 시간 감쇠(hot/steady/seasonal 등급) 반영. 축제 검증 캐시 TTL과 별개로, 스코어링 가중치에 최신성 축 추가 | `Scoring Skill`, P2 확장 데이터 모델 | 철 지난 콘텐츠의 반복 추천 방지. 시즌성 넛지·"지금 갈 이유" 기능의 품질 직결 |
| 4 | 모델 tier 명시 배정 | Converse 추상화는 유지하되, 평가 결과 기반으로 노드별 경량/고성능 tier를 배포 설정에 고정 | Converse adapter 설정, 평가 하네스 | 하나투어 Haiku(해석)/Opus(상세) 분리로 비용·품질 균형 확보. 현재는 미배정 상태라 비용 통제 기준이 없음 |
| 5 (보류) | 일자별 상세 병렬 생성 | Planner 내부 day 단위 병렬화 | `Planner_Agent` | 당일~2박3일 규모에서는 효과가 제한적이고, 사용자 주도 반경 루프와 실행 모델이 충돌할 수 있음. P2 전국 확장에서 일정 길이가 늘어나면 재검토 |

## 7. 반영하지 않는 항목

| 항목 | 결정 | 이유 |
| --- | --- | --- |
| Neptune(그래프DB) 도입 | 기존 결정 유지: Lambda 관계 탐색 보조 + 승격 옵션 | 현 규모(3~5개 도시, 국내 한정)에서 S3 vector metadata filter + DynamoDB 인접 리스트로 충분. 블로그 스스로 그래프 스키마·엣지 설계의 초기 비용을 명시. 승격 조건(3-hop 이상 상시 탐색, 대규모 실시간 그래프 쓰기)은 Agent 명세 9.1을 따름 |
| ElastiCache 캐시 계층 추가 | 도입하지 않음 | 현 트래픽에서 DynamoDB 기반 축제 검증 캐시와 사전계산으로 충분. 캐시 인프라 추가는 운영 부담 대비 효과 없음 |

## 8. 발표·대외 자료 활용 시 주의

- 하나투어 대비 차별점으로 내세울 수 있는 것은 4장 항목 중 최종 발표(2026-07-16) 시점에 동작 데모가 가능한 것으로 한정한다. 문서상 차별점은 차별점이 아니다.
- 평가·관측성 체계는 차별점 주장에서 제외한다(4장 주의 참조).
- 5장 수렴 항목은 "프로덕션 사례와 같은 결론에 도달한 설계"로 인용할 수 있다.
